This paper addresses the constrained optimization problem of learning Bayesian network structures from local scores under ancestral constraints. The proposed method is a branch-and-bound search, specifically an A* search, where nodes in the search tree represent CPDAGs. The search progresses from one CPDAG to a more refined one by selecting a specific parent set for a variable. The algorithm incorporates effective symmetry-breaking techniques. The heuristic for the A* search is derived from the URLearning system, though the details of this integration are cited from a previous paper rather than described here.

The introduction of ancestral constraints enables additional pruning of the search space. Lemma 3 in the supplementary material presents the key result for this pruning. While I believe the lemma is correct, its proof is not clear. Specifically, the statement "By the EC tree edge generation rules, Gk also contains edge Z -> W" requires further explanation to be understood.

The paper also discusses "projected constraints," which appear to be what is standardly termed "implied constraints." A set of ancestral constraints can imply edge constraints that are useful for pruning. The authors also infer "ordering constraints" on the topological ordering of variables, but the method for doing so is not adequately explained. The paper states that one "can infer Y < Z from Z not an ancestor of Y." The word "can" is problematic here. For instance, in a graph with only two disconnected nodes, Z and Y, "Z is not an ancestor of Y" does not entail that Y < Z in all valid topological orderings. What the authors seem to mean is that if "Z is not an ancestor of Y" is a constraint, one can *choose* to add the ordering constraint Y < Z, provided it is consistent with other constraints. The goal is thus to infer a maximal set of *consistent* ordering constraints, not just those that are logically entailed. The paper notes that a MAXSAT approach is used for this inference to ensure a consistent set is found.

The experimental evaluation varies the number of variables, data points, and ancestral constraints. A comparison with the GOBNILP system shows the proposed method to be significantly faster. However, as the authors acknowledge, GOBNILP is known to perform worse than A*-based approaches when the parent set cardinality is unbounded. Since the experiments in this paper do not limit parent set size, it is difficult to determine whether GOBNILP's poor performance stems from its known weakness in handling large parent sets or from a specific inefficiency in managing ancestral constraints.

Furthermore, the comparison to GOBNILP is potentially unfair due to the specific Integer Linear Programming (ILP) formulation used for ancestral constraints. The encoding, detailed in the supplement, is based on a MAXSAT encoding from [Cussens, 2008] rather than a more direct and efficient ILP encoding, such as the one in [Cussens, 2010, Section 3.1.1]. The chosen MAXSAT-style encoding is a weak formulation for an ILP solver. A much stronger formulation would use binary variables A(X,Y) to represent ancestry relationships directly, alongside the parent set indicator variables I(Y,U), which would eliminate the need for the cubic number of E(X,Y,Z) variables. The formulation would then only require transitivity constraints on the A(X,Y) variables and linking constraints to connect parentage with ancestry, such as:
A(X,Y) ≥ Σ_{U: X ∈ U} I(Y,U)
This constraint ensures that if X is a parent of Y, it is also considered an ancestor of Y. This approach, adapted from [Cussens, 2010] for a partial order, would yield a far tighter linear relaxation and likely result in much better performance for the ILP solver. While I am confident that the proposed A* method would still outperform GOBNILP even with an improved ILP formulation, the paper's central comparison is weakened by benchmarking against a suboptimal implementation of the competitor.

Finally, the paper would benefit from a comparison with the constraint programming approach of van Beek and Hoffmann (2015), which is a relevant and high-performing system that often outperforms GOBNILP. That method could likely be adapted to handle ancestral constraints, and its omission from the related work is notable.
