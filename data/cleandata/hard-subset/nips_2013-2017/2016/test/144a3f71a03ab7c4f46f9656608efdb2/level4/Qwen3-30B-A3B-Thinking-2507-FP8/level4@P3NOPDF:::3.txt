

This paper presents a method for incorporating ancestral constraints into optimal Bayesian network learning via translation to ordering constraints, leveraging the EC tree framework (Chen et al., 2015). The approach converts ancestral constraints into a maximal set of ordering constraints using a MaxSAT solver, enabling efficient score-based learning. Empirical results demonstrate significant performance gains over a modified ILP-based Gobnilp (with added linear constraints), though the comparison setup requires careful interpretation.

The problem addressed—optimal Bayesian network learning under ancestral constraints—is novel (score: 3/5), as no standard solution exists for this specific integration. However, the practical relevance is limited (impact: 2/5), as ancestral constraints typically arise in contexts where more comprehensive causal discovery methods (e.g., those accommodating unobserved confounders) are preferable. The paper’s focus on a constrained setting thus feels somewhat artificial, reducing its broader applicability.

The core contribution—translating ancestral constraints into ordering constraints compatible with decomposable scores—is conceptually sound. The MaxSAT-based translation ensures all implied constraints are captured, which is a key technical strength. However, the formalization of the constraint translation (Section 3, Rules 257–261) contains inconsistencies. Rule 3’s premise is symmetric (e.g., (X < Y) ∧ (Z < W) ≡ (Z < W) ∧ (X < Y)), yet its conclusion (Y ↛ Z) is asymmetric, creating logical ambiguity. Rule 4 also appears to exclude constraints like "X and Y do not cause each other," which contradicts the intended scope. These issues undermine confidence in the constraint set’s completeness.

Critically, the paper’s claim that ancestral constraints rarely reduce the search space (Line 310) is questionable. For typical graphs (e.g., 10–25 nodes), even sparse ancestral constraints (10–25% of possible pairs) often imply substantial structural knowledge, making the "sparse" assumption unrealistic for practical applications. This weakens the argument that the method’s efficiency gains are merely due to constraint density.

The evaluation against modified Gobnilp is misleading. Gobnilp is not designed for ancestral constraints; its performance under this mismatch does not reflect the method’s true value. A more meaningful baseline would compare against a greedy algorithm (e.g., GES) *with the same ancestral constraints*, assuming they align with the true structure. While GES cannot guarantee optimality, it would provide a realistic benchmark for efficiency gains in settings where the constraints are informative. The current comparison risks overestimating the method’s practical utility.

Minor issues require clarification:
- Line 30: Distinguish between a "standard oracle" (providing edge presence/absence) and an "omniscient oracle" (providing full structure), as the paper implies they are equivalent.
- Line 38: Replace "empower the oracle" with "leverage the oracle" for precision.
- Line 90: Specify "minimizing" (not "optimizing") the score.
- Line 164: Theorem 1 must clarify that the constraint set A derived from the CPDAG P is a *subset* of constraints satisfied by the underlying DAG G; otherwise, the "iff" condition fails.
- Lines 225–227: Theorem 3’s utility for sparse ancestral constraints is limited, as it rarely yields positive edge implications (e.g., "X → Y").
- Line 267: The "acyclic" constraint (Rule 3) is poorly defined (see Rule 3’s logical flaw above).

The paper is well-written and clearly explains the method, but the novelty is modest. The approach is technically valid and produces strong empirical results *within its constrained framework*, yet its practical significance is diminished by the artificiality of the problem setting and the questionable baseline comparison. The method would be more compelling if it demonstrated value in realistic causal discovery scenarios with partial ancestral knowledge, rather than solely against a misaligned competitor.

**Recommendation**: Weak Accept (score: 5/10). The technical contribution is sound for the specified problem, but the limited impact and methodological nuances (constraint translation, evaluation) prevent a stronger endorsement. The authors should address the logical inconsistencies in the constraint rules, clarify the oracle terminology, and reframe the evaluation against a more relevant baseline (e.g., GES with constraints) to strengthen the case for practical utility.
