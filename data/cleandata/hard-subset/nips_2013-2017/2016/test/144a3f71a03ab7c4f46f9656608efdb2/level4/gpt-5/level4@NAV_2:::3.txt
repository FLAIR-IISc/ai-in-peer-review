This paper introduces an efficient method for integrating ancestral constraints into the task of learning optimal Bayesian networks from data. The key idea is to convert a set of ancestral constraints into a (maximal) set of ordering constraints, which are then supplied to a score-based learner that searches using the EC tree framework of [Chen et al., 2015]. The approach is empirically compared against a straightforward adaptation of the ILP-based GOBNILP and demonstrates substantial speedups in the considered scenarios. The paper is generally clear and well presented, though the high-level idea is fairly direct. Overall: novel problem plus solution, solid empirical gains; not a breakthrough, but likely of interest to specialists.

The problem addressed is interesting and new in that there is no widely adopted, off-the-shelf “standard solution” for optimal BN structure learning under ancestral constraints (novelty=3). That said, part of the absence of prior work is that the setup is somewhat stylized: once ancestral information is introduced, one might naturally turn to broader causal discovery methods that also account for latent confounding, which may better reflect practical use cases (impact=2).

Methodologically, the paper’s strategy is to translate non-decomposable ancestral constraints—which do not mesh well with decomposable-score learners—into ordering constraints that can be operationalized via edge-absence constraints compatible with decomposable scores. The mapping uses a MaxSAT solver intended to render all implied ancestral relations explicit. However, some of the rules on lines 257–261 appear unusual or incomplete: for instance, the premise of acyclicity rule 3 is symmetric ((X < Y) ∧ (Z < W) is equivalent to (Z < W) ∧ (X < Y)), whereas the conclusion is not (Y -/-> Z); and rule 4 seems to omit constraints of the form “X and Y are mutually non-ancestral.” Moreover, in realistic settings one would expect only a small number of ancestral constraints, which may not dramatically shrink the search space. The resulting comparison against GOBNILP with added linear constraints is striking, but GOBNILP is not designed specifically for this use case. A more informative baseline would assume the ancestral constraints are consistent with the optimal model and compare against a greedy method such as GES which, with oracle scoring, should recover the same solution; while greedy methods lack optimality guarantees, this would better contextualize the efficiency gains claimed here.

Minor details:
- Line 30: Is there a distinction intended between a “standard” oracle and an “omniscient” oracle?
- Line 38: “Empower the oracle” might be clearer as “use in conjunction with the oracle” or “tighten the oracle’s heuristic.”
- Line 90: Clarify that “optimizing” here means minimizing the score.
- Line 164: Theorem 1 should more explicitly state that the set of constraints A satisfied by a CPDAG P in the EC tree corresponds to a subset relative to the full DAGs G in its class; otherwise the “iff” is misleading.
- Lines 225–227: Theorem 3 seems unlikely to yield many positive edge entailments when the ancestral constraint set is sparse.
- Line 267: The provided formulation is not readily recognizable as enforcing acyclicity (see also the comment above on rule 3).
- Line 310: For realistically sized graphs, specifying 10%–25% of ancestral relations typically conveys substantial prior knowledge about the target, which may be atypical in practice.
