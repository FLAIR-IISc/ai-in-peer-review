

The paper proposes leveraging fast approximate nearest-neighbor (ANN) structures and arg-top-K operations to compute attention weights in Neural Turing Machines (NTMs), demonstrating scalability to larger problems without degrading training performance. While the practical implementation of scalable attention for *writeable* memories represents a valuable contribution—particularly as prior work (e.g., Memory Networks, http://arxiv.org/pdf/1410.3916.pdf; MemNN-WSH, http://arxiv.org/pdf/1503.08895.pdf) primarily addresses *read-only* memories—the novelty is incremental. The core technical approach aligns with established scaling techniques for memory networks, and the conceptual leap is modest. Nevertheless, the authors’ successful adaptation to writeable memory systems merits publication, provided key revisions are addressed.  

**Major Concerns and Required Revisions**  
1. **Replicability and Task Specification**: The paper lacks sufficient detail to replicate results. Task construction is underspecified (e.g., no precise descriptions of the algorithmic tasks used, unlike the original NTM paper, which omitted task code). Given the absence of standardized benchmarks or released task code, the authors must provide:  
   - Clear, step-by-step task generation protocols (e.g., sequence length distributions, input/output specifications).  
   - A public code release commitment (including task generators and training scripts) to enable reproducibility. Without this, the empirical claims lack rigor.  

2. **Failure Analysis and Limitations**: The paper asserts "smoothness" as a key advantage but overlooks critical limitations of discrete arg-top-K operations (non-differentiable for $K=1$). This contrasts sharply with smooth alternatives like the NTM-ADAM model (http://arxiv.org/abs/1511.07275), which avoids discrete actions. The authors should:  
   - Analyze failure cases (e.g., tasks where arg-top-K underperforms vs. smooth methods).  
   - Discuss *when* their approach succeeds (e.g., tasks with high signal-to-noise in memory) and *why* (e.g., discrete reads compensate for noisy attention).  
   - Clarify whether $K=1$ is essential to their results or if higher $K$ would generalize better.  
   *Without this, the paper overstates the generality of its approach.*  

3. **Terminology and Claims**:  
   - Section 3.6 (on "non-synthetic" Omniglot) must be removed. Omniglot is synthetically generated (handwritten characters), not "real-world" or "non-synthetic"—this mischaracterization undermines credibility.  
   - Replace "smooth" with precise terminology (e.g., "non-differentiable discrete read operation") to avoid misleading readers.  

**Why This Matters**  
The work’s primary value lies in demonstrating *practical scalability* for writeable memories—a gap in prior literature. However, the current draft risks overstating impact by:  
- Failing to distinguish its contribution from existing ANN-based memory networks (e.g., section 3.3 of Memory Networks paper).  
- Ignoring the fundamental trade-off between discrete actions (arg-top-K) and smooth optimization, which is well-documented in the literature.  
- Using ambiguous task descriptions that hinder validation.  

**Recommendation**  
This paper presents a useful engineering advance for scalable NTMs but requires significant refinement to meet NIPS standards. The authors should address all points above, particularly task specification and failure analysis. With these revisions, the work will make a meaningful contribution to the practical deployment of memory-augmented networks. **Accept with major revisions.**  

*Specifically, the authors must:*  
- **Revise Task Descriptions**: Add subsection detailing task construction (e.g., "Task Generation Protocol: For the copy task, sequences are generated by sampling from a uniform distribution over 10 characters, with lengths drawn from $\mathcal{U}(5, 15)$").  
- **Add Failure Analysis**: Include a table comparing arg-top-K vs. smooth models on 3–4 tasks (e.g., bAbI, copy, and a synthetic task with high noise).  
- **Remove Section 3.6** and correct Omniglot terminology.  
- **Commit to Code Release** (e.g., "Code and task generators will be released at [URL] upon acceptance").  

The current work is on the cusp of being valuable; these revisions will clarify its scope, strengthen empirical claims, and ensure reproducibility—critical for NIPS.
