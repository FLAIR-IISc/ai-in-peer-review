The authors use fast approximate nearest-neighbor structures and top-K argmax operations to compute attention weights in a Neural Turing Machine. Their experiments suggest this does not harm training performance and, in fact, enables training on larger problems. The novelty is limited—see, for example, Section 3.3 of the memory networks paper http://arxiv.org/pdf/1410.3916.pdf—and there are now several memory network papers that scale to millions of memory items using hashing for efficient lookups. That said, I have not seen these tools applied to writable memories. Even if the conceptual advance is modest, the practical engineering needed to make it work is worthwhile to report.

Overall, I recommend acceptance, but I ask the authors to address the following points:

1) Reproducibility and task specification
- As written, the paper would be difficult to reproduce. The task descriptions are especially thin. Because the original NTM task code was not released and there is no single “standard” implementation of these tasks, it is essential to provide precise, self-contained descriptions of how each task is constructed.
- Please include all necessary details (e.g., data generation procedures, sequence lengths, distributions, noise models, training/validation/test splits, curriculum or difficulty schedules, and stopping criteria), as well as optimization settings and hyperparameters.
- I also ask the authors to commit to releasing the code for their experiments.

2) Limits, failure modes, and the role of discreteness
- Where does this approach fail, and does it always work? The paper often describes the model as smooth, but the argmax (or K-argmax) operation is not smooth. With K=1 in the read-only setting, the model is very similar to MemNN-WSH in http://arxiv.org/pdf/1503.08895.pdf (though that model does not use hashing), and that variant was reported to underperform fully smooth models. Is the gap due to K=1? Task specifics? Something else?
- The paper would be stronger with an analysis of failure cases and a discussion of what task properties and training setups enable learning to proceed. The model makes discrete choices, yet training appears to proceed without explicitly accounting for this discreteness (in contrast to http://arxiv.org/abs/1511.07275). That you obtain good results in this way is a valuable contribution; however, it would be more useful to understand the boundaries of when and why it works.
- Concretely, please consider ablations varying K (including K=1 vs larger K), read-only vs read–write settings, and the impact of approximate indexing vs exact attention on both performance and stability. If possible, discuss sensitivity to memory size, collisions, and noise.

3) Presentation
- I suggest removing Section 3.6.
- Please avoid calling Omniglot “non-synthetic” or “real-world.”

In summary, while the novelty is thin relative to prior memory network work on scalable lookups, applying fast approximate retrieval and top-K selection to writable memories is practically important. The empirical results are encouraging, and with stronger task specifications, code release, and an analysis of limitations and failure modes, this paper would make a solid contribution. I recommend acceptance contingent on these revisions.
