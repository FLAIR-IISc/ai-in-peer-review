Summary
The paper addresses large-scale covariance selection, targeting problems with up to millions of variables and trillions of parameters. It focuses on the CLIME estimator, which estimates the precision matrix of a Gaussian vector and is attractive because it can be decomposed column-wise and is often considered simpler to analyze than the Graphical Lasso. The authors propose a new algorithm that solves CLIME in a block-wise manner rather than column-wise, coupled with an inexact alternating direction method of multipliers. They provide convergence guarantees in terms of both objective value and optimality gap. The algorithm exploits structural properties (sparsity and low rank) to reduce computation and memory overhead and is implemented in a parallel framework that supports both shared-memory and distributed-memory systems. Experiments compare runtime against direct competitors on synthetic and real datasets, and include a detailed scalability study as a function of block size and number of cores.

Strengths
- Clear, well-written presentation with good positioning relative to existing literature and appropriately chosen references.
- A sensible and well-motivated shift from column-wise to block-wise updates that better suits modern parallel and distributed architectures.
- Solid theoretical analysis with explicit convergence guarantees (objective value and distance to optimality).
- Practical engineering that exploits sparsity and low rank to reduce computational burden, accompanied by an implementation that performs impressively at scale.
- Comprehensive experimental evaluation of runtime and scalability on both shared-memory and distributed-memory platforms, including ablations over block size and core count.

Concerns and suggestions
- Reproducibility and availability: The impact of this work will be greatly diminished if the source code is not released. Given the reliance on open-source libraries and the community’s interest in large-scale covariance estimation, public release of code, build scripts, and experiment configurations is essential.
- Evaluation metrics: The experiments prioritize runtime. For an estimator like CLIME, statistical performance matters as well. Please report estimation error (e.g., Frobenius or operator norm error relative to ground truth), support recovery metrics (precision/recall or F1 for edge detection), and downstream performance when relevant. This would help assess trade-offs between speed and accuracy.
- Baselines: While comparisons to “direct competitors” are mentioned, the paper would benefit from a clearer baseline set and stronger runtime/accuracy comparisons to prominent methods such as QUIC (for Graphical Lasso) and nodewise regression approaches, with attention to their most optimized implementations.
- Regularization and hyperparameters: Clarify how the regularization parameter is chosen at scale (e.g., cross-validation strategy, theoretical plug-in, path computation). If a path is not computed, discuss the computational impact of tuning and whether warm-starts or screening are used.
- Symmetry and positive definiteness: CLIME solutions are not necessarily symmetric; standard practice includes symmetrization. Please specify how symmetry is enforced in the block-wise and distributed setting, and whether the final estimate is guaranteed to be positive definite or post-processed to be so.
- Memory and communication: Provide more detail on memory scaling with problem size, sparsity level, and block size. In distributed settings, quantify communication overhead and how it scales with the number of workers and network bandwidth/latency. A breakdown of time by computation vs. communication would be informative.
- Choice of block size: The ablation is useful; adding guidance or an adaptive rule for selecting block sizes based on sparsity patterns, conditioning, or hardware characteristics would improve usability.
- Stopping criteria: Detail the stopping rules, primal/dual residual tolerances, and their relationship to the theoretical guarantees and practical accuracy. Empirically show how tighter tolerances affect runtime and estimation error.
- Structural exploitation: The paper mentions leveraging sparsity and low rank. Please clarify how low-rank structure is detected or imposed during optimization, and report the overhead of structural updates relative to total time. If approximate low-rank projections are used, discuss their impact on accuracy.
- Robustness and data model: The analysis appears tailored to Gaussian models. Comment on robustness to model misspecification and heavy-tailed data, and whether simple pre-processing (e.g., rank-based correlations or truncation) can be incorporated without breaking guarantees.

Minor comments
- Clarify whether “inexact direction method of multipliers” refers to an inexact alternating direction method of multipliers (ADMM); align terminology with standard references.
- Improve phrasing in several places (e.g., “in terms of” rather than “in term of”; “connection” rather than “connexion”; “sound references” rather than “sounded bibliographical references”).
- Include details on hardware (CPU model, memory per node, interconnect) and software stack (BLAS/LAPACK/MPI versions, compiler flags) to facilitate reproduction.
- If possible, release scripts and configuration files to reproduce figures and tables, and provide a minimal working example for running on a single machine.

Overall assessment
This is a strong algorithmic and systems-oriented paper that tackles an important practical problem in high-dimensional statistics. The methodological choices are well motivated, the theoretical analysis is solid, and the engineering is impressive, yielding excellent scalability. The main missing piece is comprehensive assessment of statistical accuracy alongside runtime, and a firm commitment to code release for reproducibility. Conditional on releasing the implementation and clarifying the above points—especially accuracy metrics, baseline coverage, symmetry/PD handling, and practical guidance on block size and stopping—the paper merits publication.
