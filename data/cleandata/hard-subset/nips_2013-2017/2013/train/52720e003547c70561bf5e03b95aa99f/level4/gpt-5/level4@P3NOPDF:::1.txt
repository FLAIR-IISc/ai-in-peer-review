This paper addresses sparse precision matrix estimation via CLIME and proposes a scalable variant, CLIME-ADMM, together with a distributed-memory implementation. The key algorithmic idea is to estimate the precision matrix by column blocks rather than strictly column-by-column, and to use ADMM to enable parallelism. The authors compare CLIME-ADMM against DC-QUIC, TIGER, and Flare, and present an implementation using OpenMPI and ScaLAPACK.

Strengths
- Timely problem: scalable, high-dimensional precision matrix estimation is important in several domains.
- Methodological contribution: reformulating CLIME with ADMM and processing column blocks is a clear and potentially impactful idea.
- Practical emphasis: the use of widely deployed HPC tools (MPI, ScaLAPACK) suggests the approach can leverage existing infrastructure.
- Comparative evaluation: inclusion of strong baselines (DC-QUIC, TIGER, Flare) is appropriate.

Major concerns and suggestions
1) Clarity and notation
- The notation is scattered across Sections 1–3, which hampers readability. A consolidated notation table would greatly improve accessibility.
- Some symbols appear to be undefined (e.g., rho and eta). Please define all algorithmic and hyperparameters on first use (e.g., ADMM penalty, over-relaxation/step size), and clarify how they are chosen in practice.
- Provide a compact statement of the optimization problem, constraints, and the precise per-block subproblems that ADMM solves, so readers can verify correctness and understand computational costs.

2) Block-cyclic distribution and scalability claims
- The proposed block-cyclic data (and task) distribution is interesting, but it is not obvious why it yields good load balance and scalability in this setting. Please justify this choice analytically or empirically.
- Include scaling experiments that directly target this claim:
  - Strong scaling: fix the problem size and show speedup vs. number of processes.
  - Weak scaling: grow the problem size with resources and show time per variable or per nonzero stays stable.
  - Load balance metrics: report per-process workload/idle time histograms or Gantt charts.
- Discuss when block-cyclic is preferable to block or cyclic schemes, given sparsity patterns commonly encountered in precision matrices.

3) Choice of column block size k
- The current choice of k appears ad hoc. Please provide guidance:
  - Theoretical or heuristic rationale for how k trades off parallelism, memory locality, communication volume, and convergence behavior.
  - Sensitivity analysis showing runtime and accuracy as functions of k on representative datasets.
  - A practical rule-of-thumb (e.g., based on cache size, number of cores per node, or sparsity level).

4) “Distributed” vs. “parallel” characterization
- The manuscript refers to a distributed framework but does not specify communication patterns. To substantiate the “distributed” claim, please:
  - Describe the message-passing scheme (collectives vs. point-to-point), synchronization points, and how global operations (e.g., residual norms, soft-thresholding thresholds, or feasibility checks) are computed.
  - Quantify communication costs per ADMM iteration (messages and bytes), and identify bottlenecks.
  - Clarify fault tolerance assumptions, if any. If the work is a distributed-memory parallel implementation without a new distributed optimization algorithm per se, consider describing it explicitly as a parallel implementation.

5) Experimental details and fairness
- Ensure the baselines are tuned and stopped fairly: identical stopping criteria/tolerances where applicable, consistent selection of tuning parameters (e.g., via cross-validation or the same theoretical rule), and comparable hardware.
- Report wall-clock times, iteration counts, and, if possible, solution quality metrics (e.g., sparsity, estimation error where ground truth is known) to disentangle algorithmic efficiency from implementation effects.
- Include ablations isolating the effect of the ADMM penalty, over-relaxation, and k, to help practitioners replicate strong performance.

6) Exposition and reproducibility
- Provide pseudocode for CLIME-ADMM with clear input/output, parameter list, and per-iteration operations.
- Summarize computational complexity and memory footprint per process as functions of p (dimension), k (block size), and nnz (sparsity).
- If possible, include or plan to release code/scripts to reproduce the figures.

Minor comments and typos
- Section 1: “where lambda is a tunning parameter” -> “tuning”
- Section 5.1: “As Tiger is parameter tunning free” -> “tuning”
- Section 2: “CLIME is summerized in” -> “CLIME is summarized in”
- Consistency: Use TIGER (all caps) and ScaLAPACK (capitalization).
- Define all symbols at first mention, including rho, eta, and lambda; avoid reusing symbols for different quantities.
- Consider a caption or appendix table listing all notation used.

Overall assessment
The paper tackles an important problem with a promising algorithmic variation and a practical parallel implementation. However, to meet the bar for a strong systems-and-algorithms contribution, the paper needs clearer exposition of the optimization setup and parameters, a principled treatment of the block size k, and, crucially, a detailed account of communication and scaling. Strengthening the empirical section with scaling studies, fairness checks, and ablations, alongside a consolidated notation and implementation details, would significantly improve clarity and impact.
