

The authors address sparse precision matrix estimation using CLIME, introducing a scalable variant CLIME-ADMM alongside a distributed computational framework. Empirical comparisons with state-of-the-art methods (DC-QUIC, Tiger, Flare) are provided.  

Notation is dispersed across Sections 1, 2, and 3, which impedes readability. A centralized notation table would be beneficial; for instance, $\rho$ and $\eta$ are undefined in the text.  

The block cyclic data distribution scheme is intriguing but lacks justification for its claimed load balancing and scalability. No empirical results support this assertion.  

The algorithm currently selects the column block size ad hoc. Is there theoretical guidance or intuition for choosing $k$?  

A broader concern is that the work may not constitute a *distributed* algorithm, as explicit message passing and communication mechanisms are absent. It would be more accurate to frame this as a parallel implementation.  

Minor corrections:  
- Section 1: "tunning" → "tuning"  
- Section 5.1: "parameter tunning free" → "tuning free"  
- Section 2: "summerized" → "summarized"  

The work presents a large-scale parallel algorithm for sparse precision matrix estimation via CLIME, with novelty in estimating the precision matrix by column blocks (rather than column-by-column). Empirical analysis using OpenMPI and Scalapak is provided.
