This paper addresses sparse precision matrix estimation via CLIME, proposes a scalable ADMM-based variant (CLIME-ADMM), and introduces a framework intended for distributed/parallel computation. The method estimates the precision matrix in column blocks rather than one column at a time, and the implementation leverages OpenMPI and ScaLAPACK. Empirical comparisons to DC-QUIC, TIGER, and Flare are provided.

Strengths and contributions:
- A scalable CLIME formulation using ADMM.
- A column-block strategy that may better exploit parallel resources than purely column-wise updates.
- An implementation using standard HPC tools and comparisons to strong baselines.

Major comments:
- Notation clarity: The notation is dispersed across Sections 1–3. A consolidated table of symbols would greatly improve readability. In particular, the parameters ρ and η appear without definition; please define them at first use and explain their roles in the algorithm (e.g., ADMM penalty, step sizes).
- Block-cyclic distribution: The proposed block-cyclic data layout is intriguing, but the rationale for why it ensures load balance and scalability is not adequately justified. Please provide:
  - A brief analysis of expected load balance under this scheme (e.g., for varying sparsity patterns and block sizes).
  - Empirical evidence (strong/weak scaling, per-processor utilization, and communication/computation breakdowns) demonstrating that block-cyclic placement improves performance relative to alternatives.
- Choice of column block size k: The current selection seems ad hoc. Some guidance is needed:
  - Intuition or theory for how k should scale with p, sparsity level, number of processors, cache sizes, and network characteristics.
  - A sensitivity analysis showing performance and accuracy as k varies, and whether there is a robust default.
- “Distributed” vs “parallel”: The paper claims a distributed algorithm, yet the communication patterns and message-passing mechanisms are not described. As presented, this reads more like a parallel implementation than a full distributed algorithm. Please clarify:
  - The system model (e.g., multi-node MPI with distributed memory).
  - The communication required per ADMM iteration (collectives vs point-to-point), synchronization points, and communication complexity.
  - Any overlap of computation and communication, and how fault tolerance or stragglers are handled (if applicable).
  - If the contribution is primarily a parallel/HPC implementation, consider framing it accordingly.

Suggestions to strengthen the empirical section:
- Include scaling plots (cores/nodes vs runtime), efficiency metrics, and throughput for varying matrix sizes.
- Report the impact of the block-cyclic distribution and k on runtime and memory usage.
- Provide accuracy metrics (e.g., support recovery, estimation error) alongside runtime to ensure performance is not gained at the expense of estimation quality.

Minor comments and typos:
- Section 1: “where λ is a tunning parameter” → “where λ is a tuning parameter.”
- Section 5.1: “As Tiger is parameter tunning free” → “As TIGER is parameter tuning free.”
- Section 2: “CLIME is summerized in” → “CLIME is summarized in.”
- Ensure that all symbols (including ρ and η) are defined upon first use and included in the notation table.
- Standardize the capitalization of method names (e.g., TIGER) and libraries (e.g., ScaLAPACK).

Overall assessment:
The paper presents a large-scale parallel approach to estimating sparse precision matrices using CLIME, with the main novelty being column-block estimation and an HPC-oriented implementation in OpenMPI/ScaLAPACK. The idea is promising and the comparisons to strong baselines are valuable. To meet the claim of a distributed algorithm and to improve clarity, the paper should (1) consolidate and define notation, (2) justify and empirically validate the block-cyclic distribution, (3) provide guidance on selecting the column block size k, and (4) detail the communication patterns and costs that underpin scalability. With these revisions, the contribution would be significantly clearer and more compelling.
