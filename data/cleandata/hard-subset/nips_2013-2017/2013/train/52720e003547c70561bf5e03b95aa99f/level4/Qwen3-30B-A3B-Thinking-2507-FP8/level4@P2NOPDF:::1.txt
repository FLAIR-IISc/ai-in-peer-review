

The paper introduces CLIME-ADMM, a scalable variant of CLIME for sparse precision matrix estimation, alongside a distributed computational framework. Empirical comparisons with DC-QUIC, Tiger, and Flare demonstrate its effectiveness. However, several concerns require clarification before acceptance.  

**Major Concerns:**  
1. **Notation Clarity:** Key symbols (e.g., $\rho$, $\eta$) remain undefined in the main text, appearing only sporadically across Sections 1–3. A centralized notation table would significantly improve readability and avoid confusion.  
2. **Scalability Claims:** The block cyclic data distribution is presented as enabling load balance and scalability, but the justification for these claims is unclear. Crucially, no empirical evidence (e.g., speedup, communication costs, or load metrics) supports this assertion. The authors should provide analysis or benchmarks validating scalability under distributed settings.  
3. **Column Block Size Selection:** The algorithm’s choice of column block size $k$ appears ad hoc. The authors should justify this choice with theoretical insights or empirical guidelines (e.g., sensitivity analysis across datasets) rather than arbitrary selection.  
4. **Distributed vs. Parallel Distinction:** The work lacks explicit discussion of message-passing mechanisms or communication protocols, which are fundamental to distributed algorithms. Without these, the framework should be framed as a *parallel* implementation (e.g., leveraging OpenMPI/Scalapack for concurrent computation) rather than distributed. This distinction is critical for the paper’s technical contribution.  

**Minor Corrections:**  
- Section 1: "tunning" → "tuning" (e.g., "where $\lambda$ is a tuning parameter").  
- Section 5.1: "parameter tunning free" → "tuning-free" (e.g., "Tiger is tuning-free").  
- Section 2: "summerized" → "summarized" (e.g., "CLIME is summarized in Algorithm 1").  

**Contribution Assessment:**  
The core idea—estimating the precision matrix via column blocks rather than column-wise—offers a plausible path to scalability. However, the current presentation falls short in rigorously establishing the distributed nature of the approach and validating its scalability claims. The empirical section (Section 5) demonstrates computational feasibility with OpenMPI/Scalapack but omits critical details on communication overhead or load balancing metrics. For instance, how does the block cyclic scheme reduce idle time during computation? Without addressing this, the "distributed" label is misleading.  

**Recommendation:**  
The paper addresses a relevant problem with a potentially useful algorithm, but the current version requires substantial revisions to strengthen technical claims. Specifically:  
- Add a notation table.  
- Provide theoretical justification for $k$ or empirical sensitivity analysis.  
- Clarify whether the framework is *distributed* (with communication protocols) or *parallel* (with shared-memory or task-parallel execution). If the latter, reframe the contribution accordingly.  
- Include scalability evidence (e.g., weak/strong scaling plots, communication time breakdowns).  

Without these, the paper risks overstating its contribution. If revised to address these points, the work could be a solid addition to the literature on scalable high-dimensional statistics. As-is, it lacks the rigor expected for NIPS.  

**Summary:**  
The CLIME-ADMM approach shows promise for scalable precision matrix estimation, but its distributed framework is inadequately justified. Key gaps include undefined notation, unsubstantiated scalability claims, and a misaligned "distributed" description. Addressing these would transform a promising idea into a compelling contribution. The minor grammatical corrections (e.g., "tuning," "summarized") are straightforward to implement.
