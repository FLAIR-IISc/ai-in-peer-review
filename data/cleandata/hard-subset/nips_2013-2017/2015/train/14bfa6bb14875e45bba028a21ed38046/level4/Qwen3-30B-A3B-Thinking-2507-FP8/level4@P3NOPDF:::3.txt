

This paper presents a two-stage object detection framework leveraging CNNs, building on existing R-CNN approaches. The core idea—using a Region Proposal Network (RPN) operating on CNN feature maps for efficient region generation, followed by Fast R-CNN for classification—represents a logical extension to the field. The authors demonstrate a GPU-accelerated implementation achieving ~5 fps, which is a notable practical contribution. However, significant technical concerns undermine the method’s validity and reproducibility, as detailed below.

**Major Concerns**  
1. **Insufficient Alternating Optimization**: The optimization procedure (Section 3.2) describes a single iteration of alternating updates between RPN and R-CNN. This is problematic because it implicitly assumes the initial RPN proposals are optimal, which contradicts the fundamental premise of alternating optimization (e.g., as in R-CNN). In practice, a single iteration would fail to refine proposals, leading to suboptimal bounding boxes and classification accuracy. The authors should clarify whether this is a deliberate simplification (e.g., for speed) or an omission. If the former, they must justify why convergence is unnecessary; if the latter, the method lacks theoretical grounding. Standard approaches (e.g., Faster R-CNN) use multiple iterations for refinement—this work must address why that is not required here.

2. **Ambiguous Spatial Resolution in RPN**: The RPN employs a sliding window on the top CNN feature map (Section 3.1). However, the spatial stride of this window at the *input image scale* is undefined. CNNs typically reduce spatial dimensions via max-pooling (e.g., 16× or 32× downsampling), meaning a stride of *s* pixels on the feature map corresponds to *s × 16* or *s × 32* pixels on the original image. Without specifying *s*, the method risks coarse localization—e.g., a stride of 16 pixels on the feature map would cause 256-pixel gaps in bounding box placement at the image level. This is critical for detection accuracy and should be explicitly addressed (e.g., via stride definition or anchor scaling). The current description leaves the spatial precision of proposals unclear.

3. **Undefined Notation ($h_a$)**: The notation $h_a$ appears in Lines 173–174 without definition or context. It is unclear whether this denotes a height parameter, a feature vector, or an anchor-specific variable. This omission impedes understanding of the proposal generation mechanism and violates basic mathematical rigor. The authors must define $h_a$ explicitly (e.g., "height of anchor box $a$") and ensure consistency with the equations.

**Additional Notes**  
- The claim that the RPN operates "on the top layer of CNN" (Section 3.1) requires precision. Modern CNNs (e.g., VGG) have multiple feature maps; the paper should specify *which* layer(s) are used for RPN (e.g., "conv5-3" in VGG-16) and why.  
- The speed claim ("near real-time, 5fps") lacks critical context: hardware (e.g., Titan X vs. CPU), input resolution (e.g., 600px vs. 1000px), and comparison baseline (e.g., vs. R-CNN’s 0.3 fps). Without this, the benchmark is misleading.  
- While sharing CNN weights between RPN and R-CNN is standard (Faster R-CNN), the paper does not clarify *how* this sharing is implemented (e.g., joint training vs. sequential fine-tuning), which affects reproducibility.

**Strengths**  
The two-stage design is conceptually sound and leverages existing CNN architectures efficiently. The GPU acceleration for shared computation is a practical advantage over non-shared methods like R-CNN. The paper also provides a clear high-level description of the pipeline, aiding readability.

**Recommendation**  
This work has a promising foundation but requires substantial revision to address the above issues before publication. The technical gaps—particularly the unvalidated optimization strategy, undefined spatial parameters, and unclear notation—prevent confident assessment of the method’s contribution. I recommend *rejection* pending major revisions that:  
(i) Clarify and justify the optimization procedure (e.g., through ablation studies on iteration count),  
(ii) Specify the spatial stride at the image level and anchor scaling,  
(iii) Define all notation (e.g., $h_a$) and provide a precise description of the feature map layer used for RPN.  

Without these fixes, the paper risks misleading readers about the method’s capabilities and theoretical soundness. The current draft does not sufficiently advance the state of the art beyond established work like Faster R-CNN, and the unresolved technical ambiguities overshadow its practical merits.
