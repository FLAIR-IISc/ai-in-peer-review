- Introduces a Region Proposal Network (RPN) that shares convolutional features with the detection network to produce nearly cost-free region proposals.
- RPN is a fully-convolutional network that predicts object bounds and objectness scores at each position and is trained end-to-end with the detector.
- Achieves state-of-the-art results: 73.2% mAP on PASCAL VOC 2007 and 70.4% on VOC 2012, running at ~5 fps with VGG-16.
- Produces very fast proposals (~10 ms effective proposal time), vastly faster than Selective Search (1–2 s) and competitive with EdgeBoxes.
- Key contribution: demonstrating region proposals can be generated by a deep network that shares features with the detector, removing the proposal bottleneck.
- Proposes a simple alternating optimization scheme to train the unified RPN + Fast R-CNN without full joint optimization.
- Introduces translation-invariant anchor design with k=9 anchors per location, reducing model complexity versus previous methods (e.g., MultiBox).
- Thorough experiments and ablation studies show contributions of classification/regression layers and shared features; implementation details and code are provided for reproducibility.
- Originality: reframes region proposal as a deep-learning problem and clearly differentiates from related work (MultiBox, OverFeat) by anchors, feature sharing, and training scheme.
- Significance: enables practical, near-real-time object detection with high accuracy, important for research and industry applications.
- Strengths summarized: clear problem motivation, effective solution, strong empirical results, detailed analysis, and public code release.
- Weakness 1: Alternating optimization appears to use only one iteration between RPN and Fast R-CNN—unclear if that suffices for convergence; need clarification or justification.
- Weakness 2: Unclear handling of top-layer sliding-window step size/receptive field on the input image (effective stride 16–32 px), which could hurt localization—more detail needed.
- Weakness 3: Ambiguity in the definition of h_a in equations (which anchor's height is referenced) should be clarified.
- Additional suggestion: provide analysis of trade-offs between proposal count and detection accuracy (e.g., performance with fewer proposals like 100) to show practical limits.
- Recommendation: accept with minor revisions to address the clarifications above.
