

# Review of "Faster R-CNN: Towards Real-Time Object Detection with Region Proposal Networks"

This paper introduces Region Proposal Networks (RPNs) that share convolutional features with object detection networks, enabling region proposal generation to be nearly cost-free. The authors propose a unified framework where RPNs and Fast R-CNN share features, significantly reducing the computational bottleneck of region proposals. The method achieves state-of-the-art accuracy (73.2% mAP on PASCAL VOC 2007) with real-time performance (5fps on GPU).

**Strengths:**
The paper makes a significant contribution to object detection by addressing the computational bottleneck of region proposals. The RPN design is elegant and well-motivated, with the key insight that convolutional features can be shared between proposal and detection networks. The experimental results are compelling, showing substantial speed improvements while maintaining or improving accuracy. The training scheme for alternating optimization between RPN and detection networks is practical and effective. The ablation studies provide valuable insights into the role of different components (cls vs. reg outputs, proposal count, etc.).

**Weaknesses:**
Despite the paper's significant contributions, several technical inconsistencies and omissions undermine its quality as a scientific contribution.

First, there is a critical inconsistency in the loss function definition (Equation 1). The authors state "the cls layer outputs 2k scores" (one for object, one for not-object per anchor), yet they later claim "For simplicity we implement the cls layer as a two-class softmax layer." This contradicts the loss function formulation where they use "log loss over two classes" - which is appropriate for a two-class problem, but the initial description implies a multi-class problem. This inconsistency creates confusion about how the classification layer is implemented and trained.

Second, the paper contains a significant inconsistency in the training details. On page 5, it states "In the third step, we use the detector network to initialize RPN training, but we fix the shared conv layers and only fine-tune the layers unique to RPN." However, on page 4, it states "We tune all layers of the ZF net, and conv3_1 and up for the VGG net to conserve memory [5]." This directly contradicts the "fix the shared conv layers" statement. If they are tuning conv3_1 and up for VGG, they are not fixing the shared conv layers as claimed.

Third, the paper fails to discuss relevant recent work. The bibliography includes "R-CNN minus R" by Lenc et al. (2015), which proposes a method to reduce region proposals, but this work is not mentioned or compared in the related work section. This omission is particularly problematic since the paper's core contribution is about improving region proposals, and this recent work directly addresses similar problems.

Fourth, there are minor but persistent technical inconsistencies. The anchor box sizes are written as "1282" (should be 128²), the reference to [11] is missing in the main text, and the description of stride calculation is confusing ("total stride on the last conv layer is 16 pixels on the re-scaled image, and thus is ∼10 pixels on a typical PASCAL image"). The paper also inconsistently uses "Fast R-CNN" and "Fast R-CNNs" throughout.

Fifth, the paper lacks clarity on key implementation details. The claim that "the effective running time for proposals is just 10 milliseconds" is not supported by sufficient detail - it's unclear if this is for the RPN alone or including the detection network. Similarly, the paper states "Using the expensive very deep models of [19], our detection method still has a frame rate of 5fps (including all steps) on a GPU," but doesn't specify which specific architecture from [19] is used.

**Conclusion:**
While the paper presents a valuable contribution to object detection with the RPN framework, the technical inconsistencies in the loss function, training procedure, and missing discussion of relevant recent work are substantial flaws. These issues undermine the paper's technical soundness and completeness. As a perfectionist reviewer, I believe these minor but critical flaws outweigh the paper's significant contributions and prevent it from meeting NIPS' standards for technical rigor.

The paper has strong potential but requires substantial revision to address these technical inconsistencies before it can be considered for publication at NIPS. The omission of "R-CNN minus R" is particularly concerning given the paper's focus on region proposals.

**Recommendation:** Reject
