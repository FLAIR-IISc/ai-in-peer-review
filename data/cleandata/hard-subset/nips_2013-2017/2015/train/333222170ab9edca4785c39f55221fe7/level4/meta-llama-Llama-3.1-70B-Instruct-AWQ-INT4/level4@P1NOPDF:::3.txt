This paper presents a hybrid approach that leverages both crowdsourced labels and an incrementally trained model to address prediction problems. The core idea is to rely heavily on the crowd during the system's initial stages, learn from the acquired labels, and gradually reduce the crowd's involvement as the model becomes more confident. This approach is formulated as a stochastic game based on a Conditional Random Field (CRF) prediction model, where the system and the crowd are both players. The system can issue one or more queries q for tokens x (with true label y) to elicit responses r, each associated with a utility U(q,r); the system aims to select actions that maximize the expected utility.

Queries are not issued simultaneously but rather at specific times s (with response times t), and the utility is maximized with respect to a deadline t_deadline by which an answer must be computed. This determines the number of queries sent, their rate, and so on. Computing the expected utility requires the simulation dynamics model P(y,r,t|x,q,s) to calculate the utilities as in equation (4). Given the utility values, the optimal action can be chosen; however, the introduction of continuous time makes this optimization intractable, and an approximation based on Monte Carlo Tree Search and TD learning (Algorithm 1) is used instead.

Experiments were conducted on a simplified named entity recognition (NER) task, a sentiment recognition task, and a face identification task, using four methods: the majority vote of n human judges (1, 3, 5), online learning (with true labels), the "threshold" baseline (the authors' model without continuous time, where m queries are sent at each step until the model's uncertainty is reduced below a threshold), and the authors' full model (LENSE). In terms of precision, recall, and F1 score, the full model outperforms all but the 5-vote crowdsourced version, although the "threshold" baseline performs nearly as well (and with lower latency for NER; Table 2). The authors also demonstrate that the model requires fewer crowdsourced labels over time, given a sufficiently strong model to train (Figure 3), and that compared to online learning, the accuracy is high from the first example (Figure 4), as the system can leverage crowd expertise more heavily while the model uncertainty is still high.

This paper is highly original and significant, and its results against reasonable baselines are impressive. As such, I believe the work deserves to be included in the conference. However, the paper's clarity is a major concern, which could detract from its potential impact. Many of these issues can be addressed (even in the camera-ready version) if the authors are willing to invest time and effort and adjust the tone regarding the "threshold" baseline versus the full continuous time version.

The primary issue with clarity arises from the introduction of continuous time, starting in Section 3. Although the motivation is reasonable (i.e., operating within a time and cost budget for each classification), experimentally, it seems to have little benefit, while the cost is a substantially more complex formulation and discussion (as well as optimization procedure). In fact, Table 2 suggests that the "threshold" baseline achieves significantly lower latency on the NER task while maintaining roughly equivalent performance (marginally worse); on the face ID task, it performs better (though still roughly equivalent). The authors argue (line 314) that the baseline performs better on the second task because it's a single-label prediction task with no significant interaction between labels; however, the NER task, which has such interaction, only seems to benefit marginally from this.

For many practical tasks, the "threshold" baseline will be sufficient and already represents a significant gain (in terms of being a principled hybrid learning system that outperforms active and online learning). The paper would likely have greater impact if the authors made this clear in the development, i.e., the core method could be developed fully (without introducing continuous time, making it easier to understand and implement), and the addition of continuous time could be shown as an extension.

When discussing the results, the authors could be more explicit that the "threshold" method's performance is quite strong, even in cases with sequence dependence between tasks. This is a suggestion, not a requirement, as I believe the paper is strong enough for inclusion in its current form. However, I think this would improve the paper and increase the chances of the technique being widely understood and adopted.

There are several smaller issues with respect to the results and how they are reported. First, the introduction states that the model "beats human performance," but this is not true in the 5-vote case for NER. Strangely, the 5-vote case is missing for the Face-ID and Sentiment tasks in multiple tables and figures (the right half of Tables 2, Table 3, and Figure 4); it should be included. Likewise, in Table 2, the results for the 5-vote case should be bolded, as they represent the best performance. The latency value for the Threshold in NER should also be bolded for the same reason (and the LENSE value for Face ID). More importantly, the "Threshold" baseline is missing for the sentiment experiment and does not appear in Table 3 or Figure 4; it should be included as well.

A few minor comments regarding the method description:

* While there is a discussion of t_deadline in lines 183-187, it is not clear what t_deadline was used in the experiments or what criteria were used to determine when the answer was ready to be reported. Was it an entropic threshold, as with the baseline, or reaching some t_deadline?
* In equation (2), it seems the distribution needs to be given the variable s as well (i.e., p(y,r,t|x,q,s)).
* In equation (5), the use of q in F(q) is very confusing; I assumed it referred to a query q, but in fact, it represents a nominal distribution function. Please choose a different letter, as there are plenty to choose from.
* In the description of the Threshold baseline (lines 246-252), I assume the expression should be (1 - ptheta(yi|x))*0.3^m <= (1-0.88) (note the RHS) rather than >= 0.88, as increasing m will reduce uncertainty, not increase it.
