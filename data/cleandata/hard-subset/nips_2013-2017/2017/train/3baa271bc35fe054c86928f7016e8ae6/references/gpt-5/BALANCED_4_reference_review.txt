Summary and relation to prior work
The paper addresses robust submodular maximization in a streaming setting. The task is to precompute, in one pass and with sublinear memory, a summary S that is robust to the deletion of up to m elements, so that later—after any removal E with |E| ≤ m—a good size-k solution can be extracted using only S \ E. The authors propose a two-stage method:

- Streaming stage: STAR-T builds a structured summary S organized into partitions of buckets. Partition i contains O(k/2^i) buckets of capacity min{2^i, k}, and uses a greedy-threshold rule with thresholds that decrease exponentially with i. Each incoming element is inserted into the first non-full bucket where its marginal gain exceeds the corresponding threshold.
- Query stage: Given E, run GREEDY on S \ E to obtain a k-set Z.

The main result is a single-pass streaming algorithm using O((k + m log k) log^2 k) memory (including the parallel thresholds needed to remove dependence on unknown f(OPT)) and achieving a constant-factor approximation f(Z) ≥ c f(OPT(k, V \ E)), with c → 0.149 as k grows. The analysis hinges on (i) existence of a partition with many full buckets yielding a bucket that remains valuable after deletion, or (ii) the last partition containing an underfilled bucket that either already has large value or forces OPT’s value to be captured earlier, both controlled via the exponentially decreasing thresholds. For unknown OPT, they adapt the SIEVE-STREAMING multi-guess approach to run multiple instances.

The work builds on and relates to: classical greedy (Nemhauser et al.), streaming submodular maximization via thresholding (SIEVE-STREAMING; Badanidiyuru et al.), robust non-streaming submodular maximization (Krause et al.; Orlin et al.), and the non-uniform partitioning offline method PRO (Bogunovic et al.). It also discusses concurrent work on deletion-robust streaming (Mirzasoleiman et al.), which attains a 1/2 − ε approximation with O(m k log k/ε) memory; the present paper trades a smaller approximation constant for a substantially better (logarithmic in k) dependence of memory on k.

Empirically, they evaluate on (i) dominating set on Twitter and Amazon graphs, and (ii) MovieLens personalized recommendation with a combined score/facility-location objective. STAR-T-GREEDY and STAR-T-SIEVE perform comparably to SIEVE-STREAMING even when the latter is given E in advance. They also demonstrate reusability of the same robust summary across multiple E’s.

Strengths
- Novel robust streaming design: The partitioned, exponentially thresholded structure is a clean and original extension of SIEVE-STREAMING to the robust setting. The bucketing/threshold interplay is exploited carefully in the analysis.
- Theoretical guarantees: First constant-factor guarantee for robust streaming submodular maximization with memory O((k + m log k) log^2 k). Memory depends only logarithmically on k (and linearly on m up to a log factor), which is a strong point relative to concurrent work.
- Practical two-stage workflow: The query-stage decoupling (GREEDY on S \ E) is natural and useful in interactive scenarios (“right to be forgotten”, preference shifts).
- Experiments: Demonstrate robustness to both random and adversarial deletions and to removals much larger than k, and show competitive performance to strong baselines.

Weaknesses and points needing clarification
- Approximation constant and tradeoffs: The constant 0.149 is modest. While acceptable for a first robust streaming result with strong memory bounds, a discussion of whether better constants are possible without sacrificing memory would help situate the contribution.
- Per-element processing cost: STAR-T checks all partitions and buckets until the first threshold-satisfying, non-full bucket is found. Worst-case marginal-gain oracle calls per element scale as O(w k log k), which can be substantially larger than SIEVE-STREAMING’s per-element cost. There is no runtime analysis or wall-clock reporting. This is important for streaming applicability.
- Dependence on unknown OPT and overhead: The multi-instance variant increases memory by a factor of log_{1+ε} k. The paper should clearly specify (i) the range and granularity of τ guesses, (ii) the total per-element time across instances, and (iii) how this compares empirically with baselines under a fixed memory/time budget.
- Clarity issues in thresholds: There is an inconsistency between the text (“threshold τ/2^i”) and Algorithm 1 (line 5 uses τ/min{2^i, k}). The final partition’s threshold being τ/k is consistent with the algorithm, but the exposition should be unified. Also, some arguments (e.g., Lemma 4.2) use phrasing like “on average,” whereas the guarantee is worst-case over E; the proof should use a precise counting/pigeonhole argument (which seems to be intended) rather than suggest average-case reasoning.
- Empirical reporting: Experiments do not report memory used (|S|) and time. In particular, STAR-T may store many more elements than SIEVE-STREAMING; hence, “comparable quality” may come at higher memory/time cost. Ablations over m and w (which controls memory) are missing. A fair comparison at matched memory budgets would be informative.
- Baselines: No comparison to concurrent deletion-robust streaming [18], even at a small scale, is included (perhaps due to concurrency). A brief empirical or at least synthetic comparison, or a discussion of practical regimes where STAR-T’s memory advantage dominates, would strengthen the paper.

Quality
The algorithm and analysis are technically sound, and the proofs are plausible given the sketches. However, some proof steps are relegated to the supplement; tightening the “on average” statements and unifying threshold definitions would improve rigor. Empirical evaluation shows promise but lacks runtime/memory metrics and ablations.

Clarity
Generally well written and organized, with a clear high-level picture. Please fix the threshold inconsistency (τ/2^i vs τ/min{2^i, k}), explicitly state per-element computational complexity, and more clearly explain the multi-τ instantiation (number of instances, range of τ).

Originality
The partitioned thresholding structure for robust streaming is original and non-trivial. The analysis combining bucket occupancy, deletions, and threshold levels is novel. The work is clearly differentiated from SIEVE-STREAMING, PRO, and the concurrent deletion-robust paper.

Significance
Robustness is increasingly important, and a streaming solution with sublinear memory and a constant guarantee is a valuable addition. The memory-approximation tradeoff relative to concurrent work provides a complementary tool. With better reporting of computational costs and parameter sensitivity, the impact could be substantial.

Pros for acceptance
- First constant-factor robust streaming algorithm with near-logarithmic dependence on k in memory.
- Clean, general technique (partitioned thresholds) likely of interest beyond this specific problem.
- Empirical results on real datasets showing competitive performance and practical utility.

Cons for acceptance
- Modest approximation constant.
- Missing runtime and memory comparisons; per-element cost likely higher than baselines.
- Expositional inconsistencies (thresholds) and reliance on supplemental proofs for key steps.
- No empirical comparison to concurrent deletion-robust streaming work.

Suggestions for improvement
- Report summary size |S| and wall-clock time for streaming and query stages; add per-element oracle counts.
- Provide ablations over m and w, including matched-memory comparisons to SIEVE-STREAMING.
- Clarify and unify the threshold rule throughout; tighten the proofs to avoid “on average” phrasing.
- Detail the multi-τ scheme: guess range, instance count, and overhead; discuss how to choose ε in practice.
- If possible, include a small-scale comparison to [18], or a clear empirical regime analysis demonstrating STAR-T’s advantage.
- Consider lazy evaluations to reduce per-element computation.

Recommendation: Accept (borderline strong). The theoretical contribution is solid and novel, and the empirical results are promising; with the suggested clarifications and additional reporting, this would be a strong NeurIPS paper.
