Summary
The paper tackles robust submodular maximization in the single-pass streaming setting with deletions. It proposes STAR-T, a partitioned-bucket structure with exponentially decreasing marginal thresholds, and a two-stage procedure: (i) build a robust summary S in one pass; (ii) at query time, after removal of up to m elements, run GREEDY on S\E to return a k-set. Theoretically, for w ≈ (m log k)/k buckets per partition, the summary size is claimed to be O((k + m log k) log k) (though the abstract/conclusion state O((k + m log k) log^2 k)), and the output achieves a constant-factor approximation: f(Z) ≥ c f(OPT(k, V\E)) with c → 0.149 as k grows. A parallel-instantiation variant removes the need to know f(OPT) up to a (1+ε) factor and an extra log factor in space. Experiments on dominating set (Twitter/Amazon graphs) and movie recommendation show STAR-T-GREEDY comparable to SIEVE-STREAMING even when the latter knows removals E in advance.

Relation to prior work: The approach extends SIEVE-STREAMING-style thresholding to a robust setting and borrows a partitioned structure reminiscent of PRO (Bogunovic et al., ICML’17), but fills buckets via thresholds rather than sequential greedy. It contrasts with concurrent deletion-robust streaming work (Mirzasoleiman et al., ICML’17) that attains a 1/2−ε approximation with memory O(mk log k/ε); here the trade-off is a smaller constant for reduced memory dependence on k.

Strengths
- Addresses an important practical setting (streaming, deletions) with a clean one-pass summary and efficient query-time recomputation.
- Nontrivial analysis that combines partitioning and thresholding to obtain deletion-robust guarantees with near-linear summary size in k plus polylog factors and an m log^2 k term.
- Clear articulation of the memory–approximation trade-off in comparison to concurrent work.
- Experiments include two application domains and a favorable baseline (SIEVE-STREAMING) that is allowed to know E, which is conservative.

Weaknesses and concerns
- Approximation constant c ≈ 0.149 is quite low; no discussion of whether this is an artifact of analysis vs. intrinsic. Competing streaming work achieves ~1/2 at higher memory.
- Practical dependence on τ ≈ f(OPT) is significant. While the multi-instance workaround is standard, the paper does not explain how τ was set in experiments, nor does it report the overhead (extra instances, space/time).
- Inconsistencies/clarity issues:
  - Space bound alternates between O((k + m log k) log k) (Theorem 4.1) and O((k + m log k) log^2 k) (Introduction/Conclusion).
  - Threshold description says τ/2^i in the text, but Algorithm 1 uses τ/min{2^i, k}. Explain and unify.
  - The formula for τ in Theorem 4.1 appears garbled (units don’t match).
- Experimental evaluation is thin for a streaming/robust paper:
  - No reporting of memory usage, number of maintained instances (for τ), or update/query times; no sensitivity to m, w, or stream order.
  - Robustness is illustrated with |E| = k and 2k, but the theory targets up to m; the link between m, chosen w (often set to 1), and empirical robustness is unclear. If w=1 while m is large, guarantees do not apply.
  - Removals are taken from S, which indeed hurts STAR-T, but also focuses the adversary only on the summary rather than the full ground set; the experimental protocol and adversary model should be justified more carefully.
  - No empirical comparison to the concurrent deletion-robust streaming algorithm (even if only approximate, discussing practical memory/runtime trade-offs would help).
- Reproducibility: many key details are missing (oracle calls, stopping thresholds, hyperparameters, data pre-processing).

Quality
The theoretical development appears sound in structure, but the presentation has inconsistencies and some formulas seem malformed. The constant is modest. The experimental section supports functionality but is not rigorous enough to validate robustness vs. established baselines across settings, and does not quantify the promised memory/time advantages.

Clarity
Generally understandable, but the noted inconsistencies (threshold rule, space bounds, τ specification) and some typesetting issues detract. The algorithm is specified, but practical guidance on setting τ, w, and handling unknown m is limited.

Originality
Incremental but meaningful: combines partitioned buckets (cf. PRO) with streaming threshold assignment for deletion robustness. The particular analysis to get memory O(k log k + m log^2 k) with a constant approximation appears new.

Significance
The problem is significant. The trade-off (smaller constant vs. much smaller memory than O(mk log k)) could be valuable in practice. However, without stronger empirical evidence (scaling, robustness across m, and practical overhead of τ-approximation), impact is uncertain.

Questions for authors
- How was τ chosen in experiments? If parallel instances were used, how many and with what space/time overhead?
- What are actual summary sizes, update times per element, and query times as a function of k, m, and w?
- How should practitioners set w and handle unknown m at stream time?
- Please reconcile the O((k + m log k) log k) vs. O((k + m log k) log^2 k) claims and the threshold definition.
- Can you include a comparison (empirical or analytical) to the deletion-robust streaming method with 1/2−ε guarantee?

Pros for acceptance
- Addresses an important, timely problem with a principled streaming design.
- Nontrivial theoretical guarantee with attractive memory dependence on k and m.
- Potentially useful two-stage primitive for “right to be forgotten” and interactive scenarios.

Cons for acceptance
- Low approximation constant; unclear practical competitiveness.
- Inconsistent presentation and missing critical implementation details (τ, memory/time).
- Limited and not fully convincing empirical validation; no runtime/memory reporting; no comparison to the most relevant robust streaming baseline.
- Guarantees hinge on parameters (w, m) that are not matched in practice (w=1).

Recommendation
Borderline reject. The theoretical idea is interesting and potentially impactful, but the paper needs clearer, consistent statements, stronger and more transparent experimental validation (including resource usage), and a careful discussion/comparison to concurrent robust streaming work. If these issues are addressed, I would be more inclined to support acceptance.
