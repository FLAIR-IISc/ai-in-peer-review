Summary and relation to prior work
The paper studies robust submodular maximization in a streaming, two-stage setting. In the streaming stage, the algorithm sees elements once and must store a compact summary S that is robust to the subsequent removal of up to m elements. In the query stage, given any removal set E with |E| ≤ m, the algorithm returns a size-k subset Z ⊆ S \ E with value provably within a constant factor of OPT(k, V \ E).

The main algorithmic contribution is STAR-T, which organizes the stored summary into partitions and buckets: partition i has buckets of capacity min{2^i, k} and threshold τ/min{2^i, k}, and an arriving element is placed in the first non-full bucket whose marginal exceeds the respective threshold. The query stage runs GREEDY on S \ E. The analysis yields a constant approximation factor c ≈ 0.149 (as k → ∞) provided w, the number of buckets per partition, is set as w ≥ ⌈4 m log k / k⌉. The memory then scales as |S| = O(k log k + m log^2 k). A standard “multi-instance” trick (as in SIEVE-STREAMING) removes the need to know OPT by running parallel instances with geometrically spaced τ, incurring an extra O(log k) factor.

Conceptually, STAR-T extends ideas from SIEVE-STREAMING (greedy thresholding) to a robust setting via a new exponentially decreasing threshold schedule across partitions. The robust structure resembles the non-streaming PRO approach (Bogunovic et al., ICML 2017), but STAR-T differs in how it assigns elements using thresholds rather than serially filling buckets. The work is concurrent with Mirzasoleiman et al. (ICML 2017), who obtain a 1/2 − ε guarantee but with memory O(m k log k / ε); STAR-T trades a lower approximation factor for substantially better dependence on k (logarithmic rather than linear) in memory.

Strengths
- Addresses an important and timely problem: robust data summarization in the streaming model.
- Clear, modular two-stage design; the query stage supports multiple removals without rescanning the stream.
- Novel partitioned thresholding structure with an analysis that yields a constant-factor guarantee under deletions.
- Memory scales as O(k log k + m log^2 k), improving dependence on k relative to concurrent work.
- Empirical evaluation on dominating set and personalized recommendation shows performance comparable to strong baselines, including methods that know E in advance.
- Practical variant using multiple τ instances to remove dependence on OPT follows well-established streaming paradigms.

Weaknesses
- Clarity and consistency issues:
  - The space bound is stated inconsistently: the text alternates between O((m log k + k) log^2 k) and O((k + m log k) log k). From the derivation |S| ≤ (log k + 5) w k with w ≈ Θ(m log k / k), the bound appears to be O(k log k + m log^2 k). Please reconcile all occurrences (including the Conclusion and Contributions).
  - Threshold specification is inconsistent: the prose says τ/2^i while Algorithm 1 uses τ/min{2^i, k}. Since i runs to ⌈log k⌉, these differ in the last partition; clarify intended thresholds.
  - The expression for τ in Theorem 4.1 appears malformed (missing parentheses/denominator); please restate cleanly.
- The constant 0.149 is quite small compared to 0.5 − ε in concurrent work; the practical relevance of the worst-case bound vs. the improved memory tradeoff deserves more discussion.
- The algorithm assumes knowledge of m to set w; guidance for unknown/variable m (e.g., choosing for an upper bound or dynamically adjusting) is not discussed.
- Experimental evaluation, while suggestive, is limited:
  - No comparison to the concurrent deletion-robust streaming algorithm [18] (even if concurrent, a post-submission update or empirical note would be valuable).
  - Limited sensitivity analyses: e.g., how performance varies with m, w, τ-approximation granularity, adversarial vs. random removals, and memory/time usage.
  - For dominating set, removals are chosen from S, which is harsher for STAR-T than for baselines; this should be clearly justified and complemented with experiments where E ⊆ V is chosen first and then filtered against S.
- Query-time complexity is GREEDY on S \ E; a discussion of its time/memory tradeoffs vs. SIEVE in the second stage (beyond value) would be useful for practitioners.

Quality
The high-level arguments and structure of the proof are plausible. The partition/bucket design and the two complementary case analyses (many full buckets vs. few full buckets) are sensible, and the constants come from standard GREEDY bounds. However, given the textual inconsistencies, I would like the camera-ready to carefully fix notation, thresholds, and all space/approximation constants, and to include a more explicit statement and intuition for the balancing step that yields the final 0.149 constant.

Clarity
Generally well-written and easy to follow at a high level. The noted inconsistencies (thresholds, space bounds, τ expression) should be corrected. It would help to:
- Provide a small illustrative example of how elements traverse partitions/buckets and why robustness emerges.
- Explicitly state the final memory bounds in the three cases: known τ, unknown τ with parallel instances, and practical w choices (e.g., w = 1).
- Summarize time complexity per element and per query.

Originality
The combination of partitioned buckets with exponentially decreasing thresholds for robust streaming appears novel and clearly differentiated from PRO (centralized greedy filling) and SIEVE-STREAMING (non-robust). The tradeoff against the concurrent 1/2 − ε algorithm is meaningful.

Significance
A practical and theoretically grounded advance for robust streaming summarization with attractive memory scaling. If the analysis and implementation details are solidified, this could be a useful building block for large-scale summarization systems that need to support deletions or multiple downstream queries.

Pros for acceptance
- Novel algorithmic idea and nontrivial analysis yielding constant-factor robustness in streaming with improved memory scaling in k.
- Practical, multi-query two-stage interface.
- Empirically competitive, sometimes outperforming a strong streaming baseline even when disadvantaged.

Cons for acceptance
- Several clarity/consistency issues in the main text (thresholds, space bounds, τ).
- Lower worst-case approximation factor than concurrent work; limited empirical comparison and ablations.
- Limited discussion of unknown m and of query-stage efficiency tradeoffs.

Suggestions for improvement
- Fix all notation and bound inconsistencies; restate Theorem 4.1 cleanly with an explicit τ range and final constants.
- Add sensitivity experiments varying m, w, and the number of τ-instances; report memory and wall-clock time for streaming and query stages.
- If possible, include an empirical comparison to the concurrent deletion-robust streaming method or at least a principled discussion of the tradeoffs.
- Discuss strategies for unknown or time-varying m (e.g., choosing w for an upper bound on m).
- Release code and datasets to facilitate reproducibility.

Recommendation: Weak accept (assuming the inconsistencies can be corrected)
Confidence: Medium-high
