

This paper introduces a method for generating image saliency masks that processes 100 images of size 224×224 per second on a standard GPU. The approach trains a masking model to predict the tightest bounding box containing the salient region for a target class, using a pre-trained classifier (e.g., ResNet, AlexNet) as a black box. The model leverages multi-scale feature maps (e.g., from ResNet-50), where lower-level features undergo initial localization via a feature filter, followed by upsampling blocks for mask refinement. Experiments demonstrate superior performance over existing weakly supervised localization methods on ImageNet.

The work is well-motivated, with a clear technical contribution and thorough evaluation. The related work section is sufficiently comprehensive, and the technical approach appears sound (based on the provided details). The presentation is well-organized and clearly written, with strong discussion of the design rationale and insights.

However, I have three key concerns regarding the practicality and completeness of the evaluation:

1.  **Real-time claim validity:** The reported speed (100 images/sec) likely includes image resizing to 224×224. Could the authors clarify whether this speed encompasses the entire pipeline (including resizing), and provide runtime estimates for larger input resolutions (e.g., 640×480, common in smartphone images like iPhone 6s)? If resizing is included, the speed for native-resolution inputs may be significantly lower, impacting real-world applicability.

2.  **Small-object handling:** The evaluation focuses on relatively large salient objects (as 224×224 is the input size). How does the method perform when the target object is small within the image (e.g., a bird in a landscape, where the object occupies <5% of the frame)? Will the 224×224 cropping inherently discard critical contextual information for small objects, degrading performance? Ablation studies on object size would strengthen the claim.

3.  **Table 3 comparability:** Table 3 reports metrics for the proposed method but lacks direct comparison to methods listed in Table 2 (e.g., Feed [2]). To fairly assess the improvement, Table 3 should include the relevant metrics (e.g., localization accuracy) for competing approaches from Table 2, or clearly justify why such comparisons are omitted.

Minor corrections are needed:
*   Line 273: "More over" → "Moreover"
*   Line 151: "optimise" → "optimises"

Overall, this is a strong technical contribution with compelling speed and accuracy results. Addressing the concerns above—particularly regarding input resolution scalability, small-object performance, and Table 3 comparability—would significantly enhance the paper's impact and robustness for real-world deployment. The core idea is novel and well-executed, and the results are promising for weakly supervised localization. I recommend acceptance pending these clarifications.
