**Summary**

This paper introduces the "Radon Machine," a novel parallelization scheme for regularized risk minimization algorithms. The core idea is to train a base learning algorithm on numerous small, disjoint subsets of the data in parallel, and then to iteratively aggregate the resulting hypotheses using an aggregation tree. Unlike common averaging-based approaches, the aggregation step at each node of the tree computes a Radon point of its children's hypotheses. The main theoretical contribution is a proof that this aggregation process improves the confidence bound doubly-exponentially with the height of the tree. This allows the proposed scheme to achieve the same error/confidence guarantees as a sequential algorithm in polylogarithmic time, albeit requiring a quasi-polynomial number of processors and an increased sample size. The authors connect this result to the theoretical computer science question of whether machine learning problems can be solved in Nick's Class (NC). Empirically, the Radon Machine is shown to achieve significant speed-ups over sequential base learners and competitive performance against Spark MLlib implementations on several large datasets.

**Strengths**

1.  **Originality:** The application of iterated Radon points as a mechanism for aggregating hypotheses in a parallel learning context is a highly novel and creative contribution. It provides a principled alternative to the more common and often less effective method of model averaging.
2.  **Strong Theoretical Result:** The doubly-exponential reduction in the probability of generating an ε-bad hypothesis (Theorem 3) is a powerful theoretical result. It provides a compelling argument for why this method might be superior to averaging, for which known bounds are much weaker.
3.  **Black-Box Approach:** The proposed scheme treats the base learning algorithm as a black box, making it broadly applicable without requiring algorithm-specific modifications. This is a significant practical advantage.
4.  **Ambitious Connection:** The paper laudably attempts to bridge the gap between practical parallel machine learning and fundamental theoretical questions of computational complexity, specifically NC-learnability.

**Weaknesses**

Despite the novel core idea, the paper suffers from a considerable number of flaws in clarity, technical rigor, and experimental validation. These issues, in aggregate, undermine the confidence in the presented results.

1.  **Clarity and Methodological Precision:**
    *   **Ambiguous Notation:** The key result in Theorem 3, `P(Q(rh) > ε) ≤ (rP(Q(F) > ε))^2h`, is presented using non-standard and highly ambiguous notation. The text implies a doubly-exponential relationship, suggesting the exponent is `2^h`, not `2*h`. This should be written explicitly as `(rδ)^(2^h)` to avoid confusion. This is a critical point of clarity for the paper's main theoretical claim.
    *   **Incorrect Terminology:** In Algorithm 1 (line 5), the authors describe partitioning a set `S` into "iid subsets." A fixed set cannot be partitioned into independent and identically distributed subsets. The correct term would be a "random partition." This imprecise language suggests a lack of care in the methodological description.
    *   **Incomplete Description:** The computation of the Radon point is central to the algorithm, yet the details are relegated to an appendix. The claim that it can be computed in `r^3` time is asserted in the main text without justification, making the runtime analysis difficult to verify without constantly referring to supplementary material.
    *   **Formatting:** The manuscript contains minor but distracting formatting issues. The footer "st Conference on Neural Information Processing Systems..." should be removed for a submission. There are inconsistencies in citation styles (e.g., `[25]` vs. `[e.g., 6]`).

2.  **Gaps in Theoretical Analysis and Overstated Claims:**
    *   **Proof of Proposition 2:** The proof sketch in Section 7 is unconvincing. It states that for a Radon point `r` of a set `S`, there must be two points `a, b ∈ S` with worse quality. It then claims the result follows from a union bound. A union bound over all `(r choose 2)` pairs would suggest a probability bound of `(r choose 2) * P(bad)^2 ≈ (r^2/2) * P(bad)^2`, not `r^2 * P(bad)^2` as implied by the proposition's statement `(r * P(bad))^2`. This factor-of-2 discrepancy is not addressed and may represent a technical flaw or, at minimum, a significant gap in the proof.
    *   **Connection to Nick's Class:** The paper's claim to make progress towards answering a question about Nick's Class is overstated. NC is defined for algorithms that run in polylogarithmic time on a *polynomial* number of processors. The Radon Machine, as per Theorem 4 and the surrounding analysis, requires a *quasi-polynomial* number of processors. This is a fundamentally different and weaker complexity class. While related, the paper is not addressing the question of "polynomially many processors" as stated.
    *   **Restrictive Assumptions:** The analysis relies on several strong and potentially impractical assumptions. The condition that the base learner must achieve `δ ≤ 1/2r` is particularly problematic. In high-dimensional settings, the Radon number `r = d+2` is large, requiring the base learners (trained on small data subsets) to be extremely accurate. This seems to contradict the intuition of aggregating "weak" hypotheses.

3.  **Limited Empirical Evaluation:**
    *   **Low-Dimensional Bias:** The experiments are exclusively performed on datasets with a low number of features (e.g., SUSY has 18, HIGGS has 28). This is precisely the scenario where `r = d+2` is small, making the Radon Machine most viable. The paper fails to provide any evidence of its performance in high-dimensional settings, which are common in modern machine learning. The brief dismissal of this critical issue in Section 5 with a hand-wavy reference to random projections is inadequate.
    *   **Unclear Baseline Comparison:** The comparison to Spark MLlib is presented as a direct performance benchmark. However, Spark implementations often include significant overhead for fault-tolerance, data management, and general-purpose execution, which may not be present in the authors' specific implementation. This makes it difficult to ascertain if the speed-up comes from the Radon algorithm itself or from a more lightweight implementation.

**Recommendation: Reject**

While the paper presents a fascinating and novel idea, it is not yet ready for publication at NIPS. The combination of ambiguous notation for the main theorem, methodological imprecision in the algorithm's description, a potential flaw in the key theoretical proof, overstated claims regarding computational complexity classes, and an empirical evaluation that avoids the most challenging (and common) high-dimensional scenarios constitutes a significant list of weaknesses. The core concept is promising, but the paper requires a major revision to address these fundamental issues of clarity, rigor, and validation.
