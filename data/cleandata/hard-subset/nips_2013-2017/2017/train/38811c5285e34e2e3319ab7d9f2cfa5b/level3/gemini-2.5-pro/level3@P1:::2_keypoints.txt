- Core idea: Radon Machine parallelizes regularized risk minimizers by training base learners on many disjoint subsets and recursively aggregating hypotheses using Radon points instead of simple averaging.  
- Theoretical guarantee: Aggregation via Radon points yields a doubly-exponential reduction in the probability of a poor hypothesis with tree height (Theorem 3).  
- Complexity claim: Authors argue this can reduce many algorithms' runtime from polynomial to polylogarithmic given a quasi‑polynomial number of processors (connection to NC).  
- Empirical results: Shows up to 700× speedups versus sequential baselines, better accuracy than averaging-based aggregation, and competitive or superior runtimes to Spark MLlib.

Strengths
- Practicality: Black‑box scheme requiring no changes to base learners, enabling easy application to many existing algorithms/libraries.  
- Novelty/theory: Using Radon points for aggregation is novel here, with a strong doubly‑exponential error bound supporting superiority over averaging.  
- Empirical strength: Thorough experiments demonstrating large speedups and improved predictive quality compared to averaging and strong baselines.  
- Presentation: Clear, well‑structured paper with concise algorithm description and interesting complexity‑theory framing.

Weaknesses
- Prior work overlap: Potential significant overlap with prior parallel randomized convex optimization work (e.g., Feldman et al. NIPS 2012) and insufficient comparison/citation, leaving novelty unclear.  
- Unrealistic processor assumption: Polylog runtime claim depends on quasi‑polynomial processors, which is impractical—gap between theory and real systems not fully explored.  
- High‑dimensional limitation: Radon number r = d+2 makes required subsets scale with dimensionality, limiting applicability in high‑dimensional or kernelized settings without dimensionality reduction.

Detailed comments / questions
- Prior‑art clarification: Must explicitly compare and position against work using geometric medians/center points for aggregation in parallel optimization.  
- Spark comparison details: Need specifics on which Spark MLlib algorithms/configurations were used and whether speedups stem from algorithmic benefits or simpler communication.  
- Sensitivity of h: Provide a sensitivity analysis for aggregation height h showing tradeoffs between runtime, per‑worker data, and accuracy.  
- Processor count tradeoffs: Discuss performance when available processors c << r^h and practical tradeoffs for realistic cluster sizes.  
- Proof clarity: Clarify the independence assumption in the proof of Proposition 2 (the step bounding P(Q(r) > ε) via pairs a,b) and make the logic explicit.
