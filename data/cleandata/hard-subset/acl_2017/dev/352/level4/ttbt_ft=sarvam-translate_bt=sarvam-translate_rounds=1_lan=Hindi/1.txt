This paper introduces new configurations and training objectives for neural sequence models in multi-tasking settings. As the authors state well, multi-tasking settings are important because some tasks share information, and learning multiple tasks in some scenarios can improve overall performance. The methods section is relatively clear and logical, and I like where it ended, although it could have been organized a little better. The organization I understood after reading is that there are two problems: 1) shared features end up in the private feature space, and 2) private features end up in the shared space. There is a new method for each problem. That organization will make the methods more consistent. In any case, they present a method that keeps task-specific features out of the shared representation (the adversarial loss), and the second method keeps shared features out of the task-specific representation (orthogonality constraints). The only thing I'm confused about is the adversarial system.

There is another layer after the LSTM output, D(s^kT, \thetaD), which depends on the parameters U and b. This output is considered a probability distribution compared to the real one. This means it is possible that it will only learn U and b that effectively mask the task-specific information from the LSTM output, and it doesn't seem like it can guarantee removing the task-specific information.

Before reading the evaluation section, I wrote that I expected the experiments to look like this, and it mostly did. It's an interesting idea and there are a lot of experiments that one can imagine, but I think they have the basics here to show the validity of their methods. It would be helpful to have the best-known results on these tasks.

My primary concern with this paper is the lack of deep motivation for the approach. I think it's easy to understand that problems will occur in a fully shared model due to conflicts in the feature space. Extending to partially shared features seems like a response to that issue - one would expect useful shared information to be in the shared secret space, and each task-specific space would learn features for that space. Maybe it works and maybe it doesn't, but the logic is clear to me. Conversely, the authors seem to start from the assumption that this is the problem in this "shared-private" model. I was hoping that the logical flow would be 1) fully shared clearly has this problem; 2) shared-private seems to address it; 3) in practice, shared-private does not fully address this issue for reasons a, b, c; 4) we present a method that more effectively restricts the spaces.

Table 4 helped me partially understand what is going wrong with shared-private and what your methods do; some words usually have one meaning or another, and that general tendency can probably bring them into the shared feature space. This simple explanation, an example, and a more logical flow of reasoning will help with the introduction and will make it a really good paper.

Finally, I think this research is connected to some other unreferenced MTL work [1], which does deep hierarchical MTL - supervised POS tagging at the lower level, chunking at the next level, high ccg tagging, etc. Then they finally discuss some properties that make MTL possible and conclude that MTL only works "when the tasks are sufficiently similar". The ASP-MTL paper made me think about this previous work because potentially this model can learn what is sufficiently similar - that is, if two tasks are not sufficiently similar, the shared model will not learn anything and it will return to learning two independent systems, the shared-private model compared to the baseline which can overfit and perform poorly.
