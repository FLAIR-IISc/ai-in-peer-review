**Summary and Contributions**

This paper presents a model for learning multi-prototype embeddings for entity mentions by jointly leveraging a large text corpus and a knowledge base. The core idea is to represent each ambiguous mention string (e.g., "apple") with multiple sense embeddings, where each sense is intended to correspond to a specific entity (e.g., Apple Inc. or apple the fruit). The model is trained by optimizing a joint objective that models word-context relationships, entity-context relationships (from the KB graph), and the alignment between mention senses and their corresponding entities. The authors also propose a disambiguation method based on a language model that selects the most appropriate sense embedding for a given mention in context. The learned representations and disambiguation method are evaluated on a supervised entity linking task, where they report state-of-the-art performance.

As I see them, the main contributions of this work are:
1.  The development of a multi-prototype embedding model that jointly learns representations for words, entities, and mention senses from text and a knowledge base. This is a sound approach to a well-known problem.
2.  An empirical demonstration that the proposed representations, when applied to a supervised entity linking task, outperform a strong and closely related baseline (ALIGN), suggesting the potential of the multi-prototype approach.
3.  An insightful analysis of a smoothing parameter that balances model evidence with entity popularity priors, which sheds light on a classic trade-off in entity linking.

**Strengths**

1.  **Strong Empirical Results:** The primary strength of this paper is its performance on the supervised entity linking task. The model achieves state-of-the-art results, outperforming the ALIGN model, which is a highly relevant and recent baseline. This result provides compelling evidence that the proposed multi-prototype approach is effective for capturing the semantics needed for entity disambiguation.
2.  **Direct and Relevant Baseline Comparison:** The authors compare their method directly against ALIGN, a model that also jointly learns from text and a knowledge base using very similar data sources (Wikipedia text and link graph). This makes the comparison particularly valuable, as it helps isolate the architectural contribution of the proposed multi-prototype model from differences in training data or setup.
3.  **Insightful Parameter Analysis:** The paper includes a useful analysis of the smoothing parameter α, which interpolates between the model's contextual prediction and the entity's prior popularity. This analysis is valuable because it directly addresses the well-known impact of popularity bias in entity linking and provides insight into how the model can be tuned to rely more or less on this prior. This goes beyond simply reporting a final performance number and adds to the community's understanding of these models.

**Weaknesses**

1.  **Confounded Comparison with the Key Baseline:** The most significant weakness is that the experimental comparison to the ALIGN model is not entirely fair. The authors use different hyperparameters for their model (context window size of 10, vector dimension of 500) than what was used for ALIGN (context window of 5, vector dimension of 200). Both a larger context window and higher-dimensional embeddings are known to often improve performance on their own. This confounding factor makes it impossible to determine how much of the reported performance gain is due to the novel multi-prototype architecture versus simply using a more powerful hyperparameter configuration. This weakens the central claim that the proposed model itself is superior.
2.  **Lack of Clarity in Core Model and Terminology Definitions:** Several key aspects of the model and terminology are underspecified, hindering reproducibility and a full understanding of the work.
    *   The definition of an entity's neighborhood, N(e_j), is unclear. It is not stated whether the Wikipedia link graph is treated as a directed or undirected graph when defining this neighborhood. This is a critical detail, as it significantly changes the contextual information used to learn each entity's vector.
    *   The term "mention" is used ambiguously. It is unclear if the learned embeddings are for mention *types* (i.e., unique surface strings like "apple") or for specific mention *tokens* (i.e., each occurrence of "apple" in the corpus). The description suggests the model learns a set of sense embeddings for each mention type, not for each individual instance in the text, which is a subtle but important distinction. The term "mention embedding" seems to be used interchangeably with "mention sense embedding."
3.  **Insufficient Evaluation of the Disambiguation Method:** The paper proposes a novel language-model-based disambiguation component, but its contribution is hard to assess in isolation. The experiments are performed in a supervised setting, and it is unclear how the disambiguation method would perform on its own. The paper lacks a comparison to other unsupervised or lightly-supervised entity linking methods, which would help contextualize the performance of the disambiguation strategy. Furthermore, the disambiguation is done sequentially through the document, but there is no analysis of how this ordering impacts performance.

**Questions to Authors**

1.  Could you please clarify the experimental comparison with ALIGN? Can you either provide results for your model using the same hyperparameters as ALIGN (200 dimensions, window size 5) or provide results for an ALIGN reimplementation using your chosen hyperparameters (500 dimensions, window size 10)? This is crucial for validating the claim that your model's architecture is the primary reason for the performance gain.
2.  How is the entity neighborhood N(e_j) constructed? Do you treat the Wikipedia outlink graph as directed or do you convert it to an undirected graph by considering both in-links and out-links?
3.  Could you provide a more precise definition of what a "mention embedding" represents? Is there one set of prototype embeddings for a given surface string (e.g., "Paris") that is used for all its occurrences, or does the model produce a unique representation for each specific mention in its document context?
4.  Regarding the disambiguation process, have you explored the impact of the order in which mentions are disambiguated within a document (e.g., comparing a left-to-right pass with an ordering based on mention ambiguity or confidence scores)?
