

# Peer Review: Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction

## Summary and Main Contributions
This paper proposes a method for automatically annotating grammatical error correction (GEC) output with explicit error type information. The authors address a critical gap in GEC evaluation: while CoNLL-2014 and subsequent shared tasks provided error type annotations for reference texts, they did not require systems to output error type information, preventing detailed error type analysis. The main contributions are:
1. A dataset-agnostic rule-based error typing framework that relies solely on automatically obtained linguistic features (POS tags, lemmas, etc.), eliminating the need for labeled training data.
2. A demonstration of how this approach enables detailed error type analysis of GEC systems, revealing specific strengths and weaknesses across 25 error categories.
3. A publicly available tool that standardizes error type evaluation across GEC datasets.

## Strengths

**1. Addressing a genuine gap in GEC evaluation:** The paper identifies a real problem in the field: the lack of detailed error type analysis for GEC systems. Previous work (Felice and Briscoe, 2015; Bryant and Ng, 2015; etc.) focused on overall metrics but didn't provide this granular analysis. The authors' approach directly addresses this gap, making it a valuable contribution to the field.

**2. Robust rule-based error typing:** The rule-based framework is well-motivated and well-executed. By relying only on automatically obtained features (POS tags, lemmas), it avoids the dataset dependency of machine learning approaches. The manual evaluation showing 86.3% of predictions were rated "Good" or "Acceptable" by GEC experts is compelling evidence for the quality of their approach.

**3. Significant empirical demonstration:** The authors apply their method to the CoNLL-2014 shared task data, providing the first detailed error type analysis of all participating systems. This reveals unexpected insights, such as 5 teams failing to correct any unnecessary token errors (despite these being expected to be easier to fix). The analysis of error type performance across systems (Table 6) demonstrates the framework's utility in identifying specific system strengths and weaknesses.

**4. Clear framework design:** The error type taxonomy is well-structured and flexible, allowing for different levels of granularity (e.g., evaluating just replacement errors or just noun errors). The authors provide a clear rationale for their design choices, acknowledging the trade-offs between informativeness and practicality.

## Weaknesses

**1. Insufficient comparison with existing error type annotation methods:** While the authors claim previous approaches "are not currently capable of producing individual error type scores," they don't provide a thorough comparison with existing error type annotation methods. Crucially, they don't compare their automatic annotation against the manual error type annotations used in the CoNLL-2014 shared task itself. This makes it difficult to assess whether their automatic annotation truly matches the quality of manual annotation.

**2. Limited discussion of error type complexity:** The paper acknowledges that "it is always possible to add new categories for increasingly detailed error types," but doesn't address how their framework handles complex error types that might involve multiple error categories simultaneously. For example, a correction that involves both a spelling error and a word order error would be difficult to classify under their current scheme. The authors should discuss how they would handle such cases.

**3. Underdeveloped analysis of error type distribution:** The paper presents detailed error type results for the CoNLL-2014 systems, but doesn't analyze the distribution of error types in the dataset itself. Without this, it's difficult to determine whether observed system performance is due to the systems' capabilities or the nature of the errors present in the data (e.g., if certain error types are extremely rare).

**4. Inconsistent treatment of the M2 scorer:** The authors claim the M2 scorer "tends to overestimate performance," but their explanation is somewhat superficial. They note that the M2 scorer merges edits to maximize true positives, but don't provide a quantitative analysis of how much this overestimation affects the comparison between systems. This is particularly important since they use M2 as a benchmark in their analysis.

## Questions to Authors

1. How do you plan to handle cases where a single edit spans multiple error types (e.g., a correction that involves both a spelling error and a word order error)? The paper mentions that "it is always possible to add new categories for increasingly detailed error types," but doesn't provide a concrete approach for handling such cases.

2. The paper states that "our auto annotations are qualitatively as good as human annotations" based on the lack of significant difference between auto and gold references. However, Table 4 shows that M2 scores are consistently higher than the authors' approach. Given that M2 is the de facto standard in GEC evaluation, how do you reconcile this discrepancy, and what does it mean for the interpretation of your results?

3. The authors note that "few teams attempted to correct NOUN:POSS errors" (Table 6). Is this because this error type is rare in the data, or because it's particularly difficult to handle? A more detailed analysis of error type frequency would help contextualize the results.

## Overall Assessment

This paper makes a valuable contribution to the GEC field by providing a method for detailed error type analysis that doesn't require manual annotation. The rule-based framework is well-designed, the manual evaluation provides strong evidence for its quality, and the application to CoNLL-2014 reveals interesting insights about system performance. While the paper has some weaknesses in its comparison with existing methods and discussion of error type complexity, the overall contribution is significant and would be a useful tool for the GEC community. I recommend acceptance with minor revisions to address the weaknesses noted above.
