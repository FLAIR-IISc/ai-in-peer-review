

# ACL Review

## Summary of the Paper
This paper proposes an automated method for annotating grammatical error correction (GEC) data with error type information. The authors address a gap in GEC evaluation where systems are assessed using overall scores without detailed error type analysis. Their approach consists of two components: (1) linguistically-enhanced edit extraction using a modified Damerau-Levenshtein algorithm, and (2) a rule-based framework for error typing that relies solely on automatically obtained linguistic properties (POS tags, lemmas, etc.) without requiring labeled training data.

## Main Contributions (in order of strength)

1. A dataset-agnostic, rule-based error typing framework that doesn't require labeled training data and relies only on automatically obtained linguistic properties to classify error types.

2. A demonstration of how this framework enables detailed error type analysis of GEC systems, revealing that different systems have different strengths across error types (e.g., some excel at correcting certain error categories while others struggle).

3. An automatic annotation tool for GEC data that will be released with the paper, standardizing error type evaluation across GEC datasets.

## Strengths

1. **Practical solution to a genuine evaluation gap**: The paper identifies a real limitation in current GEC evaluation practices where overall scores conceal system performance across specific error types. The authors demonstrate that their framework enables detailed error type analysis for the first time for all teams in the CoNLL-2014 shared task, revealing system-specific strengths and weaknesses that could guide future research.

2. **Strong validation of the error typing framework**: The manual evaluation with 5 GEC researchers showed that >95% of predicted error types were rated as "Good" or "Acceptable" (86.3% "Good", 10.0% "Acceptable"), despite the noise from automatic edit extraction. This high agreement across raters provides compelling evidence for the framework's quality, especially given the absence of gold-standard labels for evaluation.

3. **Clear demonstration of the framework's utility**: The application to CoNLL-2014 data provides concrete insights into system performance across 25 error types and their combinations. For example, the analysis revealed that 5 teams failed to correct any unnecessary token errors, which is surprising given the relative simplicity of this error type. This demonstrates the framework's ability to uncover meaningful patterns not visible in overall scores.

4. **Transparent and reproducible methodology**: The authors emphasize that their approach is fully deterministic and transparent, unlike machine learning approaches that can be opaque. They also provide a detailed description of their error categories and rule-based classification system, making it easy for others to replicate and build upon their work.

5. **Important clarification of evaluation metrics**: The paper demonstrates that the M2 scorer tends to overestimate performance by merging edits, while their approach provides a more accurate assessment. This has significant implications for how GEC systems are evaluated and compared, adding value beyond just the annotation framework.

## Weaknesses

1. **Lack of substantial novelty**: The core techniques (edit extraction and error typing) build directly on prior work (Felice et al., 2016; Swanson and Yamangil, 2012). The main contribution is the application of these existing techniques to a new problem (automated error type evaluation), rather than a fundamentally new technical approach. The paper doesn't clearly distinguish how their framework differs from or improves upon prior error typing approaches.

2. **Limited evaluation of the error typing framework**: The manual evaluation is limited to 200 edits from only two datasets (FCE-test and CoNLL-2014), with only 5 raters. While the results are promising, a more comprehensive evaluation comparing their framework to other error typing approaches across multiple datasets would strengthen their claims. The paper acknowledges that "the best results using a classifier were between 50-70 F1" but doesn't provide a direct comparison with these classifiers.

3. **Questionable significance of the problem addressed**: The paper claims that their method "rectifies" a situation where error type analysis was impossible, but the authors themselves state that "none of these [metrics and methodologies] are currently capable of producing individual error type scores." This suggests the problem is not a technical limitation but rather a design choice in existing metrics, which diminishes the perceived significance of the contribution.

4. **Insufficient exploration of framework limitations**: The authors acknowledge their framework is a compromise between informativeness and practicality (e.g., they don't label [could → should] as a modal error because it would be hard to differentiate from other cases). However, they don't explore how these limitations affect the practical utility of their approach or provide guidance on when the framework might fail.

5. **Lack of comparison to other error typing methods**: The paper states that their rule-based approach outperforms machine learning approaches but doesn't provide a quantitative comparison with other error typing methods. They mention that "the best results using a classifier were between 50-70 F1" but don't show how their approach compares to these results.

## Questions to Authors

1. The paper states that the error typing framework is dataset-agnostic, but the evaluation was limited to two datasets. How would you expect the framework to perform on datasets with different error type annotation schemes, particularly those that use finer-grained or different error categories?

2. The authors note that their framework "aims to be a compromise between informativeness and practicality" but don't provide guidance on when this compromise might be problematic. Could you elaborate on specific error types where the framework's simplification might lead to misleading evaluations, and how researchers could address these cases?

3. The paper claims that the M2 scorer "tends to overestimate performance" but doesn't provide statistical evidence of this overestimation across multiple datasets. Could you provide more details on how you measured this overestimation and whether it's consistent across different GEC datasets?

4. The authors state that their approach "is analogous to automating the annotation guidelines given to human annotators." Could you provide specific examples of how the rules in their framework align with standard error type annotation guidelines used in the GEC community?
