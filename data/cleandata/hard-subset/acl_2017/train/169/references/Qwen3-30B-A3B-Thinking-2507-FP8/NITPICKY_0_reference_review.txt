

# ACL Review

## Summary of the Paper
This paper proposes a method for automatically annotating grammatical error correction (GEC) data with error type information. The authors claim their approach can provide detailed error type analysis without requiring manual annotation of system outputs. The method involves two steps: 1) automatically extracting edits between original and corrected sentences using a linguistically-enhanced alignment algorithm, and 2) classifying these edits using a rule-based framework dependent only on automatically obtained properties (POS tags, lemmas, etc.). The authors demonstrate their approach by applying it to the CoNLL-2014 shared task data.

## Main Contributions (as I see them)
1. A rule-based error type classification framework that depends solely on automatically obtained linguistic properties (POS tags, lemmas, etc.) rather than requiring labeled training data.
2. A detailed error type analysis of the CoNLL-2014 shared task results using their automatic annotation approach.
3. A demonstration that their automatic annotations yield comparable results to manually annotated references.

## Strengths
1. The paper identifies a genuine gap in GEC evaluation - the lack of fine-grained error type analysis in system evaluations, which is a legitimate problem in the field.
2. The rule-based error classification approach is novel in its reliance on automatic linguistic properties rather than labeled training data, potentially avoiding the dataset-specific bias of machine learning approaches.
3. The detailed error type analysis of the CoNLL-2014 results provides valuable insights into system strengths and weaknesses across different error types.

## Weaknesses

### 1. Severe Formatting and Technical Inconsistencies (Major Issue)
The paper contains multiple formatting inconsistencies that significantly undermine its credibility as an ACL submission:

- The references section is poorly formatted with numerous errors (e.g., "Human evaluation of gram, Edward Gillian, Gillian.,? \Q2015\E, Gillian.," and "The amu system in the conll-2014 shared task: Grammatical error correction by dataintensive and feature-rich statistical machine translation, Marcin Junczys-Dowmunt, Roman Grundkiewicz., Proceedings of the Eighteenth Confer-, Junczys.Dowmunt and Grundkiewicz.,? 2014, Junczys.Dowmunt and Grundkiewicz.,"). ACL requires strict adherence to formatting guidelines, and these errors are unacceptable in a submission.

- Inconsistent use of evaluation metrics: The paper uses "F1" in some places and "F0.5" in others. Table 4 states "All scores are in terms of F0.5" but the table does not consistently label columns as F0.5. For example, in Table 5, the columns are labeled "P", "R", "F0.5" but in Table 6, the same columns are used without clear labeling.

- In Table 2, the "Code" column contains values like "ADJ" but the "Meaning" column says "Adjective." This is inconsistent with the text that claims "most of them can be prefixed with 'M:','R:' or 'U:'." The table should either include these prefixes or the text should be corrected.

### 2. Inadequate Explanation of the Rule-Based Framework (Major Issue)
The paper makes the strong claim that their approach is "dataset independent and does not require labelled training data," but provides insufficient detail about the actual rules used for classification:

- The paper states they "wrote a rule to classify an edit based only on its automatic POS tags" but never provides the actual rules. How do the rules map POS tags to error types? For example, how does the system determine that "big → wide" should be classified as ADJ rather than ADV or NOUN?

- The paper claims "we extended our approach to classify errors that are not well-characterised by POS alone (such as Spelling or Word Order)" but doesn't explain how these exceptions are handled. Without these details, it's impossible to replicate their work.

- The paper states "our approach is analogous to automating the annotation guidelines given to human annotators" but never provides the annotation guidelines they're automating, making this claim unsubstantiated.

### 3. Flawed Manual Evaluation Description (Major Issue)
The paper describes a manual evaluation of their classifier but provides insufficient detail to assess its validity:

- The paper states "all 5 raters individually considered at least 95% of our rule-based error types to be either 'Good' or 'Acceptable'" but Table 3 shows rater 3 had 83% "Good" and 13% "Acceptable" (96% total), which is technically correct but misleading as it's not 95%+ for all raters. The paper should clarify that rater 3 had a lower score.

- The paper claims "inter-annotator agreement was good at 0.724 κfree" but fails to explain what "κfree" means. This is not standard terminology, and the paper should either use standard kappa notation (e.g., Cohen's kappa) or explain what "κfree" refers to.

- The paper doesn't explain how the 200 edits were selected or how the raters were instructed to handle edits with automatic boundaries. This lack of detail makes the evaluation difficult to replicate or assess.

### 4. Inconsistent Terminology and Labeling (Minor but Significant Issue)
The paper uses inconsistent terminology throughout, which creates confusion:

- The paper refers to "Operation Tier" in the text but uses "Type" in Table 5, creating inconsistency.
- The paper claims "25 error types" but Table 2 lists 24 (excluding UNK), and Appendix A mentions UNK as a possible error type without clear explanation.
- In Table 6, the paper claims "The highest F-score for each type is highlighted" but the table provided does not show any highlighting, making this claim unsubstantiated.

### 5. Unsubstantiated Claims About the M2 Scorer (Minor Issue)
The paper claims "the M2 scorer in fact tends to overestimate performance" but doesn't provide sufficient evidence or cite where exactly Felice and Briscoe (2015) and Napoles et al. (2015) make this claim. The explanation for why M2 overestimates performance is vague and lacks sufficient detail.

## Questions to Authors
1. The paper claims "our approach is analogous to automating the annotation guidelines given to human annotators." Can you provide the specific annotation guidelines you're automating?

2. In Table 2, the codes are listed without prefixes (M:, R:, U:), but the text states these can be prefixed. Can you provide an example of a full error code (e.g., R:ADJ) and explain how the prefixes are applied in practice?

3. The paper states "we took inspiration from Swanson and Yamangil's (2012) observation that most error types are based on part-of-speech (POS) categories, and wrote a rule to classify an edit based only on its automatic POS tags." Can you provide the specific rules used for classification?

4. The references section contains multiple formatting errors. Can you please ensure all references are properly formatted according to ACL guidelines?

## Recommendation
**Reject**

While the paper addresses a legitimate problem in GEC evaluation and proposes a potentially valuable method, the numerous formatting inconsistencies, technical errors, and lack of sufficient detail about the rule-based framework significantly undermine the paper's credibility. The reference formatting alone is unacceptable for an ACL submission, and the lack of detail about the core technical contribution makes it difficult to assess the paper's novelty or validity. The minor flaws identified here are not just minor issues but represent significant problems with the paper's presentation and technical rigor. Even if the method has merit, the paper as submitted does not meet the standards expected for ACL.
