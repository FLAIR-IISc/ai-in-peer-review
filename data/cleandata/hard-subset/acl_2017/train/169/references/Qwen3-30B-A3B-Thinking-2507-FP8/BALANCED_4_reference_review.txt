

# Review for "Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction"

## Summary of the Paper
This paper proposes a method for automatically annotating grammatical error correction (GEC) system outputs with error type information. The authors address a significant gap in GEC research: while reference data in shared tasks like CoNLL-2014 was annotated with error types, system outputs were not, making detailed error type analysis impossible. Their approach involves two main steps: 1) extracting edits between original and corrected sentences using a linguistically-enhanced alignment algorithm, and 2) classifying these edits into error types using a rule-based framework that depends only on automatically obtained properties (POS tags, lemmas) rather than labeled training data. The authors demonstrate their method by applying it to CoNLL-2014 shared task data, revealing new insights about system performance across different error types.

## Main Contributions (as I see them)
1. A rule-based error typing framework that is dataset-independent and does not require labeled training data, enabling consistent error type evaluation across different datasets.
2. A tool for automatic error type annotation that can be applied to any GEC system output, facilitating more detailed error analysis.
3. A demonstration of this tool's application to the CoNLL-2014 shared task data, revealing new insights about system performance across different error types.

## Strengths
1. **Addresses a genuine gap in GEC research**: The paper identifies and solves a real problem in GEC evaluation. Prior to this work, detailed error type analysis of GEC systems was impossible because system outputs weren't annotated with error types. This is a significant contribution to the field that will enable more nuanced understanding of system performance.

2. **Innovative dataset-independent error typing**: The rule-based approach is well-motivated and avoids the limitations of machine learning approaches (which require different classifiers for different datasets and may not be compatible across datasets). The small-scale manual evaluation showing that 86.3% of predicted error types were rated "Good" or "Acceptable" by experts provides compelling evidence for the quality of their approach.

3. **Clear demonstration of practical value**: The application to CoNLL-2014 data reveals interesting insights about system performance across different error types that would have been impossible without their tool. The detailed error type analysis (Tables 5-8) provides valuable information for the GEC community about which systems excel at which types of errors.

4. **High-quality tool development and release**: The authors have developed and will release a tool that will be valuable for the GEC community, enabling more detailed error analysis for future work.

5. **Strong writing and organization**: The paper is well-structured and clearly written, making the technical contributions accessible to readers.

## Weaknesses
1. **Limited novelty of the approach**: While the specific combination of edit extraction and rule-based error typing is novel, the individual components (edit extraction and rule-based error typing) are based on prior work (Felice et al., 2016; Swanson and Yamangil, 2012). The paper doesn't sufficiently articulate how their specific implementation is superior or different from existing methods, making the contribution appear incremental.

2. **Insufficient evaluation of the error typing system**: The manual evaluation is limited to 200 examples from two datasets, and the authors don't provide a detailed analysis of where errors occur or why. It would be valuable to know how often errors in POS tagging lead to misclassification, and how this affects overall performance. A comparison to machine learning approaches (even if not directly comparable) would strengthen the evaluation.

3. **Unsubstantiated claim of "dataset independence"**: The paper claims the framework is "dataset-independent," but this is not fully demonstrated. The evaluation is limited to CoNLL-2014 and FCE-test, and the authors don't show how the framework would work on a completely new dataset with different error types or annotation standards. The paper acknowledges this limitation but doesn't address it adequately.

4. **Lack of discussion about limitations**: The paper mentions that errors in POS tagging can lead to misclassification but doesn't analyze how often this occurs or how it affects the overall error type evaluation. The authors don't discuss how their framework would handle new error types that don't fit into their 25-category scheme.

5. **Limited insight into practical implications**: While the paper shows that different systems have different strengths and weaknesses, it doesn't sufficiently articulate how these insights can be used to improve GEC systems. The paper could better explain how the error type analysis can guide future research.

## Questions to Authors
1. How would your error typing framework handle error types that are not covered by the 25 categories in your framework (e.g., errors that would require new categories to describe them properly)? What systematic process would you recommend for expanding the framework?

2. The paper claims that the M2 scorer "tends to overestimate performance" compared to your approach. Could you provide more specific examples or analysis of when and why this happens, and how it might affect the interpretation of GEC results?

3. The paper mentions that "it is always possible to add new categories for increasingly detailed error types," but doesn't provide a systematic way to do this. How would you recommend expanding the framework for new error types, and what would be the implications for consistency across datasets?

4. The authors state that "our approach is analogous to automating the annotation guidelines given to human annotators." Could you elaborate on how this automation differs from the guidelines used in the CoNLL-2014 shared task?

## Additional Comments
- The paper would benefit from a more detailed discussion of how the error type analysis could be used to improve GEC systems, beyond just evaluation.
- A comparison to other error typing approaches (even if they are not directly comparable) would strengthen the evaluation of the proposed method.
- The paper could provide more insight into the practical implications of the error type analysis for GEC researchers and practitioners.

## Decision
This paper makes a valuable contribution to the GEC field by addressing a significant gap in evaluation methodology. While the novelty is somewhat limited, the practical utility of the tool and the insights revealed by its application to CoNLL-2014 data are substantial. The paper would be significantly strengthened by addressing the weaknesses noted above, particularly by providing a more thorough evaluation of the error typing system and a clearer articulation of the novelty of the approach.

I recommend **weak acceptance**. The paper is well-written, addresses an important problem, and provides a valuable tool for the community. With the suggested improvements, particularly in the evaluation and discussion of novelty, it would be a strong contribution to ACL.
