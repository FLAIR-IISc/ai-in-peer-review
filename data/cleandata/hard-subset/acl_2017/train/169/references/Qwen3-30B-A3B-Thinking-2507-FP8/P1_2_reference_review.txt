

# Peer Review for ACL Submission

## Summary and Main Contributions

This paper addresses a significant gap in Grammatical Error Correction (GEC) evaluation by proposing an automatic method to annotate system outputs with error type information. The authors present a two-stage pipeline: (1) linguistically-enhanced edit extraction using Felice et al.'s (2016) alignment algorithm, and (2) rule-based error typing based solely on automatically obtained linguistic properties (POS tags, lemmas, etc.). The main contributions, in order of strength, are:

1. **A dataset-agnostic error typing framework**: The authors introduce a rule-based error classification system that doesn't require labeled training data or depend on specific datasets, enabling consistent error type evaluation across different corpora.

2. **An automatic annotation pipeline**: The authors provide a complete method to automatically annotate GEC data with error type information, which has been missing from the field since the CoNLL-2014 shared task.

3. **Detailed error type analysis of CoNLL-2014 systems**: The paper demonstrates the utility of their approach by conducting the first comprehensive error type analysis of all systems in the CoNLL-2014 shared task, revealing system strengths and weaknesses by error type.

## Strengths

1. **Addresses a significant gap in GEC evaluation**: The paper correctly identifies that while GEC systems are evaluated on overall metrics, the inability to analyze performance by error type has limited our understanding of system capabilities. The authors' approach directly fills this gap, enabling researchers to identify which error types each system handles well or poorly.

2. **Practical and reproducible methodology**: The rule-based error typing framework is simple, dataset-agnostic, and relies only on automatically obtained properties. This avoids the need for labeled training data, making it immediately applicable across different datasets. The authors also plan to release their tool, which will be valuable for the community.

3. **Empirical validation of error typing quality**: The small-scale human evaluation (86.3% "Good" or "Acceptable" ratings) provides strong evidence that their error typing approach produces high-quality annotations. This validation is particularly important given the potential for automatic edit extraction to introduce noise.

4. **Insights into system performance**: The detailed error type analysis reveals valuable information about system strengths and weaknesses (e.g., CAMB excelled in 10 error categories, while PKU performed best on ADJ errors). This level of detail is crucial for guiding future research directions.

5. **Clear demonstration of limitations in existing metrics**: The paper effectively shows that the M2 scorer tends to overestimate performance by merging edits in a way that artificially inflates scores, which is an important insight for the field.

## Weaknesses

1. **Limited comparison with previous error typing approaches**: While the authors mention previous work, they don't provide a thorough comparison of their rule-based approach with previous machine learning methods. Without showing how their method compares in terms of F1 scores (despite different error schemes), it's difficult to assess the novelty and practical value of their approach.

2. **Insufficient evaluation of error typing reliability**: The human evaluation of 200 examples is too small to confidently claim that the error typing is reliable. A larger-scale evaluation (e.g., 1,000+ examples) would provide stronger evidence for the quality of their annotations.

3. **Lack of discussion on practical impact**: While the paper demonstrates how the approach can be used for evaluation, it doesn't clearly articulate how this would impact the development of better GEC systems. How would researchers use this detailed error type information to improve their models?

4. **Questionable claim about M2 scorer overestimation**: The paper claims the M2 scorer overestimates performance, but the evidence provided is limited to a single example. To support this claim, the authors should show systematic differences across multiple examples, perhaps by comparing M2 scores with their own approach on a larger set of examples.

5. **Potential limitations of the error type framework**: The authors acknowledge their framework is a compromise between informativeness and practicality, but they don't thoroughly discuss potential limitations. For example, their approach labels [could → should] as a tense error, which might be better classified as a modal error. This suggests their framework might not capture all nuances of error types.

## Questions to Authors

1. How does your rule-based error typing approach compare to previous machine learning approaches in terms of performance (F1 scores) when using the same error scheme? If previous approaches used different schemes, how would you standardize the comparison?

2. Given that your error typing relies on automatic POS tagging, what is the impact of POS tagging errors on your error type classification? Could you provide an analysis of error types where POS tagging errors seem to have the biggest impact?

3. The paper claims the M2 scorer overestimates performance, but the evidence is limited to a single example. Could you provide more systematic evidence of this overestimation across multiple examples?

4. In Table 5, 5 teams failed to correct any unnecessary token errors. Could you elaborate on why this might be the case, and whether this suggests a fundamental limitation in current GEC approaches?

5. How might this error type analysis approach be integrated into the development process of GEC systems to help improve them, beyond just evaluation?

## Overall Recommendation

This paper makes a valuable contribution to the GEC community by providing a practical method for detailed error type analysis. While the approach has some limitations in evaluation and comparison, the potential utility of the method for future GEC research is significant. With minor revisions addressing the weaknesses identified above, this paper would make a strong contribution to ACL. I recommend acceptance with minor revisions.
