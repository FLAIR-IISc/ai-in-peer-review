

# Review: "Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction"

## Summary and Main Contributions

This paper proposes a method for automatically annotating grammatical error correction (GEC) system outputs with error type information. The authors address a significant gap in GEC evaluation: while systems are evaluated on overall scores, detailed error type analysis is impossible because systems don't annotate their output with error types. Their approach consists of two main components: (1) linguistically-enhanced edit extraction using a modified Damerau-Levenshtein algorithm, and (2) a rule-based error typing framework that classifies edits based solely on automatically obtained linguistic features (POS tags, lemmas, etc.) without requiring labeled training data.

**Main Contributions (in order of strength):**

1. **A dataset-independent rule-based error typing framework that requires no labeled training data:** This is the paper's most significant contribution. By relying only on automatically obtained linguistic features, the framework enables consistent error type evaluation across different datasets without the need for retraining or dataset-specific classifiers.

2. **A detailed error type analysis of CoNLL-2014 results demonstrating the value of their approach:** The authors apply their method to the CoNLL-2014 shared task data, revealing nuanced performance differences between systems that would be hidden in overall scores, such as which systems excel at specific error types.

3. **A standardized error type framework with 25 categories that can be combined flexibly (e.g., M:ADJ for missing adjective errors, R:VERB:FORM for replacement verb form errors):** This provides a comprehensive and granular way to report error type performance, enabling more detailed analysis of GEC system strengths and weaknesses.

## Strengths

1. **Addresses a clear and important gap in GEC evaluation:** The paper correctly identifies that detailed error type analysis has been impossible for GEC systems due to the lack of error type annotations in system outputs. This is a genuine problem in the field that needs to be addressed, and the authors' approach directly solves it.

2. **The rule-based error typing framework is practical and innovative:** The authors' approach to error typing is dataset-independent, requires no labeled training data, and is transparent and deterministic. This is a significant advantage over machine learning approaches that would need retraining for each dataset and are often black boxes. The small-scale manual evaluation showing 86.3% "Good" or "Acceptable" ratings from experts provides reasonable evidence for the framework's quality.

3. **Demonstrates clear value through application to CoNLL-2014:** The authors successfully apply their method to the CoNLL-2014 data, revealing new insights about system performance that were previously hidden. For example, they show that CAMB excelled at missing token errors while CUUI was most successful at unnecessary token errors, suggesting complementary strengths that could inform future system development.

4. **The approach is transparent and deterministic:** Unlike machine learning classifiers, the rule-based approach allows researchers to precisely understand why a particular error type was assigned, which is valuable for debugging and improving GEC systems.

5. **The tool will be released with the paper:** This practical contribution will enable other researchers to apply the method to their own data, facilitating wider adoption and further research.

## Weaknesses

1. **Lack of rigorous comparison with existing error type annotation approaches:** The paper mentions that their rule-based approach outperforms machine learning classifiers (which had 50-70 F1), but the comparison is not fair because the metrics and error schemes differ. The authors don't provide a direct comparison with existing error type annotation methods on the same dataset and evaluation metric, making it difficult to assess the true improvement offered by their approach.

2. **Limited evaluation of the error typing framework's performance:** The manual evaluation is small-scale (200 edits), and the paper doesn't provide detailed error analysis to show where the framework fails or how it compares to human annotation quality in more rigorous terms. The paper states that the error types were "Good" or "Acceptable" in 95% of cases, but doesn't explain what "Good" or "Acceptable" means in terms of specific error types or cases where errors occurred.

3. **Unclear novelty of the error type framework:** The paper presents a new error type framework with 25 categories, but doesn't clearly explain how this framework differs from or improves upon existing error type schemes (e.g., the CoNLL-2014 error type scheme). The authors mention they chose not to create new categories for narrow distinctions (e.g., [could → should] is labeled as a tense error rather than a modal error), but don't provide a clear rationale for these design choices or how they compare to existing schemes.

4. **Insufficient discussion of limitations:** The paper acknowledges that the approach relies on POS taggers, which can make mistakes, but doesn't explore how robust the approach is to different levels of POS tagging quality or how it handles cases where the POS tagger fails. For example, how would the approach handle cases where the POS tagger mislabels words that are central to error type classification?

5. **Lack of comparison with current state-of-the-art evaluation methods:** While the authors mention that the M2 scorer tends to overestimate performance, they don't provide a thorough comparison between their evaluation approach and other existing evaluation methods. This makes it difficult to assess how much their approach improves upon current practices.

## Questions to Authors

1. The paper claims the error typing framework is dataset-independent, but how does it handle error types that might be specific to certain languages or domains? For example, how would it handle error types that are common in learner English but not in other language contexts?

2. The authors state that the M2 scorer tends to overestimate performance, but they don't provide a detailed analysis of why this happens. Could you provide more details on the specific cases where the M2 scorer overestimates, and how your approach avoids this issue?

3. In the error typing framework, the authors mention that they chose not to create new categories for narrow distinctions (e.g., [could → should] is labeled as a tense error rather than a modal error). Could you explain the specific criteria they used to decide when to create a new category versus making a compromise?

4. The paper states that the rule-based approach is "always possible to determine precisely why an edit was assigned to a particular error category," but how does this transparency help with improving GEC systems? Could you provide an example where this transparency led to actionable insights for system improvement?

5. The authors mention that the tool will be released with the paper. Could you clarify the scope of the tool? For example, does it work with any GEC system output, or does it require specific input formats?

## Overall Recommendation

The paper addresses a genuine problem in GEC evaluation and presents a practical approach to solving it. The rule-based error typing framework is a significant contribution that could have a meaningful impact on how GEC systems are evaluated and improved. However, the paper has several weaknesses that need to be addressed before it can be accepted for publication.

The main weaknesses are the lack of rigorous comparison with existing methods, insufficient evaluation of the error typing framework's performance, and unclear novelty of the error type framework. These issues need to be addressed to demonstrate the true value and contribution of the work.

**Recommendation:** **Weak Accept** (if the authors address the weaknesses in a major revision) or **Weak Reject** (if the authors cannot sufficiently address the weaknesses).

The paper has strong potential to make a meaningful contribution to the field, but the current version lacks sufficient evidence to support the claims about the framework's performance and novelty. With a major revision addressing these weaknesses, the paper could be a strong contribution to ACL.
