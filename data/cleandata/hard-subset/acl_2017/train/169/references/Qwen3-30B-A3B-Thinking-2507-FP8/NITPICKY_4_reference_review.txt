

# Review of "Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction"

## Summary of the Paper
This paper proposes a method to automatically annotate grammatical error correction (GEC) system outputs with error type information. The authors present a two-step approach: (1) extracting edits between original and corrected sentences using a linguistically-enhanced alignment algorithm, and (2) classifying these edits using a new rule-based framework specifically designed for error type evaluation. The paper demonstrates the value of this approach by applying it to the CoNLL-2014 shared task data to provide detailed error type analysis, which was previously impossible due to the lack of explicit error type annotations in system outputs.

## Main Contributions (in order of strength)
1. A dataset-independent rule-based error typing framework that relies solely on automatically obtained properties (POS tags, lemmas), eliminating the need for labeled training data
2. A detailed error type analysis of all systems in the CoNLL-2014 shared task, revealing which systems excel at correcting specific error types
3. A practical tool for automatic error type annotation that will be released with the paper

## Strengths

1. **Novel approach to error type evaluation**: The paper's rule-based framework is a significant contribution because it solves a real problem in the GEC community—lack of standardized error type evaluation across different datasets. Unlike previous machine learning approaches that require dataset-specific training, this framework is entirely dataset-independent, making cross-dataset comparisons possible.

2. **Practical demonstration of value**: The authors successfully apply their method to the CoNLL-2014 shared task, providing the first detailed error type analysis of all participating systems. This analysis reveals that different systems have different strengths and weaknesses across error types, which is valuable for future GEC research.

3. **Comprehensive error type framework**: The proposed error type framework is well-structured, with 25 error types that can be combined with operation types (Missing, Unnecessary, Replacement) to provide extremely detailed analysis. This flexibility allows researchers to focus on specific aspects of error correction performance.

4. **Small-scale manual evaluation**: The authors conducted a manual evaluation of their classifier, showing that raters considered >95% of predicted error types as "Good" or "Acceptable." This provides reasonable evidence for the quality of their rule-based approach.

5. **Clear demonstration of M2 scorer limitations**: The paper convincingly demonstrates that the M2 scorer tends to overestimate performance by favoring multiple edits over single edits, which is an important contribution to the field.

## Weaknesses

1. **Numerous formatting inconsistencies**: The paper contains multiple formatting issues that undermine its professionalism and readability. Examples include:
   - Inconsistent spacing around hyphens (e.g., "M:ADJ U:ADJ R:ADJ" vs. "M: ADJ U: ADJ R: ADJ")
   - Inconsistent table formatting (e.g., Table 3 headers "Rater Good Acceptable Bad" with percentage values, rather than categorical values)
   - Inconsistent use of "et al." in citations (e.g., "Napoles et al\.,? 2015" vs. "Napoles et al., 2015")
   - Inconsistent reference formatting (e.g., "Junczys.Dowmunt and Grundkiewicz.,? 2014" should be "Junczys-Dowmunt and Grundkiewicz, 2014")

2. **Missing technical details for reproducibility**: The paper lacks crucial technical details necessary for reproduction:
   - The exact version of spaCy (v1.6) is not specified (e.g., 1.6.0 vs. 1.6.1)
   - The exact version of NLTK is not specified
   - The specific rules used to differentiate Missing, Unnecessary, and Replacement errors are not provided
   - The paper states "Damerau-Levenshtein is an extension of Levenshtein that also handles transpositions; e.g. AB→BA" but fails to cite a reference for this common knowledge

3. **Inconsistency in error type framework**: The paper claims "most of them can be prefixed with 'M:', 'R:' or 'U:'" but Table 2 shows error types like "MORPH", "ORTH", "SPELL", and "WO" that don't follow this pattern, creating confusion about the framework's structure.

4. **Lack of concrete evidence for M2 scorer overestimation**: While the paper claims "the M2 scorer in fact tends to overestimate performance," it provides only a general explanation without specific examples or quantitative evidence. A concrete example with actual numbers would strengthen this claim.

5. **Inconsistent terminology**: The paper uses "edit spans" and "edit boundaries" interchangeably without precise definitions, which could lead to confusion for readers.

## Questions to Authors

1. Could you provide the exact version numbers for spaCy (v1.6) and NLTK used in your experiments? This is crucial for reproducibility.

2. Could you clarify why certain error types (like "MORPH", "ORTH", "SPELL", "WO") don't have the 'M:', 'R:', or 'U:' prefixes, despite the paper stating that "most of them can be prefixed with 'M:', 'R:' or 'U:'"?

3. Could you provide more details about the rule used to differentiate Missing, Unnecessary, and Replacement errors? This would help readers understand the implementation.

## Overall Recommendation
**Reject**

While the paper presents a valuable contribution to the GEC community with a practical framework for error type annotation, the numerous formatting inconsistencies, missing technical details for reproducibility, and inconsistencies in the error type framework are substantial enough to warrant rejection. As a perfectionist reviewer, I believe these minor flaws significantly undermine the paper's quality and professionalism, making it difficult to recommend acceptance despite the valuable contribution.

The authors should address these issues before considering publication in ACL. The technical details missing for reproducibility alone are a serious concern for a computational linguistics paper. The formatting inconsistencies also suggest a lack of attention to detail that would be expected in an ACL publication.
