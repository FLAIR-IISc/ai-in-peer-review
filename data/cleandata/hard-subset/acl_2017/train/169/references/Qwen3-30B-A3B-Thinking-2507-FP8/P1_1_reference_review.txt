

# Peer Review for ACL

## Summary and Main Contributions

This paper addresses a significant gap in Grammatical Error Correction (GEC) evaluation: the lack of methods to automatically annotate system outputs with error type information. The authors propose a two-step method: (1) linguistically-enhanced edit extraction using a Damerau-Levenshtein algorithm with linguistic features, and (2) a rule-based error typing framework that classifies edits based solely on automatically obtained properties like POS tags and lemmas.

The primary contribution is a dataset-agnostic method for detailed error type evaluation in GEC that doesn't require labeled training data, enabling more granular analysis of system performance. The secondary contribution is a demonstration of this method's value through a detailed analysis of the CoNLL-2014 shared task systems, revealing which approaches excel at which error types. The tertiary contribution is a tool that will be released with the paper to facilitate this analysis.

## Strengths

1. **Addresses a genuine evaluation gap**: The paper correctly identifies that previous GEC evaluation metrics couldn't provide detailed error type analysis because systems weren't required to output error type annotations. This is a significant contribution to the field, as detailed error type analysis is crucial for understanding system strengths and weaknesses.

2. **Practical dataset-agnostic approach**: The rule-based error typing framework is dataset-agnostic and doesn't require labeled training data, unlike previous ML-based approaches. This is a major practical advantage that makes the method applicable across different GEC datasets without retraining.

3. **Strong validation through expert evaluation**: The paper's manual evaluation by 5 GEC experts shows high agreement (86.3% "Good" or "Acceptable"), providing strong evidence for the validity of the approach. This is particularly impressive given that edit boundaries were determined automatically.

4. **Demonstrates clear value through analysis**: The CoNLL-2014 analysis reveals meaningful insights about which systems excel at which error types, such as CAMB's strength in Missing errors and CUUI's strength in ORTH errors. This demonstrates the practical utility of the method.

5. **Validates automatic reference processing**: The paper shows that automatic processing of reference data produces results comparable to manual annotation (p > .05 in bootstrap tests), which is crucial for the method's adoption as a standard evaluation approach.

## Weaknesses

1. **Lack of comparison to existing error type annotation methods**: The paper claims previous approaches used ML classifiers but doesn't provide a clear comparison of accuracy or practicality between their rule-based approach and existing methods. Without this comparison, it's difficult to assess the novelty and superiority of their approach.

2. **Insufficient handling of ambiguous cases**: The paper doesn't adequately address how the rule-based system handles ambiguous cases where an edit could be classified as multiple error types. For example, how would the system handle "I have eat" → "I have eaten" (could be VERB:INFL or VERB:TENSE)? This is a critical limitation for a comprehensive error typing framework.

3. **Limited analysis of error propagation from linguistic tools**: The method relies on automatic POS tagging and lemmatization, but the paper doesn't analyze how errors in these tools might propagate to error type classification. For instance, if a POS tagger mislabels a token, how does this affect the error type assignment? This is particularly important for out-of-domain text.

4. **Lack of generalizability testing**: The paper only demonstrates the method on CoNLL-2014 data. Without testing on additional datasets (e.g., FCE, BEA), it's unclear how well the method generalizes across different error types, domains, or writing levels.

5. **Limited discussion of practical implications**: While the paper shows which systems excel at which error types, it doesn't provide concrete guidance on how this analysis can be used to improve GEC systems. The paper claims this information "should help researchers," but doesn't explain how.

## Questions to Authors

1. How does your rule-based system handle ambiguous cases where an edit could be classified as multiple error types (e.g., "I have eat" → "I have eaten" for VERB:INFL vs. VERB:TENSE)? Can you provide specific examples and your approach to resolving such ambiguities?

2. The paper states your method is dataset-agnostic, but it relies on automatic POS tagging. How do you handle cases where POS taggers have errors, especially for out-of-domain text or non-native English? Have you conducted any error analysis on how these errors propagate to error type classification?

3. The paper demonstrates your method on CoNLL-2014 data but doesn't show results on other datasets. Could you provide preliminary results on another GEC dataset (e.g., FCE) to demonstrate the method's generalizability?

## Overall Assessment

This paper makes a valuable contribution to GEC evaluation by providing a practical method for detailed error type analysis. The rule-based approach is dataset-agnostic and validated through expert evaluation, addressing a genuine gap in the field. While the paper has some weaknesses regarding comparison to existing methods, handling of ambiguous cases, and generalizability, the core contribution is significant and well-validated. The detailed error type analysis of the CoNLL-2014 systems provides meaningful insights that could guide future GEC research. The paper is well-written, the method is clearly described, and the tool will be a useful resource for the community. I recommend acceptance with minor revisions addressing the weaknesses noted above.
