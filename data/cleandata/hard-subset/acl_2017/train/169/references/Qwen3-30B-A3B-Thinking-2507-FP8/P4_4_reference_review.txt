

# Review of "Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction"

## Summary and Main Contributions

This paper proposes a method to automatically annotate grammatical error correction (GEC) system outputs with error type information, addressing a gap in GEC evaluation where previous work (like CoNLL-2014) could only provide overall scores without detailed error type analysis. The approach consists of two main steps: (1) extracting edits between original and corrected sentences using a linguistically-enhanced alignment algorithm, and (2) classifying these edits into error types using a rule-based framework that depends solely on automatically obtained linguistic properties (POS tags, lemmas, etc.).

My assessment of the paper's main contributions (not as claimed by authors):

1. A practical tool for automatically annotating GEC system outputs with error type information, enabling detailed error type analysis.
2. A dataset-agnostic rule-based error typing framework that doesn't require labeled training data.
3. A comprehensive application of this tool to the CoNLL-2014 shared task data, providing the first detailed error type analysis for all participating teams.

## Strengths

1. **Addresses a genuine gap in GEC evaluation**: The paper correctly identifies that previous GEC evaluation methods (like M2 scorer) cannot provide detailed error type analysis, which is crucial for understanding system strengths and weaknesses. The authors demonstrate the practical value of their approach by applying it to the CoNLL-2014 data, revealing that different systems excel at different error types (e.g., CAMB excelled at most error categories while CUUI was strong at ORTH errors). This level of detailed analysis is valuable for guiding future research and system development.

2. **Dataset-agnostic error typing framework**: The rule-based approach is a significant technical contribution because it avoids the limitations of previous machine learning-based approaches that required different classifiers for different datasets. The authors demonstrate that their framework is robust through a manual evaluation where 5 GEC researchers rated over 86% of predicted error types as "Good" or "Acceptable" despite noise from automatic edit extraction. This shows the framework's practical utility and reliability across different datasets.

3. **Demonstrates practical utility with concrete results**: The authors successfully apply their method to the CoNLL-2014 shared task data, providing detailed error type performance metrics for all participating teams. This analysis reveals important insights, such as the fact that 5 teams failed to correctly resolve any unnecessary token errors at all, and that different approaches (MT, classifiers, language models) complement different error types. This application provides concrete evidence of the method's value.

## Weaknesses

1. **Insufficient demonstration of novelty**: While the authors claim their rule-based approach is novel, they don't sufficiently distinguish it from previous work. The rule-based approach seems like a natural extension of Swanson and Yamangil's (2012) observation that most error types are based on POS categories, rather than a fundamentally new contribution. The paper doesn't provide a clear comparison showing how their approach is superior to previous methods in a way that hasn't been attempted before.

2. **Limited evaluation of the error typing framework**: The manual evaluation (200 examples) is insufficient to establish the framework's reliability. The paper doesn't compare their rule-based approach against previous machine learning classifiers on the same dataset, making it impossible to determine if their approach is actually better. Additionally, they don't analyze error cases where the framework failed (beyond noting that these were usually due to POS tagger errors), which would help understand the approach's limitations.

3. **Lack of evidence that the approach improves GEC performance**: The paper demonstrates that their tool enables more detailed analysis, but doesn't show that this analysis leads to better GEC systems. The value of the approach is assumed rather than proven. There's no evidence that using this detailed error type analysis has led to improved system development or performance.

## Questions to Authors

1. Could you provide a quantitative comparison between your rule-based error typing framework and previous machine learning-based approaches on the same dataset? This would help establish whether your method is actually superior to existing approaches.

2. The paper mentions that the error scheme is a "compromise between informativeness and practicality" (e.g., labeling [could → should] as a tense error rather than a modal error). Could you elaborate on how you determined this compromise and whether you've considered alternative error schemes that might be more informative without being impractical?

## Additional Comments

The paper is well-written and clearly articulates the problem and proposed solution. The framework design is comprehensive and well-motivated. However, the paper would benefit from a more thorough comparison with existing error typing approaches and a clearer demonstration of the method's superiority over alternatives. The authors should also more explicitly address the limitations of their error typing framework and how these limitations might impact the interpretation of their results.
