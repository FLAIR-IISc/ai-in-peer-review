

# Peer Review: Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction

## Summary
This paper addresses a critical gap in Grammatical Error Correction (GEC) evaluation: the inability to analyze system performance by error type due to inconsistent annotation practices. The authors propose a two-stage method: 1) linguistically-enhanced edit extraction using Felice et al.'s (2016) alignment algorithm, and 2) rule-based error typing that classifies edits using only automatically obtained properties (POS tags, lemmas, etc.). They demonstrate the approach by applying it to the CoNLL-2014 shared task data, enabling detailed error type analysis of all participating systems.

## Primary Contributions
1. A rule-based error typing framework that is dataset-independent and requires no labeled training data, enabling consistent error type evaluation across different datasets.
2. A practical tool for automatic annotation that allows detailed error type analysis of GEC systems, which was previously impossible due to inconsistent annotation practices.
3. A comprehensive analysis of GEC system performance by error type across the CoNLL-2014 shared task, revealing nuanced strengths and weaknesses of different approaches.

## Strengths

1. **Addressing a Genuine Gap in GEC Evaluation**: The paper correctly identifies a significant limitation in GEC research - the inability to perform detailed error type analysis due to inconsistent annotation practices. Previous work has focused on overall correction quality metrics (like M2), but this paper provides a practical solution for analyzing error type performance, which is crucial for targeted system improvement.

2. **Methodological Soundness and Validation**: The rule-based error typing approach is well-justified and avoids the dataset-specific issues of previous machine learning approaches. The small-scale manual evaluation (with 5 raters) showing >85% "Good" or "Acceptable" ratings for predicted error types provides strong evidence for the approach's validity.

3. **Impactful Demonstration**: The application to the CoNLL-2014 shared task is particularly valuable as it's the largest publicly available GEC dataset. The detailed error type analysis reveals nuanced insights (e.g., CAMB excelling at most error types, 5 teams failing to correct unnecessary token errors at all) that would be impossible with previous metrics.

4. **Reproducibility and Community Value**: The promise to release the tool with the paper significantly increases the paper's utility for the community. The hierarchical error type framework (operation tier + error type) provides flexibility for different levels of analysis.

## Weaknesses

1. **Insufficient Comparison to Previous Automatic Annotation Methods**: While the paper claims superiority of their rule-based approach over ML methods, it doesn't directly compare to other rule-based approaches like Swanson and Yamangil (2012) or Xue and Hwa (2014). A more thorough comparison would strengthen their claim about the novelty and superiority of their method.

2. **Limited Evaluation of the Complete Pipeline**: The paper focuses on the error typing component's validation but doesn't comprehensively evaluate the entire tool pipeline (alignment + error typing). How does the final annotated output compare to human annotations? What's the error rate in the complete pipeline?

3. **Underdeveloped Analysis of System Performance**: The paper identifies which systems excel at which error types but doesn't deeply analyze *why* certain systems excel at specific error types. For example, why did CAMB's hybrid MT approach outperform others across multiple error types, while CUUI's classifier approach excelled at specific error types like VERB:FORM?

4. **Questionable Claim about M2 Scorer**: The paper claims M2 "overestimates" performance, but the evidence is somewhat anecdotal. They don't provide systematic comparison of M2 scores versus their method across multiple datasets or error types, which weakens this important claim.

## Questions to Authors

1. How does your rule-based error typing framework specifically improve upon Swanson and Yamangil's (2012) rule-based approach? What limitations of their method does your framework address?

2. The paper states that "our auto annotations are qualitatively as good as human annotations," but doesn't provide a direct comparison of final annotated output quality. How would you measure this, and would you consider a gold standard evaluation on a subset of CoNLL-2014 data?

3. The paper mentions that "most error types are based on part-of-speech categories," but doesn't discuss how your framework handles errors requiring semantic understanding (e.g., "I like to eat" vs. "I like eating"). Could you provide examples and explain your framework's handling of such cases?

4. Why did CAMB (the most successful team) attempt to correct errors from every category while other teams failed to address certain error types? What specific design choices enabled this comprehensive coverage?

## Overall Recommendation
This paper makes a valuable contribution to GEC evaluation by providing a practical method for detailed error type analysis. The rule-based error typing framework is well-designed, validated, and addresses a genuine gap in the field. While the demonstration on CoNLL-2014 is compelling, the paper would be strengthened by a more thorough comparison to previous annotation methods and a deeper analysis of why certain systems excel at specific error types. With these improvements, this work would significantly benefit the GEC community. I recommend acceptance with minor revisions.
