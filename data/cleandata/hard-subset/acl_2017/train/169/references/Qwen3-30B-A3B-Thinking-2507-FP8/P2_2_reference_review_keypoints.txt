- Summary: Proposes an automatic, dataset-independent two-step method (linguistically-enhanced edit extraction + rule-based error type classification) to annotate GEC outputs with error types without labeled training data.

- Contribution 1: Introduces a dataset-independent rule-based error type classification framework enabling detailed error-type evaluation without dataset-specific training.

- Contribution 2: Demonstrates detailed error-type analysis on CoNLL-2014, revealing nuanced system strengths and weaknesses.

- Contribution 3: Plans to release an open-source tool for automatic error type annotation to standardize GEC evaluation.

- Strength 1: Addresses a real gap in GEC evaluation by enabling fine-grained error-type analysis beyond overall scores.

- Strength 2: Strong manual validation: 86.3% of predicted error types rated "Good" or "Acceptable" by 5 experts, supporting the approach's effectiveness.

- Strength 3: Practical demonstration on a large public dataset (CoNLL-2014) that uncovers surprising system behaviors (e.g., teams failing to correct unnecessary token errors).

- Strength 4: Clear technical presentation, including detailed framework tables and multi-granularity analysis examples.

- Strength 5: Useful analysis of M2 scorer limitations, explaining why it can overestimate performance by merging multiple edits.

- Weakness 1: Limited novelty in edit extraction—relies on Felice et al. (2016) alignment method without major improvements.

- Weakness 2: No comparison with alternative automatic annotation approaches (e.g., ML-based methods), leaving relative merits unclear.

- Weakness 3: Practicality concerns over framework complexity (25 error types, 55 combinations) and whether added granularity yields meaningful insights.

- Weakness 4: Insufficient discussion of limitations, such as handling unseen error types or datasets with different error distributions.

- Weakness 5: Limited discussion of how the method concretely helps GEC system development and improvement in practice.

- Question 1: Request for concrete examples of challenging-to-distinguish error types and justification for the chosen categorization.

- Question 2: Request for specific examples showing how M2 overestimates performance and how the proposed method avoids this.

- Question 3: How to handle inconsistent human-annotated guidelines across datasets given the framework is based on such guidelines.

- Overall assessment: Meaningful, well-validated, and practical contribution that advances GEC evaluation; recommendation for acceptance despite some limitations.
