- Paper presents a method to automatically annotate GEC system outputs with error types using linguistically-enhanced edit extraction plus a rule-based typing framework and re-evaluates CoNLL-2014.  
- Core contribution: a dataset-agnostic, rule-based error-typing system using only automatic linguistic features (POS, lemmas, stems).  
- Contribution: detailed re-analysis of CoNLL-2014, producing per-system error-type breakdowns.  
- Contribution: intends to release a publicly available annotation tool for the GEC community.  
- Strength: rule-based approach is novel and avoids dataset-label dependence, showing high agreement with expert raters (>95% “Good”/“Acceptable”).  
- Strength: framework supports fine-grained, flexible error categories (e.g., R:NOUN) enabling nuanced analysis.  
- Strength: thorough re-evaluation provides insightful observations about error types and complementary strengths of systems.  
- Strength: tackles a core GEC evaluation gap by enabling automatic error-type breakdowns.  
- Strength: focus on edit-level and multi-token edits gives deeper understanding of system corrections.  
- Weakness: heavy reliance on underlying NLP tools (spaCy, stemmer, Hunspell) with no detailed sensitivity/error analysis of that pipeline.  
- Weakness: coverage limited to ~25 error types, likely missing stylistic and discourse-level errors and broader real-world complexity.  
- Weakness: human comparison is limited (5 raters, 200 edits); larger evaluation and comparison to inter-annotator agreement needed.  
- Weakness: omits analysis of error prevention (systems introducing new errors) to fully assess system quality.  
- Weakness: discussion of M2 scorer limitations is insufficient given notable differences from their edit comparison approach.  
- Question: provide detailed error analysis of the underlying NLP pipeline and its impact on typing accuracy?  
- Question: what is the computational cost and scalability of the annotation pipeline?  
- Question: how easily can the rule-based framework be extended to new error types or annotation schemes?  
- Question: have you considered metrics for systems’ tendency to introduce new errors (error prevention)?  
- Question: how does automatic annotation agreement compare to human inter-annotator agreement?  
- Additional: code release is highly anticipated for adoption.  
- Additional: some large tables (e.g., Table 6) are hard to interpret; visualization could help.  
- Additional: reference list appears incomplete.  
- Overall: valuable, well-validated contribution to GEC evaluation; recommend acceptance with minor revisions addressing the raised questions.
