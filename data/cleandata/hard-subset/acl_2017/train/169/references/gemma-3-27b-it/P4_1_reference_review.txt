This paper introduces a method for automatic annotation of grammatical errors in text, specifically aimed at facilitating more granular evaluation of Grammatical Error Correction (GEC) systems. The core idea is to combine linguistically-enhanced edit extraction with a rule-based error typing framework. The authors claim to provide a dataset-agnostic error typing system, and demonstrate its utility through a re-analysis of the CoNLL-2014 shared task data.

The primary contribution is the development and release of a tool for automatically annotating GEC system outputs with error type information. This addresses a significant gap in the field, as existing evaluation metrics often lack the granularity needed to understand system strengths and weaknesses.  A second contribution lies in the rule-based error typing framework itself, designed to be independent of specific training data and based on readily available linguistic features (POS tags, lemmas). Finally, the re-evaluation of the CoNLL-2014 data using this tool provides a valuable, detailed analysis of the performance of different systems on various error types.

**Strengths:**

1. **Addresses a clear need:** The lack of detailed error type analysis in GEC is a well-recognized problem. This work directly tackles this limitation by providing a tool and methodology to annotate corrections, enabling a more fine-grained evaluation. The ability to automatically annotate system outputs, especially when gold-standard annotations are unavailable, is a significant benefit.
2. **Dataset-agnostic approach:** The rule-based error typing system is a strength. By relying on automatically obtained features (POS tags, lemmas), the framework avoids the biases and limitations of machine learning approaches trained on specific datasets. This makes the tool more widely applicable.  The evaluation showing >95% agreement with human raters further validates the approach.
3. **Detailed analysis of CoNLL-2014:**  The re-analysis of the CoNLL-2014 shared task data provides valuable insights into the performance of different systems. Identifying specific error types where systems excel or struggle is a significant contribution and could guide future research. The finding that different approaches are better suited to different error types is particularly useful.
4. **Practicality and accessibility:** The authors explicitly state they will release the tool, increasing the potential impact of the work. This commitment to reproducibility and accessibility is commendable.

**Weaknesses:**

1. **Limited novelty of core techniques:** While the *combination* of linguistically-enhanced alignment and rule-based error typing is useful, neither technique is novel in itself. The edit extraction relies on existing work (Felice et al., 2016), and rule-based systems for error typing have been explored previously (Swanson & Yamangil, 2012). The novelty resides primarily in the specific implementation and its application to comprehensive error type annotation.
2. **Reliance on accuracy of POS tagging:** The rule-based error typing framework is heavily reliant on the accuracy of POS tagging. The authors acknowledge this in the evaluation section (where errors were often traced back to tagger mistakes), but the sensitivity of the system to POS tagging errors is a potential limitation.  The impact of different taggers or tagger errors is not extensively investigated.
3. **Evaluation limitations:**  The manual evaluation, while useful, is small-scale (200 edits) and relies on subjective ratings.  A more rigorous quantitative evaluation, even if challenging given the lack of gold-standard annotations, would strengthen the claims about the framework's accuracy. The bootstrap testing showing no significant difference between gold and auto references is helpful but doesn’t address the absolute accuracy of the automatic annotation.
4. **Lack of comparison with existing tools:** The paper doesn’t provide a direct comparison with any existing (even if limited) tools or approaches for error type annotation. Establishing a benchmark and showing clear improvements over existing methods would enhance the impact of the work.

**Questions to Authors:**

1. How robust is the error typing framework to errors in POS tagging? Have you investigated the impact of using different POS taggers on the accuracy of the annotations?
2. Could you elaborate on the limitations of the current error type scheme? Are there specific error types that are particularly difficult to classify accurately with your rule-based approach?
3. What are the computational costs associated with running the annotation tool on large datasets?
4. Will the released tool include functionality for customizing the error type scheme or adding new error types?

**Additional Comments:**

* Table 9 (complete list of error codes) is helpful but could be formatted for better readability.
* The section on “Detection vs. Correction” is interesting, but feels slightly disjointed from the rest of the paper. It could be more tightly integrated into the overall analysis.
* Consider discussing potential applications of this tool beyond re-evaluating existing datasets, such as automatic error analysis of learner writing or providing feedback to language learners.
