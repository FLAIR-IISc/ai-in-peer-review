This paper presents a method for automatically annotating grammatical error correction (GEC) system outputs with error types, aiming to address the limitations of previous evaluations which lacked detailed error-type analysis. The core idea is to combine linguistically-enhanced edit extraction with a rule-based error typing framework. The authors claim to provide a dataset-agnostic approach that is more transparent and consistent than machine learning-based methods.

**Contributions:**

1. **Automated Error Type Annotation Pipeline:** The primary contribution is a complete pipeline for automatically annotating GEC corrections with error type information, consisting of edit extraction and rule-based error typing. This addresses a significant gap in GEC evaluation, providing the means for more granular analysis of system performance.
2. **Dataset-Agnostic Rule-Based Error Typing:** The proposed rule-based error typing framework, relying solely on automatically obtained linguistic features (POS tags, lemmas, etc.), is a notable contribution. This circumvents the need for labeled training data, making the method broadly applicable and avoiding biases introduced by dataset-specific annotations.
3. **Re-analysis of CoNLL-2014 Shared Task Data:** Applying the developed tool to the CoNLL-2014 shared task data and presenting a detailed error type analysis offers valuable insights into the strengths and weaknesses of various GEC systems. This detailed analysis, facilitated by the tool, is a worthwhile contribution to the field.

**Strengths:**

1. **Addressing a Critical Gap:** The lack of detailed error type analysis in GEC has been a longstanding issue. This work directly tackles that problem, offering a solution that enables more nuanced evaluation of system performance. The provided tool, promised to be released with the paper, is a valuable resource for the GEC community.
2. **Strong Justification for Rule-Based Approach:** The authors convincingly argue for the benefits of a rule-based approach, highlighting the issues of bias and domain dependence inherent in machine learning classifiers. The focus on interpretability and consistency further strengthens this argument.
3. **Thorough Evaluation:** The evaluation is comprehensive, including a manual validation of the error typing framework (demonstrating high agreement with human raters), re-evaluation of the CoNLL-2014 data, and detailed analysis of individual error types and multi-token edits.
4. **Practical Insights from CoNLL-2014 Analysis:** The re-analysis of the CoNLL-2014 data reveals interesting patterns in system performance, providing actionable insights for future GEC research.  The identification of specific error types where different approaches excel is particularly valuable.
5. **Clear and Well-Structured Presentation:** The paper is well-written and logically organized. The methodology is clearly explained, and the results are presented in a concise and informative manner. The inclusion of the complete error type list in Table 2 is helpful.

**Weaknesses:**

1. **Limited Novelty of Core Components:** While the *combination* of linguistically-enhanced edit extraction and rule-based error typing is a valuable contribution, neither component is entirely novel. The edit extraction relies on previous work (Felice et al., 2016), and rule-based error typing, while effective, is a relatively established technique. The paper could benefit from a more nuanced discussion of how this work advances beyond existing methods.
2. **Reliance on Accurate POS Tagging:** The accuracy of the entire pipeline is heavily dependent on the quality of the POS tagger. The authors acknowledge this in the evaluation section, but a more detailed error analysis demonstrating the impact of POS tagging errors on the final results would strengthen the paper.
3. **Simplistic Comparison with M2 Scorer:** The critique of the M2 scorer is valid, but the comparison feels somewhat superficial. A more in-depth investigation of the discrepancies between the two evaluation approaches, and a discussion of the trade-offs involved, would be beneficial.
4. **Limited Scope of Error Type Analysis:** While the analysis of the CoNLL-2014 data is detailed, it focuses primarily on overall performance at the error type level. A more in-depth analysis of specific error patterns within each type (e.g., common misspellings, frequently confused word pairs) could yield further insights.

**Questions to Authors:**

1.  Could you elaborate on the computational cost of the error typing framework compared to machine learning approaches? Is there a significant difference in processing time, and if so, how does it impact the scalability of the approach?
2.  The paper mentions the potential for adding new error categories. What criteria would you use to determine when to add a new category, and how would you balance the need for informativeness with the desire for simplicity?
3.  How robust is the framework to different POS taggers? Have you evaluated its performance with alternative POS tagging tools, or considered incorporating uncertainty estimates from the tagger into the error typing process?

*   The discussion of multi-token edits is interesting, but could be expanded. Are there specific types of multi-token errors that are particularly challenging for current GEC systems?
