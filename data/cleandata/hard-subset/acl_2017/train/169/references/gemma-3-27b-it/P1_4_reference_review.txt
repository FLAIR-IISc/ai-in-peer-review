## Peer Review: Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction

**Summary:**

This paper addresses the problem of detailed error type analysis in Grammatical Error Correction (GEC), motivated by the limitations of the CoNLL-2014 shared task where systems were evaluated without error-type specific feedback. The authors propose a pipeline consisting of linguistically-enhanced edit extraction (leveraging Felice et al., 2016) followed by a rule-based error type classification system. The core contribution is a dataset-agnostic, rule-based framework for automatically annotating GEC system outputs with error types, enabling more granular evaluation. They demonstrate the utility of their tool by re-evaluating the CoNLL-2014 systems, providing a detailed error type breakdown.

**Main Contributions (as I see them):**

1. **A practical tool for post-hoc error analysis in GEC:** The primary contribution is a readily usable tool that allows researchers to automatically annotate GEC system outputs with error type information, facilitating a much more detailed analysis than previously possible.
2. **A novel, dataset-agnostic rule-based error typing framework:** The rule-based approach avoids the data-dependency issues of machine learning classifiers and offers transparency and consistency in error type assignment.
3. **Detailed re-evaluation of the CoNLL-2014 systems:**  The application of the tool to the CoNLL-2014 data provides valuable insights into the strengths and weaknesses of different GEC approaches, offering a benchmark for future research.

**Strengths:**

1. **Addresses a significant gap in GEC evaluation:** The lack of granular error type feedback has been a long-standing issue in the field. This work directly tackles this problem by providing a solution to automatically annotate and evaluate system performance on specific error types.
2. **Strong methodology and justification:** The authors build upon established work in edit extraction (Felice et al., 2016) and carefully justify their choice of a rule-based approach over machine learning, highlighting the benefits of dataset independence and transparency.
3. **Thorough evaluation with insightful results:**  The re-evaluation of the CoNLL-2014 systems is well-executed and yields interesting observations about the performance of different systems across various error types.  The comparison of performance at different granularity levels (e.g., general vs. specific error types) is particularly valuable.
4. **Practicality and reproducibility:**  The authors explicitly state that the tool will be released, promoting reproducibility and wider adoption within the GEC community.
5. **Clear and well-written paper:** The paper is clearly structured, logically organized, and easy to follow. The explanations are concise and insightful.

**Weaknesses:**

1. **Reliance on the accuracy of initial edit extraction:** The entire framework hinges on the quality of the edit extraction step. While the authors utilize a strong alignment algorithm (Felice et al., 2016), errors in edit boundary detection will inevitably propagate to the error typing stage, potentially impacting the reliability of the analysis.  The authors acknowledge this but do not fully quantify the impact of edit extraction errors.
2. **Limited evaluation of the rule-based classifier:** While the manual evaluation with GEC researchers is a good starting point, it is relatively small-scale (200 edits) and focused on "appropriateness" rather than precision/recall. A more rigorous evaluation against a gold standard (even a small one) would strengthen the claims about the classifier's accuracy.
3. **Potential for oversimplification in error typing:**  The rule-based approach, by its nature, may struggle to capture nuanced errors that require deeper linguistic understanding. While the authors acknowledge this and strive for a balance between informativeness and practicality, it's a potential limitation.
4. **Limited discussion of error type schema design choices:**  The paper briefly mentions trade-offs in error type granularity but could benefit from a more detailed discussion of the rationale behind the chosen schema and potential alternative designs.

**Questions to Authors:**

1. **Can you provide any quantitative analysis of the errors introduced by the edit extraction step?** Specifically, how often does the edit extraction algorithm make mistakes (e.g., incorrect boundaries, missed edits), and how might these errors impact the downstream error type classification?
2. **What steps were taken to ensure the consistency and reliability of the manual evaluation?**  Were clear guidelines provided to the raters, and was inter-rater reliability assessed beyond the reported κ-free score?
3. **Could you elaborate on how the rule-based framework handles ambiguous cases where an edit could potentially be classified into multiple error types?** What tie-breaking strategies are employed?
4. **Given the limitations of a purely rule-based approach, have you considered incorporating any machine learning components to handle more complex error types or refine the classification rules?**  Perhaps a hybrid approach could leverage the strengths of both rule-based and machine learning methods.

**Additional Comments:**

* The tables are very helpful for a quick understanding of the results.
* The decision to reprocess the reference data with the automatic annotator to create a consistent evaluation framework is commendable.
* The discussion of the limitations of the M2 scorer is valuable.



Overall, this is a strong paper that addresses an important problem in GEC with a practical and well-motivated solution. While there are some minor weaknesses, the strengths of the work clearly outweigh them. I recommend acceptance with minor revisions.
