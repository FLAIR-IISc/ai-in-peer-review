This paper presents a method for automatically annotating grammatical error correction (GEC) system outputs with error types, addressing a limitation of previous evaluations like CoNLL-2014 where detailed error type analysis was hampered by unannotated hypotheses. The core idea is to leverage linguistically-enhanced edit extraction followed by a rule-based error typing framework. The authors claim to provide a dataset-agnostic approach, relying solely on automatic markup, and release a tool for this purpose. 

The primary contribution of this work is the rule-based error typing framework, which, despite its simplicity, demonstrates surprising effectiveness. The ability to automatically annotate corrections with error types without relying on labeled training data is valuable.  A secondary contribution is the application of this framework to re-evaluate the CoNLL-2014 shared task data, providing a more granular analysis of system performance than previously possible. Finally, the release of the annotation tool itself is a positive step towards reproducibility and further research.

**Strengths:**

1. **Practicality and Accessibility:** The method is straightforward and relies on readily available tools (spaCy, NLTK, Hunspell). This makes it easily adoptable by other researchers and practitioners without requiring extensive resources or specialized expertise. The release of the tool is a significant asset.
2. **Dataset Independence:** The rule-based approach’s reliance on automatic markup is a major strength. This mitigates the issues of domain adaptation and annotator bias inherent in machine learning-based approaches. This is especially important in GEC, where annotation guidelines can vary significantly.
3. **Detailed Analysis Enabled:** The automated annotation facilitates detailed error type analysis, as demonstrated by the CoNLL-2014 re-evaluation. This allows for a more nuanced understanding of system strengths and weaknesses, potentially guiding future development.  The breakdown of performance across different error categories (Table 6) is particularly insightful.
4. **Good Empirical Validation:** The manual evaluation (Table 3), while small-scale, provides encouraging evidence of the framework's accuracy. The high agreement among raters suggests the classifications are reasonably aligned with human judgment. The bootstrapping tests for the auto vs. gold reference comparison are also a strong addition.

**Weaknesses:**

1. **Limited Novelty of Components:** While the *combination* of linguistically-enhanced edit extraction and rule-based typing is valuable, neither component is entirely novel. The edit extraction relies on prior work (Felice et al., 2016), and rule-based classification, while effective, is a relatively standard technique. The novelty lies in applying it specifically to error *type* evaluation.
2. **Error Category Granularity:**  The 25 error categories (Table 2) feel somewhat arbitrary. The authors acknowledge this in section 3.2, but a more principled approach to defining the error schema might improve its robustness and interpretability.  The explanation for decisions about granularity (e.g., combining tense and modality) isn’t fully convincing.
3. **Sensitivity to POS Tagging Accuracy:** The framework's reliance on POS tags makes it vulnerable to errors in the POS tagger. The authors acknowledge this (Table 3, “ring” example), but a more thorough error analysis of how POS tagging errors propagate to error type classification would be beneficial. This is particularly important given that the framework is "dataset-agnostic" – an imperfect POS tagger could introduce systematic biases.
4. **Simplified Evaluation Metric:** The edit comparison approach for evaluating the CoNLL-2014 data is straightforward, but it doesn't fully capture the complexity of GEC evaluation. Comparing against the M2 scorer highlights this, as the M2 scorer accounts for more nuanced alignment and correction scenarios. While the authors argue the M2 scorer overestimates performance, dismissing it entirely feels too strong.



**Questions to Authors:**

1.  Could you provide a more detailed error analysis showing the impact of POS tagging errors on the final error type classifications? Specifically, what proportion of "Bad" ratings in Table 3 were attributable to POS tagging errors?
2.  The choice of 25 error categories seems somewhat arbitrary. What principles guided the selection of these categories?  Were any other categorization schemes considered?
3.  You mention the possibility of extending the framework with more fine-grained error types. What are the main challenges to doing so, and how might they be addressed?
4.  How does the performance of the framework vary across different types of learner writing (e.g., different proficiency levels, L1 backgrounds)? Is the rule-based approach equally effective in all contexts?

**Additional Comments:**

*   Table 7 (detailed error type performance) would be more useful if it included precision, recall, and F0.5 scores for *all* teams, not just a representative selection.
*   The discussion of the M2 scorer’s potential inflation of scores is valid, but could be presented more objectively.
*   Consider adding a discussion of potential limitations of the rule-based approach, beyond the dependence on POS tagging. For instance, how well does it handle complex errors involving multiple error types?



Overall, this is a solid and valuable contribution to the field of GEC. The proposed method offers a practical and accessible way to automatically annotate error types, enabling more detailed and nuanced evaluation of GEC systems. While the novelty of the individual components is limited, the integration and application of these components address a significant need in the community. The weaknesses are not fatal, and addressing the questions above would further strengthen the paper. I recommend acceptance.
