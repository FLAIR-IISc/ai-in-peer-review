This paper addresses the problem of automatically annotating grammatical error correction (GEC) system outputs with error types, a task hindered by the lack of standardized annotation in previous shared tasks like CoNLL-2014. The authors propose a two-step process: linguistically-enhanced edit extraction followed by rule-based error type classification.

The primary contribution is a dataset-agnostic, rule-based framework for automatically assigning error types to edits in GEC system outputs. This is valuable because it bypasses the need for labeled training data specific to different datasets, a significant limitation of previous machine learning approaches. A second notable contribution is the application of this framework to re-evaluate the CoNLL-2014 shared task submissions, providing a more granular analysis of system performance than previously possible. Finally, the authors offer a publicly available tool implementing their approach, facilitating further research in this area.

**Strengths:**

1. **Novelty of Dataset-Agnostic Error Typing:** The core idea of a rule-based system relying solely on automatic linguistic features (POS tags, lemmas, etc.) for error typing is genuinely novel. Existing approaches largely rely on supervised learning and are therefore dataset-dependent. This approach removes that dependency, making comparisons across datasets and systems more meaningful.
2. **Comprehensive Re-evaluation of CoNLL-2014:** The application of the tool to the CoNLL-2014 data is a strong validation of its utility. Providing a more detailed breakdown of system performance beyond overall scores is insightful and valuable to the GEC research community. The careful consideration of evaluation methodology, including re-annotating the references for consistency, demonstrates thoroughness.
3. **Practical Tool Release:** The commitment to releasing the tool is a significant positive aspect. It enables reproducibility and allows other researchers to build upon this work. The modularity of the error type framework (allowing evaluation at different levels of granularity) adds to its practical value.
4. **Strong Validation via Human Evaluation:** The manual evaluation of the rule-based classifier, showing >95% "Good" or "Acceptable" ratings from experts, provides strong evidence for the accuracy and reliability of the approach, especially considering the inherent noise from automatic edit extraction.

**Weaknesses:**

1. **Limited Discussion of Error Type Scheme Design:** While the paper presents a 25-error type scheme, the rationale behind the specific choices and the trade-offs involved are not fully discussed. The example provided concerning the distinction between tense and modal errors highlights potential ambiguity and the subjective nature of categorization. A more in-depth justification for the chosen taxonomy would strengthen the work.
2. **Performance Discrepancy with M2 Scorer:** The observation that the proposed evaluation method yields lower scores compared to the standard M2 scorer requires more detailed investigation. While the authors correctly point out limitations of the M2 scorer, the substantial difference in scores might raise concerns about comparability with existing literature. A more rigorous comparison, potentially with other evaluation metrics, would be beneficial.
3. **Superficial Analysis of Detailed Error Type Results:** The presentation of detailed error type results (Table 6) is extensive but lacks a cohesive analysis. While observations are made, a deeper interpretation of the patterns and potential reasons for the observed performance differences across error types and systems is missing. A more focused discussion linking the error types to the strengths and weaknesses of different GEC approaches would be more impactful.
4. **Lack of Error Analysis for Rule-Based System:** While the rule-based system demonstrated strong performance in the human evaluation, the paper does not provide a detailed error analysis of its failure cases. Identifying common errors made by the system and the reasons behind them could lead to improvements and a better understanding of its limitations.

**Questions to Authors:**

1. Can you elaborate on the design process of the error type scheme? What criteria were used to select the 25 error types, and what trade-offs were considered?
2. Could you provide more details on the differences between your evaluation method and the M2 scorer? For example, how does your method handle edge cases or ambiguous edits differently?
3. You mention that some systems detected more errors than they could correct. Can you analyze the types of errors these systems were detecting but failing to fix? Is there a pattern related to the error type or complexity of the correction?
4.  What is the computational cost of the automatic annotation tool? Specifically, how does the runtime scale with the size of the input corpus?
