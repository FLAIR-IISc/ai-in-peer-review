This paper addresses the important problem of automatically annotating grammatical error correction (GEC) data with error types, a capability lacking in previous shared task evaluations like CoNLL-2014. The authors propose a two-step approach: linguistically-enhanced edit extraction followed by rule-based error typing. The primary contribution is a practical and surprisingly effective rule-based system for automatically assigning error types, achieving high agreement with human raters (over 95% rated as “Good” or “Acceptable”). A secondary contribution is the application of this tool to re-evaluate the CoNLL-2014 shared task data, offering new insights into system performance at a granular level. The release of the annotation tool itself is also a valuable contribution to the GEC community.

**Strengths:**

1. **High-Quality Rule-Based System:** The core of the paper – the rule-based error typing framework – is a strong point. The authors astutely recognize the limitations of machine learning approaches reliant on potentially inconsistent training data and instead leverage automatically obtainable linguistic features (POS tags, lemmas, etc.). This results in a dataset-agnostic system, a significant advantage. The reported 95%+ agreement with human raters provides compelling evidence of its accuracy, particularly given the automated edit extraction stage introduces noise.
2. **Novel and Valuable Re-evaluation of CoNLL-2014:** Applying the tool to the CoNLL-2014 dataset delivers new insights.  The detailed breakdown of performance by error type is far more informative than overall F0.5 scores. The findings highlighting different system strengths (e.g., CAMB excels at missing edits, CUUI at replacements) are valuable for the GEC research community. The comparison of auto vs. gold references and the justification for using auto references for fairer evaluation is also well-reasoned.
3. **Practical Contribution: Tool Release:**  Releasing the annotation tool is a substantial and practical contribution. This allows other researchers to leverage the authors’ work, build upon it, and standardize error type annotation across GEC datasets.  This fosters reproducibility and enables more meaningful comparisons of different GEC systems.
4. **Clear Presentation and Comprehensive Analysis:** The paper is well-written and clearly explains the methodology. The comprehensive analysis of the CoNLL-2014 results, including the tables breaking down performance by error type and the discussion of detection vs. correction, demonstrates a thorough understanding of the data and the challenges of GEC.

**Weaknesses:**

1. **Limited Scope of Novelty in Edit Extraction:** The edit extraction step relies on Felice et al. (2016), which is acknowledged. While effective, this component is not a novel contribution of *this* paper. The paper's novelty resides primarily in the error typing framework. More discussion around limitations of relying on an external edit extraction algorithm would be beneficial. For example, how sensitive is the error typing to inaccuracies in the initial edit boundaries?
2. **Overly Optimistic Rater Evaluation:** While the 95%+ agreement with human raters is encouraging, the evaluation methodology has limitations. The raters were explicitly told the edit boundaries were automatically determined. This could have introduced bias, as raters might have been more lenient knowing the boundaries were potentially inaccurate.  A blind evaluation (where raters are unaware of the source of the boundaries) would have provided a more robust assessment.
3. **Lack of Comparison to Other Error Typing Approaches:** The paper focuses heavily on justifying the rule-based approach *against* machine learning methods. However, a more direct comparison to other existing rule-based error typing systems (if any exist) would strengthen the argument for the novelty and superiority of the proposed framework. A literature review specifically addressing other rule-based systems would be valuable.
4. **Granularity of Error Types:** The 25 error types, while comprehensive, may be overly granular for some analyses. The authors acknowledge this in the discussion of modal vs. tense errors. The practical utility of distinguishing between such finely-grained categories needs further justification. Providing examples of how this granularity directly informs improvements to GEC systems would be helpful.

**Questions to Authors:**

1. How sensitive is the error type classification to errors in the automatically extracted edit boundaries?  Have you investigated the impact of boundary inaccuracies on the final classification results?
2. Was the decision to use a rule-based approach driven primarily by the limitations of existing training data for GEC, or were there other factors (e.g., interpretability, computational efficiency)?
3. Are there plans to integrate the annotation tool with other GEC datasets besides CoNLL-2014? If so, have you encountered any challenges related to dataset-specific annotation schemes?
4. Could you elaborate on the potential trade-offs between the granularity of error types and their practical utility for GEC system development?



**Additional Comments:**

* The table listing the error categories (Table 2) is difficult to read in its current format. Consider breaking it up into multiple tables or using a more visually appealing layout.
* The discussion of the M2 scorer’s overestimation of performance is insightful. It would be valuable to briefly explain *why* the M2 scorer tends to inflate scores.
*  A link to the released tool would be helpful to include in the paper.
