This paper proposes a method for automatically annotating grammatical error correction (GEC) system outputs with error types, addressing a gap identified following the CoNLL-2014 shared task. The authors leverage existing linguistically-enhanced alignment techniques (Felice et al., 2016) and introduce a rule-based framework for error type classification. The core contribution is the tool itself, intended to facilitate more detailed evaluation of GEC systems. A secondary contribution is the re-analysis of the CoNLL-2014 data using this automated annotation, providing insights into system performance at a more granular level.

**Strengths:**

1. **Addresses a Practical Problem:** The lack of detailed error type analysis for GEC systems is a valid concern, and this work directly tackles that issue. Providing a tool to automatically annotate outputs enables more nuanced evaluation than relying solely on aggregate scores.
2. **Sound Methodological Foundation:** Building on established alignment techniques (Felice et al., 2016) is a reasonable approach. While the rule-based classification is simpler than machine learning alternatives, it is arguably more transparent and avoids the pitfalls of training data bias.
3. **Comprehensive Error Type Breakdown:** The detailed error type categorization (25 types, with combinations for operation type) is commendable and allows for a rich analysis of system strengths and weaknesses. The decision to derive categories solely from automatic markup is also sensible given the goal of dataset independence.
4. **Demonstrated Applicability:** Re-analyzing the CoNLL-2014 data and presenting detailed results for each participating system provides concrete evidence of the tool’s utility and generates potentially valuable insights for the GEC community.

**Weaknesses:**

1. **Limited Validation of the Error Type Classification:** The manual evaluation, while present, is small-scale (200 edits) and relies on subjective ratings ("Good," "Acceptable," "Bad"). While the results are encouraging, they don't provide a robust assessment of the classifier’s accuracy. A comparison against a gold standard (even a small one) would significantly strengthen the work. The reliance on POS tagger accuracy as the primary source of errors identified in the manual evaluation is concerning; errors in the tool are thus indirectly attributed to another system.
2. **Overreliance on Existing Work:** The method heavily depends on the quality of the underlying alignment algorithm (Felice et al., 2016). Any errors in alignment will propagate to the error type classification. The paper doesn’t sufficiently discuss the potential impact of alignment errors on the final results. More validation of the alignment *specifically within the context of error type assignment* is needed.
3. **Questionable Reprocessing of Gold References:** The decision to re-process the gold reference data using the *same* automatic pipeline as the system outputs, to create comparable annotations, feels counterintuitive. It effectively lowers the bar for evaluation and undermines the value of having a “gold” standard. While the authors demonstrate non-significant differences in overall scores, this approach introduces systematic biases that are not fully addressed.  The stated motivation – ensuring edit boundaries match – is weak justification for discarding the accuracy of human annotation.
4. **Lack of Comparison to Existing Error Typing Methods:** While the paper acknowledges prior work in error type classification (Swanson and Yamangil, 2012; Xue and Hwa, 2014), it does not provide a direct comparison of the proposed rule-based approach against these methods, even acknowledging the incompatibility of error schemes. A comparison, even with adjustments for differing schemes, would help establish the relative merits of the new framework.

**Questions to Authors:**

1.  What steps were taken to evaluate the alignment quality, and how might alignment errors affect the accuracy of the error type assignments?
2.  Why was it deemed necessary to re-process the gold reference data using the automatic pipeline, and what are the potential biases introduced by this approach? Could the evaluation be performed by mapping the auto-annotated hypotheses to the original gold reference error types, even if imperfectly?
3.  Could you provide more details on the specific errors identified by the human raters as "Bad," beyond simply attributing them to POS tagging errors? Were there systematic issues with the rule-based classification logic itself?
4.  Given that the tool will be released publicly, how will you address the potential for errors in the underlying alignment algorithm or POS tagger to affect the accuracy of the automated annotations for other researchers?

**Additional Comments:**

Table 9 is helpful, but could be more clearly formatted. It's difficult to quickly grasp the valid combinations. The discussion of the M2 scorer and its biases is insightful. The detailed analysis in Section 4.3 regarding the combination of different system approaches is valuable. However, the conclusions drawn from this analysis feel somewhat tentative given the limitations in the validation of the underlying annotation process.
