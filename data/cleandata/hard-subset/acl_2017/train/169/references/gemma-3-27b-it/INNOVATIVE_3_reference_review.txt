This paper tackles the important problem of evaluating Grammatical Error Correction (GEC) systems beyond overall scores, specifically by automatically annotating system outputs with error types. The primary contribution is a rule-based framework for error type classification that operates solely on automatically obtainable linguistic features, removing the need for labeled training data and, critically, offering dataset independence. A secondary contribution is the application of this framework to re-evaluate the CoNLL-2014 shared task results, providing a more granular analysis of system performance. Finally, the authors provide a readily available tool to facilitate this annotation process.

**Strengths:**

1. **Novelty & Conceptual Leap:** The core idea of dataset-independent error type annotation is genuinely novel and represents a significant departure from reliance on labeled data. The reliance on automatic linguistic features is elegant and addresses a key bottleneck in GEC evaluation – the scarcity and inconsistency of high-quality annotated corpora. This is the strongest aspect of the paper.
2. **Practical Impact & Tool Release:** The practical value of the developed tool is high. Providing a means to automatically annotate GEC system outputs unlocks more detailed analysis, fostering a better understanding of system strengths and weaknesses. The commitment to releasing the tool is commendable and facilitates reproducibility and wider adoption.
3. **Detailed Analysis of CoNLL-2014:** The re-evaluation of the CoNLL-2014 results, while using an established dataset, yields insightful observations about the performance of different systems on specific error types. The demonstration that the approach doesn’t drastically alter overall rankings when applied to the references adds credibility.
4. **Robustness Evaluation:** The manual evaluation, although small-scale, demonstrates that the rule-based system aligns well with human judgment (>95% “Good” or “Acceptable” ratings), mitigating concerns about the accuracy of automatic annotation. The good inter-annotator agreement further strengthens this point.

**Weaknesses:**

1. **Limited Scope of Evaluation:** While the CoNLL-2014 re-evaluation is valuable, it’s the only empirical demonstration provided. Applying the tool to more diverse datasets, and ideally, more recent GEC systems, would strengthen the claims of generalizability and robustness.
2. **Simplistic Error Scheme:** While the framework aims for practicality, the 25 error types feel somewhat coarse-grained. The discussion in Section 3.2 acknowledges this trade-off, but the potential for more nuanced categorization should be explored. There’s a risk of collapsing distinct error patterns into single categories.
3. **Lack of Comparison to ML-based Approaches:** The paper positions the rule-based approach as superior to ML-based classifiers due to dataset dependence. However, a direct comparison on a shared test set, even with potential domain adaptation techniques for the ML models, would be more convincing. The authors acknowledge previous F1 scores, but a head-to-head comparison is missing.
4. **Underdeveloped Discussion of Error Detection vs. Correction:** While acknowledging the importance of error *detection* (Section 4.6), this aspect is not thoroughly explored. A more detailed analysis of systems’ ability to *identify* errors, independent of correction accuracy, would provide further insights.

**Questions to Authors:**

1.  What are the computational costs associated with running the annotation tool on large datasets? Specifically, what is the runtime for annotating the entire CoNLL-2014 test set, and how does this scale with dataset size?
2.  You mention a potential for adding more granular error categories. What are some specific examples of error distinctions currently collapsed within your framework that you believe could be fruitfully separated?
3.  Could you elaborate on potential strategies for incorporating error detection performance into a more comprehensive GEC evaluation metric?
4.  Regarding the observation that several teams struggled with unnecessary token errors, do you have hypotheses about why this is the case (e.g., features used, training data)?



**Additional Comments:**

Table 2 is a bit difficult to read; consider a more visually clear format for presentation. The reference list is formatted inconsistently (e.g., capitalization).



**Overall Assessment:**

Despite some minor methodological limitations, this paper presents a significant contribution to the field of GEC evaluation. The novelty of the dataset-independent error typing framework, coupled with the practical utility of the released tool, outweigh the weaknesses. I recommend **acceptance** with minor revisions. The authors have identified an important problem and proposed a creative and potentially impactful solution. While a more comprehensive evaluation would be beneficial, the current work provides a solid foundation for future research.
