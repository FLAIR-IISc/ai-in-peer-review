This paper tackles the important problem of evaluating Grammatical Error Correction (GEC) systems beyond overall scores, specifically by automatically annotating system outputs with error types. The primary contribution is a rule-based framework for error type classification, operating solely on automatically derived linguistic features (POS tags, lemmas, etc.), making it dataset-agnostic. A secondary contribution is the application of this framework to re-evaluate the CoNLL-2014 shared task results, offering a more granular analysis. The authors also release a tool implementing this framework, which is valuable for the community.

**Strengths:**

1. **Novelty of the Dataset-Agnostic Approach:** The core idea of bypassing the need for labeled training data for error typing is genuinely novel and appealing. Existing approaches, relying on supervised learning, suffer from domain dependence and the issues of differing annotation schemes. This work offers a path toward standardized, reproducible error type evaluation.
2. **Practicality and Accessibility:**  The rule-based approach, while potentially less accurate than a sophisticated machine learning model, is transparent, interpretable, and crucially, readily available as a tool. This practicality significantly enhances the impact of the work. The release of the tool is a strong plus.
3. **Insightful Re-evaluation of CoNLL-2014:** Applying the framework to the CoNLL-2014 data generates valuable insights into the strengths and weaknesses of different GEC systems, highlighting complementary approaches (e.g., CUUI for unnecessary edits, CAMB for missing edits). This analysis demonstrates the usefulness of the proposed method.
4. **Detailed Evaluation Capabilities:** The framework’s granularity – allowing evaluation at different levels of specificity (general error types, operation tiers, detailed types) – is a significant advantage. This level of detail enables more targeted improvements to GEC systems.

**Weaknesses:**

1. **Limited Evaluation of the Rule-Based Classifier:** The evaluation of the rule-based classifier relies heavily on a small-scale manual evaluation with only five raters. While the results are encouraging (high “Good”/“Acceptable” ratings), this is not a rigorous validation. A comparison against human annotations on a larger, held-out dataset would be much more convincing, even if it requires some manual annotation effort.
2. **Potential for Error Propagation:** The entire pipeline relies on the accuracy of automatic linguistic tools (POS tagging, lemmatization, etc.). Errors in these tools will inevitably propagate to the error typing stage, potentially skewing the evaluation results. While acknowledged, the paper doesn't thoroughly investigate the sensitivity of the framework to these errors.
3. **Oversimplification of Error Categories:** The paper concedes that the error scheme represents a compromise between informativeness and practicality. While understandable, certain categories feel overly broad or potentially overlapping.  A more nuanced categorization scheme, even if more complex, might offer more actionable insights.
4. **Lack of Comparison to Existing Automatic Annotation Methods:** The introduction mentions previous attempts at automatic error annotation (Swanson & Yamangil, Xue & Hwa, Felice et al.). However, there's no direct comparison of the performance of the proposed framework against these baselines. Demonstrating a clear improvement over existing methods would strengthen the contribution.



**Questions to Authors:**

1.  Could you provide a more detailed error analysis, specifically identifying the types of errors where the rule-based system performs poorly, and attributing these errors to deficiencies in the underlying linguistic tools (e.g., POS tagging)?
2.  What are the computational costs associated with running the annotation tool on large datasets? Is it scalable to datasets significantly larger than CoNLL-2014?
3.  How does the performance of the framework vary across different types of learner writing (e.g., different L1 backgrounds, proficiency levels)?
4.  Could you provide some examples of situations where the framework assigns different error types to the same edit, and explain why these discrepancies occur?

**Additional Comments:**

Table 9 is useful, but would be more accessible if formatted for easier reading (e.g., using a more compact layout or a different visual separation).  It may also be helpful to provide a more detailed discussion of the limitations of the framework, and potential directions for future research. For instance, exploring the integration of shallow neural networks to augment the rule-based system could be a promising avenue.



Despite the weaknesses, I lean towards acceptance. The novelty of the dataset-agnostic approach, combined with the practical value of the released tool and the insightful analysis of the CoNLL-2014 data, outweigh the limitations. The paper introduces a significant new concept and direction for GEC evaluation, and the relatively minor flaws can be addressed in future work.
