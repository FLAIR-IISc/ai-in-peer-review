This paper presents a method for automatically annotating grammatical error correction (GEC) system outputs with error types, aiming to address the lack of detailed error analysis in previous evaluations like CoNLL-2014. The core contribution is a rule-based error typing framework, built upon linguistically-enhanced edit extraction, that operates solely on automatic markup (POS tags, lemmas, etc.), making it dataset-agnostic. The authors apply this to re-evaluate the CoNLL-2014 systems and provide a detailed breakdown of performance by error type.

**Strengths:**

1. **Addressing a Real Gap:** The paper directly addresses a significant limitation in GEC evaluation – the inability to perform detailed error type analysis of system outputs. Providing a method to annotate system hypotheses is valuable for researchers seeking to understand where their models struggle.
2. **Dataset Agnosticism:** The rule-based approach, relying only on automatic markup, is a key strength. It avoids the biases inherent in machine learning models trained on specific annotated datasets. This generality is highly desirable in a field where datasets vary considerably in annotation schemes.
3. **Detailed CoNLL-2014 Analysis:**  The application of the tool to the CoNLL-2014 data provides a useful, concrete demonstration of its capabilities and generates insights into the performance of various systems across different error types. The findings regarding the relative strengths of different approaches (e.g., MT vs. classifiers) are potentially valuable.
4. **Transparency & Determinism:** The rule-based nature ensures transparency in error classification and avoids the “black box” problem of machine learning models. The deterministic nature also promotes consistency.

**Weaknesses:**

1. **Overstated Classifier Evaluation:** The manual evaluation of the classifier (Table 3) is presented as overwhelmingly positive, with “Good” or “Acceptable” ratings exceeding 95%. However, the description notes that ratings were performed *despite* automatic edit extraction errors. This is a crucial caveat and strongly suggests the positive ratings may be inflated. A more rigorous evaluation isolating the classifier’s performance *given correct edit boundaries* is necessary.
2. **Insufficiently Justified Rule Design:** While the authors state their rules are based on POS categories and inspired by Swanson and Yamangil (2012), the rationale behind many specific rules (e.g., for distinguishing error types within a POS category) is lacking. A more detailed explanation and justification for each rule would strengthen the framework’s credibility. The decision to compromise between “informativeness and practicality” feels ad hoc without more justification.
3. **Limited Novelty of Edit Extraction:**  The adoption of Felice et al. (2016)’s edit extraction method is acknowledged, but its contribution within *this* paper feels incremental. The focus is overwhelmingly on error typing, and the edit extraction component is treated largely as a pre-processing step.  More discussion of potential limitations of the chosen edit extraction method would be valuable.
4. **Questionable Auto vs. Gold Reference Comparison:** The decision to re-process the reference data with the automatic system before comparison (Section 4.1) is problematic. While the authors attempt to justify this by arguing for consistent processing, it fundamentally undermines the evaluation. Comparing against a "gold" standard processed *automatically* defeats the purpose of having a gold standard in the first place.  The lack of statistical significance in the bootstrap testing is concerning.
5. **Formatting & Table Presentation:** The tables are difficult to parse. Table 2 lacks clear visual separation between error codes and descriptions. Table 6 is overly large and unmanageable, making it difficult to draw meaningful conclusions. A more concise presentation, perhaps focusing on the most significant findings, is needed.  Also, there is an inconsistency in referencing the shared task paper (Ng et al., 2014) versus other papers.

**Questions to Authors:**

1.  Can you provide a more detailed analysis of the impact of automatic edit extraction errors on the manual evaluation of the classifier (Table 3)? Specifically, what percentage of “Bad” ratings were directly attributable to incorrect edit boundaries?
2.  Please provide a more detailed explanation of the rationale behind the specific rules used for error type classification, particularly for cases where classification relies on more than just POS tags.
3.  Why was the decision made to re-process the gold reference data automatically (Section 4.1)? What are the implications of this choice for the validity of the evaluation?
4.  Can you discuss the limitations of relying solely on POS tags and other automatic features for error typing? Are there error types that are inherently difficult to classify accurately using this approach?
5.  The reference list is incomplete. Please add complete references for all citations.

**Additional Comments:**

*   The paper is generally well-written, but a careful proofread is recommended to address minor grammatical errors (e.g., “Human evaluation of gram…” in the references).
*   Consider including a discussion of potential applications of the tool beyond re-evaluating existing systems, such as debugging and improving GEC models.
*   The appendix (Table 9) is helpful, but could be presented more clearly.



Overall, while the paper addresses an important problem and presents a potentially useful tool, several weaknesses – particularly related to the evaluation and justification of the rule-based framework – prevent me from recommending acceptance at this time. The analysis, while detailed, feels somewhat superficial due to the questionable evaluation methodology. A major revision addressing these concerns is required.
