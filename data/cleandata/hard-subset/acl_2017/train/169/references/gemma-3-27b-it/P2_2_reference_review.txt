## Review for ACL

**Summary:** This paper presents a method for automatically annotating grammatical error correction (GEC) system outputs with error type information. The approach involves linguistically-enhanced edit extraction followed by rule-based error type classification, aiming to address the lack of detailed error type evaluation in prior GEC work (particularly stemming from the CoNLL-2014 shared task). The authors release a tool implementing this annotation pipeline and demonstrate its use by re-evaluating the CoNLL-2014 systems.

**Main Contributions:**

1. **Automated Error Type Annotation Pipeline:** The core contribution is a practical pipeline for automatically adding error type annotations to GEC system outputs. This addresses a significant gap in the field, enabling more granular evaluation than previously possible.
2. **Rule-Based Error Typing Framework:** The proposed rule-based framework for error classification is a valuable contribution.  Its dataset-agnostic nature and transparent decision-making process are compelling advantages over machine learning-based approaches.
3. **Re-evaluation of CoNLL-2014 Systems:** Applying the pipeline to the CoNLL-2014 data and presenting a detailed analysis of system performance across different error types provides useful insights and establishes a baseline for future research.

**Strengths:**

1. **Addresses a Clear Need:** The paper directly tackles a known limitation in GEC evaluation: the lack of fine-grained error type analysis.  The motivation is well-articulated and the problem statement is important.
2. **Sound Methodology:**  Leveraging a strong edit extraction method (Felice et al., 2016) and building a transparent, rule-based classification framework is a sensible and well-justified approach. The rule-based nature is particularly attractive, avoiding the biases and data dependency of machine learning models.
3. **Strong Validation:** The manual evaluation with GEC researchers (Table 3) provides compelling evidence of the classifier's accuracy and reliability, despite the noise introduced by automatic edit extraction.  The high agreement among raters is particularly encouraging.
4. **Detailed Analysis:** The re-evaluation of the CoNLL-2014 systems (Tables 4-8) demonstrates the practical value of the annotation pipeline, revealing interesting insights into the strengths and weaknesses of different systems across various error types. The observation about the difficulty of correcting multi-token errors is particularly noteworthy.
5. **Tool Release:**  Making the annotation tool publicly available enhances the reproducibility of the work and encourages further research in this area.

**Weaknesses:**

1. **Limited Novelty in Core Techniques:** While the combination of existing techniques (linguistically-enhanced alignment + rule-based classification) is effective, the individual components are not entirely novel. The primary novelty lies in the *application* of these techniques to the specific problem of automated error type annotation for GEC.
2. **Reliance on External Tools:** The performance of the entire pipeline is dependent on the accuracy of external tools like spaCy, Hunspell, and the Lancaster Stemmer. Errors in these tools can propagate through the pipeline and affect the final annotation quality.  The paper acknowledges this in the POS tagging section but could benefit from a more detailed discussion of potential error sources and their impact.
3. **Evaluation Metric Caveats:** The paper acknowledges that the chosen evaluation metric (direct edit comparison) might underestimate performance compared to the M2 scorer. While the rationale for choosing this metric is sound, it's important to recognize its limitations and potentially explore alternative evaluation strategies.
4. **Depth of Analysis Could Be Increased:** While the analysis of CoNLL-2014 systems is detailed, a more in-depth investigation of specific error cases and the reasons behind misclassifications could further strengthen the findings. For example, a manual error analysis of the misclassified edits in Table 3 would be valuable.




**Questions to Authors:**

1.  **Error Propagation:** Could you provide a more quantitative assessment of the impact of errors in the external tools (spaCy, Hunspell, Lancaster Stemmer) on the final annotation accuracy?  Have you considered any strategies to mitigate these errors?
2.  **Error Type Disambiguation:**  The paper acknowledges the challenges of defining error categories (Section 3.2). Could you elaborate on how you handled ambiguous cases during the rule creation process?  Are there specific error types that consistently prove difficult to classify, and if so, why?
3.  **Scalability:** How does the performance of the annotation pipeline scale with the size of the input corpus?  Are there any performance bottlenecks that need to be addressed for larger-scale applications?




**Additional Comments:**

*   Table 9 (complete list of error types) is helpful and well-organized.
*   The paper is generally well-written and clearly presented. The motivation and contributions are clearly stated.
*   The inclusion of the Kappa statistic for inter-annotator agreement is a good practice.




**Overall Assessment:**

This is a strong and valuable contribution to the field of GEC. While the individual techniques aren't groundbreaking, the combination, the well-defined framework, the thorough evaluation, and the tool release make this a significant piece of work. The weaknesses are relatively minor and can be addressed during the author response period. I recommend **acceptance** with minor revisions.
