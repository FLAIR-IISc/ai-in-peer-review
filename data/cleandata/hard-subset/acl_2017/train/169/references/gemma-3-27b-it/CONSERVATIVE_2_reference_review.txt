This paper proposes a method for automatically annotating grammatical error correction (GEC) system outputs with error types, aiming to address the lack of detailed error analysis in previous evaluations like CoNLL-2014. The core idea involves linguistically-informed edit extraction followed by rule-based error classification based on automatic POS tags and other features. The authors claim to achieve high agreement with human annotators ( >95% “Good” or “Acceptable” ratings) and demonstrate the utility of their approach through re-analysis of the CoNLL-2014 shared task data.

The primary contribution of this work is the release of a tool for automatically annotating GEC outputs, which will facilitate more granular evaluation of GEC systems. A secondary contribution is a rule-based error typing framework that, while not particularly novel in its components (POS tagging, stemming, etc.), attempts to provide a dataset-agnostic approach to error categorization. A tertiary, though potentially valuable contribution, is a re-evaluation of the CoNLL-2014 systems using this new annotation scheme, highlighting potential strengths and weaknesses of different approaches.

**Strengths:**

1. **Practical Tool:** The development and planned release of an automated annotation tool is genuinely useful to the GEC community. Detailed error analysis is hampered by the labor-intensive nature of manual annotation, and a reliable automated system would lower this barrier.
2. **Dataset Independence:** The authors correctly identify a crucial issue with previous error typing approaches – their dependence on specific annotation guidelines. The attempt to create a dataset-agnostic system based on automatic linguistic features is a laudable goal, even if its full success is debatable (see weaknesses).
3. **Detailed Analysis:** The re-evaluation of CoNLL-2014 systems provides some interesting insights into the strengths and weaknesses of different approaches *within the context of this particular annotation scheme*.  The observation about differing strengths across systems (e.g., CUUI for unnecessary errors, CAMB for missing errors) is potentially valuable for future research.
4. **Acceptable Accuracy:**  The human evaluation, while small-scale, suggests that the rule-based system achieves reasonably good agreement with human annotators, particularly when accounting for errors in automatic edit extraction.

**Weaknesses:**

1. **Lack of Novelty:** The core techniques employed – edit extraction based on alignment algorithms and rule-based classification – are well-established in the GEC literature. The paper does not present any fundamentally new algorithmic ideas. The primary innovation lies in the *combination* of existing techniques and the specific rule set designed for this purpose, but this feels incremental rather than groundbreaking.
2. **Insufficient Baseline Comparison:**  The paper mentions previous work on automatic error typing (Swanson & Yamangil, Xue & Hwa, Felice et al.) but provides only a cursory comparison.  A more thorough comparison against these baselines, ideally using the same evaluation methodology, is essential to demonstrate the advantages of the proposed approach. The claim of "high scores awarded by the raters" is not compelling without a quantifiable comparison to existing methods.
3. **Reliance on Automatic Linguistic Tools:** The system’s performance is heavily dependent on the accuracy of the underlying linguistic tools (POS tagger, lemmatizer, stemmer). Errors in these tools directly translate into incorrect error type assignments. While the authors acknowledge this, they do not provide a detailed error analysis identifying the impact of specific linguistic tool errors.  The example provided ("ring -> rings" being misclassified) highlights this vulnerability.
4. **Oversimplification of Error Types:** While aiming for dataset independence, the rule-based system necessarily makes simplifying assumptions about error categorization.  The authors acknowledge this trade-off (Section 3.2), but the resulting error types may be too coarse-grained to capture the nuances of real-world grammatical errors. The discussion of modal vs. tense errors illustrates this limitation.
5. **Questionable Evaluation Methodology:** Reprocessing the gold standard references with the *automatic* annotation pipeline to enable direct comparison seems dubious. The entire point of having gold standard annotations is to provide a reliable reference.  Introducing the same automated errors into the reference data undermines the validity of the evaluation. The authors' justification ("no significant impact on the results") is not convincing.

**Questions to Authors:**

1.  Could you provide a more detailed comparison of your system’s performance against existing automatic error typing approaches (e.g., Swanson & Yamangil, Xue & Hwa) using the same evaluation methodology (human evaluation or, ideally, against a shared dataset with gold standard error types)?
2.  Can you provide a more detailed error analysis identifying the types of linguistic tool errors that most frequently lead to misclassification? What steps could be taken to mitigate these errors?
3.  Why was it necessary to re-annotate the gold standard references using your automatic pipeline? Wouldn't it be more appropriate to evaluate your system against the original gold standard annotations, even if it requires a different evaluation metric?
4.  Could you provide examples of error cases where the rule-based system struggles to provide an accurate error type assignment, and explain why?

**Additional Comments:**

The table listing the error types (Table 2) is somewhat difficult to parse. A more visually clear presentation might be helpful. It would also be valuable to know the distribution of error types in the CoNLL-2014 dataset after automatic annotation.
