## Peer Review: Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction

**Summary:**

This paper presents a method for automatically annotating grammatical error correction (GEC) system outputs with error type information. The approach involves a two-step process: linguistically-enhanced edit extraction followed by rule-based error type classification. The authors argue this addresses a gap in GEC evaluation, allowing for more granular analysis beyond overall scores and enabling comparison of system performance across different error types. They evaluate their system on the CoNLL-2014 data, demonstrating its ability to provide consistent, dataset-independent error type annotations.

**Main Contributions:**

1. **Automated Error Type Annotation Pipeline:** The primary contribution is a functional pipeline for automatically annotating GEC corrections with error types, bridging a significant gap in the field. This offers a practical solution to analyzing system behavior beyond aggregate metrics.
2. **Dataset-Independent Rule-Based Error Typing:** The development of a rule-based error typing framework that relies solely on automatic linguistic analysis (POS tags, lemmas, stemming) is a valuable contribution. This avoids the bias inherent in training classifiers on specific, potentially inconsistently annotated datasets.
3. **Detailed Analysis of CoNLL-2014 Systems:** The application of this framework to the CoNLL-2014 shared task data provides a novel and insightful analysis of the strengths and weaknesses of different GEC systems, revealing varying performance across different error types.

**Strengths:**

1. **Addresses a Critical Gap in GEC Evaluation:**  The ability to automatically annotate corrections with error types is extremely valuable. Existing metrics rely on often opaque alignment algorithms, making it difficult to understand *why* a system succeeds or fails. This work directly tackles this issue, opening avenues for more targeted system development.
2. **Strong Justification for Rule-Based Approach:** The authors convincingly argue against machine learning-based approaches for error typing due to dataset dependency and potential bias. The rule-based approach, coupled with readily available NLP tools, offers a more robust and generalizable solution.  The 95%+ “Good” or “Acceptable” rating from expert reviewers strongly supports the quality of the rules.
3. **Comprehensive Evaluation on a Standard Dataset:**  The application of the tool to the CoNLL-2014 dataset is a strong point. This allows for direct comparison of systems and provides concrete insights into their performance profiles.  The detailed breakdown of performance across error types (Tables 5, 6, 7, 8) is highly informative.
4. **Clear and Well-Structured Presentation:** The paper is well-written, logically organized, and provides sufficient detail for reproducibility. The inclusion of the error type table (Table 2) is helpful for understanding the granularity of the analysis.

**Weaknesses:**

1. **Limited Novelty in Edit Extraction:** The edit extraction component relies on existing work (Felice et al., 2016) and does not present novel contributions in that area. While a solid foundation, this diminishes the overall novelty of the presented work. The paper acknowledges this, but the dependence on a pre-existing method is a limitation.
2. **Error Scheme Design Trade-offs:** The authors acknowledge the inherent difficulty in designing a comprehensive error scheme. While the current scheme is well-defined, the granularity level feels somewhat arbitrary in places (e.g., the distinction between certain VERB subtypes). Further justification for specific categorization choices would strengthen the work.
3. **Lack of Comparison to Human Annotation Baseline:** While the expert evaluation of the rule-based classifier is positive, it would be beneficial to compare the automatic annotations directly to a small-scale human-annotated gold standard.  This would provide a more quantitative assessment of the annotation accuracy.
4. **Over-Reliance on F0.5 Score:** The paper primarily uses the F0.5 score, which prioritizes precision.  While appropriate for GEC, reporting additional metrics (e.g., F1) would provide a more balanced evaluation.



**Questions to Authors:**

1. **Edit Extraction Error Impact:** How sensitive is the overall error type analysis to errors in the initial edit extraction phase? Did you conduct any experiments to assess the impact of edit extraction errors on the final error type classification results?
2. **Error Scheme Refinement:** Given the inherent trade-offs in error scheme design, are you planning to release the framework in a modular way, allowing users to customize or extend the error type categories?
3. **Human Baseline Details:**  Could you provide more detail about the expertise level of the five GEC researchers involved in the manual evaluation?  Were they explicitly instructed to follow a specific annotation guideline?
4. **Zero-Shot Generalization:** While the approach is dataset-independent, have you tested its performance on datasets significantly different from CoNLL-2014 (e.g., learner corpora from different L1 backgrounds)?

**Additional Comments:**

*   The link to the tool should be prominently displayed in the final version of the paper.
*   Consider adding a table summarizing the linguistic tools (spaCy version, Hunspell dictionary, etc.) used in the pipeline for better reproducibility.
* The discussion around M2 scorer vs. the proposed approach is well-argued.



Overall, this is a strong and valuable contribution to the field of GEC.  The proposed method addresses a critical need for more detailed and interpretable evaluation of GEC systems. While the novelty of the edit extraction component is limited, the rule-based error typing framework and the detailed analysis of CoNLL-2014 systems are significant achievements. I recommend acceptance with minor revisions to address the weaknesses outlined above.
