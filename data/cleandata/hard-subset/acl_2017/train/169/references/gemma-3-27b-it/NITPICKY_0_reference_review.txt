This paper presents a method for automatically annotating grammatical error correction (GEC) system outputs with error types, aiming to address the lack of detailed error analysis in previous GEC evaluations like CoNLL-2014. The core contribution is a rule-based error typing framework built on automatically obtained linguistic features (POS tags, lemmas, etc.), coupled with an existing linguistically-enhanced edit extraction method. A secondary contribution is the application of this framework to re-analyze the CoNLL-2014 shared task data, offering new insights into team performance at a granular level. Finally, the authors promise to release the tool used for this annotation.

**Strengths:**

1. **Addressing a Genuine Gap:** The paper tackles a significant problem in GEC evaluation – the inability to dissect system performance by error type beyond simple recall scores. Providing a means to automatically annotate corrections facilitates more nuanced analysis.
2. **Dataset Independence:** The rule-based approach, relying solely on automatic linguistic markup, is a strength. This avoids the biases introduced by training classifiers on specific, potentially idiosyncratic, datasets, a point the authors rightly emphasize. This portability is valuable.
3. **Detailed Analysis of CoNLL-2014:** Re-evaluating the CoNLL-2014 systems with the new annotation scheme yields interesting observations about the strengths and weaknesses of different approaches (e.g., MT for missing edits, classifiers for determiner replacements). This retrospective analysis is a useful application of the developed tool.
4. **High Rater Agreement:** The manual evaluation demonstrating >95% agreement from experts on the appropriateness of the error type assignments is compelling, lending credence to the rule-based system’s accuracy, despite the inherent noise from automated edit extraction.

**Weaknesses:**

1. **Overstated Novelty:** While the *combination* of linguistically-enhanced edit extraction with rule-based error typing is a reasonable approach, neither component is particularly novel. Felice et al. (2016) is extensively cited for the alignment, and rule-based systems for error typing have precedents (though perhaps not with the same dataset independence). The paper needs to be more modest in its claims of innovation.
2. **Insufficient Error Category Detail & Justification:** Table 2 shows 25 error types, but the descriptions are often terse and lack sufficient detail. The justification for *why* certain errors are categorized as they are (e.g., why [could → should] is a 'tense' error, not 'modal') feels arbitrary and could benefit from more rigorous linguistic motivation. The explanation regarding the compromise between informativeness and practicality is unconvincing.
3. **Methodological Flaws in Evaluation:** Reprocessing the gold reference data through the *same* automatic pipeline as the hypothesis is questionable. While the authors attempt to justify this by citing non-significant differences, it fundamentally undermines the validity of comparison to a true gold standard. The fact that the M2 scorer still yields different results suggests the automatic annotation introduces errors. The statement that the M2 scorer “exploits its dynamic edit boundary prediction” appears critical rather than descriptive.
4. **Lack of Quantitative Comparison to Baseline Classifiers:** The paper mentions previous work using machine learning classifiers for error typing, but provides no quantitative comparison. Reporting results alongside a baseline classifier (even if trained on a different dataset) would demonstrate the true advantage of the rule-based approach. The authors simply state their approach “bypasses bias effects altogether”, which is a strong claim requiring empirical evidence.
5. **Formatting and Presentation Issues:** The table formatting is inconsistent (varying column widths, missing separators). The references are not formatted correctly according to ACL guidelines. Minor details, but indicative of a lack of meticulousness.



**Questions to Authors:**

1.  Can you provide a more detailed justification for the error category definitions and the rationale behind grouping certain errors together (e.g., the decision not to label [could → should] as a modal error)?
2.  Why was it necessary to reprocess the gold reference data through the automatic annotation pipeline? What specific issues were you attempting to address, and how does this impact the validity of the evaluation?
3.  Can you provide quantitative results comparing the performance of your rule-based approach to a baseline classifier trained on a publicly available GEC dataset?
4.  The analysis in Section 4.6 highlights differences in detection vs. correction. Can you elaborate on why certain systems perform well at detection but poorly at correction? What insights does this provide about the limitations of these systems?
5.  Regarding Table 7, why is the Precision for CUUI’s Replacement Determiner errors so much higher than its Recall? What does this suggest about the nature of errors being corrected?




**Additional Comments:**

*   The claim that the tool will be released with the paper is positive, but details regarding the tool’s accessibility and documentation should be provided.
*   Consider including an ablation study to assess the contribution of different linguistic features to the accuracy of the error typing framework.
* The paper is generally well-written, but suffers from occasional instances of overly strong claims and insufficiently supported arguments. A more cautious and nuanced tone would improve the paper's credibility.
