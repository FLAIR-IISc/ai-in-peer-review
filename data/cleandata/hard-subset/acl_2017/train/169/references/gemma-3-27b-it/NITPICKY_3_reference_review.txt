This paper proposes a method for automatically annotating grammatical error correction (GEC) system outputs with error types. The authors address the issue that the CoNLL-2014 shared task evaluated systems based on error type, but did not require systems to *produce* error type annotations, hindering detailed analysis. The core contribution is a pipeline consisting of linguistically-enhanced edit extraction followed by rule-based error typing. A secondary contribution is the application of this pipeline to re-evaluate the CoNLL-2014 systems, offering new insights into team performance. A final contribution is the release of the annotation tool.

**Strengths:**

1. **Addresses a Real Problem:** The paper tackles a valid issue in GEC evaluation – the lack of standardized error type annotations for system outputs. This makes detailed analysis difficult and limits our understanding of system strengths and weaknesses.
2. **Clear Methodology:** The pipeline is clearly described, leveraging existing work on edit extraction (Felice et al., 2016) and presenting a novel rule-based error typing framework. The rationale behind the rule-based approach – dataset independence – is sensible.
3. **Detailed CoNLL-2014 Analysis:** The re-evaluation of the CoNLL-2014 systems provides interesting, granular data about team performance across different error types. The tables presented are comprehensive and offer concrete insights.
4. **Tool Release:** Releasing the annotation tool is a valuable contribution to the GEC community, potentially facilitating further research in this area.

**Weaknesses:**

1. **Overstated Novelty:** While the combination of existing techniques and the rule-based framework is a reasonable approach, the *novelty* is incremental.  The core idea of annotating system outputs isn’t new; the specific implementation details and the dataset-agnostic design are the main contributions, and these aren't groundbreaking.
2. **Rule-Based Limitations:** The rule-based approach, while aiming for dataset independence, inevitably suffers from brittleness. The authors acknowledge this by noting the trade-off between informativeness and practicality when defining the error categories. The evaluation, though positive, relies on a relatively small sample size (200 edits) and doesn't fully demonstrate the system’s generalization capability. The reliance on accurate POS tagging is a significant vulnerability, as highlighted by the raters’ feedback.
3. **Evaluation Concerns:** Reprocessing the gold references with the automatic pipeline to enable a direct comparison is a questionable decision. While the authors demonstrate no *statistical* difference, it introduces a potential bias by applying a potentially imperfect automatic annotation process to the gold standard.  The argument that M2 overestimates performance is not convincingly supported; simply stating it is a known issue isn't sufficient. A more rigorous comparison, perhaps examining agreement with human annotations on a larger scale, is needed.
4. **Inconsistent Detail:** The paper provides an *extremely* detailed error type list (Table 2) but offers limited discussion on the reasoning behind specific choices and potential ambiguities. For instance, the distinction between “modal” and “tense” errors appears somewhat arbitrary, as the authors concede. A more in-depth discussion of the challenges in defining a comprehensive and unambiguous error typology would be beneficial.
5. **Formatting and Redundancy:** Numerous minor formatting issues exist (e.g., inconsistent spacing, line breaks in references). Additionally, certain sections, particularly the introduction, contain repetitive phrasing and could be more concise.

**Questions to Authors:**

1.  How robust is the rule-based classifier to variations in POS tagger accuracy? Have you investigated the impact of using different POS taggers or fine-tuning the existing one?
2.  Could you provide more details on the rationale behind the specific error type categories defined in Table 2, especially those that seem ambiguous or overlap (e.g., the distinction between "tense" and "modal" errors)?
3.  Why was the decision made to re-annotate the gold references with the automatic pipeline? What is the potential impact of this decision on the validity of the CoNLL-2014 re-evaluation results?
4.  The paper mentions that some teams didn't attempt to correct certain error types.  Is there any analysis of *why* these teams avoided those error types?  Could this reveal limitations in their approaches?
5. Could you share the code for the annotation tool publicly, as promised?



**Additional Comments:**

*   The references are not consistently formatted.
*   The sentence structure in several places is awkward or unclear.
*   Table 6 is very large and difficult to parse. Consider summarizing the key findings instead of presenting the entire table.
*   The use of the term “UNK” in Table 9’s error type list should be better explained – is it only for uncorrected errors, or can it also indicate errors that the system cannot classify?
