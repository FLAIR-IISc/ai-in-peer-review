## Review for ACL: Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction

**Summary:**

This paper tackles the problem of evaluating Grammatical Error Correction (GEC) systems beyond overall scores, by presenting a method for automatically annotating system outputs with error types. The approach combines linguistically-enhanced edit extraction (Felice et al., 2016) with a novel, rule-based error typing framework. The authors demonstrate the utility of their tool by re-evaluating the CoNLL-2014 shared task results, providing a detailed breakdown of system performance across different error types.

**Main Contributions:**

1. **A Dataset-Agnostic Error Typing Framework:** The core contribution is a rule-based system for error type classification that relies *solely* on automatically obtained linguistic features (POS tags, lemmas, stems) rather than requiring labeled training data. This addresses a key limitation in GEC evaluation, where datasets are often annotated inconsistently.
2. **Detailed Re-evaluation of CoNLL-2014:** The paper provides a valuable re-analysis of the CoNLL-2014 GEC shared task, generating error type breakdowns for each system and highlighting strengths and weaknesses in different approaches. This is a concrete application and demonstration of the tool's utility.
3. **A Publicly Available Tool:** The authors intend to release the annotation tool, which will be a valuable resource for the GEC community.

**Strengths:**

1. **Novelty of the Rule-Based Approach:** The rule-based error typing is a strong point. Machine learning approaches suffer from domain dependence and labeling issues.  This dataset-agnostic approach is a significant advantage, especially given the heterogeneity of GEC datasets. The reported high agreement with expert raters ( >95% “Good” or “Acceptable”) validates the approach.
2. **Practicality & Granularity:**  The framework’s flexibility, allowing evaluation at different levels of granularity (e.g., ‘R:NOUN’ for replacement noun errors), is commendable. This facilitates nuanced analysis of system performance. The detailed breakdown of error types (Table 2 & 6) offers a rich understanding of individual system capabilities.
3. **Thorough Evaluation & Insightful Analysis:** The re-evaluation of the CoNLL-2014 data is comprehensive. The observations about the challenges in correcting specific error types (e.g., unnecessary tokens, multi-token edits) and the complementary strengths of different system types (CAMB vs. CUUI) are insightful and contribute to a better understanding of the GEC landscape.
4. **Addressing a Core Evaluation Problem:** The paper directly addresses a longstanding issue in GEC evaluation – the lack of detailed error type breakdowns. The ability to automatically annotate system outputs with error types is a crucial step towards more informative and actionable evaluations.
5. **Focus on Edit Level Analysis:** The emphasis on analyzing edits rather than just overall scores, and the exploration of multi-token edits, are valuable additions to the field. This allows for a deeper understanding of *how* systems are making corrections, not just *if* they are correct.

**Weaknesses:**

1. **Reliance on Underlying NLP Tools:** The performance of the rule-based classifier is heavily dependent on the accuracy of the underlying NLP tools (spaCy, Lancaster Stemmer, Hunspell). While the authors acknowledge this in the error analysis (e.g., POS tagging errors), the paper lacks a detailed error analysis *of the NLP pipeline itself*. How robust is the system to variations in POS tagger performance? A sensitivity analysis would strengthen the work.
2. **Limited Error Type Coverage:** While 25 error types are a good starting point, the framework might be insufficient to capture the full complexity of real-world GEC errors. The discussion around adding categories highlights this trade-off. The framework seems particularly weak on stylistic or discourse-level errors, focusing primarily on grammatical correctness.
3. **Comparison to Human Annotation:**  The manual evaluation with 5 raters is helpful, but is limited in scope (200 edits). A more extensive evaluation against human annotation, perhaps on a larger, publicly available dataset, would provide a more robust assessment of the system’s accuracy. How does the system compare to inter-annotator agreement on the same dataset?
4. **Lack of Error Prevention Analysis:** The paper focuses on *correcting* errors. It would be valuable to also consider the error *prevention* capabilities of the systems.  Are certain systems better at avoiding introducing new errors during correction?  Analyzing this aspect would provide a more complete picture of system quality.
5. **M2 Scorer Limitations:**  While the authors acknowledge the M2 scorer’s inflation of scores, the discussion could be expanded. The differences between their edit comparison approach and M2 are significant, and further exploration of these discrepancies, and their implications for GEC evaluation, would be beneficial.

**Questions to Authors:**

1. **NLP Pipeline Error Analysis:** Could you provide a more detailed analysis of the errors made by the underlying NLP tools (spaCy, etc.) and how these errors impact the accuracy of the error type classification?
2. **Scalability & Computational Cost:** What is the computational cost of the annotation pipeline? How does it scale with the size of the input text?
3. **Extensibility:** How easily can the rule-based framework be extended to incorporate new error types or adapt to different annotation schemes?
4. **Error Prevention Metrics:**  Have you considered incorporating metrics to assess a system’s ability to *avoid* introducing new errors during correction?
5. **Inter-Annotator Agreement:** How does the agreement between your automatic annotation and multiple human annotators compare to the inter-annotator agreement among the human annotators themselves?



**Additional Comments:**

*   The code release is highly anticipated and will be crucial for the wider adoption of this work.
*   The tables are generally clear and well-organized, but some (e.g., Table 6) are very large and difficult to interpret quickly. Consider using visualization techniques to highlight key findings.
*   The reference list appears incomplete.



**Overall Assessment:**

This paper presents a valuable contribution to the field of GEC evaluation. The proposed rule-based error typing framework is novel, practical, and well-validated. The detailed re-evaluation of the CoNLL-2014 data provides useful insights into the strengths and weaknesses of different GEC approaches. While some weaknesses remain, the strengths of the work outweigh them, and I recommend acceptance with minor revisions addressing the questions and suggestions above.
