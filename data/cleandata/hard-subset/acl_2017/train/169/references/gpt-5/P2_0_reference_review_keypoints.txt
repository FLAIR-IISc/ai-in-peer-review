- Primary contribution: A dataset-agnostic evaluation framework and tool that auto-annotates GEC hypothesis outputs with explicit edit spans and rule-based error types using linguistic markup.
- Framework components: Linguistically enhanced edit extraction (based on Felice et al., 2016) and a deterministic rule-based error type classifier driven by POS/lemma/UD tags and lexicons.
- Multi-granularity evaluation: Enables per-error-type evaluation at operation, POS, and morphology tiers.
- Empirical case study: Comprehensive re-analysis of CoNLL-2014 systems revealing strengths/weaknesses (e.g., problems with unnecessary deletions and multi-token edits).
- Reference auto-processing: Shows auto-processed references yield similar system rankings/scores as gold references (no significant bootstrap differences), justifying a fully automatic pipeline.
- Practical impact: Addresses the lack of standardized per-type analysis for unannotated hypotheses, aiding error analysis, model selection, and diagnostics.
- Design rationale: Rule-based, dataset-agnostic typing improves cross-corpus comparability, auditability, and explainability versus ML classifiers.
- Human agreement evidence: High Good/Acceptable rates (>95%) across 5 experts and κfree=0.724 suggest rules align with expert intuitions.
- Thorough analyses: Detailed operation-tier, per-type (24 categories), multi-token handling, and detection vs. correction breakdowns give actionable insights.
- Evaluation protocol strengths: Auto-processing both sides, separating extraction from scoring, and critique of M2 scorer dynamics are well-motivated and useful.
- Reproducibility intent: Taxonomy and pipeline components are listed and tool release is promised, making it a valuable resource.
- Limited validation scale: Manual error-typing validation is small (200 edits, 5 raters) and English-only, so dataset-agnostic claims need broader support.
- Dependence on NLP markup: Misclassifications often stem from POS/tagging errors; robustness to different taggers/parsers is unquantified and could bias per-type evaluation.
- Engineering vs. novelty: Contribution is mainly engineering/integration and rule design rather than new algorithms; paper should be framed as an evaluation/resource work and fully document rules.
- Metric claims need more evidence: Lack of correlation analyses with human judgments for the exact-span metric vs. M2 means claims about annotation parity may be overstated.
- Language portability unclear: Uses English-specific resources and taxonomy; portability to other languages is not demonstrated or discussed in detail.
- Requested reproducibility items: Authors asked to release code, rule files, and auto-annotated CoNLL-2014 files and to provide a timeline.
- Robustness question: Reviewer requests sensitivity analysis of type assignments to different tagger/parser choices (e.g., spaCy vs. UDPipe/Stanza).
- Error analysis request: Asked for quantitative breakdown of “OTHER” and “UNK” categories and analysis of common misclassifications.
- Comparison to human-labeled corpora: Reviewer asks for direct comparisons to corpora with gold error types (e.g., FCE) beyond the small rating study.
- Multi-token span flexibility: Suggestion to consider constrained matching that tolerates minor boundary differences while preserving type correctness.
- Overall assessment: Useful, timely, and well-argued GEC evaluation contribution with actionable insights; broader validation and public release would make it a valuable community tool.
