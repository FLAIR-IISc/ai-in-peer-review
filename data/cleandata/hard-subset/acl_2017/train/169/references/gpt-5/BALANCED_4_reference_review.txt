Summary
This paper proposes a two-step, fully automatic pipeline to annotate grammatical error correction (GEC) outputs with explicit edits and error types: (1) linguistically-enhanced edit extraction via Damerau–Levenshtein alignment with POS/lemma-informed costs; (2) a deterministic, rule-based error type classifier driven by automatic markup (POS, lemmas, stems, dependencies, Hunspell). The framework is dataset-agnostic and aims to enable detailed, comparable error-type evaluation for GEC systems. A small manual evaluation (200 edits, 5 raters) indicates high perceived appropriateness of predicted error types. The authors apply the tool to all CoNLL-2014 system outputs, demonstrate that reprocessing references automatically yields system rankings comparable to those using gold references, and provide analyses by operation (Missing/Replacement/Unnecessary), by fine-grained error type, multi-token edits, and detection vs. correction.

Main contributions (as I see them)
1) A practical, deterministic framework for automatic error-type annotation and evaluation of GEC outputs, including a released tool. The main value lies in making type-level evaluation accessible and consistent across datasets/systems.
2) A large-scale empirical analysis of CoNLL-2014 systems at multiple granularity levels (operation tier, error categories, multi-token edits, detection vs. correction), yielding actionable insights about complementary strengths and persistent weaknesses.
3) Evidence that using automatically reprocessed references instead of gold references does not significantly change system-level scores, facilitating streamlined evaluation pipelines.

Strengths
- Practical impact and tooling: The framework closes a well-known gap since CoNLL-2014: teams were not required to produce error-type annotations, hindering type-level analyses. Making a reproducible, deterministic tool available is highly valuable for the community and future shared tasks.
- Dataset-agnostic design and transparency: Rules rely only on automatic markup (POS/lemmas/deps/lexicon), avoiding dependence on corpus-specific annotation schemes or trained classifiers. Determinism improves interpretability and reproducibility, and facilitates error analysis; one can always trace why a label was assigned.
- Convincing utility via thorough analyses: The CoNLL-2014 case study is substantive. The paper surfaces non-obvious findings (e.g., widespread difficulty with Unnecessary edits by several systems; classifier approaches excelling on VERB:FORM and ORTH; MT better on multi-token edits; detection–correction gaps). These insights are immediately useful to system developers.
- Sensible evaluation protocol design: Separating edit extraction from evaluation simplifies scoring and enables direct span-and-correction matching. The bootstrap test showing no significant difference between gold vs. auto references for overall F0.5 lends credibility to the pipeline, and the critique of M2’s dynamic boundary choices is careful and grounded.
- Human validation of the classifier: While modest, the 5-rater evaluation shows high acceptability (86.3% “Good”, 10.0% “Acceptable”) and reasonable inter-rater agreement (κ_free=0.724), supporting the rules’ face validity.

Weaknesses
- Limited validation granularity and scope: The manual evaluation is small (200 edits) and reports only coarse Good/Acceptable/Bad judgments, not per-type precision/recall or confusion patterns. The study is restricted to English learner data; claims of dataset-agnosticism would be strengthened by testing on multiple corpora/genres and with diverse error distributions.
- Dependence on upstream automatic markup: The system’s correctness is sensitive to POS/dependency tagging and tokenization, as the authors note (e.g., ring/rings NOUN vs. VERB). No robustness analysis is provided across different taggers/parsers, nor an ablation to quantify how tagging errors propagate to type misclassification.
- Under-specified rule set and coverage: The classifier’s rules are described at a high level. Without more formal specification, examples, or coverage statistics per rule, it’s hard to assess edge cases, rule conflicts, and long-tail categories (e.g., MODAL vs. TENSE conflation). Some design choices (e.g., SPELL including dialectal variants like color→colour) could inflate certain categories or conflate orthography with variant preference.
- Conflation of extraction and typing errors in evaluation: Many reported results (e.g., type-level F-scores) rest on automatically determined spans on both hypothesis and reference. It remains unclear how much error stems from boundary extraction vs. type assignment. There is no evaluation using gold spans to isolate classifier performance.
- Generalization beyond CoNLL-2014 and metrics: The empirical analysis is limited to CoNLL-2014 system outputs and F0.5. It would be informative to see whether insights hold for other test sets (e.g., FCE, JFLEG), and how conclusions interact with other metrics (e.g., detection-only evaluation per type, alternative match criteria).

Questions to Authors
- Rules and precedence: How are rule conflicts resolved when multiple categories are plausible? Is there a deterministic precedence order you can share? Will the released tool include the full rule set and unit tests?
- Sensitivity to taggers/parsers: Have you tried alternative POS/dependency taggers? How stable are type assignments across taggers, and how large is the induced variance in system-level scores?
- Boundary vs. type errors: Can you report classifier performance on gold edit spans (e.g., from FCE/NUCLE), to disentangle extraction from typing? Conversely, how well does your aligner approximate human spans on CoNLL-2014?
- Dialect and lexicon choices: Using GB Hunspell flags American vs. British variants as SPELL. Do you allow configurable dialects or lexicons? How does this affect SPELL/ORTH vs. OTHER categories?
- Extensibility: How easy is it to add new categories (e.g., dedicated MODAL, collocations) or adapt rules to languages beyond English? What are the practical steps and expected effort?
- Detection breakdown: Your tool can compute detection-only scores per type. Could you provide or release this analysis for CoNLL-2014?

Additional comments and suggestions
- Provide per-type confusion matrices from the human evaluation to identify systematic mislabels.
- Report coverage and frequency of each rule firing, and ablations for key signals (POS, lemma, deps, Hunspell).
- Consider releasing outputs in a standard, machine-readable format with comprehensive metadata (span indices, operation, type, rule fired, tagger outputs), to facilitate community use.
- Explore robustness by running the pipeline with multiple taggers/parsers and reporting variance.
- Revisit multi-token edits: analyze which phenomena dominate (e.g., phrase insertion/deletion, reordering) and where rules fail; this can guide future modeling.

Recommendation: Weak Accept. The technical novelty is moderate, but the paper provides a timely, useful tool and a thorough, insightful analysis that the GEC community can directly use. Strengthening empirical validation and detailing the rules would further solidify the contribution.
