Summary and main contributions
- A two-stage pipeline for automatic annotation of grammatical error correction (GEC) outputs: (i) linguistically-informed edit extraction via a Damerau–Levenshtein alignment with merging rules, and (ii) a deterministic, rule-based error type classifier driven by automatic POS/lemma/stem/parse signals.
- A typed evaluation of CoNLL-2014 system outputs at multiple granularities (operation tier and fine-grained categories), including analyses of multi-token edits and detection vs. correction.
- A claim that automatically reprocessing the gold reference to match the hypothesis processing pipeline yields comparable aggregate scores to standard M2 evaluation, arguing their automatic reference is “as good as” human reference for evaluation purposes.

Strengths
- Practical utility and scope: Producing consistent, typed annotations of system outputs at scale is valuable to the community and allows analyses that were previously cumbersome or impossible (operation-tier breakdowns, fine-grained error-type comparisons, multi-token edit performance).
- Deterministic, interpretable classifier: A transparent rule-based approach that does not require training data is attractive for reproducibility, explanation, and dataset-agnostic deployment (within the same language).
- Sensible engineering choices: Using a linguistically-enhanced aligner and coarse-grained UD POS mapping aligns with intuition for edit extraction and type assignment; using only automatic signals avoids reliance on idiosyncratic annotation schemes.
- Empirical insights: The breakdowns highlight complementary strengths across system families (e.g., MT vs. classifiers), surprising weaknesses (e.g., missing handling of unnecessary tokens), and the general difficulty of multi-token and fluency-oriented edits—information useful for system design.
- Initial human assessment: The small manual study indicates that, in context, most assigned types are judged Good/Acceptable by multiple expert raters despite automatic edit boundaries.

Weaknesses
- Validation of error typing is insufficient and conflates tasks: The 200-edit, 5-rater study asks for perceived appropriateness, not accuracy against gold error types. There is no quantitative evaluation of type assignment against any gold-typed corpus (even if partial mappings were needed) nor per-type reliability/confusion analyses. The outcome metric (Good/Acceptable/Bad) masks systematic category confusions; kappa is computed on the 3-way appropriateness labels rather than on type labels.
- Overreach in claims about auto vs. gold references: The bootstrap test showing no significant difference in overall F0.5 when reprocessing references does not demonstrate that automatic alignment and type assignment are “qualitatively as good as” human annotations. Aggregate score stability can persist despite substantial span/type mismatches. No span-level or type-level agreement statistics between the gold reference edits and the auto-processed reference are reported.
- Heavy dependence on noisy automatic NLP for learner text without robustness analysis: The method hinges on POS tags, lemmas, stems, and dependency parses on non-native text, where error rates are non-trivial. While a few anecdotal POS failures are acknowledged, there is no sensitivity analysis (e.g., ablating POS vs. parse, swapping taggers/parsers, measuring performance degradation) nor error attribution quantifying how much each component affects type decisions.
- Incomplete methodological specification and reproducibility: The classifier’s rule inventory, precedence, conflict resolution, and decision logic are not fully specified; Appendix A lists allowed label combinations but not the concrete rule set. Without a released tool at review time and precise implementation details (e.g., spaCy models, UD mapping version, Hunspell config, WO detection rules), replication is unlikely.
- Language and scheme generality overstated: The approach is “dataset-agnostic,” but English-specific resources (Penn/UD mapping, determiner categories, SVA, GB Hunspell) and categories (articles, SVA, tense handling) limit cross-lingual generality. Even within English, the GB dictionary choice could misclassify US spellings as errors. The paper does not discuss portability or how types map across schemes.
- Evaluation design issues: 
  - The critique that M2 “overestimates” is asserted largely by example; a controlled, paired analysis quantifying TP/FP deltas and effect sizes is missing.
  - Multiple bootstrap tests (per team) lack correction for multiple comparisons; sampling unit (sentence vs. edit) and resampling protocol are not specified.
  - Multi-token analysis reports F0.5 but not sample sizes per team/category; Table 8 mentions “roughly 220” edits but no per-type confidence intervals.
- Presentation/clarity problems: Numerous formatting errors, broken references, footnote handling, hyphenation artifacts, and inconsistent notation impede readability and raise concerns about carefulness.

Questions to authors
- Can you provide span-level and type-level agreement between gold reference annotations and your auto-processed reference (exact match, macro/micro, per-type), along with confusion matrices?
- What is the full, deterministic rule set for the classifier, including precedence and tie-breaking? How is ambiguity between, e.g., MORPH vs. VERB:FORM vs. ADV handled?
- Sensitivity analysis: How does performance change with different taggers/parsers, or when removing parsing? What fraction of “Bad” judgments stems from POS vs. parsing vs. rule design?
- Can you evaluate type assignment against a corpus with gold types via a mapping (even partial), to provide an actual accuracy metric per category?
- How is Word Order detected operationally? Does it depend on dependency arc patterns or token reordering heuristics?
- What population and unit were used for bootstrap testing? Were multiple comparisons corrected?
- Will the tool, trained models, and exact versions (spaCy model name, UD mapping script, Hunspell dictionary) be released at camera ready?

Additional comments
- References are incomplete/inconsistent (dangling punctuation, repeated author strings with “.?”, truncated venue names, missing years/pages). Please fix all entries and conform to ACL style.
- Footnote markers appear inline (e.g., “Damerau-Levenshtein1”) with the note embedded awkwardly; consolidate using proper footnotes.
- Numerous typographical issues: stray hyphenations from PDF extraction (e.g., “in- dependent”), inconsistent quotes (“Good’”), inconsistent use of en dash vs hyphen, sporadic italics/code formatting for labels.
- Table captions and column heads need tightening; report sample sizes and, where feasible, confidence intervals. Highlighting “highest F” is mentioned but not visible here—ensure consistent styling.
- Clarify that “UNK” indicates detected-but-uncorrected and explain how it is produced; current wording is confusing.
- Justify the GB Hunspell choice; consider dual US/GB lexicons or explicit dialect handling.
- Minor reproducibility: specify max-unchanged-words parameter for M2 in all comparisons; state any deviations from defaults in all tools.

Overall, while the goal and tool are useful and the analyses are informative, the paper falls short on validation rigor, methodological detail, and presentation quality. I recommend rejection unless the authors can substantially strengthen the empirical validation of the error typing and the auto-vs-gold alignment claims, and clean up the exposition.
