Summary and contributions
This paper presents an automatic pipeline to annotate grammatical error correction (GEC) outputs with explicit edit spans and error types, and to evaluate systems by error category. The approach has two parts: (i) linguistically-informed edit extraction via a Damerau–Levenshtein aligner that prefers linguistically similar tokens; and (ii) a deterministic, rule-based error type classifier that uses only automatic annotations (POS, lemma, stems, dependencies, dictionary lookup). The authors reprocess references with the same pipeline to avoid span/type mismatches and then evaluate by exact edit matching. They validate the classifier with a small human study and apply the tool to provide a detailed CoNLL-2014 system analysis (operation tier, per-type scores, multi-token edits, and detection vs. correction).

Contributions (as I see them):
1) A deterministic, dataset-agnostic rule-based classifier and taxonomy for error typing that allows flexible aggregation (operation tier × POS/morphology) without training data.
2) An evaluation protocol that standardizes hypothesis and reference processing to enable per-type precision/recall/F0.5 and facilitates analyses not possible with M2 alone.
3) A comprehensive, novel error-type analysis of CoNLL-2014 systems revealing complementary strengths, systematic weaknesses (e.g., unnecessary edits, multi-token edits), and a useful detection vs. correction breakdown.

Strengths
- Clear practical value and enabling infrastructure: The pipeline fills a gap by allowing detailed error-type evaluation of unannotated system outputs, something not supported by common GEC metrics. The dataset-agnostic, transparent rules are a strong engineering contribution with direct utility for the community.
- Transparent and reproducible reasoning: Rules depend only on automatic markup, making decisions traceable, consistent, and explainable (vs. opaque ML classifiers). This design also avoids domain-mismatch issues in prior learned error-typing approaches.
- Empirical validation that is fit-for-purpose: Although modest, the human study shows high appropriateness of error labels (86% good, 10% acceptable, 0.724 κfree) even with noisy automatic edit extraction—evidence that the rule set is reasonable in practice.
- Insightful analyses that advance understanding: The per-type breakdowns yield actionable findings (e.g., CUUI excels at R:MORPH/VERB:FORM; CAMB at M:DET and coverage breadth; multiple teams failing on unnecessary edits; MT best on multi-token edits). The detection vs. correction comparison is particularly informative for diagnosing pipeline bottlenecks.
- Sensible evaluation setup and critical discussion of M2: Reprocessing references to remove span-type mismatch is justified empirically (no significant difference vs. gold for overall F0.5), and the paper offers a useful critique of how M2’s dynamic boundaries can inflate scores.

Weaknesses
- Limited intrinsic validation of error typing: The human study is small (200 edits), subjective (appropriateness ratings), and not stratified by error type, operation, or edit length. There is no quantitative agreement with a gold error-typed corpus, nor ablations to bound sensitivity to POS/lemma/parse errors. This undermines strong claims of dataset-agnostic reliability.
- Heavy dependence on automatic linguistic markup without robustness analysis: The authors acknowledge POS/parsing errors drive “Bad” labels, but do not quantify how much each component (POS, lemma, dependency parse, stemming, dictionary) contributes to classification accuracy. There is no test with alternative taggers/parsers or on different domains/registers.
- Strict exact-match evaluation inherits reference bias: While the method is appropriate for per-type comparisons, exact span and string match against a single reference can under-credit plausible alternative corrections and stylistic rewrites (a known limitation in GEC). The paper does not assess correlation with human judgments nor explore multi-reference scenarios for per-type scoring.
- Taxonomy design choices and coverage not fully justified: Some categories conflate phenomena (e.g., modals folded into tense), and rules for difficult distinctions (e.g., VERB:FORM vs. VERB:TENSE, ADJ vs. ADV) are not documented. Without rule details/examples and mapping to prior schemes (e.g., FCE/NUCLE), it is hard to assess generality and interpretability across studies.
- Incomplete reporting and polish: The references section is error-prone and inconsistent; implementation details (full rules, parameters, handling of dialectal variants, multiword tokens) are sparse. Claims about M2 inflation would benefit from controlled experiments quantifying the effect of the max-unchanged-words parameter and dynamic boundary choices.

Questions to authors
- Release: Will you release the full tool (code, models, rule set), the auto-annotated CoNLL-2014 outputs, and scripts to reproduce all tables? Under what license?
- Robustness: How sensitive is error typing to the choice/accuracy of POS tagger and parser? Can you report an ablation (e.g., POS only vs. POS+lemma vs. full features) and test with a different tagger/parser?
- Validation: Can you provide a per-type breakdown of “Good/Acceptable/Bad” judgments, and a small gold-typed set to report precision/recall by type? Alternatively, compare against an existing manually typed resource.
- Taxonomy and rules: Please document the decision rules with examples, especially for ambiguous cases (e.g., VERB:FORM vs. VERB:TENSE, NOUN vs. VERB tags in noisy learner text). How are contractions, dialectal spellings (color/colour), and non-words handled?
- Evaluation fairness: For per-type scoring, how do you handle multiple valid references or partial credit when span boundaries differ but error type and locus match? Could a relaxed overlap match be used for analysis (not leaderboard scoring)?
- Generalization: Have you tried the tool on other corpora (e.g., FCE full, JFLEG) and obtained similar conclusions about per-type behavior and multi-token edits?
- Efficiency: What is the runtime and memory footprint per sentence for annotation and scoring? Is the tool practical for large-scale benchmarks?

Overall, this is a useful, well-motivated engineering contribution that enables richer GEC analysis, with convincing downstream insights. Strengthening intrinsic validation, documenting the rules/taxonomy, and adding robustness experiments would substantially improve the paper.
