Summary
The paper presents a deterministic pipeline to automatically annotate grammatical error correction (GEC) outputs with edit spans and error types, enabling per-type evaluation without requiring manually labeled error types. The pipeline has two steps: (i) linguistically informed edit extraction via a Damerau-Levenshtein aligner augmented with POS/lemma features, and (ii) a rule-based classifier that assigns edits to a new, dataset-agnostic error type scheme (25 base types; 55 valid type-operation combinations). A small human study (5 raters, 200 edits) judges the assigned types as Good/Acceptable in >95% of cases. The tool is applied to all systems from CoNLL-2014 to provide detailed per-type, operation-tier (Missing/Replacement/Unnecessary), multi-token, and detection vs. correction analyses. The authors also show that reprocessing references with the same automatic aligner yields overall scores statistically indistinguishable from gold references, and argue that the M2 scorer inflates performance due to dynamic edit-boundary choices.

Main contributions (as I see them)
- A practical, deterministic, dataset-agnostic error typing framework and tool that labels edits using only automatic linguistic markup (POS, lemma, morphology, dependency, spelling), enabling detailed per-type evaluation of unannotated system outputs.
- A comprehensive retrospective analysis of CoNLL-2014 systems across operation tiers, fine-grained error types, multi-token edits, and detection vs. correction, revealing actionable strengths/weaknesses of different approaches (e.g., classifiers vs MT).
- Empirical observation that auto-reannotating references aligns overall system scores with gold (no significant difference), plus a critical examination of M2’s dynamic matching behavior that likely inflates scores.

Strengths
- Practical impact and tooling: The paper addresses a real gap in GEC evaluation—lack of per-type scoring for hypotheses—by releasing a tool that can annotate any parallel data without training. This is immediately useful for researchers to diagnose systems and for shared tasks to report richer metrics.
- Deterministic, transparent rules: By eschewing supervised classifiers, the method avoids domain-specific training biases and offers traceable, explainable labeling decisions. This is important for standardization and reproducibility.
- Human validation: Despite noisy automatic alignments, 86.3% of labels were rated Good and only 3.7% Bad; inter-annotator agreement (κ_free=0.724) supports that the rules generally assign appropriate categories. This is stronger evidence than purely automatic proxy metrics.
- Insightful analyses: The CoNLL-2014 breakdowns surface meaningful findings: widespread neglect of Unnecessary deletions by several teams; classifiers excelling on VERB:FORM and ORTH; MT dominating multi-token edits but all systems weak there; differing detection vs. correction profiles. These insights can guide system design (e.g., combining CAMB’s missing-token strength with CUUI’s replacement strength).
- Clean separation of alignment and scoring: The paper argues convincingly that evaluation should compare explicit edits rather than embed alignment into the metric, and empirically shows auto- vs gold-reference parity using bootstrap tests, while highlighting how M2’s dynamic boundaries can inflate scores.

Weaknesses
- Limited validation of the error-type classifier: The human study is small (200 edits) and measures appropriateness rather than accuracy against a gold error-type scheme. There is no evaluation on corpora with established error-type labels, nor per-type agreement analysis, making it hard to assess reliability for fine-grained categories.
- Insufficient disclosure of rules and taxonomy mapping: The paper lists categories but does not fully specify the rule set or decision criteria for disambiguating categories (e.g., between VERB:TENSE vs VERB:FORM vs MORPH; WO vs MORPH). This hampers reproducibility and adoption, and makes it difficult to audit edge cases.
- Dependence on automatic preprocessing: The approach critically relies on POS/lemmatization/dependencies; the authors note mislabeling often traces to tagger errors. However, there is no sensitivity analysis to different taggers/parsers, domains, or parameter settings, nor calibration of how preprocessing error rates translate to label reliability across categories.
- Stability and significance of per-type analyses: While overall auto vs gold parity is tested, there is no statistical assessment for per-type scores, which may be more sensitive to alignment and POS errors. Some strong conclusions (e.g., “five teams failed to resolve any unnecessary token errors”) could be confounded by alignment/operation decisions and would benefit from robustness checks.
- Novelty: The core edit extraction is borrowed; the main novelty is the rule-based error typing and its application to shared-task outputs. This is a useful engineering contribution but conceptually incremental relative to prior edit classifiers; the methodological advances are more in standardization and tooling than in new algorithms.

Questions to authors
- Can you publish the complete rule inventory (or decision tree) used to assign error types, including precedence/overrides and examples for each rule? This is key for reproducibility and community adoption.
- How sensitive are error-type assignments to the choice and quality of POS/lemmatizer/parser? Have you tried alternative taggers or an ablation (e.g., POS only vs POS+dep) to quantify robustness?
- How do you disambiguate between VERB:TENSE, VERB:FORM, VERB:INFL, and MORPH in cases with overlapping surface cues? Similarly, how is WO distinguished from PREP/ADV placement changes?
- What is the definition and assignment procedure for UNK in the automatic setting? Under what conditions does the system “detect an error but is unable to correct it” without gold?
- Can you release the auto-annotated CoNLL-2014 hypotheses and auto-reannotated references, along with scripts to reproduce Tables 4–8?
- Do per-type results remain stable when varying the aligner’s parameters (e.g., max unchanged words) or when using gold vs auto POS on the references?

Additional comments
- The reference list contains many incomplete or malformed entries; please fix citations. Consider evaluating on additional datasets (e.g., JFLEG/BEA) to support the “dataset-agnostic” claim and include per-type confidence intervals. Upgrading tooling beyond spaCy v1.6 and documenting versions will aid longevity. Overall, this is a valuable, practical contribution for the GEC community.
