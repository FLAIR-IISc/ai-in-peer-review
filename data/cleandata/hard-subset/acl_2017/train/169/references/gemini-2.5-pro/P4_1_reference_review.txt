### Summary and Contributions

This paper introduces a methodology and a tool for automatically annotating Grammatical Error Correction (GEC) data with explicit edit spans and error types. The primary motivation is to enable detailed, error-level evaluation of GEC systems, a capability lacking in standard evaluation practices, as exemplified by the CoNLL-2014 shared task. The proposed method first uses a linguistically-enhanced alignment algorithm to extract edits between original and corrected sentences. It then classifies these edits into a newly proposed schema of 25 error types using a rule-based system that relies on automatic linguistic markup (e.g., POS tags, lemmas). To demonstrate its utility, the tool is applied to re-evaluate all participating systems from the CoNLL-2014 shared task, providing the first comprehensive, fine-grained analysis of their performance across different error categories.

In my view, the main contributions of this work are:

1.  **A comprehensive, fine-grained performance analysis of the CoNLL-2014 GEC shared task systems.** The detailed breakdown of system performance across 25 error types, edit operations, multi-token edits, and detection vs. correction provides significant and novel insights into the specific strengths and weaknesses of different GEC architectures. This analysis serves as a valuable resource for the GEC community.
2.  **A dataset-agnostic framework and tool for automatic error type annotation.** The paper presents a complete, rule-based pipeline for annotating GEC corpora. Its independence from labeled training data makes it broadly applicable and avoids the dataset-specific biases of machine learning classifiers. The planned release of the tool is a practical contribution to the field.
3.  **A standardized GEC evaluation methodology.** The authors propose a workflow where both the reference and system hypotheses are processed through the same automatic annotation pipeline. This ensures that comparisons are made on a consistent basis and simplifies the evaluation logic, addressing certain methodological complexities present in existing metrics.

### Strengths

1.  **High practical utility and significant community impact.** The paper's most significant strength is its detailed re-evaluation of the CoNLL-2014 shared task. This analysis provides an unprecedented level of insight, revealing, for example, that certain systems excelled at specific error types (e.g., CAMB on missing tokens, CUUI on unnecessary tokens) while many systems failed entirely on others. Such findings are crucial for guiding future research, particularly in the development of hybrid systems that combine complementary strengths. The release of the annotation tool will enable standardized, detailed analysis for future GEC research.
2.  **Sound and well-motivated technical approach.** The two-stage process of edit extraction followed by rule-based classification is clear and logical. The choice to build upon a validated alignment algorithm (Felice et al., 2016) is sound. The justification for a rule-based, rather than a machine learning-based, classifier is well-argued in terms of achieving dataset independence and full interpretability. Furthermore, the methodology of reprocessing the gold-standard reference annotations with the same automatic pipeline is a crucial step to ensure a fair and consistent evaluation, a point well-supported by the results in Table 4.
3.  **Thorough and multi-faceted experimental analysis.** The authors effectively demonstrate the value of their tool through a comprehensive set of experiments. The analysis is conducted at multiple levels of granularity: from high-level operations (Missing, Unnecessary, Replacement) to 25 specific error categories, and also includes dedicated analyses of multi-token edits and the distinction between error detection and correction. This thoroughness provides a rich set of findings and robustly supports the paper's claims about the tool's utility.
4.  **Clarity of presentation.** The paper is well-structured and clearly written. The motivation is established effectively in the introduction, the methodology is described in sufficient detail, and the results are presented in logically organized tables and figures. The discussion of the results is insightful, drawing clear and actionable conclusions from the data.

### Weaknesses

1.  **Limited quantitative evaluation of the error classifier.** The performance of the rule-based error type classifier is assessed via a small-scale manual study where five raters judged the appropriateness of 200 classifications. While the reported agreement and positive ratings (>95% "Good" or "Acceptable") are encouraging, this evaluation is subjective and limited in scope. It does not provide a clear, objective measure of the classifier's accuracy (e.g., precision/recall) on its 25 categories. The argument that no gold standard exists is valid, but a small, manually annotated test set could have been created specifically for a more rigorous, quantitative evaluation of the classifier itself.
2.  **Potential impact of cascading errors is not fully explored.** The entire annotation pipeline relies on a sequence of automatic tools (aligner, POS tagger, lemmatizer, etc.). The paper acknowledges that errors from these tools, particularly the POS tagger, can lead to incorrect error type classification. However, the extent and impact of these cascading errors are not systematically analyzed. While reprocessing the reference standardizes the *evaluation*, it does not eliminate the underlying noise in the *analysis*. A more detailed discussion or sensitivity analysis regarding how upstream errors affect the final error type distributions would strengthen the paper's claims of reliability.
3.  **Novelty is primarily in application rather than core technique.** The paper effectively combines and extends existing ideas. The edit extraction component is adopted directly from prior work (Felice et al., 2016), and the concept of rule-based classification based on linguistic features was previously explored (Swanson and Yamangil, 2012). The main novelty lies in the development of a new, more comprehensive rule set and error schema, and its large-scale application to provide a new analysis of a major shared task. While this is a valuable engineering and scientific contribution, the work does not introduce fundamentally new algorithms.
4.  **Critique of the M2 scorer lacks a discussion of trade-offs.** The paper criticizes the M2 scorer for potentially inflating scores by dynamically optimizing edit boundaries to maximize true positives and minimize false positives. While this is a valid point, the paper's own approach has the opposite characteristic: its evaluation is rigidly tied to the output of a single, fixed alignment tool. If that tool produces a suboptimal segmentation, a system may be unduly penalized. A more balanced discussion acknowledging the trade-offs between M2's flexible matching and the proposed method's rigid but consistent matching would be beneficial.

### Questions to Authors

1.  The manual evaluation of the error classifier in Section 3.3 is insightful. Could you provide a qualitative breakdown of the 3.7% of edits rated as "Bad"? Were these errors concentrated in specific error categories (e.g., semantically complex ones like 'OTHER' or 'MORPH' versus syntactically clear ones like 'VERB:SVA'), or were they primarily due to specific, recurring POS tagging mistakes?
2.  The paper shows that reprocessing the gold reference does not significantly alter overall system rankings. However, could this process systematically obscure certain classes of errors? For instance, if the automatic aligner is known to perform poorly on complex, multi-token edits (e.g., paraphrases), reprocessing the reference would remove these from the gold standard, effectively making the evaluation blind to a GEC system's ability to handle them. Can you comment on which error types, if any, are most likely to be "lost" during this reprocessing step?
3.  Could you briefly situate your 25-type error schema with respect to other GEC error taxonomies used in the literature (e.g., the original, more limited schema from the CoNLL-2013/14 tasks)? A short discussion of the key design principles and differences would help clarify the rationale and advantages of the proposed framework.
