### Summary and Contributions

This paper introduces a method and a tool for automatically annotating parallel sentences from Grammatical Error Correction (GEC) corpora with explicit edit spans and error types. The method employs a two-stage process: first, it extracts edits using an existing linguistically-enhanced alignment algorithm, and second, it classifies these edits using a novel, rule-based framework that relies on automatic linguistic features like POS tags and lemmas. The primary utility of this approach is to enable fine-grained, diagnostic evaluation of GEC systems. The authors demonstrate this by conducting a comprehensive error analysis of all system outputs from the CoNLL-2014 shared task, providing detailed insights into system performance across 25 distinct error categories.

The main contributions of this work are:

1.  **A comprehensive, fine-grained performance analysis of the CoNLL-2014 shared task systems.** This is the most significant contribution, offering the first detailed, public breakdown of how different GEC architectures perform on specific error types. This analysis provides valuable diagnostic information that can guide future research by highlighting the complementary strengths of different approaches.
2.  **A rule-based, dataset-agnostic tool for automatic error type annotation.** The proposed tool is a valuable resource for the GEC community. Its primary strength lies in its rule-based design, which ensures transparency, consistency, and applicability across different datasets without the need for labeled training data, a notable advantage over previous machine learning-based methods.

### Strengths

1.  **Addresses a Clear Need in GEC Evaluation.** The work directly tackles a well-known limitation of standard GEC metrics like M², which consolidate system performance into a single score, obscuring critical details. By enabling a detailed breakdown by error type, this work provides a much-needed mechanism for diagnostic evaluation. This allows researchers to understand the specific strengths and weaknesses of their systems, fostering more targeted improvements. The release of the accompanying tool is a significant practical contribution to the field.
2.  **Thorough and Insightful Experimental Analysis.** The application of the proposed tool to the CoNLL-2014 shared task data is exemplary. The paper presents a multi-faceted analysis, breaking down performance by operation type (Missing, Unnecessary, Replacement), general error categories (Table 6), multi-token edits, and the distinction between error detection and correction. This comprehensive analysis yields valuable insights, such as identifying that hybrid MT systems excel at missing token errors while classifier-based systems are stronger on replacement and unnecessary token errors, substantiating the value of the proposed methodology.
3.  **Sound and Well-Motivated Methodology.** The technical approach is logical and clearly justified. The use of a rule-based classifier is motivated by the need for a dataset-independent and transparent evaluation standard. The authors astutely recognize the need for a consistent annotation process between the reference and hypothesis files. Their experiment showing that re-annotating the gold reference with their automatic tool does not significantly alter system rankings (Table 4) provides strong evidence that their method is valid for comparative evaluation.
4.  **Effective Validation of the Error Classifier.** In the absence of a gold standard for their new error typology, the authors conduct a manual evaluation with five GEC researchers. The results are compelling: over 96% of the automatically assigned categories were deemed "Good" or "Acceptable", and the inter-annotator agreement was high (0.724 κfree). This provides solid evidence for the quality and reliability of the rule-based classifier, which is the core novel component of the methodology.

### Weaknesses

1.  **Limited Justification for the Error Typology Design.** The paper introduces a new error scheme with 25 categories but provides minimal rationale for its design. While Section 3.2 gives a brief example regarding modal verbs, it does not discuss how this new typology relates to or improves upon previous schemes, such as the one used in the original CoNLL-2014 annotation. A more detailed discussion of the principles behind the typology's design, its coverage, and its potential limitations would strengthen the work's contribution and provide better context for the presented analysis.
2.  **Underexplored Impact of Upstream NLP Tool Errors.** The methodology is fundamentally dependent on the quality of upstream tools (spaCy, NLTK) for POS tagging, lemmatization, and parsing. The authors acknowledge that errors from these tools are the main source of incorrect classifications. However, the paper would be more robust if it included a quantitative analysis of this error propagation. For instance, an analysis of the manually evaluated "Bad" cases could reveal how frequently a specific type of upstream error (e.g., POS tag confusion) leads to a misclassification. This would give potential users a clearer understanding of the tool's reliability and failure modes.
3.  **Potential Rigidity of the Proposed Evaluation Metric.** In Section 4, the paper proposes a "simpler edit comparison approach" that requires an exact match of the edit's span and correction string. While this simplifies evaluation, it may be overly strict. The paper criticizes the M² scorer for its flexibility, but that flexibility is intended to handle cases where multiple corrections are valid (e.g., synonyms) or where annotators define boundaries slightly differently. The proposed strict matching could unfairly penalize systems that produce valid but non-identical corrections. A discussion of this trade-off between simplicity and flexibility is warranted.

### Questions to Authors

1.  Could you elaborate on the design principles behind your 25-category error typology? How does it compare to other established error schemes in GEC, and what guided your decisions to include, exclude, or merge certain categories?
2.  Your manual evaluation found that POS tagging errors were a primary cause of "Bad" classifications. Are you able to provide a more quantitative breakdown of this? For example, of the 3.7% of edits rated "Bad", what proportion can be directly attributed to an identifiable error in the upstream POS tagger or parser?
3.  The proposed evaluation metric relies on an exact match against a single, automatically generated reference annotation. This contrasts with the more flexible matching in M², which can handle multiple references and minor variations. Have you analyzed the impact of this rigidity? Could it potentially penalize plausible corrections that do not perfectly align with the single reference generated by your tool?
