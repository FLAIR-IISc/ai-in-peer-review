This article presents a new method for evaluating grammatical error correction (GEC) systems. This method makes it possible to evaluate the performance of GEC systems based on the type of error, which is important not only from a recall perspective but also from an accuracy perspective. Previously, this was not generally possible because the system's output was usually not annotated with error categories.

Strength:
- The proposed evaluation is an important step for analyzing the behavior of GEC systems.
- This article includes evaluations of various systems.
- This method has several advantages over previous work:
- It calculates accuracy based on the type of error.
- It is independent of manual error annotations.
- It can evaluate performance in multi-token errors.
- The error tags automatically selected for the previously computed error span are in most cases approved by human experts.

Weaknesses:
- An important part – the rules for determining the type of error are not described.
- There is a lack of future work on how to improve classifier evaluation and error analysis.
- The evaluation is only done for English, and it is not clear how difficult it would be to use this method in other languages.

Classifier and Classifier Evaluation
====================================
It is not clear on what basis the error types are created. Are they based on previous work?

Although the method is generally independent of alignment algorithms, the rules are not likely, but the authors have not provided any detailed information on this. Error types are a major part of this article, and a reader should at least get a glimpse of what the rule for determining the error type looks like.

Unfortunately, this article does not apply the proposed evaluation in languages other than English. There is also no detailed explanation of what changes are needed to run the classifier in other languages. I assume that the rules used to determine the edit boundary and the error tag are dependent on the language/pre-processing pipeline and therefore need to be adapted. Also, the error types may need to be changed. The authors have not provided any detailed information on the rules for determining the error types (how many/how complex are the error types overall?) so that an estimate can be made of the effort required to use this method in other languages.

The error spans calculated in the pre-processing step seem inherently continuous (which also applies to the M2 scorer), which is problematic because errors can only be correctly tagged if the error span is isolated. For example, in German, preposition-ending verbs are separated from each other in the main clause: [1st element] [verb] [other elements] [verb prefix]. Will the classifier be able to tag isolated edit spans?

The authors wrote that at least 95% of the automatically assigned error tags were automatically considered appropriate by all human judges "despite the disruption caused by automated editing extraction" (295). I would be more cautious about this judgment, as raters may be more forgiving in the case of disturbed boundaries. Also, the system is not asked to select tags without knowing the output, but in the case of disturbed boundaries, they may be more biased towards the system output. In addition, there was no rating option between "bad (not appropriate)" and "appropriate," which could have led raters to choose "appropriate" instead of "bad." To make the evaluation more reliable, the authors should further evaluate how human judges rate the classifier output if the boundaries are created manually, i.e., without the disruption caused by the flawed boundaries.

There is a lack of error analysis in classifier evaluation. It is only mentioned that "bad" is usually derived from a false POS tag. Among the questions I would like to see are: when did the raters choose "bad," when did they choose "appropriate"? Does expert rating indicate the possibility of improving the classifier?

Gold Reference vs. Auto Reference
=================================
It is not clear what data was used to accurately test significance. Did you test at an F0.5 score? If so, I don't think it's a good idea, because it is an emerging measure with weak discriminative power (functionality may be completely different in terms of recall and accuracy, but the F0.5 score may be the same). Also, at the beginning of section 4.1, the authors mention a discrepancy between automatic and reference in terms of alignment and classification, but as far as I understand, the comparison between gold and reference is only in terms of boundaries and not in terms of classification.

Error Type Evaluation
=====================
I don't think it's surprising that the 5 groups (~line 473) failed to correct any unnecessary token errors. There is at least a simple explanation for why they cannot handle additional words for at least two systems. The most obvious is UFC: their rule-based method works on POS tags (Ng et al., 2014) and it is not possible to determine additional words based only on POS. Rozovskaya & Roth (2016) gave an explanation of why AMU performs poorly in additional words.

The authors did not analyze the results or comment in Table 6 whether the systems were designed to handle error types. For some error types, there is a simple mapping of error types between the gold standard and the auto reference, for example for word order errors. It is still unclear whether the systems have completely failed in specific error types or whether they were not designed to correct them (for example CUUI is reported with accuracy + recall=0.0, although it does not target word order errors). In the case of CUUI (and there are probably similar cases), it also points to an error in classification that has not been analyzed or discussed.

Please also report the raw values for TP, FP, TN, FN in the appendix to Table 6. This makes it easier to compare the systems using other measures. Also, it seems that the results in Table 6 for some error types and systems are based only on a few instances. This will be clarified when reporting the raw values.

Your article states that "all groups except 2 (IITB and IPN) scored the best in at least 1 category, which suggests that different methods of GEC complement different error types." (606) It is good to mention here that this is consistent with previous research.

Multi-token error analysis is helpful for future work, but the result needs further explanation: some systems are probably inherently unable to correct such errors, but no system is trained in student data and fluent (in the sense of Sagaguchi et al., 2016) correction parallel corpus.

Other
=====
- The authors should mention that for some GEC methods, it was not previously impossible to provide error annotations, such as systems with submodules for each error type. Admittedly, the system must be adapted to include submodules responsible for changes in the system output. Still, the proposed method enables comparison of GEC systems for which creating an output tagged with error tags is not directly a unified way for other systems.
- Reference: Some titles do not have capitalization. The URL of Sagaguchi et al. (2016) needs to be covered. Page information is missing for Efron and Tibshirani (1993).

Author Response
===============
I agree that your method is not "seriously flawed" and I think this review actually highlights some positive aspects. The method is good, but the article is not yet ready.

A major reason for the lack of rules and descriptions for classifying errors is the lack of rules and descriptions. This is not just a matter of additional examples. If the rules are not seen as a single implementation, they need to be described in order to be replicable or adaptable.

Generalization in other languages should not be a secondary issue. If the method is designed only in one language, it will be a serious limitation. Even if you are not adapted for another language, your method should be clear enough so that others can guess how much such adaptation will work and how well it can work. Simply saying that most research targets only the problem that is targeted for ESL makes it more serious.

You wrote that some systems that deal with error types are "usually clear from the table". I don't think it's that easy - see the CUUI example mentioned above and the unnecessary token errors. There are five systems that do not correct them (table 5) and so it should be clear that they did not try to deal with them. However, the article you wrote "has no clear explanation of why these groups have difficulty with this error type".
