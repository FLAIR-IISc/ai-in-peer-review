- **Strength:** An application useful for teachers and students; supports fine-grained comparison of GEC systems.
- **Weaknesses:** The system's description is very superficial; the evaluation is not satisfactory.
- **General Discussion:**

The paper presents a method for automatically enriching the output of GEC systems with error types. This is a very useful application because both teachers and students can benefit from this information (and many GEC systems only output a modified version, without explicitly specifying the error type). It allows for a finer-grained comparison of GEC systems, in terms of general accuracy and for recall and accuracy in terms of error type-specific statistics. Unfortunately, the system's description remains very superficial. Some rules have been created in the core of the system (by hand?). But the paper does not provide any details about these rules. The authors should, for example, show examples of such rules, mention the number of rules, how complex they are, how they are organized (can any initial rule prevent the application of a subsequent rule?), etc. Instead of presenting a relevant description of the system, several pages of the paper are dedicated to the evaluation of the participating systems in CoNLL-2014. Table 6 (which takes up a whole page) lists the results of all the systems, and the text repeats a lot of information and statistics that can be read from the table. The evaluation of the proposed systems is not satisfactory in several cases. Firstly, instead of simply rating the system's output, reviewers should have independently rated 200 test sentences using a gold standard. Instead of assigning a specific number of tags, it should be possible to create a gold standard for short test sentences. It is highly possible that the method adopted in the paper creates a rating that is good enough for the reviewers (see the comparison between Marcus et al. (1993) when reviewing the agreement and when reviewing from scratch). Secondly, it is stated that "all 5 raters individually considered at least 95% of our rule-based error types to be 'good' or 'acceptable'". Considering multiple ratings individually and not averaging them is not a common practice. If "bad" scores are assigned to each different edit (we cannot know their distribution from the paper), then some edits were considered "bad" by some reviewers - this sounds much worse than the average of 3.7% calculated in the paper. Thirdly, no information is provided about the test data, for example, how many error categories there are, or which error categories are included (according to the categories rated "good" by the reviewers). Fourthly, what does "the boundary of the edits may be unusual" mean? More specific details and examples are needed here. Could this be problematic for the system's application? The authors say that their system is less domain-dependent than systems that require training data. I am not sure if this is true. For example, I think Hunspell's vocabulary probably does not cover all domains in the same detailed manner, and the hand-made rules may also be domain-dependent - and completely language-dependent, which is a clear flaw compared to machine learning methods. Moreover, the test data used here (FCE-test, CoNLL-2014) is only from one domain: student essays. It is still unclear why a new set of error categories was designed. A reason for the tags has been given: being able to easily search non-specific categories (e.g., generally "noun"). I think the tagset presented in Nichols (2003) also supports such a search. Or what is the reason for not using the CoNLL-2014 tagset? Then the CoNLL gold standard can be used to evaluate. In short, the main purpose of the paper remains somewhat unclear. Is it about a new system? But its most important details have been omitted. Is it about a new set of error categories? But no motivation or discussion is provided.
