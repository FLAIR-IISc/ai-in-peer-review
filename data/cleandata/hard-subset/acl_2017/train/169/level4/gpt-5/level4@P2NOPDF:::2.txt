The paper introduces a new method for evaluating grammatical error correction (GEC) systems that supports per–error-type precision as well as recall, even when system outputs lack error labels. This fills a gap in existing evaluations that generally cannot compute precision by error category.

Strengths
- The proposed evaluation is a valuable step toward analyzing GEC systems by error type.
- The paper evaluates a broad set of systems.
- Advantages over prior work include:
  - computing precision by error type,
  - avoiding reliance on manual error annotation,
  - handling multi-token errors.
- Human experts largely accept the automatically assigned error tags for precomputed spans.

Weaknesses
- The core rules for deriving error types are not described.
- The classifier evaluation lacks a detailed error analysis and concrete directions for improving the classifier.
- The evaluation is limited to English; portability to other languages is unclear.

Classifier and classifier evaluation
- The basis for the error categories is unclear. Are they derived from prior taxonomies?
- While the general approach is alignment-agnostic, the rules likely are not; no details are given. Given that the category design and assignment rules are central, readers should see at least a few representative rules.
- The paper does not evaluate beyond English nor discuss what would be required to port the system to other languages. Rules for edit boundaries and category assignment likely depend on language and preprocessing, and categories themselves may need to change. The paper should report how many rules exist overall and per error type, and their complexity, to help estimate the effort for adaptation.
- Preprocessing appears to yield only continuous error spans (as in M2). Some errors are inherently discontinuous. For instance, in German, separable verb prefixes lead to discontinuous spans. Can the classifier handle discontinuous edits?
- The authors report that all judges rated at least 95% of automatic tags as appropriate despite noisy automatic spans. This may overstate reliability: raters may be more lenient when spans are noisy, and seeing the system output can bias tag choices. The binary scale (Appropriate vs. Bad) also pressures raters toward “Appropriate.” A stronger evaluation would include judgments on edits with gold, manually defined boundaries.
- The classifier evaluation lacks an in-depth error analysis. Beyond noting that “Bad” often stems from POS tagging errors, the paper should report when raters chose “Bad” versus “Appropriate,” and whether these judgments suggest concrete improvements to the classifier.

Gold reference vs. auto reference
- It is unclear which metric was used for significance testing. If tests were run on F0.5, this is problematic: F0.5 is a composite with weak discriminative power and can mask divergent precision/recall. Also, Section 4.1 mentions mismatches in alignment and classification, but the gold-vs-auto comparison appears to cover boundaries only, not category assignment. Please clarify.

Error type evaluation
- It is not surprising that five teams failed to correct unnecessary-token errors. Some systems are inherently unsuited for this, e.g., UFC’s POS-rule-based system (Ng et al., 2014) cannot robustly detect superfluous words from POS alone. Rozovskaya and Roth (2016) similarly explain AMU’s difficulty with such errors.
- The paper does not analyze whether systems were designed to address each error type. For some types (e.g., word order), there is a straightforward mapping between gold and auto categories. It should be clarified whether systems truly failed on certain types or simply did not target them. For instance, CUUI shows precision+recall=0.0 on word order despite not addressing that error class, which also suggests possible misclassification that is neither analyzed nor discussed.
- Please provide raw TP, FP, TN, and FN per error type (e.g., in an appendix) for Table 6. This would enable comparisons under alternative metrics and reveal when results are based on very few instances.
- You note: “All but two teams (IITB and IPN) achieved the best score in at least one category, suggesting that different approaches complement different error types.” It would help to state that this aligns with prior findings.
- The multi-token error analysis is useful but needs more interpretation. Some systems may be inherently unable to correct multi-token errors. Moreover, none of the systems appear to be trained on parallel learner–fluent data in the sense of Sakaguchi et al. (2016), which may limit performance.

Other comments
- For some GEC approaches, producing error-tagged outputs was already possible (e.g., systems with dedicated modules per error type), though often with additional engineering to attribute changes to modules. Your method’s main advantage is enabling unified comparison across systems that do not natively output error tags.
- References: some titles lack proper capitalization; the Sakaguchi et al. (2016) URL should be wrapped; Efron and Tibshirani (1993) lacks page numbers.

Author response
- I agree the approach is not fatally flawed and has several merits, but the paper is not yet ready. The classification rules are foundational; without describing them, the work is hard to reproduce, scrutinize, or adapt. This requires more than a few examples.
- Generalization beyond English should not be an afterthought. Even without running multilingual experiments, the paper should be transparent enough for others to estimate the effort and feasibility of adapting the method. Arguing that most research targets ESL only reinforces a limitation rather than addressing it.
- It is not “usually obvious from the tables” which error types each system targets. The CUUI example and unnecessary-token errors show the risk of misinterpretation: five systems do not correct unnecessary tokens (Table 5), making it evident they did not attempt that type, yet the paper states, “There is also no obvious explanation as to why these teams had difficulty with this error type.”
