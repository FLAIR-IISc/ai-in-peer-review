Review- The paper introduces a new method for assessing grammatical error correction (GEC) systems. The approach enables per–error-type evaluation not only of recall but also of precision—previously generally infeasible because system outputs typically lack error-category annotations.
Strengths:
 - The proposed evaluation is an important step toward analyzing how GEC systems behave.
 - The paper evaluates a diverse set of systems.
 - The method offers multiple advantages over prior work:
   - it estimates precision by error type
   - it does not rely on manual error annotation
   - it can evaluate performance on multi-token errors
 - Human experts largely endorsed the automatically assigned error labels for pre-computed error spans
Weaknesses:
 - A critical component—the rules used to derive error types—is not described.
 - The classifier evaluation lacks a substantive error analysis and, consequently, concrete directions for improving the classifier.
 - The evaluation is limited to English, and the effort required to port the approach to other languages is unclear.
Classifier and Classifier Evaluation
====================================
The basis for the error taxonomy is not specified. Is it derived from existing literature?
While the overall approach is nominally independent of the alignment algorithm, the rules likely are not; however, the paper provides no details. Given that the error categories are central to the contribution, readers should at least see examples of rules that assign error types.
The paper does not apply the proposed evaluation beyond English, nor does it discuss what modifications are needed to run the classifier in other languages. It seems likely that both the rules for determining edit boundaries and those for assigning error tags depend on the language and/or the preprocessing pipeline and would therefore require adaptation. The set of error categories might also need revision. The authors do not specify details about the rules for assigning categories (e.g., how many rules overall/per error type, their complexity), making it hard to estimate the effort to use the approach in another language.
The pre-processing step appears to produce inherently continuous error spans (as with the M2 scorer), which is problematic for errors that require discontinuous spans to tag correctly. In German, for instance, verbs with separable prefixes are split in main clauses: [first constituent] [verb] [other constituents] [verb prefix]. Can the classifier handle discontinuous edit spans?
The authors state that all human judges rated at least 95% of the automatically assigned error tags as appropriate “despite the degree of noise introduced by automatic edit extraction” (295). I would be more cautious here: raters might have been more lenient when boundaries were noisy. Moreover, they did not select tags without seeing the system output and thus could have been biased toward it when boundaries were imperfect. Additionally, the lack of an intermediate option between “Bad (Not Appropriate)” and “Appropriate” may have nudged raters toward “Appropriate.” To strengthen the evaluation, please also assess how judges rate the classifier when boundaries are created manually, i.e., without noise from automatic boundary errors.
The classifier evaluation lacks a detailed error analysis. The paper only notes that “Bad” ratings are typically due to incorrect POS tags. Questions that should be addressed include: Under what circumstances did raters choose “Bad” versus “Appropriate”? Do the expert ratings suggest concrete ways to improve the classifier?
Gold Reference vs. Auto Reference
=================================
It is not clear which data the significance test used. Were tests conducted on F0.5 scores? If so, this is problematic because F0.5 is a derived metric with limited discriminative power (recall and precision can differ substantially yet yield identical F0.5). Also, at the start of Section 4.1 you discuss mismatches between automatic and reference in both alignment and classification, but as far as I can tell the gold vs. auto comparison addresses only boundary differences, not classification differences.
Error Type Evaluation
=====================
It is not surprising that five teams (~line 473) failed to correct any unnecessary-token errors. For at least two systems there are straightforward reasons they cannot handle superfluous words. Most notably, UFC’s rule-based approach operates on POS tags (Ng et al., 2014), and POS alone is insufficient to detect superfluous tokens. Rozovskaya & Roth (2016) discuss why AMU performs poorly on unnecessary words.
The paper does not analyze or comment on Table 6 with respect to whether systems were designed to handle specific error types. For some categories there is a direct mapping between the gold standard and the auto reference (e.g., word order errors). It remains unclear whether systems truly failed on certain types or simply were not designed to address them. For instance, CUUI is reported with precision+recall=0.0 for word order, although it does not target that error type. In CUUI’s case (and likely others), this also hints at a classification issue that is neither analyzed nor discussed.
Please include raw TP, FP, TN, and FN counts in an appendix for Table 6. This would facilitate comparisons using alternative metrics. It also appears that some system–error-type cells in Table 6 are based on very few instances, which would become evident with raw counts.
You state, “All but 2 teams (IITB and IPN) achieved the best score in at least 1 category, which suggests that different approaches to GEC complement different error types.” (606) It would be helpful to note that this observation aligns with prior work.
The multi-token error analysis is useful for guiding future research, but the findings need more interpretation. Some systems are likely inherently unable to correct such errors, and none of the evaluated systems were trained on parallel corpora of learner data paired with fluent corrections (in the sense of Sakaguchi et al., 2016).
Other
=====
- The paper should acknowledge that for some GEC approaches, providing error annotations was already feasible—for instance, systems with dedicated submodules per error type. Although such systems would need instrumentation to track which submodule produced a given change, your approach makes it possible to compare systems that cannot easily produce error-tagged output with others in a unified manner.
- References: Some titles are missing proper capitalization; the URL for Sakaguchi et al. (2016) should be wrapped; page numbers are missing for Efron and Tibshirani (1993).
Author response
===============
I agree that your approach is not “fatally flawed,” and this review highlights several positive aspects. The approach is solid, but the paper is not yet ready.
The core of the work lies in the rules that classify errors, and the lack of their description is a major shortcoming. This is not merely a call for more examples: if the rules are meant to be more than a one-off implementation, they must be described so others can replicate or adapt them.
Generalization to other languages should not be an afterthought. It would be a serious limitation if the approach is, by design, English-only. Even without presenting a cross-lingual adaptation, the paper should be transparent enough for others to estimate the required effort and the likely effectiveness of such an extension. Simply noting that most research targets ESL risks entrenching the issue.
You suggest that the error types each system targets would be “usually obvious from the tables.” I do not think it is that straightforward—see the CUUI example and the unnecessary-token errors. Five systems do not correct these (Table 5), which strongly indicates they did not attempt them. Yet the paper states, “There is also no obvious explanation as to why these teams had difficulty with this error type.”
