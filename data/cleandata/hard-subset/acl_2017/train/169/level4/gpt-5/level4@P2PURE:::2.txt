The paper proposes a novel method for evaluating grammatical error correction (GEC) systems. It enables per–error-type assessment of both recall and precision—previously not generally possible because system outputs typically lack error-category annotations.

Strengths:
- The proposed evaluation is an important step toward a deeper analysis of GEC system behavior.
- The paper evaluates a diverse set of systems.
- Compared with prior work, the approach:
  - computes precision by error type;
  - does not depend on manual error annotation;
  - can assess performance on multi-token errors.
- Human experts largely approve the automatically selected error tags for pre-computed error spans.

Weaknesses:
- A central component—the rules that map edits to error types—is not described.
- The classifier evaluation lacks a thorough error analysis and, consequently, concrete directions for improving the classifier.
- The evaluation is limited to English, and it is unclear how difficult it would be to apply the approach to other languages.

Classifier and Classifier Evaluation
It is unclear how the error categories were devised. Are they based on prior work?
While the overall approach is nominally independent of the alignment algorithm, the rules likely are not; yet the paper provides no details. Because the error categories are a major contribution, the reader should at least see what a rule assigning an error type looks like.
The paper does not apply the proposed evaluation to languages other than English, nor does it describe what changes would be needed to run the classifier elsewhere. It seems likely that the rules for determining edit boundaries and error tags depend on both the language and the preprocessing pipeline and would therefore require adaptation. The set of error categories might also need revision. The paper does not provide details on the rules for assigning categories (e.g., how many rules exist overall/per error type, how complex they are), making it hard to estimate the effort required to port the approach to another language.
The error spans computed in preprocessing appear to be inherently continuous (as with the M2 scorer), which is problematic because some errors can only be tagged accurately with discontinuous spans. In German, for instance, separable-prefix verbs are split in main clauses: [1st constituent] [verb] [other constituents] [verb prefix]. Would the classifier be able to tag discontinuous edit spans?
The authors state that all human judges rated at least 95% of the automatically assigned error tags as appropriate “despite the degree of noise introduced by automatic edit extraction” (295). I would be more cautious about this conclusion: raters may have been more forgiving when boundaries were noisy; they were not asked to select a tag without seeing the system output and thus may have been biased toward the system’s choice under noisy boundaries; and the absence of a rating option between “Bad (Not Appropriate)” and “Appropriate” could have pushed judgments toward “Appropriate.” To strengthen the evaluation, the authors should also assess how human judges rate the classifier’s output when boundaries are manually created, i.e., without boundary noise.
The classifier evaluation lacks a detailed error analysis. The paper notes only that “Bad” is usually due to an incorrect POS tag. Questions that should be addressed include: Under what conditions did raters choose “Bad” vs. “Appropriate”? Do expert ratings suggest concrete avenues for improving the classifier?

Gold Reference vs. Auto Reference
It is unclear which data the significance test used. Was the test performed on F0.5 scores? If so, this seems inappropriate, since F0.5 is a derived measure with weak discriminative power (recall and precision can differ substantially yet yield the same F0.5). Furthermore, Section 4.1 mentions mismatch between automatic and reference in terms of alignment and classification, but as far as I can tell, the gold vs. reference comparison addresses only boundaries, not classification.

Error Type Evaluation
I do not find it surprising that five teams (~line 473) failed to correct any unnecessary token errors. For at least two systems there are straightforward reasons they cannot handle superfluous words. Most obviously, UFC’s rule-based approach operates on POS tags (Ng et al., 2014), and superfluous words cannot be reliably identified based on POS alone. Rozovskaya & Roth (2016) explain why AMU performs poorly on superfluous words.
The paper does not analyze or discuss the results in Table 6 with respect to whether systems were designed to handle each error type. For some types, there is a straightforward mapping between the gold-standard error type and the auto reference (e.g., word order). It remains unclear whether systems failed outright on particular types or simply were not designed to correct them. For example, CUUI is reported with precision+recall=0.0 even though it does not target word-order errors; this likely indicates a classification error that is neither analyzed nor discussed.
Please also report raw TP, FP, TN, and FN in the appendix for Table 6. This would facilitate comparison using other measures and would make it clear when results for certain error types and systems are based on very small counts.
You write, “All but 2 teams (IITB and IPN) achieved the best score in at least 1 category, which suggests that different approaches to GEC complement different error types.” (606) It would be helpful to note that this aligns with prior research.
The multi-token error analysis is useful for future work, but the results need more interpretation: some systems are likely inherently unable to correct such errors, and none of the systems were trained on a parallel corpus of learner data and fluent (in the sense of Sakaguchi et al., 2016) corrections.

Other
- The paper should note that for some GEC approaches, producing error annotations was not previously impossible—for example, systems with separate submodules for individual error types. Admittedly, such systems would need to be adapted to log which submodule triggered a change. Still, the proposed approach enables fair, unified comparison across systems for which error-tagged output is not straightforward to produce.
- References: Some titles lack capitalization; the URL for Sakaguchi et al. (2016) needs wrapping; page information is missing for Efron and Tibshirani (1993).

Author response
I agree that the approach is not “fatally flawed,” and this review highlights several positive aspects. The approach is sound, but the paper is not yet ready.
The rules for classifying errors are foundational to the paper, and the lack of description is a major shortcoming. This is not merely a request for a few extra examples: if the rules are not a one-off implementation, they must be documented to allow replication and adaptation.
Generalization beyond English should not be an afterthought. It would be a serious limitation if the approach worked only for one language by design. Even without conducting a cross-lingual adaptation, the paper should be transparent enough for readers to estimate the required effort and the likely effectiveness of such an adaptation. Simply noting that most research targets ESL reinforces the problem.
You write that the error types certain systems tackle would be “usually obvious from the tables.” I do not think it is that straightforward—see the CUUI example above and the case of unnecessary token errors. Five systems fail to correct them (Table 5), which should make it clear they did not attempt to address them. Yet the paper states, “There is also no obvious explanation as to why these teams had difficulty with this error type.”
