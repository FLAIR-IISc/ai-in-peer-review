The paper proposes a novel approach to evaluating grammatical error correction (GEC) systems. It enables assessment of error-type performance not only in terms of recall but also precision—previously not generally possible because system outputs typically lack error-category annotations.

Strengths:
- The proposed evaluation is an important step toward analyzing GEC system behavior.
- The paper evaluates a variety of systems.
- The approach offers several advantages over prior work:
  - it computes precision by error type,
  - it is independent of manual error annotation,
  - it can assess performance on multi-token errors.
- The automatically selected error tags for pre-computed error spans are largely endorsed by human experts.

Weaknesses:
- A key component—the rules used to derive error types—is not described.
- The classifier evaluation lacks a thorough error analysis and, as a result, provides no concrete directions for improving the classifier.
- The evaluation is conducted only for English, and it is unclear how difficult it would be to apply the approach to other languages.

Classifier and classifier evaluation
It is unclear how the error categories were devised. Are they based on previous work?

Although the approach is in principle independent of the alignment algorithm, the rules used for assigning error categories likely are not. The paper does not provide details. Because error categories are a major part of the contribution, readers should at least see examples of rules for assigning error types.

The paper does not apply the proposed evaluation to languages other than English, nor does it explain what changes would be needed to run the classifier on other languages. It seems likely that the rules for determining edit boundaries and assigning error tags depend on the language and the pre-processing pipeline, and would therefore require adaptation. The error categories themselves might also need revision. The authors provide no details on the rules for assigning error categories (e.g., how many rules there are overall or per error type, how complex they are), making it difficult to estimate the effort required to apply the approach to another language.

The error spans computed in the pre-processing step seem to be inherently continuous (as with the M2 scorer), which is problematic for errors that can only be tagged accurately with discontinuous spans. In German, for example, verbs with separable prefixes are split in main clauses: [1st constituent] [verb] [other constituents] [verb prefix]. Would the classifier be able to tag discontinuous edit spans?

The paper states that all human judges rated at least 95% of the automatically assigned error tags as appropriate “despite the degree of noise introduced by automatic edit extraction” (p. 295). I would be cautious with this conclusion. Raters might have been more forgiving when boundaries were noisy. Moreover, raters were not asked to select a tag without seeing the system output; when boundaries were noisy, this could bias them toward the system output. Additionally, there was no rating option between “Bad (Not Appropriate)” and “Appropriate,” which might have pushed raters toward “Appropriate.” To strengthen the evaluation, the authors should also report how human judges rate the classifier output when boundaries are manually created (i.e., without noise from faulty boundaries).

The classifier evaluation lacks a thorough error analysis. The paper only mentions that “Bad” ratings are usually traced to incorrect POS tags. Questions that should be addressed include: Under what conditions did raters select “Bad” versus “Appropriate”? Do expert ratings point to concrete avenues for improving the classifier?

Gold reference vs. auto reference
It is unclear exactly what data the significance tests were performed on. Were tests conducted on F0.5 scores? If so, this is problematic because F0.5 is a derived measure with weak discriminative power—systems can have very different precision and recall yet the same F0.5. In addition, at the beginning of Section 4.1, the authors refer to mismatch between automatic and reference in terms of alignment and classification, but as far as I can tell, the comparison between gold and reference is only with respect to boundaries, not classification.

Error type evaluation
I do not find it surprising that five teams (~line 473) failed to correct any unnecessary token errors. For at least two systems there are straightforward explanations for their inability to handle superfluous words. The most obvious is UFC: its rule-based approach operates on POS tags (Ng et al., 2014), and determining superfluous words based on POS alone is not feasible. Rozovskaya and Roth (2016) also explain why AMU performs poorly on superfluous words.

The paper does not analyze or comment on the results in Table 6 with respect to whether systems were designed to handle the given error types. For some error types there is a straightforward mapping between the gold-standard error type and the auto reference (e.g., word order errors). It remains unclear whether systems failed completely on specific error types or were simply not designed to correct them. For example, CUUI is reported with precision + recall = 0.0, although it does not target word order errors. In the CUUI case (and likely others), this also points to potential errors in classification, which are neither analyzed nor discussed.

Please report the raw counts of TP, FP, TN, and FN in the appendix for Table 6. This will facilitate comparison across systems using other measures. It also appears that for some error types and systems, the results in Table 6 are based on very few instances; reporting raw values would make this explicit.

You write, “All but 2 teams (IITB and IPN) achieved the best score in at least 1 category, which suggests that different approaches to GEC complement different error types” (p. 606). It would be helpful to note that this aligns with previous research.

The multi-token error analysis is useful for future work, but the results need more interpretation. Some systems are likely inherently unable to correct such errors, and none of the systems were trained on a parallel corpus of learner data paired with fluent (in the sense of Sakaguchi et al., 2016) corrections.

Other
- The paper should acknowledge that for some GEC approaches, producing error annotations was not previously impossible—for example, systems with submodules dedicated to particular error types. Admittedly, systems would need to be adapted to indicate which submodule produced a given change. Still, the proposed approach enables a unified comparison across GEC systems for which producing error-tagged output is not straightforward.
- References: Some titles lack proper capitalization; the URL for Sakaguchi et al. (2016) should be wrapped; page information is missing for Efron and Tibshirani (1993).

Author response
I agree that the approach is not “fatally flawed,” and this review highlights several positive aspects. The approach is good, but the paper is not yet ready.

The foundation of the paper is the set of rules for classifying errors; the lack of description is a major issue. This is not merely a call for a few additional examples. If the rules are to be more than a one-off implementation, they must be described so that others can replicate or adapt them.

Generalization to other languages should not be an afterthought. It would be a serious limitation if the approach worked only for one language by design. Even without performing an adaptation to other languages, the paper should be transparent enough for others to estimate the effort required and the likely effectiveness of such adaptations. Simply noting that most research targets ESL reinforces the problem rather than addressing it.

You write that the error types targeted by certain systems would be “usually obvious from the tables.” I do not think it is that simple—see the CUUI example above, as well as the unnecessary token errors. There are five systems that do not correct them (Table 5), which should make it obvious that those systems did not attempt to handle them. Yet the paper states that “There is also no obvious explanation as to why these teams had difficulty with this error type,” which is inconsistent with what is known about these systems’ designs.
