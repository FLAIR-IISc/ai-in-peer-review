- Introduces a method to evaluate GEC systems with per-error-type precision and recall even without labeled system outputs, filling a gap in existing evaluations.
- Valuable step toward analyzing GEC systems by error type and evaluated on a broad set of systems.
- Advantages: computes precision by error type, avoids manual error annotation, and handles multi-token errors.
- Human experts largely accepted automatically assigned error tags for precomputed spans.
- Core rules for deriving error types are not described, hindering reproducibility and scrutiny.
- Classifier evaluation lacks detailed error analysis and concrete guidance for improving the classifier.
- Portability beyond English is unclear; paper does not discuss required adaptations for other languages.
- Unclear whether error categories are derived from prior taxonomies or newly designed.
- Though alignment-agnostic in concept, the actual rules likely depend on alignment/preprocessing and are not provided.
- Paper should report the number and complexity of rules overall and per error type to estimate adaptation effort.
- Preprocessing appears to produce only continuous spans; it's unclear if the classifier can handle inherently discontinuous edits.
- The judged reliability (95% appropriate) may be overstated due to noisy spans, bias from seeing system outputs, and a forced-binary rating scale.
- Classifier evaluation should report when raters chose “Bad” vs. “Appropriate” and whether those judgments indicate actionable improvements.
- Significance testing metric is unclear; using F0.5 is problematic because it can mask diverging precision and recall.
- Gold-vs-auto comparison seems to cover boundaries only and may not clarify category assignment mismatches.
- Some systems’ failures (e.g., unnecessary-token errors) are unsurprising given design limitations; paper lacks analysis of which systems targeted which error types.
- Instances of zero precision/recall (e.g., CUUI on word order) may reflect systems not addressing those types or misclassification, but this isn’t analyzed.
- Provide raw TP/FP/TN/FN per error type (e.g., appendix) to enable alternative metrics and reveal sparse-instance issues.
- The finding that different teams excel in different categories aligns with prior work and should be stated explicitly.
- Multi-token error analysis is useful but needs deeper interpretation, including system training-data limitations.
- Some systems already produce error-tagged outputs; the proposed method’s main advantage is enabling unified comparison across systems without native tags.
- Several reference formatting issues and missing bibliographic details need fixing.
- Author response concedes merits but stresses the paper isn’t ready: classification rules are foundational and require full description.
- Generalization beyond English must be addressed transparently so others can estimate adaptation effort, not dismissed as ESL-focused.
