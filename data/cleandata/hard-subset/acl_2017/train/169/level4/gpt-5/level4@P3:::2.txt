Summary
This paper proposes an automatic method to annotate unlabelled grammatical error correction (GEC) output with edit spans and error types. The approach first extracts edits using a linguistically enhanced alignment (Damerau–Levenshtein with POS/lemma costs), then assigns error types via a deterministic, dataset-agnostic rule-based classifier built on automatic linguistic markup. The authors validate the classifier with a small human study (>95% Good/Acceptable) and apply the tool to the CoNLL-2014 shared task outputs to perform, for the first time, a detailed precision/recall error-type analysis (including multi-token edits and detection vs correction). They also compare evaluation using gold vs automatically reprocessed references and find no significant difference.

Contributions (as I see them)
1) A practical, dataset-independent pipeline that enables per-error-type precision and recall for GEC, including multi-token edits and a simple, transparent edit-comparison scorer. This fills an important gap in GEC evaluation.
2) A comprehensive error-type analysis of CoNLL-2014 system outputs, surfacing complementary strengths across approaches (MT, classifiers, rules) and highlighting under-addressed categories.
3) A deterministic rule-based error-type classifier, validated by experts, that avoids domain-specific training and improves interpretability.

Strengths
- Enabling per-type precision: Prior work largely measured only recall by type. The proposed annotation of system hypotheses makes per-type precision straightforward, allowing finer-grained diagnosis and fairer cross-system comparisons.
- Dataset-agnostic, interpretable classification: The rule-based classifier relies only on automatic tags, lemmas, and parses, requiring no in-domain training data. Determinism and traceability of rules improve transparency compared to black-box classifiers; human raters judged >95% of labels Good/Acceptable despite noisy spans.
- Broad empirical analysis: Applying the framework to all CoNLL-2014 systems reveals actionable findings (e.g., strengths of CUUI for replacement and orthography, CAMB for missing tokens, weaknesses on unnecessary tokens, challenges for multi-token edits). This provides a valuable resource for the community.
- Multi-token and detection/correction evaluation: Fixing edit spans enables analysis of multi-token edits—crucial for fluency-oriented correction—and separates detection from correction, illuminating where systems fail (e.g., some detect many errors but correct few).

Weaknesses
- Insufficient description of the rule set: The core contribution—the rules that map markup to error types—is under-specified. Readers cannot assess coverage, complexity, or replicability (e.g., representative patterns, priority/ordering, features used, conflict resolution). This limits scientific transparency and inhibits adaptation.
- Limited classifier evaluation and error analysis: The human study is small and conflates boundary noise with typing quality. There is no confusion analysis (per-type confusions, POS/parse errors, dictionary issues), no breakdown by operation (M/R/U) or edit length, and no analysis of failure modes beyond POS tagging errors. Stronger validation on manual spans would isolate classifier accuracy.
- Language portability unclear: The approach depends on POS tags, lemmas, dependency parses, and a spelling dictionary. The paper does not estimate the effort or feasibility of porting to other languages (e.g., agglutinative morphology, separable verbs, non-Latin scripts), nor whether categories or rules require revision. Discontinuous error spans (e.g., German separable verb prefixes) are not discussed.
- Evaluation methodology details: The bootstrap significance test uses F0.5 (a derived metric) without reporting per-type or per-sentence statistics; it’s unclear whether classification mismatches were also reprocessed in the gold vs auto comparison. Reporting raw counts (TP/FP/FN) per type would greatly aid interpretation and alternative metrics.
- Interpretation gaps: Some conclusions overlook system design constraints (e.g., systems not targeting unnecessary tokens or word order). Lack of cross-referencing to system capabilities can misattribute poor performance to algorithmic difficulty rather than design choices or scope.

Questions to Authors
- Rule inventory and design: Please provide a compact but informative description of the rule set: the hierarchy of rules, the features used (UD POS vs PTB tags, lemmas, dependency relations, Hunspell), representative rules for key categories (e.g., VERB:FORM vs VERB:TENSE; NOUN:NUM vs NOUN:INFL), tie-breaking, and how Missing/Unnecessary/Replacement are decided. How sensitive are rules to the specific tagger/parser?
- Portability: What modifications are required to port the classifier to another language? Which components are language-specific (tagset mapping, morphology, dictionary)? How would you handle discontinuous edits (e.g., German separable verbs) or languages with rich morphology/compounding? What is your estimate of engineering effort?
- Classifier validation: Can you rerun the human study on manually segmented edits to decouple boundary errors from typing accuracy? Please provide a confusion matrix by error type and an analysis of common mislabels, including the proportion caused by tagging/parsing vs rule logic.
- Gold vs auto references: What exactly was bootstrapped (document-level F0.5, per-sentence scores), and were error types reclassified in both conditions? Please report raw TP/FP/FN per team and type for both references.
- Detection vs correction by type: Can you provide per-type detection metrics and compare them to correction metrics to diagnose where systems fail to propose good fixes?

Additional comments
- Some systems could, in principle, output their own error tags; your method’s key advantage is unifying evaluation across systems that cannot easily do so. Please clarify this positioning.
- For Table 6 and other detailed tables, include raw counts (TP/FP/FN) in the appendix to support alternative analyses and reveal small-sample effects.
- When discussing surprising failures (e.g., unnecessary tokens), cross-reference system design (e.g., whether that category was targeted) and prior analyses (e.g., Rozovskaya & Roth, 2016).
- Minor editorial issues: fix capitalization in references; add missing page ranges (e.g., Efron & Tibshirani, 1993); wrap long URLs (e.g., Sakaguchi et al., 2016).

Overall assessment
This is a valuable and much-needed step toward richer GEC evaluation, with an immediately useful tool and compelling analysis. However, the paper needs substantially more detail about the rule-based classifier, stronger evaluation and error analysis, and a clearer discussion of portability beyond English. With these revisions, the contribution would be significantly strengthened.
