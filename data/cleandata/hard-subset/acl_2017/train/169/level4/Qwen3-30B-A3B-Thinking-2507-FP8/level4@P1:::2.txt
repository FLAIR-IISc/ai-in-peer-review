

Summary: This paper introduces a method for automatically annotating grammatical error correction (GEC) system outputs with error type information, enabling detailed error type analysis previously impossible due to unannotated system outputs. The approach combines a linguistically-enhanced alignment algorithm for edit extraction with a dataset-independent rule-based classifier for error typing. The authors validate their classifier through human evaluation (95%+ "Good" or "Acceptable" ratings) and apply it to analyze all systems from the CoNLL-2014 shared task.

Main Contributions:
1. A novel method for automatically annotating GEC system outputs with error type information, enabling detailed error type analysis beyond simple recall metrics.
2. A rule-based error typing framework that is dataset-independent and relies solely on automatically obtained linguistic features (POS tags, lemmas).
3. The first comprehensive error type analysis of all systems in the CoNLL-2014 shared task, revealing distinct system strengths across error categories.

Strengths:
1. The paper addresses a significant gap in GEC evaluation by enabling precision-based error type analysis, which was previously impossible due to the lack of error annotations in system outputs.
2. The rule-based approach offers clear advantages over previous machine learning methods: it's dataset-independent, transparent (unlike black-box classifiers), and requires no labeled training data.
3. The human evaluation provides strong validation, with all five raters rating over 95% of predicted error types as "Good" or "Acceptable" despite noise from automatic edit extraction.

Weaknesses:
1. The paper lacks sufficient detail about the error classification rules. It's unclear how the error categories were derived, what the rules look like, or how many rules exist. This severely limits replicability and adaptation to other systems or languages.
2. The classifier evaluation is superficial. While the authors note "Bad" ratings usually stem from POS tagger errors, they don't analyze when and why errors occurred or suggest concrete improvements for the classifier.
3. The evaluation is limited to English without discussion of language generalization. The rules for error classification likely depend on language-specific features, yet the paper doesn't address what adaptations would be needed for other languages.
4. The error span extraction appears inherently continuous, which would fail for languages with discontinuous errors (e.g., German separable verbs). The paper doesn't address this limitation.
5. The human evaluation protocol is potentially biased. Raters weren't asked to evaluate error types without knowing system outputs, which could have led to inflated ratings, and the absence of a middle rating option between "Bad" and "Appropriate" likely contributed to overly positive assessments.

Questions to Authors:
1. Could you provide concrete examples of your error classification rules to demonstrate their structure and complexity?
2. How would your approach handle discontinuous error spans (e.g., German separable verbs), and what adaptations would be necessary for such cases?
3. The paper notes five teams failed to correct unnecessary token errors, but for UFC (rule-based POS approach) and AMU (as noted by Rozovskaya & Roth, 2016), there are clear reasons for this failure. Could you clarify whether these systems were designed to handle these error types?
4. Could you provide raw TP/FP/TN/FN counts for Table 6 to facilitate comparison using alternative metrics, especially for error types with very low counts?
