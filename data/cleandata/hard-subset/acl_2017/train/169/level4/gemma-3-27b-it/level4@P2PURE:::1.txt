- Strengths: Useful application for teachers and learners; supports
fine-grained comparison of GEC systems.
- Weaknesses: System description is highly superficial; evaluation is not
satisfying.
- General Discussion:
The paper presents a method for automatically enriching GEC system output with error type information. This is a valuable application, as it would benefit both teachers and learners by explicitly identifying error types—information often absent in standard GEC outputs. Furthermore, it enables more nuanced comparisons of GEC systems, providing precision metrics overall and recall/precision figures for specific error types. 

However, the system description lacks sufficient detail. The core of the system appears to be a set of manually created rules, but the paper provides no specifics regarding these rules. The authors should, for instance, provide illustrative examples, quantify the number of rules, describe their complexity, and explain their ordering (and whether early rules might preclude the application of later ones). Instead of detailing the system, a significant portion of the paper is dedicated to presenting results from the CoNLL-2014 shared task, with Table 6 (spanning an entire page) simply listing system performance, and the accompanying text largely reiterating information readily apparent from the table.

The evaluation of the proposed system is problematic in several respects. First, annotators should have independently annotated a gold standard for the 200 test sentences, rather than simply evaluating the system’s output. Given the relatively small test set size and a fixed tagset, creating a gold standard should be feasible. The current approach likely yields inflated ratings compared to assessment against a true gold standard (see Marcus et al., 1993, for a discussion of agreement in pre-annotated vs. scratch annotation scenarios). 

Second, reporting individual rater scores and averaging them is atypical. The paper states that "all 5 raters individually considered at least 95% of our rule-based error types to be either "Good" or "Acceptable"". However, if the 18.5% of edits flagged as "bad" by at least one annotator were distributed across different edits (a distribution not provided in the paper), this represents a considerably more substantial error rate than the reported average of 3.7%. 

Third, the paper lacks information about the test data, such as the number of error categories represented and which categories are covered by the annotators' "good" ratings. Fourth, the meaning of “unusual edit boundaries” is unclear, and examples or a more precise explanation are needed. The authors should also address whether this poses a problem for system application.

The authors claim their system is less domain-dependent than those relying on training data. This claim is questionable, as resources like Hunspell's vocabulary are domain-specific, and manually created rules are inherently domain- and language-dependent – a significant limitation compared to machine learning approaches. Furthermore, the test data (FCE and CoNLL-2014) are limited to student essays, representing a single domain.

The rationale for designing a new set of error categories is also unclear. While the authors mention enabling searches for underspecified categories (e.g., "NOUN"), existing tagsets like Nicholls (2003) seem to already support such functionality. Alternatively, leveraging the CoNLL-2014 tagset would have allowed for evaluation against the existing CoNLL gold standard.

In summary, the paper’s central motivation remains ambiguous. Is it presenting a new system? Critical details are omitted. Is it introducing a novel error category set? The justification is weak. Is it evaluating CoNLL-2014 systems? The presentation is superficial.
