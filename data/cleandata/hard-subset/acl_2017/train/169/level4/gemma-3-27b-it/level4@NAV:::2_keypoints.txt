- Proposed evaluation enables per-error-type precision and recall for GEC systems, addressing prior lack of error-category annotations.
- Evaluation is a significant advancement for analyzing GEC system behavior.
- Paper presents a comprehensive comparison across many GEC systems.
- Method computes precision by error type, requires no manual error annotation, and handles multi-token errors.
- Human experts largely agreed with automatically assigned error tags for pre-computed spans.

- Rules for deriving error types are not described in the paper.
- Classifier evaluation lacks thorough error analysis and guidance for improvement.
- Evaluation is limited to English and adaptation challenges for other languages are not discussed.

- Rationale and pedigree of error category design are unspecified; unclear if categories build on prior work.
- Rules for mapping edits to error types are likely alignment-dependent but are omitted; at least one rule example is essential.
- No discussion of cross-lingual adaptation effort, complexity, or required changes to edit boundaries and tags.
- Continuous pre-processing spans (M2-style) prevent handling of discontinuous edits (e.g., German separable prefixes).
- Claim of ≥95% tag appropriateness by human judges is suspect due to possible rater leniency, bias from system outputs, and lack of intermediate rating options.
- Classifier error analysis is shallow—e.g., “Bad” ratings correlate with POS errors but circumstances and actionable insights are unexplored.

- Data used for significance testing is unclear.
- Using F0.5 for significance testing is questionable because it can mask differing precision/recall trade-offs.
- Comparison between gold and auto references focuses on boundary discrepancies and omits classification differences.

- Five teams failing to correct unnecessary tokens is unsurprising given system designs (e.g., UFC uses only POS tags; AMU known limitations).
- Paper fails to analyze Table 6 results in light of system design and whether failures reflect capability gaps or design choices.
- CUUI’s 0.0 precision/recall for word order suggests a possible classification error that is not investigated.
- Authors should provide raw TP/FP/TN/FN counts for Table 6 and flag low-instance error types/systems.
- Claim that different systems excel at different error types aligns with prior work and should be acknowledged.
- Multi-token error analysis is valuable but needs interpretation; many systems may be inherently unable to correct these due to lack of suitable training data.

- Acknowledge that some GEC systems can output error-tagged results and the proposed approach offers a unified comparison framework.
- References and formatting need fixes (capitalization, URLs, missing page info).

- Reviewer: approach promising but paper not ready due to missing rule descriptions and replication details.
- Rules must be fully described for replicability and adaptation beyond a one-off implementation.
- Generalization to other languages must be addressed or made transparent to estimate adaptation effort.
- It is not obvious from tables which systems target which error types; claims about system difficulties (e.g., unnecessary tokens) need clearer justification.
