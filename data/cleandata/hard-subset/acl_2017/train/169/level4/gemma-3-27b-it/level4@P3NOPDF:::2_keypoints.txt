- Novel evaluation enables assessing GEC performance by error type with both recall and precision without requiring manual error annotations.
- Paper provides a comprehensive evaluation across a variety of GEC systems.
- Approach offers key advantages: computes precision by error type, is independent of manual annotation, and handles multi-token errors.
- Human experts largely validated automatically selected error tags for pre-computed error spans.
- Rules used to derive error types are not described, undermining replicability and adaptation.
- Classifier evaluation lacks detailed error analysis and actionable directions for improvement.
- Evaluation is limited to English; cross-lingual extension and required adaptation effort are not discussed.
- Basis for error categories is unclear and not linked clearly to prior work or alignment algorithm choices.
- No example or description of rules for edit boundaries and error-tag assignment is provided.
- Unknown number/complexity of rules makes estimating cross-lingual adaptation effort difficult.
- Pre-processing yields continuous spans, which may fail for discontinuous edits (e.g., German separable verbs); classifier’s handling of discontinuity is unclear.
- The 95% human-approval claim may be biased by noisy boundaries, rater leniency, lack of intermediate rating options, and knowledge of system output.
- Human validation should be repeated on manually created (noise-free) boundaries to be robust.
- Classifier evaluation notes POS-tag correlation but lacks finer-grained error analysis (when raters chose “Bad” vs “Appropriate” and what that implies for fixes).
- Significance-test details are unclear; using F0.5 for significance is questionable due to its limited discriminative power.
- Comparison between gold and auto-reference appears focused on boundaries and not on classification differences.
- Lack of analysis on whether systems were designed to handle each error type (e.g., unnecessary token errors) leads to ambiguous interpretation of failures.
- Cases like CUUI (precision+recall=0.0 for word order) suggest possible classification errors or mismatches with system design that are not investigated.
- Raw contingency counts (TP/FP/TN/FN) should be provided (e.g., in an appendix) and small-sample notes included for Table 6.
- The claim about different systems complementing different error types should be tied to prior work for support.
- Multi-token error analysis is valuable but needs interpretation regarding system capabilities and training data limitations.
- Authors should acknowledge some systems could produce error annotations and discuss how the proposed framework can unify comparisons across such systems.
- Several reference formatting issues and missing bibliographic details need correction.
- Reviewer agrees the approach is promising but emphasizes the paper is not ready without transparent rules and consideration of cross-lingual generalization; system-design vs. classifier-failure distinctions must be addressed.
