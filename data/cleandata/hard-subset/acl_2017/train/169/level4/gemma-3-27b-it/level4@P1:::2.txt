The paper presents a novel approach for evaluating grammatical error correction (GEC) systems. This approach makes it possible to assess the performance of GEC systems by error type, not only in terms of recall but also in terms of precision, which was previously difficult to achieve generally since system output is usually not annotated with error categories.

**Strengths:**

- The proposed evaluation is an important step forward for analyzing GEC system behavior.
- The paper includes evaluation for a variety of systems.
- The approach has several advantages over previous work:
    - It computes precision by error type.
    - It is independent of manual error annotation.
    - It can assess performance on multi-token errors.
- The automatically selected error tags for pre-computed error spans are largely approved by human experts.

**Weaknesses:**

- A key component – the rules used to derive error types – are not described in detail.
- The classifier evaluation lacks a thorough error analysis and, as a result, lacks suggestions for future work on improving the classifier.
- The evaluation was performed only for English, and it is unclear how difficult it would be to adapt the approach to another language.

**Classifier and Classifier Evaluation**

It is unclear on what basis the error categories were devised. Are they based on previous work? Although the approach is generally independent of the alignment algorithm, the rules are likely dependent on it, but the authors do not provide enough detail. The error categories are a central part of the paper, and the reader should at least get an overview of what a rule for assigning an error type looks like.

Unfortunately, the paper does not apply the proposed evaluation to languages other than English. It also does not elaborate on what changes would be necessary to run the classifier on other languages. I assume that the rules for determining edit boundaries, as well as for determining the error tags, depend on the language and the pre-processing pipeline, and therefore need to be adapted. Additionally, the error categories themselves might need to be changed. The authors do not provide enough detail about the rules for assigning error categories to estimate the effort necessary to use the approach on another language.

The error spans computed in the pre-processing step seem inherently continuous (which is also the case with the M2 scorer), which is problematic since some errors can only be tagged accurately with discontinuous edit spans. For example, in German, verbs with separable prefixes are separated in the main clause: [1st constituent] [verb] [other constituents] [verb prefix]. Would the classifier be able to tag discontinuous edit spans?

The authors state that all human judges rated at least 95% of the automatically assigned error tags as appropriate "despite the degree of noise introduced by automatic edit extraction" (line 295). I am cautious about this judgment, as the raters might have been more forgiving given the noisy boundaries. Additionally, they were not asked to select a tag without knowing the system output and might have been biased by it. Moreover, there was no rating option between "Bad (Not Appropriate)" and "Appropriate," which might have led raters to select "Appropriate" over "Bad." To improve the evaluation, the authors should also evaluate how human judges rate the classifier output if the boundaries were manually created, i.e., without the noise introduced by faulty boundaries.

The classifier evaluation lacks a thorough error analysis. It is only mentioned that "Bad" ratings are usually traced back to an incorrect POS tag. I would like to see answers to the following questions: When did raters select "Bad," and when did they select "Acceptable"? Does the rating by experts suggest possibilities for improving the classifier?

**Gold Reference vs. Auto Reference**

It is unclear on what data the significance test was performed exactly. Did you test on the F0.5 scores? If so, I don't think this is a good idea, since it is a derived measure with weak discriminative power (performance in terms of recall and precision can be very different but still result in the same F0.5 score). Also, at the beginning of Section 4.1, the authors refer to the mismatch between automatic and reference in terms of alignment and classification, but as far as I can tell, the comparison between gold and reference is only in terms of boundaries, not classification.

**Error Type Evaluation**

I do not think it is surprising that five teams (line 473) failed to correct any unnecessary token errors. For at least two of the systems, there is a straightforward explanation: Their approaches cannot handle superfluous words. For example, UFC’s rule-based approach works on POS tags (Ng et al., 2014), and it is impossible to determine superfluous words based on POS tags alone. Rozovskaya & Roth (2016) provide an explanation for AMU’s poor performance on superfluous words.

The authors do not analyze or comment on the results in Table 6 with respect to whether the systems were designed to handle the error type. For some error types, there is a straightforward mapping between the error type in the gold standard and in the auto reference, such as for word order errors. It remains unclear whether the systems failed completely on specific error types or were simply not designed to correct them (for example, CUUI). The CUUI case (and there are likely similar cases) also points to an error in the classification, which is neither analyzed nor discussed.

Please report raw values for TP, FP, TN, and FN in the appendix for Table 6. This would make it easier to compare the systems using other measures. Also, it seems that for some error types and systems, the results in Table 6 are based only on a few instances. This should also be clarified when reporting the raw values.

You write, “All but 2 teams (IITB and IPN) achieved the best score in at least 1 category, which suggests that different approaches to GEC complement different error types” (line 606). It would be helpful to mention here that this is in line with previous research.

The multi-token error analysis is helpful for future work, but the result needs more interpretation. Some systems are likely inherently unable to correct such errors, but none of the systems were trained on a parallel corpus of learner data and fluent (in the sense of Sakaguchi et al., 2016) corrections.

**Other**

- The authors should mention that, for some GEC approaches, producing error annotations was not impossible before, e.g., systems with submodules for one error type each. Admittedly, the system would need to be adapted to include the submodule responsible for a change in the system output. Still, the proposed approach enables comparing GEC systems for which producing an error-tagged output is not straightforward with other systems in a unified way.
- References: Some titles lack capitalization. The URL for Sakaguchi et al. (2016) needs to be wrapped. Page information is missing for Efron and Tibshirani (1993).

The authors should provide more details on the rule-based classification scheme for error types. Describing the rules with examples would significantly improve the reproducibility and understandability of the work. Furthermore, exploring the generalizability of the approach to other languages should be addressed, even if a full implementation is beyond the scope of the paper. Finally, a more in-depth error analysis of the classifier's performance would be valuable for guiding future improvements.
