### Summary and Contributions

This paper introduces a method for automatically annotating the output of Grammatical Error Correction (GEC) systems with error type information. The process consists of two stages: first, it uses an existing linguistically-enhanced alignment algorithm to extract edits between original and corrected sentences. Second, it classifies these edits using a new, dataset-independent, rule-based classifier. The primary value of this work is demonstrated through its application to the CoNLL-2014 shared task output, enabling, for the first time, a detailed analysis of system performance in terms of precision, recall, and F-score for specific error types.

In my view, the main contributions of this work are:

1.  **A Detailed, Fine-Grained Analysis of CoNLL-2014 GEC Systems:** The paper provides the first comprehensive, per-error-type evaluation of the CoNLL-2014 shared task systems that includes precision and F-score metrics. This analysis reveals specific strengths and weaknesses of different approaches (e.g., SMT-based vs. classifier-based) for different error types, which is a valuable resource for the GEC community.

2.  **A Method and Tool for Automatic Error Annotation:** The authors present and plan to release a complete pipeline for annotating unannotated GEC system output. This tool addresses a significant gap in GEC evaluation, where the inability to automatically classify system corrections has historically limited error-specific analysis to recall only.

### Strengths

1.  **Addresses a Clear Need in GEC Evaluation:** The paper correctly identifies a major limitation in standard GEC evaluation. By enabling the automatic annotation of system hypotheses, this work allows for the calculation of precision and F-scores at the error-type level. This is a crucial step forward, as it provides a much more nuanced and insightful view of a system's capabilities beyond a single aggregate score.

2.  **Provides Valuable Insights into GEC System Performance:** The detailed analysis of the CoNLL-2014 systems is a significant strength. The results presented in Tables 5, 6, and 8 offer a wealth of information that can guide future GEC research. For example, the finding that certain systems excel at specific error types (e.g., CUUI's classifier for orthography, CAMB's hybrid MT for prepositions) strongly supports the case for developing hybrid or ensemble systems. The analysis of multi-token edits is also particularly insightful, highlighting a key area where current systems struggle.

3.  **Publicly Released Tool:** The commitment to release the annotation tool is a valuable contribution. This will allow other researchers to replicate the analysis, evaluate their own systems with greater detail, and potentially build upon the proposed framework.

### Weaknesses

1.  **The Core Method is Critically Underspecified:** The paper's primary methodological novelty—the rule-based error type classifier—is described in a highly superficial manner. The authors state that it uses rules based on POS tags and other linguistic features, but they provide no examples of these rules, no information on their number or complexity, and no discussion of how they are ordered or how conflicts are resolved. This lack of detail makes the method irreproducible and difficult to assess critically. A significant portion of the paper is dedicated to presenting the CoNLL-2014 evaluation results (e.g., Table 6 occupies a full page), which comes at the expense of adequately describing the core technical contribution.

2.  **The Classifier's Evaluation is Methodologically Weak:** The manual evaluation of the classifier is not fully convincing for two main reasons.
    *   First, the protocol of asking raters to judge pre-annotated output ("Good", "Acceptable", "Bad") is known to produce inflated agreement scores compared to having annotators independently assign labels from scratch (cf. Marcus et al., 1993, "Building a large annotated corpus of English: The Penn Treebank"). A more rigorous evaluation would have involved creating a small, independently annotated gold standard for the 200 edits and measuring the classifier's accuracy against it.
    *   Second, the reporting of results is potentially misleading. The paper states that "all 5 raters individually considered at least 95% of our rule-based error types to be either 'Good' or 'Acceptable'." However, this way of averaging individual raters' "Bad" scores can obscure the true level of disagreement. If each rater's 2-4.5% "Bad" ratings were for different items, it is possible that up to 18.5% of the edits were considered "Bad" by at least one expert. Reporting this latter figure would provide a much more transparent view of the classifier's quality.

3.  **Insufficient Justification for a New Error Taxonomy:** The paper introduces a new set of 25 error categories without adequately motivating why existing schemas, particularly the one used in the CoNLL-2014 shared task, were insufficient. Using the CoNLL-2014 tagset would have offered a clear advantage: the ability to directly evaluate the automatic classifier against the human-annotated gold standard reference. Instead, the authors opt to reprocess the gold reference with their automatic pipeline (Section 4.1), which sidesteps this direct and valuable comparison.

### Questions to Authors

1.  Could you please provide more detail about the rule-based classifier? For instance, could you share a few representative rules for different error types (e.g., one for `VERB:SVA` and one for `DET`), state the approximate number of rules in the system, and explain how rule ordering or potential conflicts are handled?

2.  Regarding the manual evaluation of the classifier: Could you report the percentage of edits that were marked as "Bad" by *at least one* annotator? This would provide a clearer picture of the upper bound of perceived errors in your system's output.

3.  What was the main motivation for developing a new error schema rather than using or adapting the existing CoNLL-2014 schema? Using the shared task's schema would seem to enable a direct, automatic evaluation of your classifier against the original gold standard, which would be a very powerful validation of your approach. Was this considered?
