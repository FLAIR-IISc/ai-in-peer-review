- Summary: Proposes a two-step automatic pipeline (linguistically-enhanced alignment + rule-based error-type classifier) to enable fine-grained GEC evaluation beyond recall, validated by human judges and applied to CoNLL-2014.  
- Contribution 1: A complete evaluation pipeline that computes precision, recall, and F-scores for 25 error types, yielding deeper insights than previous metrics.  
- Contribution 2: A dataset-independent, rule-based error-type classifier using standard NLP preprocessing (POS, lemmas, etc.), shown to be accurate via manual evaluation.  
- Strength 1: Enables much deeper, system-specific analysis (e.g., per-type strengths/weaknesses) demonstrated on CoNLL-2014 results.  
- Strength 2: Strong empirical validation: five GEC researchers rated ≥95% of automatic type assignments as "Good" or "Acceptable."  
- Strength 3: Broad applicability across system types (MT, classifiers, rules, hybrids) and commitment to public tool release for reproducibility.  
- Weakness 1: The paper omits the actual classification rules, treating the classifier as a black box and harming reproducibility and critical evaluation.  
- Weakness 2: Error analysis is superficial—no detailed breakdown of the 3.7% "Bad" cases or which error types are most misclassified and why.  
- Weakness 3: Entirely English-focused (specific tools and resources used) with no discussion of adapting to other languages or morphological/syntactic differences.  
- Question 1: Request detailed or representative rules and an explanation of how rules interact with edit extraction and alignment errors.  
- Question 2: Ask how the 25 error categories were developed and whether they map to existing GEC annotation schemes (e.g., CoNLL-2014).  
- Question 3: Suggest evaluating classifier performance on gold-standard edit boundaries to separate classification quality from alignment noise.  
- Question 4: Ask how the framework handles discontinuous edits (e.g., separable verbs) and whether the alignment supports such cases.  
- Question 5: Request clarification of the bootstrap significance test setup (Table 4) and how automatic type classification is validated against manual classification.
