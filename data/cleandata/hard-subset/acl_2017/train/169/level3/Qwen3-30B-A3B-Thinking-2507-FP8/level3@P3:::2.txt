

# Review of "Automatic Annotation and Evaluation of Error Types for Grammatical Error Correction"

## Summary
This paper introduces a novel method for automatically annotating grammatical error correction (GEC) system outputs with error type information. Prior to this work, error type performance for GEC systems could only be measured in terms of recall because system outputs lacked error type annotations. The authors propose a two-stage approach: first, they automatically extract edits between original and corrected sentences using linguistically-enhanced alignment; second, they classify these edits into error types using a rule-based classifier dependent only on automatically obtained properties (POS tags, lemmas, etc.). The method was validated through human evaluation (95%+ "Good" or "Acceptable" ratings) and applied to the CoNLL-2014 shared task data for the first detailed error type analysis of GEC systems.

## Main Contributions
1. A novel evaluation method enabling measurement of GEC performance by error type including precision, not just recall.
2. A method independent of manual error annotation that can handle multi-token errors.
3. Demonstration of broad applicability through evaluation across multiple GEC systems in a major shared task.

## Strengths
1. **Significant methodological contribution**: The paper solves a critical problem in GEC evaluation by enabling precision-based error type analysis. This is a major advancement over previous work that could only measure recall, allowing researchers to identify specific system weaknesses (e.g., "This system is poor at correcting noun number errors but good at verb tense errors").

2. **Robust validation approach**: The human evaluation of the classifier (with 5 raters and 200 samples) provides strong evidence that the automatically assigned error types are highly reliable (95%+ "Good" or "Acceptable" ratings), addressing a key concern about the method's validity.

3. **Practical utility demonstrated**: The application to CoNLL-2014 data reveals meaningful insights about system strengths and weaknesses across different error types, providing actionable information for future system development (e.g., "CAMB's hybrid MT approach significantly outperformed others in missing token errors, while CUUI's classifier approach excelled at replacement noun errors").

## Weaknesses
1. **Lack of reproducibility**: The paper fails to describe the specific rules used for error type classification, making replication impossible. The rules are crucial for understanding how the classifier works, yet they're not provided in sufficient detail (e.g., no examples of how specific error types like "NOUN:POSS" are distinguished from others).

2. **Incomplete evaluation of the classifier**: The human rating of automatic tags is limited to overall "Good/Good" ratings without systematic analysis of "Bad" labels. The paper states "whenever raters judged an edit as 'Bad', this could usually be traced back to a mistake made by the POS tagger," but provides no evidence of this analysis or concrete examples of when and why raters chose "Bad."

3. **Limited language scope**: The evaluation is restricted to English, and the paper does not address how the approach would generalize to other languages with different morphological structures (e.g., German with separable verb prefixes), which is a significant limitation for a method claiming to be dataset-independent.

4. **Insufficient statistical reporting**: The paper omits raw TP/FP/TN/FN counts (Table 6), making it impossible to assess the statistical significance of results. Some error type results (e.g., "CONJ" for IPN) are based on very few instances (only 1 TP), but this is not clarified.

5. **Lack of error type framework justification**: The paper does not explain how the error categories were devised or whether they follow prior work (e.g., the FCE error types), making it difficult to assess the framework's comprehensiveness.

## Questions to Authors
1. Could you provide the specific rules used for error type classification, including examples of how edge cases are handled (e.g., "could → should" being labeled as tense error rather than modal error)?

2. How would your approach handle discontinuous edit spans (e.g., separable verb prefixes in German), which the paper does not address?

3. The paper states that "the M2 scorer in fact tends to overestimate performance," but this claim isn't supported with concrete evidence. Could you provide specific examples demonstrating this overestimation?

4. Why did 5 teams fail to correct any unnecessary token errors (Table 5), and what does this reveal about GEC system design rather than evaluation limitations?

## Recommendation
While the paper presents a promising approach with significant potential to advance GEC evaluation, the lack of detail about the rule-based classifier and the English-only scope currently limit its impact. The authors should provide full rule descriptions to enable replication and address the language generalization issue. With these improvements, the paper would make a strong contribution to the field. As it stands, it is not ready for acceptance but has the potential to be a valuable contribution after addressing these critical weaknesses.
