Contributions:
- Proposes a novel evaluation method that measures GEC performance by error type, including precision, not just recall.
- Introduces a two-stage pipeline: linguistically-enhanced alignment to extract edits, then a rule-based error-type classifier.
- Presents a dataset-independent approach that handles multi-token errors and shows broad applicability across systems.

Strengths:
- Fully automatic, dataset-independent method relying only on POS tags and lemmas.
- Delivers the first detailed error-type performance evaluation for all CoNLL-2014 teams.
- Provides a comprehensive error-type analysis (e.g., detailed determiner-error breakdown between systems).

Weaknesses / limitations:
- Rules for deriving error types are not described, threatening reproducibility.
- Classifier evaluation lacks detailed error analysis and concrete improvement directions.
- Evaluation is limited to English; adaptation to other languages is not addressed.
- Significance test setup is unclear and reliance on F0.5 for discrimination may be weak.
- Gold vs. auto-reference comparison appears limited to boundaries, not full classification agreement.
- Some observed failures may stem from system design rather than evaluation errors, but this is unanalyzed.
- Raw TP/FP/TN/FN counts are not reported, preventing assessment of sample sizes and alternative metrics.
- Some reported results may be based on very few instances, but instance counts are not clarified.

Requested fixes / suggestions:
- Publish detailed rules for error-type derivation to ensure reproducibility.
- Provide a more comprehensive error analysis and concrete directions to improve the classifier.
- Investigate and report how the approach can be adapted to other languages.
- Clarify the statistical significance testing setup and consider alternative metrics to F0.5.
- Expand the comparison between gold and automatic references to include classification agreement, not just boundaries.
- Analyze how system design choices contribute to failures on particular error types.
- Report raw TP/FP/TN/FN counts per system and error type to enable further analysis.
- Clarify the number of instances used for each system/error-type result.

Questions to authors:
- What are the exact rules used to derive error types?
- How do you plan to adapt the approach to other languages?
- What is the significance test setup and rationale for using F0.5?
- How will you address the limitations of F0.5 as a discrimination metric?
- Can you provide more details on the gold vs. auto reference comparison?
- How will you analyze the relationship between system design and evaluation failure?
- Will you report raw TP/FP/TN/FN counts for each system/error type?
- How will you address results based on very few instances for some system/error types?

Additional comments:
- The linguistically-enhanced alignment algorithm needs a more detailed description.
- Include more examples of error types and their classifications.
- Discuss implications of the results for GEC system design and evaluation more thoroughly.
