

Summary: This paper addresses a significant gap in Grammatical Error Correction (GEC) evaluation by proposing a method to automatically annotate system outputs with error type information. The authors develop a linguistically-enhanced alignment algorithm to extract edits between original and corrected sentences, followed by a rule-based classifier that categorizes these edits into error types using only automatically obtained linguistic properties (POS tags, lemmas, etc.). The method was validated through human evaluation (95%+ agreement) and applied to the CoNLL-2014 shared task data to provide the first detailed error type analysis of GEC systems.

Main contributions (in order of strength):
1. A dataset-independent rule-based error typing framework that relies solely on automatically obtained linguistic properties, eliminating the need for labeled training data.
2. A detailed error type analysis of CoNLL-2014 systems revealing system-specific strengths and weaknesses across 25 error categories.
3. A practical tool for automatic error type annotation that will be released with the paper, enabling standardized error type evaluation.

Strengths:
1. The paper solves a genuine, long-standing problem in GEC evaluation - the inability to analyze error types without manual annotation. This is a significant contribution to the field.
2. The rule-based classifier's dataset independence is a major advantage over previous ML-based approaches that require domain-specific training.
3. The human evaluation showing >95% agreement with human raters provides strong validation for the method's reliability.
4. The detailed error type analysis reveals valuable insights about system performance (e.g., MT approaches excel at NOUN errors while classifiers are better for VERB:FORM), which could guide future system development.

Weaknesses:
1. The evaluation is limited to the CoNLL-2014 dataset, which is now outdated (2014), and the method's applicability to modern neural GEC systems is not demonstrated.
2. The paper doesn't address how the method would handle error types not included in their 25-category framework or how to extend it for new error types.
3. The comparison with existing error type annotation methods is insufficient; the paper doesn't show how much better this approach is than previous work.
4. The paper doesn't discuss limitations of the rule-based approach, such as handling ambiguous errors or errors spanning multiple categories.

Questions to Authors:
1. How would your method handle error types that don't fit neatly into your 25-category framework, such as complex errors involving multiple categories?
2. Have you considered adapting your approach for modern neural GEC systems, which produce different error patterns than the rule-based systems in the CoNLL-2014 task?
