

Summary of Paper and Main Contributions
The paper proposes a method to automatically annotate grammatical error correction (GEC) system outputs with error type information, enabling detailed error type analysis that was previously impossible. The authors address the gap where system outputs were not annotated with error type information, making it difficult to assess which error types individual systems perform well on. The main contributions of this work are: 1) A method for automatic annotation of error types in GEC system outputs using linguistically-enhanced alignment and a rule-based classifier; 2) A dataset-independent error typing framework that relies solely on automatically obtained linguistic properties (POS tags, lemmas, etc.) without requiring labeled training data; 3) A detailed error type analysis of systems participating in the CoNLL-2014 shared task, revealing system-specific strengths and weaknesses.

Strengths
1. The paper addresses a clear and significant gap in the GEC evaluation landscape. Prior to this work, it was impossible to conduct detailed error type analysis of GEC systems because system outputs weren't annotated with error type information. The authors demonstrate that their method enables this analysis for the first time, providing valuable insights into system performance across different error types.

2. The rule-based classifier is dataset-independent and relies only on automatically obtained linguistic properties, making it practical for immediate application across different datasets without retraining. The human evaluation showing >95% of predicted error types were rated as "Good" or "Acceptable" by experts provides strong validation of the approach's effectiveness.

3. The application of the method to the CoNLL-2014 data provides concrete evidence of the approach's value, revealing important system-specific insights (e.g., that 5 teams failed to correct unnecessary token errors at all, while other teams excelled at specific error types). The analysis demonstrates that different systems have different strengths, suggesting opportunities for combining approaches to improve overall performance.

4. The paper clearly explains the technical approach to edit extraction and error typing, with sufficient detail for reproducibility. The authors provide a detailed error classification framework (25 error types) and validate their approach against the gold standard through a human evaluation, addressing a key concern about the reliability of automatic annotation.

5. The authors demonstrate that their automatic reference processing produces results comparable to gold references (with no statistically significant difference), validating the quality of their automatic annotation approach. They also provide a thorough explanation of how their method differs from and improves upon existing evaluation metrics like M2.

Weaknesses
1. The paper doesn't sufficiently establish the novelty of the core method. The edit extraction component relies directly on Felice et al. (2016), which is cited as foundational work, and the error typing approach builds on Swanson and Yamangil (2012) with minor modifications. The paper would benefit from a clearer explanation of what is truly novel in their specific implementation and how it differs from prior work.

2. The error classification framework (25 error types) appears somewhat arbitrary and lacks a thorough justification for the specific categories chosen. For example, the paper notes that it's "impractical to create new categories for increasingly detailed error types," but doesn't provide evidence that their current framework is optimal or that the chosen categories capture meaningful distinctions.

3. The human evaluation of the classifier was limited to 200 randomly selected edits, which may not be sufficient to establish robustness across diverse error types. The paper doesn't discuss potential biases in the selection of these samples or how they represent the full range of error types in GEC.

4. The paper doesn't clearly demonstrate how the detailed error type analysis provides insights that couldn't be obtained through other means. The authors claim that their method enables "a more detailed evaluation" but don't sufficiently compare their insights to what could be learned from other evaluation approaches.

5. The analysis of system performance in the CoNLL-2014 task doesn't convincingly demonstrate that the error type analysis reveals meaningful system differences beyond what could be inferred from overall scores. For instance, the observation that "5 teams failed to correctly resolve any unnecessary token errors" could potentially be explained by other factors beyond the error type analysis.

Questions to Authors
1. How does your error classification framework specifically address limitations of previous error type schemes (e.g., those used in FCE or CoNLL-2014), and why is your particular set of 25 error types preferable to alternatives?

2. Given that your classifier relies on POS tags, which can be error-prone, how do you account for potential error propagation from the POS tagging step to the error type classification, especially for error types that depend on precise POS information?

3. Could you provide more details about the statistical significance of the differences between teams' performance across error types, particularly for error types where the differences appear small but might be practically meaningful?
