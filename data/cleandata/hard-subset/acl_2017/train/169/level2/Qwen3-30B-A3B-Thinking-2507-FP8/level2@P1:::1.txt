

Summary: This paper proposes a method for automatically annotating grammatical error correction (GEC) system outputs with error type information, addressing a significant gap in GEC evaluation where error type analysis was previously impossible due to lack of annotations on system outputs. The main contributions are: (1) a linguistically-enhanced alignment approach for automatic edit extraction, (2) a rule-based error typing framework that is dataset-independent and relies only on automatically obtained linguistic properties, and (3) the first detailed error type analysis of GEC systems using the CoNLL-2014 shared task data.

Strengths:
1. The paper addresses a genuine and important gap in GEC evaluation. Prior to this work, researchers could only measure overall GEC performance without detailed error type analysis, which is crucial for understanding system strengths and weaknesses. This contribution significantly advances the field by enabling more nuanced evaluation.

2. The rule-based error typing approach is well-motivated and well-executed. By relying only on automatically obtained linguistic properties (POS tags, lemmas, etc.) rather than labeled training data, the method avoids the dataset dependency that plagues previous approaches. The small-scale human evaluation (5 raters, 200 examples) showing >95% "Good" or "Acceptable" ratings provides strong evidence for the classifier's quality.

3. The application to the CoNLL-2014 shared task provides valuable insights into system performance across different error types. The detailed analysis (Tables 5-8) reveals that different systems have complementary strengths (e.g., CAMB excelled at missing token errors while CUUI was better at replacement errors), which could guide future system design.

4. The paper clearly demonstrates the utility of the approach by showing that automatic reference processing yields results comparable to gold-standard references (Table 4), and the method also provides a more transparent alternative to the M2 scorer that inflates scores by merging edits (Section 4.1).

Weaknesses:
1. The paper lacks sufficient evidence to fully justify why the automatic edit extraction and error typing are sufficient for meaningful error type analysis. While the human evaluation of the classifier is good, the paper doesn't address how errors in edit extraction (which it acknowledges can be "unusual") might propagate to error type classification. A more thorough analysis of error types where the edit extraction was likely incorrect would strengthen the argument.

2. The claim that the framework is "dataset-independent" is under-supported. The evaluation was limited to CoNLL-2014 and FCE-test data, and the paper doesn't demonstrate how the approach would handle new error types or datasets with different annotation schemes. The authors should clarify the boundaries of the framework's applicability.

3. The paper doesn't adequately address the practical limitations of the approach. For example, how does the method handle cases where the automatic POS tagger makes errors (which the authors acknowledge as a source of error in the classifier)? The paper mentions this but doesn't quantify its impact or propose solutions.

4. The detailed analysis in Tables 5-8 is valuable, but the paper doesn't sufficiently discuss the implications of the findings for future work. For instance, why did 5 teams fail to correct unnecessary token errors at all (Section 4.2)? What specific system design choices might explain these differences? More discussion of these findings would strengthen the contribution.

Questions to Authors:
1. The paper claims the approach is dataset-independent, but the human evaluation only used CoNLL-2014 and FCE-test data. How would the framework handle error types not present in the current 25-category scheme (e.g., errors that might emerge in new datasets or domains)?

2. In Section 4.1, the authors show that automatic reference processing yields comparable results to gold references. However, they don't discuss how this might affect error type analysis for systems that handle multi-token edits differently. Could you provide more detail on how the edit extraction process handles cases where human annotators might have split edits differently?

3. The paper notes that many teams didn't employ spell checkers, which would have improved performance. Given that spelling errors are common in learner text, why do you think most teams didn't incorporate spell checkers into their systems, and how might this impact the generalizability of your error type analysis?
