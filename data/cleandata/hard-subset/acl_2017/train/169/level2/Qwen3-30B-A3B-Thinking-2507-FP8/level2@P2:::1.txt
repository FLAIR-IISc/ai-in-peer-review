

Review:

Summary: The paper presents a method to automatically annotate error types in grammatical error correction (GEC) system outputs using linguistically-enhanced alignment and a rule-based classifier. This enables detailed error type analysis of GEC systems for the first time, as system outputs were previously unannotated.

Main Contributions:
1. A method to automatically extract edits between original and corrected sentences using linguistically-enhanced alignment, followed by a rule-based classifier to assign error types.
2. A detailed error type analysis of all teams in the CoNLL-2014 shared task, revealing nuanced strengths and weaknesses of different approaches.

Strengths:
1. The paper addresses a genuine and significant gap in GEC evaluation methodology. Prior work could not perform detailed error type analysis of system outputs because they weren't annotated with error types, which the authors convincingly argue is a meaningful limitation. The ability to analyze error types separately is valuable for understanding system behavior.
2. The approach is well-motivated and practical. The rule-based classifier is dataset-independent and relies only on automatically obtained information (POS tags, lemmas), which solves the problem of needing different classifiers for different datasets. The human evaluation showing 95%+ of predicted error types were rated "Good" or "Acceptable" by experts is strong validation.
3. The demonstration on the CoNLL-2014 data provides valuable insights into GEC systems' error type performance. The analysis reveals that different systems have complementary strengths (e.g., CAMB excelled at detection, CUUI at replacement errors), which could inform future system development.

Weaknesses:
1. The paper fails to adequately explain why this problem wasn't addressed before. It states "system output is not annotated" but doesn't explain why it was never done or why it's so difficult. This weakens the claim of novelty and significance. Previous work (e.g., Swanson and Yamangil, 2012) attempted error typing, but the paper doesn't properly contextualize how this work differs from or improves upon them.
2. The evaluation of the classifier is insufficiently comprehensive. The human evaluation was limited to 200 edits from only two datasets (FCE-test and CoNLL-2014). It's unclear if the classifier would generalize well to other datasets or error types. The claim that "our approach is dataset independent" is not sufficiently supported.
3. The comparison with previous work is weak. The authors mention that previous machine learning approaches had "50-70 F1" but don't provide a direct comparison on the same datasets. They claim their approach is superior without demonstrating how it compares to existing methods on the same evaluation metrics.
4. The paper doesn't adequately address limitations of the approach. It acknowledges that POS tagging errors can affect performance but doesn't discuss how these errors propagate through the system or how they might be mitigated. This is particularly important given the high sensitivity of the method to POS tagging accuracy.

Questions to Authors:
1. How did you determine the 25 error types in your framework? What was the process for ensuring these types were comprehensive and useful for analysis?
2. The paper states that "our approach is analogous to automating the annotation guidelines given to human annotators." How do your rules map to the actual annotation guidelines used in previous GEC datasets?
3. In Table 7, CUUI's classifier approach performs better at Replacement Determiner errors than CAMB's MT approach, but CAMB has higher overall scores. How would you explain this discrepancy, and what does it suggest about the relationship between overall performance and error type performance?
