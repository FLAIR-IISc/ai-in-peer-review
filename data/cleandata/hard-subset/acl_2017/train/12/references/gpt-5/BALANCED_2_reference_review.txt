Summary and main contributions
This paper presents SynTime, a lightweight, rule-based system for recognizing temporal expressions using syntactic types and simple boundary-expansion heuristics. The authors first analyze temporal expressions across four corpora (TimeBank, Gigaword, WikiWars, and a Twitter dataset) and report four empirical observations (brevity, small vocabulary of time tokens, prevalence of at least one time token per expression, and shared syntactic behavior). SynTime defines token types based on syntactic behavior (time tokens, modifiers, numerals) and applies heuristic expansion rules around identified time tokens. Experiments on TimeBank, WikiWars, and a manually annotated Tweets dataset show consistently high recall and strong F1 scores, often surpassing HeidelTime, SUTime, and UWTime.

Contributions, in order of strength:
1) A simple syntactic-type-based temporal tagger that achieves consistently high recall and strong F1 across three datasets using boundary-expansion heuristics (practical, interpretable, and efficient).
2) An empirical analysis of temporal expressions across four corpora that motivates the type system and heuristics (brevity, limited lexicon, syntactic regularities).
3) A manually annotated tweets corpus for temporal expression recognition (if released; otherwise weaker as a contribution).

Strengths
- Strong empirical performance with especially high recall: SynTime(-I/-E) achieves best or second-best results on most metrics across datasets. On TimeBank, strict/relaxed F1 of 92.1/95.0; on Tweets, 91.7/95.9, outperforming baselines substantially (e.g., +11.37 F1 strict over the best baseline claimed). High recall aligns with Observation 3 (93%+ expressions contain a time token), supporting the design.
- Simple, interpretable, and efficient method: The system relies on a small set of syntactic types and boundary-expansion rules. Compared with fixed, longer rules in traditional taggers, the heuristic approach is flexible and extensible, making domain adaptation (via a few lexicon additions) practical. Table 5 indicates only a handful of additions are needed per domain.
- Data-driven motivation: The cross-corpus analysis (length distributions, token vocab sizes, POS behavior) grounds the design choices. The observation that temporal expressions are short and use a small set of time tokens motivates the “identify time token first, then expand” strategy.
- Practical domain adaptation result: SynTime-E improves notably on WikiWars via modest lexicon expansion (+3.25 recall strict, +2.98 recall relaxed), demonstrating that the approach scales to domain-specific vocabularies with low manual cost.
- Clarity of the algorithmic pipeline: The steps (token identification, segment identification, expression extraction) and merging criteria (adjacent, overlapping via LINKAGE/COMMA conditions) are clearly described and illustrated, aiding reproducibility.

Weaknesses
- Novelty relative to prior rule-based systems is limited: SUTime and HeidelTime already use token-level regexes, expansion from tokens to chunks, POS cues, and flexible pattern composition. While framing types as “syntactic” vs. “semantic” is conceptually neat, the practical distinction may be modest. The paper would benefit from a sharper articulation and ablation showing which parts of SynTime (syntactic type definitions, boundary rules) yield gains over existing expansion strategies.
- Tweet dataset construction risks bias and limits validity: Tweets were first filtered by SUTime to contain at least one temporal expression, then partially sampled and annotated. This enriches the dataset with SUTime-detectable expressions and may bias both lexicon coverage and pattern distributions, undermining the claim that SynTime “exceptionally” outperforms baselines on tweets. A truly unbiased evaluation should start from a random sample with complete manual annotation.
- Comparability of baselines and significance: Some TimeBank/WikiWars numbers are borrowed from prior papers, which may differ in preprocessing, versions, and splits; others are newly run. There is no statistical significance testing. This weakens “significant outperforms” claims. A consistent evaluation setup with uniform preprocessing and tests (e.g., bootstrap resampling) is needed.
- Reliance on SUTime resources: SynTime initialization uses SUTime’s token regexes; this increases coverage but makes it difficult to attribute performance to the proposed type system vs. inherited resources. An ablation replacing SUTime lexicon with a minimal lexicon (or learned lexicon) would clarify SynTime’s independent contribution.
- Claims about the principle of least effort are speculative: While the descriptive stats (short length, small token set) are interesting, the link to Zipf’s principle is not empirically tested. The paper should temper the claim or provide stronger evidence.
- Reproducibility and engineering details: Runtime claims (“real time”) lack measurements. POS mapping to types and ambiguity filters are only sketched. It is unclear whether code and the tweet annotations will be released. These gaps hinder replication.

Questions to authors
- Dataset construction: For Tweets, what fraction of the final test set came from SUTime-positive tweets vs. randomly sampled negatives? Can you provide results on a fully random, exhaustively annotated sample?
- Baseline setup: Which versions/configurations of HeidelTime and SUTime were used? Were they run under the same tokenization/POS tagging as SynTime? For TimeBank/WikiWars, did you re-run baselines under your setup or only cite prior numbers?
- Statistical testing: Are the improvements over baselines statistically significant? Please report tests and confidence intervals.
- Ablations: Can you report ablations for (a) without SUTime lexicon initialization, (b) without POS disambiguation, (c) without specific heuristics (e.g., COMMA/LINKAGE merge), and (d) using only “semantic” vs. only “syntactic” modifier grouping?
- Resource release: Will you release SynTime code, initialization lexicon, and the annotated tweet dataset?
- Robustness: How sensitive is performance to POS tagging errors (especially on tweets)? Have you tried more robust taggers for social media text, or character-level fallback for ambiguous tokens?
- Prepositions and TimeML compliance: You note WikiWars has more prepositions/descriptive expressions that TimeML often excludes. Can you quantify the error types and discuss a variant that optionally includes common prepositions for downstream IR tasks?

Additional comments
- Clarify train/dev/test splits for TimeBank/WikiWars and whether SynTime-E used any in-domain expansion on those test sets (risk of leakage).
- Provide runtime benchmarks and memory footprint to substantiate “real time.”
- Fix typos and reference formatting inconsistencies; ensure all cited resources are properly referenced and accessible.

Overall, this is a solid, practical contribution with strong recall and clear engineering, but it needs stronger evidence of novelty, cleaner evaluation, and better reproducibility to meet a high bar.
