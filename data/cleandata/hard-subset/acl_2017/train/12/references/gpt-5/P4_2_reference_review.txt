Summary and contributions
This paper presents SynTime, a lightweight, rule-based system for time expression recognition that relies on syntactic types and simple boundary-expansion heuristics. The authors first analyze four corpora (TimeBank, Gigaword, WikiWars, Tweets) and observe that time expressions are short, rely on a small vocabulary, almost always contain a “time token,” and show consistent syntactic behavior. SynTime defines syntactic types for time tokens, modifiers, and numerals, initializes its lexicon from SUTime’s token-level regexes, and applies simple, position-aware heuristics to expand from time tokens to full expressions. Experiments on TimeBank, WikiWars, and a manually annotated Tweets set compare SynTime against HeidelTime, SUTime, and UWTime and report higher F1 on TimeBank and Tweets, comparable results on WikiWars, and consistently strong recall.

As assessed, the main contributions are:
1) A simple, type-based boundary-expansion framework for time expression recognition that achieves strong recall and state-of-the-art or competitive F1 across domains while remaining fast and lightweight.
2) An empirical analysis of time expressions across multiple corpora that motivates the design (short length, small lexicon, high prevalence of time tokens, syntactic regularities).
3) A manually annotated Tweets dataset and comparative evaluation against strong baselines, including both rule-based and learning-based systems.

Strengths
- High recall and strong overall performance across datasets: SynTime-I and SynTime-E attain the best recalls on all three evaluation datasets and top or near-top F1 in most settings. On Tweets, strict F1 reaches ~91.7% and relaxed F1 ~95.9%, improving over baselines by notable margins; on TimeBank, strict/relaxed F1 ~92/95 also surpass baselines. This supports the claim that the type-first, boundary-expansion approach covers diverse surface forms effectively.
- Clear, reproducible rule design with interpretable types: The system defines explicit syntactic types (15 token types, 5 modifier types, numerals) and concise, position-based rules for left/right expansion and merging. The algorithm is described with concrete examples (Figure 3), making the approach understandable and easy to implement. The design decision to separate token identification and expansion is well motivated by the analyses.
- Practicality and efficiency: SynTime operates without training, initializes from existing token regex resources, and can be expanded with a small number of domain-specific additions. The expansion study (SynTime-I vs. SynTime-E) shows modest effort yields measurable gains (e.g., on WikiWars, recall +3.25% strict). The lightweight nature suggests favorable runtime, suitable for real-time applications.
- Empirical analysis informing design: The cross-corpus analyses (length distributions, token coverage, POS roles) are consistent and plausible, providing empirical grounding for the “time token + modifier/numeral” recognition strategy. The observation that >93% of expressions contain a time token supports the token-first pipeline.
- Breadth of comparisons: The evaluation includes two strong rule-based systems (HeidelTime, SUTime) and a learning-based system (UWTime), with strict and relaxed metrics following TempEval-3. This strengthens the evidence for general competitiveness.

Weaknesses
- Novelty relative to existing rule-based systems is limited: While the paper repositions types as “syntactic” and uses simple boundary expansion, similar ideas (token-first recognition, expansion to spans, position-aware rules) exist in SUTime and related systems. The distinction between semantic vs. syntactic typing is not clearly shown to be substantively different in practice. An ablation demonstrating that “syntactic types + boundary heuristics” materially outperform a comparable “semantic type + heuristics” or SUTime-like pattern set would better establish novelty.
- Potential dataset bias for Tweets: The Tweets corpus was assembled by first tagging 4000 tweets with SUTime and retaining only those with a detected time expression (942), with a follow-up sample suggesting a 3% miss rate. This preselection likely skews the evaluation set toward expressions that SUTime-like token lexicons can detect, and may underrepresent hard negatives or unseen forms. This could inflate recall for any method initialized from SUTime’s lexicon (including SynTime). Evaluating on an unbiased, fully annotated random sample would address this concern.
- Fairness and configuration transparency of baselines: SynTime initializes from SUTime’s token regexes and further expands with manual additions, but it is unclear whether baselines received comparable tuning or domain adaptation (e.g., HeidelTime has tuning resources; UWTime’s configurations and features vary). Some baseline results are borrowed from prior papers; tool versions and settings (e.g., DCT handling, language resources, Twitter-specific models) are not fully specified. Without uniform tuning and documented configurations, claims of significant superiority are difficult to verify.
- Limited analysis of errors and scope: The paper notes POS tagging errors and the exclusion of prepositions per TimeML but provides little systematic error analysis (e.g., over-merge/under-merge patterns, ambiguity with tokens like “May,” descriptive spans in WikiWars). There is no statistical significance testing for reported improvements. Additionally, there is no ablation isolating the contributions of POS gating, type inventory granularity, or merging rules.
- Reproducibility and resource release: Code, the final lexicon/type lists, and the annotated Tweets dataset do not appear to be released. Given manual lexicon expansion decisions (“generic,” “no ambiguity”), reproducibility would benefit from releasing the exact resources and annotation guidelines.

Questions to authors
- Can you release the SynTime code, the exact token/multitoken regexes, type inventories, and the manually annotated Tweets dataset with guidelines? This would address reproducibility.
- How were baselines configured on Tweets and WikiWars (versions, resources, domain adaptations)? Did you enable any Twitter-specific models or retune lexicons for HeidelTime/SUTime? Please clarify UWTime’s training details (features, hyperparameters, train/dev splits).
- Can you report results on a fully annotated, randomly sampled Tweet set (without SUTime preselection) to remove selection bias, or provide an analysis estimating its impact?
- Did you conduct significance testing for the reported improvements? If so, please provide tests and p-values.
- Ablation requests: (a) performance without POS gating; (b) performance with only token identification (no expansion); (c) replacing syntactic vs. semantic modifier types while keeping the same heuristics to quantify the advantage of “syntactic” typing.
- Runtime: Please report throughput (docs/sec or tokens/sec) and memory usage relative to baselines to substantiate the “real time” claim.

Overall, the paper is clear and presents a simple, effective approach with strong recall and competitive F1. However, the methodological novelty over existing rule-based taggers is modest, and evaluation design and baseline configuration details limit the strength of the empirical claims. Addressing fairness, bias, and ablations would substantially strengthen the work.
