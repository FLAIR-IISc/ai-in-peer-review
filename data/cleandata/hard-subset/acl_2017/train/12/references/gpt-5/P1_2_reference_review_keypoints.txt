- Proposes SynTime, a rule-based time expression recognizer using a small set of syntactic token types and boundary-expansion heuristics.
- Contributions: type-based rule formulation with strong recall, corpus-driven analysis motivating design choices, and a lightweight extensible rule framework.

- Strong empirical performance: top/second-best results and >92% relaxed-recall across datasets, very strong results on Tweets.
- Simple and interpretable design: small intuitive type inventory and easy-to-understand boundary-expansion/merging heuristics.
- Data-driven motivation: corpus analysis (length, token coverage, POS patterns) supports token-first strategy.
- Lightweight domain adaptation: small manual lexicon additions yield meaningful recall gains.
- Clear scope: focuses on recognition (not normalization), justified by existing normalization resources.

- Limited methodological novelty: largely reuses token regexes/expansion heuristics and initial SUTime resources without clear new capabilities.
- Potential evaluation bias in Tweets: dataset preselected by SUTime risks inflating recall and not testing on a random gold sample.
- Missing significance testing and detailed error analysis: no statistical tests, per-type performance, or ablations reported.
- Reproducibility concerns: code, full type inventories/regexes, added tokens, and the annotated Tweets dataset are not released.
- Unsupported claims about efficiency and generality: no runtime/memory data and TimeML-centric design may limit portability (e.g., handling prepositions).
- Incomplete baselines: lacks comparison to modern neural sequence-labeling baselines (e.g., BiLSTM-CRF or transformer taggers).

- How much does performance depend on initializing from SUTime’s token regexes versus building from scratch or a reduced lexicon?
- Please provide statistical significance testing (bootstrap or randomization) for reported improvements.
- What are the exact POS mappings per type and how sensitive is performance to POS tagging errors or different taggers?
- Can you release or re-evaluate on a randomly sampled, fully annotated Tweets test set independent of any system prefiltering?
- Do you have runtime measurements (tokens/second) and comparisons to SUTime/HeidelTime on the same hardware?
- Can you provide error breakdowns by type (dates, durations, times) and typical failure modes (prepositional phrases, ambiguous tokens)?
