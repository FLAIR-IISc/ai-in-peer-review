Summary and contributions
This paper proposes SynTime, a simple rule-based time expression recognizer that uses a small set of syntactic token types (time tokens, modifiers, numerals) and heuristic boundary-expansion rules to identify temporal mentions. The authors analyze four corpora (TimeBank, Gigaword, WikiWars, Tweets) to motivate design choices (expressions are short; small, recurrent vocabulary; most spans contain a time token; consistent POS behavior). SynTime is initialized from SUTime’s token-level regexes and differs mainly in how it composes spans via type- and position-based heuristics. Evaluated on TimeBank, WikiWars, and Tweets against HeidelTime, SUTime, and UWTime, SynTime achieves strong recall and competitive-to-best F1, with especially large gains on tweets.

As I see it, the main contributions are:
1) A very simple, high-recall, type-based rule system for temporal mention recognition that achieves strong F1 on TimeBank and Tweets relative to established taggers.
2) A cross-corpus empirical analysis of temporal expression properties (length, vocabulary size, token presence, POS behavior) that usefully motivates the rules.
3) A lightweight adaptation story: minimal lexicon additions (often single digits of tokens) suffice to boost recall in a new domain (WikiWars).

Strengths
- Strong empirical performance with high recall: SynTime-I/E obtain the best or second-best F1 in most settings, with especially notable gains on Tweets (strict F1 91.7 vs 79–80 for baselines; relaxed F1 95.9 vs 86–90). Recall is consistently high (>92% relaxed), aligning with the “time token presence” observation and validating the type-first, expansion-later design.
- Simplicity, interpretability, and practicality: The approach requires no training, uses a tiny ruleset and lexicon, and runs in real time (even though speed numbers are not reported). The boundary-expansion heuristics are easy to understand and modify, and the manual expansion needed is small (e.g., 3–19 type entries), which is attractive for practitioners.
- Sensible, data-driven design: The cross-corpus analysis (short spans, small vocabulary, frequent time tokens, consistent POS patterns) directly informs the system architecture and explains why the method is robust, especially on noisy tweets where temporal expressions are often one token and formal variants dominate.
- Domain adaptation with minimal effort: On WikiWars, adding a small number of domain-specific tokens materially improves recall (e.g., +3.25 strict recall), demonstrating that the approach can be ported with little annotation overhead.
- Clear algorithmic description: The three-stage pipeline (token identification → segment expansion → expression merging) is well specified, with examples for adjacent/overlapping segments and linkage/comma handling that clarify edge cases.

Weaknesses
- Limited novelty relative to prior rule-based systems: The core ideas (token classes, regex over tokens, expansion from nuclei) are close to SUTime/HeidelTime. The “syntactic types” vs “semantic types” distinction is not convincingly shown to be substantively different (SUTime already uses TokensRegex types and positional constraints). The contribution feels incremental, largely an alternative but similar rule design with SUTime’s lexicon.
- Potential evaluation bias in Tweets dataset creation: Tweets were pre-filtered by SUTime and only then fully annotated; the authors estimate SUTime misses ~3%, based on a small sample of negatives (500/3058). This risks sampling bias toward SUTime-positive phenomena and may advantage systems using the same lexicon (SynTime initializes from SUTime’s tokens). A corpus constructed independently of any tagger would strengthen claims, especially for the reported large gains on tweets.
- Missing statistical significance and limited error analysis: No significance tests are reported for F1 differences, and there is little qualitative analysis of remaining errors (e.g., which phenomena are missed, effect of linkages/prepositions, failure modes on WikiWars). Without this, it is hard to attribute gains to specific rule choices or assess robustness.
- Scope and reproducibility gaps: The work focuses on recognition only (normalization omitted), evaluated only on English, and relies on general-purpose POS tagging (which is brittle on tweets). There are no runtime or ablation studies (e.g., removing POS filters, changing token windows, or using a tweet POS tagger). Details about POS constraints per type, the exact added lexicon entries, versions/configs of baselines, and resource/code release are missing, which impacts reproducibility and fair comparison.
- Fairness of baseline comparison: HeidelTime/SUTime appear to be used off-the-shelf; SynTime-E is manually adapted per dataset. It is unclear whether comparable light tuning was allowed for the baselines. Also, UWTime is trained on TimeBank+Tweets, but model and feature calibration to tweets is not detailed. Reporting versions and ensuring comparable tuning would strengthen the empirical case.

Questions to authors
- Dataset: Can you release a tweet corpus constructed without SUTime pre-filtering? Alternatively, can you report results on a random sample of tweets fully annotated to remove selection bias?
- Baselines: Which versions/configurations of HeidelTime and SUTime were used? Were they tuned to the same training data used for SynTime-E’s lexicon expansion? If not, can you report results with comparable light tuning?
- POS tagging: Which POS tagger was used for tweets (Stanford News vs a tweet-specific tagger like ARK)? How sensitive is SynTime to POS errors on tweets? Please provide an ablation without POS filtering.
- Significance and error analysis: Are the F1 improvements statistically significant (e.g., bootstrap or approximate randomization)? Could you add a short error breakdown (e.g., prepositional phrases, ranges, holidays, descriptive expressions) to clarify remaining gaps?
- Resources and reproducibility: Will you release code, the SynTime type definitions and regex lists (including added tokens), and detailed instructions to reproduce the experiments? Also, please provide runtime numbers to substantiate the “real-time” claim.

Overall, this is a well-motivated, practical system with strong recall and clear gains on tweets, but with limited novelty and some concerns about evaluation bias and reproducibility. Addressing the questions above (especially dataset construction, baseline tuning, and significance) would materially strengthen the submission.
