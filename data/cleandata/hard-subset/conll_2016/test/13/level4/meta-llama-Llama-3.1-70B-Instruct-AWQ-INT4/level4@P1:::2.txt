This paper presents a model for event entity linking that leverages sentential features from Convolutional Neural Networks (CNNs) instead of relying on external knowledge sources. The proposed model consists of two parts: the first part learns event mention representations, and the second part calculates a coreference score given two event entity mentions.

The paper is well-written and easy to follow, with a thorough analysis of the ACE corpus regarding argument sharing between event coreferences. The analysis on the size impact of the dataset is also a strong motivation for creating the ACE++ dataset.

However, there are several major issues that need to be addressed:

1. The authors fail to provide a clear motivation and analysis of the pros and cons of using CNNs for generating mention representations. It is not discussed why CNNs were chosen over other models, such as Recurrent Neural Networks (RNNs), and there are no comparisons to other architectures. Given the modest improvement over the state-of-the-art (only 2-3 points on F1 score), more evidence is needed to demonstrate the superiority of this architecture.

2. The novelty of tackling event linking with sentential features is not clearly established, as using CNNs for classification tasks is not new. The authors should explicitly compare their approach to existing continuous space methods for event linking and provide a more thorough comparison to other methods in Table 3.

3. There is a lack of information regarding the collection of the ACE++ dataset. A major issue with the ACE dataset is its limited number of event types, making it too constrained and biased. The authors should provide details on the event types covered in ACE++ to support their claim that their approach is more generalizable. Making the ACE++ dataset publicly available would also facilitate future comparisons.

Some minor issues:

- It would be useful to include the performance of the model without gold references in Table 3 for a more comprehensive comparison.

- Exploring how the model can be augmented with a vanilla coreference resolution system could provide additional insights. For example, leveraging the off-the-shelf CoreNLP system to link 'It' to 'bombing' in the specific example in line 687 could be used as an event entity linking baseline.

- Given the relatively small size of the ACE dataset, testing the model on other available resources, such as EventCorefBank, would be beneficial. This could also motivate working on entity and event coreference simultaneously.

- Table 5 shows that pairwise features have been effective, suggesting that feature engineering is still crucial for having a competitive model. The authors could provide more information on how the current set of features was chosen and what other features were tried.

Main Contributions:
1. The authors propose a novel approach to event entity linking that leverages sentential features from CNNs, which achieves state-of-the-art performance on the ACE dataset.
2. The creation of the ACE++ dataset, which expands on the ACE dataset and provides a more comprehensive evaluation of the model.
3. The thorough analysis of the ACE corpus and the impact of dataset size on model performance.

Strengths:

1. The paper presents a well-motivated and well-designed approach to event entity linking that addresses the limitations of previous methods.
2. The use of CNNs for generating mention representations is a novel contribution to the field.
3. The creation of the ACE++ dataset is a significant contribution to the research community.

Weaknesses:

1. The lack of clear motivation and analysis of the pros and cons of using CNNs for generating mention representations.
2. The limited novelty of tackling event linking with sentential features.
3. The lack of information regarding the collection of the ACE++ dataset.

Questions to Authors:

1. Can you provide more information on how the ACE++ dataset was collected and what event types it covers?
2. How did you choose the specific architecture and hyperparameters for the CNNs, and what other architectures were considered?
3. Can you provide more details on how the pairwise features were selected and what other features were tried?
4. Have you considered testing the model on other datasets, such as EventCorefBank, and if so, what were the results?
