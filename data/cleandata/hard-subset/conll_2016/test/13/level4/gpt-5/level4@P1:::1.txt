Summary and contributions
This paper addresses within-document event coreference (event linking) using a neural model that eschews external semantic resources. The approach builds sentence-level representations with a CNN over word and position embeddings, applies piecewise max-pooling, and concatenates these sentential features with local lexical features (the trigger and its immediate neighbors). Together with a small set of pairwise features, a single-layer neural network computes a distributed similarity vector, which is then fed to a logistic classifier for a coreference score. The system is evaluated on ACE 2005 and an expanded dataset (ACE++), achieving performance comparable to prior feature-rich systems.

As I see it, the main contributions are:
1) A resource-light neural architecture for event linking that combines sentence-level CNN features with minimal pairwise features, achieving state-of-the-art or on-par performance without WordNet/FrameNet-style external resources.
2) An empirical analysis showing that embeddings alone are insufficient; both local lexical and sentential features materially improve performance. The ablations and clustering experiments provide useful insight into which components matter.
3) A careful error analysis highlighting remaining challenges (e.g., pronoun resolution, annotation ambiguity), pointing to concrete future directions.

Strengths
- Competitive performance without external semantic resources: The system matches or exceeds prior state-of-the-art results on several coreference metrics (e.g., BLANC, MUC, positive-links) while avoiding heavy reliance on domain-specific knowledge bases. This improves portability across domains where such resources are unavailable or incomplete.

- Clear evidence that sentential and local features are complementary: The ablations (Table 5) demonstrate that pairwise features alone perform poorly; adding trigger-local features boosts performance, and adding sentential features boosts it further. This supports the claim that sentence-level context is valuable for event linking.

- Sensible analysis of clustering strategies: The comparison between transitive closure and BestLink (Table 4) shows BestLink improves precision but hurts recall enough to reduce F1 in this model. This analysis helps position the method relative to prior work that recommended BestLink.

- Scalable benefits from more data: Moving from ACE to ACE++ notably increases recall with only a small precision drop, suggesting the model benefits from larger training sets and is not overfitting to ACE.

- Useful error analysis: The qualitative study identifies key failure modes (e.g., pronoun-based event references) and annotation inconsistencies that plausibly cap performance. This is constructive and situates the model in the broader landscape of event coreference challenges.

Weaknesses
- Evaluation uses gold triggers; fairness concerns in comparisons: While the paper argues that gold triggers isolate the linking task, several comparisons involve systems that use predicted triggers, making cross-paper results less directly comparable. Reporting performance with predicted triggers would strengthen the empirical claims.

- Non-standard train/test splits: Different works use different ACE splits. Departing from previously used splits makes results harder to compare. Reproducing standard splits or providing results on prior splits would increase comparability.

- Methodological clarity: Several implementation details are unclear and limit reproducibility:
  • Position embeddings are “generated in a way similar to word embeddings,” but details are missing. Are they randomly initialized and learned? Are they shared across sentences irrespective of lexical content? Why should the same relative position embedding apply next to different words?
  • Piecewise pooling and left/right treatment: How exactly are left/right contexts operationalized in the CNN and pooling? Does the left/right distinction only affect max-pooling, or earlier layers as well?
  • Choice of local context window: Appending only one word before/after the trigger seems arbitrary; a rationale or sensitivity study would help.
  • Use of ve: The event-mention representation ve is defined (Eq. 3) but appears unused later; subsequent equations use vsent+lex instead. Please clarify how ve is fed to the similarity layer.
  • Pairwise features: Most are binary; how is distance encoded? Are these features treated as fixed inputs or learned embeddings/parameters?

- Dataset availability and comparisons: ACE++ is not publicly available. Without plans to release it or compare competing systems on the same data, results on ACE++ are hard to contextualize.

- Statistical significance: Several results are close. Significance testing (e.g., bootstrap on BLANC/MUC/B-CUBED) would substantiate claims of superiority or parity.

Questions to authors
- Predicted triggers: Can you report performance when triggers are predicted (and, if possible, matched to the same predictor used by compared systems) to enable fairer comparisons?

- Position embeddings: Please specify initialization, training, and parameter sharing. Are positions relative to the trigger mapped to a fixed index set (e.g., …, −3, −2, −1, 0, +1, +2, …) with a learned embedding table? Any clipping? Why is word-independent position embedding appropriate here?

- Left/right handling: Beyond piecewise max-pooling, do filters differ for left vs. right context or share weights? How is the trigger boundary enforced during convolution windows?

- ve vs. vsent+lex: Is ve actually the representation passed to the similarity layer (Eq. 4), or is vsent+lex used directly? Please clarify the exact vector concatenated with pairwise features.

- Pairwise feature encoding: How is distance represented (raw, binned, or embedded)? Are these features normalized and are their parameters learned?

- Square nonlinearity: Is using square(.) for similarity novel in this context? Did you compare against absolute value, ReLU, tanh, or cosine more extensively? Any intuition for when square helps beyond polarity invariance?

- Generality: Could the architecture be applied to entity coreference? If so, could you report results on OntoNotes to compare with a broader literature?

Additional comments
- Dataset/splits: If ACE++ cannot be released, consider releasing document IDs or feature representations to facilitate replication. Also, adopting or additionally reporting on established ACE splits would aid comparability.

- Significance testing: Please include significance tests for the main metrics.

- Related work: Please cite neural coreference work such as Wiseman et al. (2015), “Learning Anaphoricity and Antecedent Ranking Features for Coreference Resolution.”
