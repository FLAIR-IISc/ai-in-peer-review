Summary and contributions
- The paper proposes an MDL-based method to discover correspondence rules among multiple languages directly from orthographic forms, without assuming a particular distributional form for correspondences. It encodes a model of multiway string correspondences and an alignment procedure framed as shortest-path search, and uses an EM-like training loop that merges rules to minimize total description length.
- It introduces analyses based on normalized compression distance (NCD) derived from the learned MDL code to produce a phylogenetic tree for 13 Slavic languages, and a four-language analysis (CS-PL-RU-BG) with an ad hoc “shared Description Length” (sDL) measure to quantify similarities shared by specific subsets of languages.
- It proposes “data over-weighting” (α>1 in L(M)+αL(D|M)) as a knob to force discovery of longer/more detailed rules when data are sparse.

Strengths
- Conceptual framing: Positioning correspondence discovery as two-part MDL over a set of multi-language string rules is conceptually clean and orthogonal to popular probabilistic priors. The ability to plug in expert rules for initialization is practical.
- Interpretable patterns: The qualitative examples include linguistically plausible correspondences (e.g., cz–č, g–h, palatalizations, liquid metathesis), demonstrating the method can discover heterogeneous types of regularities (orthographic, phonological, morphological).
- Multiway capability: Unlike many pairwise approaches, the model is designed to handle N>2 languages jointly. The four-way examples show how rules can be partially specified across languages and still be learned and deployed, which is potentially useful for multilingual reconstruction or transfer.
- Phylogeny sanity check: The NCD-based tree roughly aligns with the standard West/East/South Slavic partition and captures expected outliers (BG/MK due to morphology, Sorbian, Polish), suggesting the compressor induced by the learned code reflects typological structure.
- Deterministic training path: The stated deterministic EM/merge scheme is attractive for reproducibility and for integrating linguistic priors.

Weaknesses
- Methodological underspecification and potential MDL inconsistencies: The code design is not fully derived or justified. In L(Π), the mixture of LN(|ΠCi|+1), ∑π L(π), and LN(TΠ)+log((TΠ−1 choose |Π|−1)) is asserted without a clear derivation or references to refined MDL for multinomials. It is unclear whether usage counts are double-accounted between L(Π) and L(D|M). The text states “none of the symbols of any alphabets will have a non-zero count,” which appears incorrect. The definitions of the “data-to-model code” and “weak number composition” are vague; no proofs or worked examples ensure Kraft-compliance or show that L(M)+L(D|M) corresponds to a valid prefix code.
- Alignment/EM algorithm details are insufficient: The graph construction for Dijkstra, edge definitions, admissible rule application, insertions/deletions (empty strings), and partial order reduction are not precisely specified. Complexity claims (E-step O(|D|·|R|^2), M-step O(|D|·A^2/2)) are not derived and are difficult to reconcile with Dijkstra’s complexity. The candidate-generation for merges, stopping criteria, and convergence behavior are underspecified; no pseudocode or ablations (null vs seeded initialization) are provided.
- Evaluation is limited and largely qualitative: The phylogeny is not evaluated quantitatively (no correlation with a gold taxonomy, no bootstrap support, no statistical tests). There are no comparisons to established baselines (e.g., LexStat/ASJP/ALINE/PHMMs/PairHidden Markov Models, PMI-based sound correspondence discovery, or contemporary multilingual sound change models). The four-way sDL analysis is descriptive, and the metric sums model-encoding lengths L(π) rather than usage-weighted contributions, which may not meaningfully quantify shared structure.
- Data quality and reproducibility: The datasets are small (Swadesh lists; 207–778 entries), from Wiktionary, and preprocessing is not described (tokenization, casing, diacritics, handling of multiple variants). No train/test split or robustness checks (e.g., sensitivity to list composition) are provided. Code and data are not released; the method’s current scalability is modest (“up to five languages” in hours on 4GB), with no profiling or memory/runtime curves.
- Over-weighting undermines objectivity: While α-scaling is practical, it departs from MDL’s objective criterion and introduces user tuning without guidance. The procedure “increase until no singleton rules” is ad hoc and risks cherry-picking. No sensitivity analysis over α, nor a principled criterion for selecting α, is provided. The deterministic claim becomes fragile once subjective reweighting is used.
- Writing, notation, and formatting issues: Numerous typos and spacing errors (e.g., “seekmodels”, “them odel class”), inconsistent notation (Π, R, A, TΠ) and definitions, dangling footnote markers (“bits1”), and several unclear sentences hamper clarity. References are incomplete and inconsistently formatted (years/authors missing or malformed; escaped characters; missing venues). Figures are referenced but not fully described; some table/figure captions lack detail.

Questions to Authors
- MDL code derivation: Please provide a rigorous derivation (or explicit references) for all components of L(M) and L(D|M), including the role of the multinomial composition terms. Are you using refined MDL/NML or two-part plug-in codes? How do you avoid double-counting counts when also using Shannon-optimal code lengths for rule usage in L(D|M)?
- Rule application semantics: Precisely define how a rule π = (π1,…,πN) applies during alignment. Are empty strings allowed? How are insertions/deletions penalized? Can rules overlap or cross? How do you prevent trivial “whole-word” rules from dominating for α>1?
- EM/merging procedure: What is the candidate set of merges per M-step? Is there any beam/pruning? What is the stopping rule? Do you guarantee monotone decrease of total description length? Please include pseudocode and complexity derivations consistent with the actual implementation.
- Sensitivity and robustness: Report sensitivity to initialization (null vs seeded), α, and data selection (e.g., leave-one-language-out, bootstrapping words). Do the phylogeny and sDL patterns persist?
- Baselines and metrics: Compare against standard cognate-based similarity baselines (e.g., LexStat distances, ALINE-based distances, PMI-based correspondences) and report quantitative agreement with a reference Slavic taxonomy (e.g., cophenetic correlation, quartet distances, Robinson–Foulds distance), with confidence estimates.
- sDL definition: Why use the sum of model code lengths L(π) rather than usage-weighted contributions or data code contributions to quantify “shared structure”? Can you justify sDL theoretically, and show it correlates with linguistic similarity more robustly than alternatives?
- Preprocessing and data: Specify orthographic normalization, handling of diacritics/digraphs, tokenization, variant selection, and treatment of multiword entries. Provide the exact lists used, and release code/data for reproducibility.
- Scalability: Provide empirical runtime/memory as a function of N, average word length, and number of rules; any batching or index structures? What prevents scaling beyond five languages today?

Overall, the idea is promising and the qualitative examples are appealing, but the paper requires substantial methodological clarification, stronger evaluation with baselines, and improved presentation to meet CoNLL standards.
