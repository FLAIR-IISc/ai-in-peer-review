### Summary of the Paper
This paper presents a method for discovering string-level correspondence rules among multiple languages. The approach is based on the Minimum Description Length (MDL) principle, which posits that the best model is the one that provides the most succinct description of the data. The authors propose an MDL cost function (a "code") for a set of cognates and a set of correspondence rules. They then describe a deterministic, greedy algorithm, akin to Expectation-Maximization, to infer a good set of rules from the data. The method is evaluated on a dataset of 13 Slavic languages. The authors perform pairwise comparisons to construct a phylogenetic tree, which aligns with known linguistic classifications. They also showcase a four-language analysis, introducing a "shared Description Length" (sDL) metric to quantify similarities between arbitrary subsets of the analyzed languages.

### Contributions
1.  **A Method for N-Way Correspondence Discovery:** The primary contribution is an algorithm that can discover correspondence rules simultaneously for more than two languages (N>2). This contrasts with many existing methods that are limited to pairwise analysis.
2.  **A Granular, Information-Theoretic Similarity Metric:** The paper introduces the "shared Description Length" (sDL), which leverages the MDL formulation to quantify the amount of shared structure between any subset of the languages being analyzed. This provides a more detailed view of linguistic similarity than a single pairwise distance matrix.

### Strengths
1.  **Principled, Assumption-Free Framework:** The use of the MDL principle is well-motivated for the task of discovering correspondences without imposing arbitrary prior assumptions on their shape or distribution. This allows the patterns to emerge directly from the data, lending objectivity to the results.
2.  **Novel Multi-Language Analysis:** The ability to analyze more than two languages at once is a significant strength. The sDL analysis in Section 5.3.1 is particularly insightful, revealing subgroup structures (e.g., CS-PL vs. RU-BG) and other subtle relationships within a single model, which would be difficult to capture with pairwise methods alone.
3.  **Empirically Plausible Results:** The pairwise analysis produces a phylogenetic tree (Figure 1) that successfully groups the Slavic languages according to established linguistic families (West, East, South). This provides strong external validation that the learned string-level similarities are linguistically meaningful.

### Weaknesses
The paper, while presenting promising ideas, is undermined by a significant number of flaws in its presentation, methodology, and experimental details.

1.  **Poor Presentation and Formatting:** The paper is in a state that is unacceptable for a conference submission.
    *   **References:** The reference section is entirely malformed. It contains BibTeX artifacts (`\Q2004\E`), misplaced brackets (`Saitou, Nei1987]`), incomplete entries, and inconsistent formatting. This is not a minor issue; it prevents readers from verifying the cited work and reflects a lack of care in the paper's preparation.
    *   **Typos and Grammar:** The text is littered with errors, such as missing spaces ("two ormore"), hyphenation inconsistencies ("stringlevel" vs. "string-level"), and grammatical mistakes ("Much of these focus").
    *   **Inconsistent Labeling:** Figures and Tables are inconsistently labeled (e.g., "F 1:", "T 2:", but "Figure 2").
    *   **Structural Error:** The paper's roadmap in the introduction omits Section 3 entirely, stating the paper moves from Section 2 to Section 4.

2.  **Methodological Obscurity:** Several core components of the proposed method are either ambiguous or insufficiently explained.
    *   **MDL Formulation:** The description of the cost function `L(M)` in Section 4.1 is confusing. It appears to include data-dependent counts (`TΠ = ∑ count(π)`), which violates the principle of a two-part code where `L(M)` should describe the model independent of its usage on the data `D`. The log-binomial term for rule counts typically belongs in the encoding of the data, `L(D|M)`, not the model. This potential misapplication of the MDL principle calls the theoretical soundness of the objective function into question.
    *   **Unexplained Details:** The paper mentions using "partial order reduction" to make alignment graphs smaller, a potentially crucial optimization for scalability, but provides no details on how this is achieved.
    *   **Complexity Analysis:** The time complexity for the E and M steps is given in terms of variables `R` ("maximum number of possible rule applications in a single data entry") and `A` ("maximum number of rules used to align a single data entry"), which are poorly defined and non-standard, making the analysis difficult to interpret or verify.

3.  **Weak Optimization and Scalability:**
    *   The model is trained using a greedy agglomerative search (merging the two rules that give the best compression). This is a standard heuristic but is susceptible to local optima. The claim that rules are learned "ordered by statistical significance" is only locally true at each step of this specific greedy path.
    *   The reported performance ("up to five languages within a few hours") reveals a significant scalability problem, which severely limits the utility of the method for its primary selling point: the analysis of *multiple* languages.

4.  **Ad-Hoc "Data Over-Weighting":** The introduction of the `α` parameter in Section 4.5 to "over-weight" the data term is an ad-hoc modification that fundamentally breaks the MDL principle the paper is built upon. While the authors are transparent about this, it reads as a post-hoc fix for the model's inability to find linguistically desirable (i.e., longer) rules on its own. This weakens the central claim of objectivity.

### Questions to Authors
1.  Could you please clarify the formulation of the MDL cost function? Specifically, why are the rule usage counts (`TΠ`) and the associated data-to-model code part of `L(M)` instead of `L(D|M)`? This seems to conflate the cost of defining the model with the cost of using it to encode the data.
2.  Could you provide a concrete definition for the variables `R` and `A` used in your complexity analysis and explain how the complexity of the M-step depends on the dataset size `|D|`?
3.  Could you elaborate on the "partial order reduction" technique mentioned in Section 4.3? How is it implemented and what is its concrete impact on the search space size?
