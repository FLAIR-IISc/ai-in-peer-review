- Task: MC resolution in Wikipedia — identify all mentions (proper, nominal, pronominal) that refer to an article’s main concept.

- Method: Two linear SVMs (pronominal vs non-pronominal) using Wikipedia markup (titles/redirects/links), Freebase attributes, and POS/position/distance features.

- Dataset: Constructed a 30-article Wikipedia coreference resource (WCR) with OntoNotes-style annotations and showed OntoNotes-trained CR systems drop on this genre.

- Results: MC classifier outperforms baselines (F1 ≈ 89 vs 73–76) and adding it as a high-precision sieve to Dcoref yields ≈ +4 CoNLL F1 on WCR.

- Strength — task framing: Clear, well-scoped, practically useful subtask; binary classification is intuitive and effective.

- Strength — empirical gains: Large, consistent improvements including thoughtful feature ablations showing which signals matter.

- Strength — modeling: Simple, reproducible linear models that leverage Wikipedia structure without heavy pipelines or entity linking.

- Strength — impact on coref: Early, high-precision MC decisions benefit full clustering, improving Dcoref results on WCR.

- Strength — dataset insight: Documents a realistic domain shift where OntoNotes-trained systems degrade, motivating Wikipedia-specific approaches.

- Weakness — dataset size/availability: WCR is small (30 docs), split protocol unclear, and public availability/reproducibility is uncertain.

- Weakness — potential leakage: Cross-validation protocol unspecified; article-level features risk leakage if folds are not document-level.

- Weakness — baseline fairness: No straightforward supervised or rule-based baseline using the same Wikipedia signals to isolate learning gains from signal access.

- Weakness — resource dependence: Relies on Freebase and internal links; robustness when Freebase is unavailable or for low-link articles is not evaluated.

- Weakness — automatic singleton labels: ~13k auto-extracted singletons added to training lack explanation on how they were labeled, risking label noise.

- Weakness — limited CR evaluation: Dcoref++ sieve evaluated only on WCR; effects on OntoNotes or recall tradeoffs from forbidding later sieves are untested and no significance tests reported.

- Question — cross-validation: Were folds created at the document or mention level to rule out leakage from article-level features?

- Question — data/code release: Will WCR (texts, annotations, features) and code be released, and can an anonymized link be provided for review?

- Question — singleton labeling: How were the 13,400 automatically extracted mentions labeled as MC/non-MC, and what fraction might be false negatives?

- Question — resource ablations: What is performance using Wikipedia-only vs Freebase-only vs combined features, and can Wikidata substitute Freebase?

- Question — stronger baselines: Was a supervised baseline using only simple title/variant/string/frequency features tested, and what was its F1?

- Question — generalization: Does adding the MC sieve help or harm performance on OntoNotes or other Wikipedia subsets; does forbidding later sieves reduce recall?

- Question — inferred type extraction: How is the MC type inferred from the first sentence, and how are appositives/multiple/noisy types handled?

- Additional suggestion: Report confidence intervals or significance tests given small document count.

- Additional suggestion: Provide error breakdowns by entity type (PER/ORG/LOC/MISC) and by link density to assess robustness.

- Additional suggestion: Clarify definitions for “position tier” and the “distance” feature computation and thresholds.

- Overall: Valuable, underexplored task addressed with a simple effective method and promising gains, but needs clearer evaluation protocol, stronger baselines, and ensured data/code availability.
