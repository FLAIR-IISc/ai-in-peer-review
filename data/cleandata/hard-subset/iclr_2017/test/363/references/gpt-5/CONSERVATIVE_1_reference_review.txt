1) Summary and contributions
The paper studies a simple “compare-aggregate” architecture for matching text sequences. The model attends from one sequence to another, compares each target token with an attention-weighted source representation via a chosen comparison function, and aggregates with a CNN for prediction. The main claimed contributions are: (a) an empirical demonstration that the compare-aggregate paradigm works across four tasks (MovieQA, InsuranceQA, WikiQA, SNLI), and (b) a systematic comparison of six comparison functions (NN, NTN, Euclidean+Cosine, SUB, MULT, SUBMULT+NN), with the finding that element-wise operations (SUB/MULT and their combination) are strong, often outperforming more flexible neural comparison modules. The paper also releases code.

2) Decision: Reject
Key reasons: (1) The experimental methodology does not convincingly support the central claims due to weak/uneven baselines, lack of controlled comparisons, and missing statistical rigor. (2) The novelty is limited relative to prior compare-aggregate/attention models; the main addition is a comparison-function sweep whose conclusions are confounded by design choices and not deeply analyzed.

3) Supporting arguments
- Baselines and fairness. The paper does not re-implement baselines and instead compares to reported numbers. This risks discrepancies in preprocessing, embeddings, hyperparameters, and training regimes. In particular, many prior models fine-tune embeddings, while this work freezes GloVe; conversely, the authors use one-directional attention plus a CNN aggregator not present in several baselines. On MovieQA, the compared baselines are relatively weak (mostly similarity-based TFIDF/Word2Vec), so “SOTA” claims are not persuasive without comparison to stronger neural baselines developed for that benchmark.

- Confounded comparison-function study. The core claim—that SUB/MULT are superior—relies on comparisons where the output dimensionality differs across functions (e.g., EUCCOS yields a 2-d vector, SUB/MULT yield l-d). This alone can change downstream performance via the CNN aggregator. Without equalizing dimensions (e.g., projecting each comparison output to a fixed dimension via a shared linear layer), the study does not isolate the effect of the comparison operation itself. Further, the aggregation choice (CNN) is fixed; prior work (e.g., Decomposable Attention) aggregates differently. It is unclear whether the observed gains stem from the comparison function or interactions with the specific aggregator.

- Inconsistent architecture across tasks. For SNLI, an LSTM is added in preprocessing, while for others the preprocessing is order-insensitive. This undermines the “general” architecture claim and complicates cross-task conclusions about the comparison functions. If the architecture changes by task, the effect of the comparison module cannot be cleanly separated.

- Missing rigor: variance and significance. Results are presented as single numbers, with no multiple seeds, standard deviations, or statistical significance tests. Given the modest margins in some settings, this makes it hard to judge robustness. Hyperparameter tuning appears limited (dropout, and LR/batch size only for WikiQA), and it is unclear if each comparison function was tuned equivalently.

- Analysis depth. The visualization is anecdotal; there is no systematic error analysis, no control for lexical overlap, and no ablations quantifying the contributions of attention vs preprocessing vs aggregation vs comparison. The claim that order-insensitive preprocessing suffices on some tasks could reflect dataset artifacts (e.g., overlap heuristics), which should be probed.

- Positioning and novelty. The architecture is very close to established compare-aggregate/attention models (e.g., Parikh et al., match-LSTM). The main novelty is a small set of hand-designed comparison functions and a claim that element-wise operations are effective—an idea already present in prior pairwise/heuristic matching work. Without a more careful experimental design and stronger baselines, the incremental contribution is not compelling for ICLR.

4) Suggestions to improve the paper (helpful, not decisive)
- Equalize comparison outputs. For each comparison function, project to a common dimensionality with the same linear layer, and report results with/without this projection to isolate the operation’s effect.

- Stronger, consistent baselines. Re-implement representative baselines under the same preprocessing, embeddings, and training protocol. Include bi-directional attention (as in Decomposable Attention/ESIM) and a variant with an LSTM aggregator to test aggregation effects.

- Rigor and robustness. Report mean±std over multiple seeds and statistical tests. Tune hyperparameters per comparison function fairly. Include parameter counts and training time for each variant.

- Ablations. Quantify the contributions of preprocessing, attention, comparison, and aggregation across all tasks. Test order-sensitive vs order-insensitive preprocessing on QA tasks to substantiate claims.

- Controls for lexical overlap. Evaluate on splits or adversarial sets controlling overlap to verify the model is not exploiting superficial cues.

- SNLI consistency. Use the same preprocessing (order-insensitive vs LSTM) across tasks, or clearly justify and analyze the differences.

- Clarify notation/typos and improve citation formatting.

Questions for the authors
- How do you control for the dimensionality mismatch among comparison functions (e.g., 2-d EUCCOS vs l-d SUB/MULT)? Have you tried projecting all comparison outputs to a fixed l before aggregation?

- Are the reported improvements robust across random seeds and hyperparameter variations? Please provide variance or significance analyses.

- Why use one-directional attention only? How does the model compare to a two-way (bi-attention) variant under the same setup?

- How much do results change if embeddings are fine-tuned? Do your conclusions about SUB/MULT still hold?

- For SNLI, what is the exact LSTM preprocessing and why is it not used for the other tasks? Does using LSTM preprocessing on QA tasks change the findings?

- On MovieQA, can you compare against stronger neural baselines trained on the same split and with the same resources?

- Can you provide detailed ablations quantifying the relative contributions of preprocessing, attention, comparison function, and aggregator on each dataset?
