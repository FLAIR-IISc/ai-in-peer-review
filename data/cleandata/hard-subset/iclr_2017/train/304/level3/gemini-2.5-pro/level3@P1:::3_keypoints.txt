- Introduces recursion into neural programming (NPI) to improve generalization by training on explicit recursive execution traces.
- Hypothesis: existing models learn spurious correlations (e.g., input length) rather than true program semantics.
- Augmented NPI is tested on four tasks — grade-school addition, bubble sort, topological sort, quicksort — showing dramatic generalization improvements.
- Recursive models achieve perfect accuracy on complex, unseen inputs after training on very few simple examples.
- Presents a novel verification procedure enabled by recursion that permits formal proofs of perfect generalization for learned programs.
- Decision: Accept — reviewer recommends publication.
- Major reason 1: identifies a fundamental limitation (poor generalization) and offers an elegant, effective solution (recursion) with strong empirical results and sample efficiency.
- Major reason 2: provides provable guarantees via verification, advancing beyond purely empirical evaluation in neural program synthesis.
- Key mechanism: recursion resets controller hidden state at each call, preventing dependence on input length and encouraging state-independent reduction-step policies.
- Empirical evidence: stark performance gap between non-recursive and recursive models, e.g., recursive topological sort generalizes from a single 5-node training graph.
- Verification insight: recursion converts infinite-input correctness into checking a finite set of base cases and reduction rules, enabling tractable verification.
- Feedback 1: clarify how the verification set V is constructed (provide concrete analytic derivations and specific states/transitions, e.g., for topological sort).
- Feedback 2: acknowledge and discuss the strong supervision requirement (explicit recursive traces) and potential paths to reduce supervision.
- Feedback 3: reiterate scope limits of the verification approach for very large or continuous input domains and specify which problem classes are provable.
- Question 1: how exactly is V constructed for topological sort and quicksort, and how is coverage/absence of corner cases ensured?
- Question 2: has the model been trained on non-recursive traces to see if NPI can discover recursion from iterative examples?
- Question 3: can the NPI architecture learn more complex recursion patterns, such as mutual recursion between multiple functions?
