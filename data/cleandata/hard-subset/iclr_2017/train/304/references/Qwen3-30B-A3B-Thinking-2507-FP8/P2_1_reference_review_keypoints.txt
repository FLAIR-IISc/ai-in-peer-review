- Paper identifies spurious dependencies (e.g., input length) as the reason neural programs fail to generalize and proposes recursion as the remedy.
- Main contributions: propose recursion as necessary for generalization; show recursion gives provable perfect-generalization guarantees; show NPI supports recursion; provide a verification procedure checking base cases and reductions; empirically demonstrate perfect generalization on four tasks with tiny training sets.
- Decision: Accept.
- Key acceptance reasons: novel theoretical contribution establishing recursion for provable generalization; strong empirical results showing perfect generalization with very small training sets.
- Theoretical point: recursion reduces verification to a finite set (base cases + reduction rules), making provable generalization tractable.
- Verification procedure: elegant method to prove perfect generalization by checking only base cases and reduction rules.
- Empirical evidence: recursive NPIs achieve perfect generalization on addition, bubble sort, topological sort, and quicksort from very small training sets; non-recursive models fail on longer inputs.
- Claim novelty: first work to provide provable guarantees of perfect generalization for neural programs; previous NPI work didn’t leverage recursion.
- Literature placement: correctly cites and builds on prior neural programming work but could compare to more recent architectures.
- Improvement suggestion 1: add comparisons to more recent neural programming approaches (e.g., Neural GPUs, NTMs).
- Improvement suggestion 2: explicitly report verification set sizes relative to training set sizes to demonstrate efficiency.
- Improvement suggestion 3: include a concrete limitations section discussing infeasibility for large input domains and possible mitigations.
- Improvement suggestion 4: provide more details on how recursive training traces are constructed for reproducibility.
- Question 1: clarify whether any prior work offered similar provable guarantees for limited cases or specific architectures.
- Question 2: explain how verification sets are constructed to ensure coverage of all base cases and reduction rules (e.g., topological sort’s size-73 set).
- Question 3: clarify the relationship between training set sizes and verification set sizes across experiments (examples given: very small training sets vs large verification sets).
- Question 4: explain how the verification procedure handles non-tail recursion (e.g., quicksort) versus tail-recursive tasks.
