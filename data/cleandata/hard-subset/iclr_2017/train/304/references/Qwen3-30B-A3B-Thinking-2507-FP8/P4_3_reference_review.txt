

# Review of "MAKING NEURAL PROGRAMMING ARCHITECTURES GENERALIZE VIA RECURSION"

## 1. Summary of Claims
The paper proposes that incorporating recursion into neural programming architectures enables provable perfect generalization, which has been a fundamental limitation of previous neural programming approaches. The authors argue that recursion is a critical abstraction in programming that reduces problem complexity and makes it tractable to prove generalization properties. They demonstrate this by modifying the training procedure for the Neural Programmer-Interpreter (NPI) architecture to learn recursive programs, showing empirical results where recursive neural programs achieve 100% accuracy on inputs of arbitrary complexity (e.g., addition with 5000+ digits, sorting graphs with 70+ vertices) after training on extremely small datasets (as few as 4-6 examples). Crucially, the authors provide a verification procedure that proves these recursive programs generalize perfectly by testing on a finite verification set covering all base cases and reduction rules.

## 2. Decision
Accept. The paper makes a significant theoretical contribution by introducing recursion as a key abstraction for neural programs and demonstrating provable perfect generalization, which is novel to the field.

Key reasons: 
1) The paper successfully demonstrates that recursion enables provable perfect generalization, a fundamental limitation previously unaddressed in neural programming literature.
2) The empirical results are scientifically rigorous, showing consistent 100% accuracy on inputs of arbitrary complexity after training on very small datasets.

## 3. Supporting Arguments
The paper's main contribution is the novel concept of neural recursive programs and the verification procedure that enables provable generalization. This addresses a critical gap in the literature: existing neural programming approaches (including NPI) fail to generalize beyond a threshold complexity level because they learn spurious dependencies on training inputs (e.g., input length) rather than true program semantics. The authors correctly situate their work within the literature by referencing previous neural programming approaches and clearly articulating how their work extends the field.

The empirical results are compelling and well-supported. For addition, the recursive model generalizes perfectly to 5000+ digit problems after training on just 5 examples, while the non-recursive version fails. For bubble sort, the recursive model sorts arrays of any length (tested up to 120 elements) while non-recursive fails beyond length 8. For topological sort, the recursive model handles graphs up to 70 vertices after training on 6 examples, while non-recursive fails at size 5. For quicksort, the recursive model sorts any array length while non-recursive fails at length 11.

The verification procedure is a strong theoretical contribution. The authors correctly argue that recursion reduces the problem of generalization to verifying a finite set of base cases and reduction rules, making verification tractable. This is a significant advance over previous approaches, which cannot provide any provable generalization guarantees due to the complex, interdependent nature of neural program memory updates.

## 4. Additional Feedback
- The paper would benefit from a more detailed explanation of how the verification set is constructed for addition (20,181 problems), including why this specific number was chosen and how it covers all necessary cases.
- A concrete example of the verification procedure applied to one task would help readers better understand the process.
- The paper mentions that the verification procedure may not scale to very large input domains, but a more detailed discussion of the limitations and potential extensions would strengthen the contribution.

## 5. Questions for Authors
1) For addition, you state the verification set contains 20,181 input problems. Could you provide a more detailed explanation of how these were constructed to cover all base cases and reduction rules, and why this specific number was chosen?

2) In the verification procedure, you mention the verification set can be larger than the training set. For bubble sort, the training set had 100 traces (max length 2) while the verification set consisted of one array of size 10. How does the verification set size scale with task complexity, and what factors influence this scaling?

3) You state that for tasks with very large input domains (e.g., MNIST), the verification procedure may be infeasible. What specific modifications would be needed to extend your verification procedure to such domains, and what would be the practical limitations?

4) For quicksort, you note the recursion is "not purely tail recursive." How does this non-tail-recursive nature affect the learning process compared to the tail-recursive tasks (addition, bubble sort, topological sort), and why is this more challenging to learn?
