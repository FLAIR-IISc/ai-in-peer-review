### Review of "MAKING NEURAL PROGRAMMING ARCHITECTURES GENERALIZE VIA RECURSION"

**1. Summary of Contributions**

This paper introduces the concept of recursion as a crucial element for improving the generalization capabilities of neural program synthesis models. The authors propose that by explicitly structuring programs recursively, models can learn the true underlying semantics from a small number of examples, rather than spurious correlations related to input complexity (e.g., length). They instantiate this idea within the Neural Programmer-Interpreter (NPI) framework by modifying the training execution traces to be recursive. The paper's primary claims are twofold: first, that these learned recursive programs demonstrate perfect empirical generalization on several algorithmic tasks (addition, bubble sort, topological sort, quicksort) from very few training examples; and second, that recursion makes it tractable to create a verification procedure that provides "provably perfect generalization" guarantees, a novel contribution in the neural programming space.

**2. Recommendation and Key Reasons**

**Recommendation:** Reject

While the paper presents a conceptually appealing idea and impressive empirical results, I recommend rejection due to two primary, significant concerns:

1.  **Overstated "Provable Generalization" Claims:** The paper's claim of providing "provably perfect generalization" is not supported with sufficient rigor. The proposed "proof" is a verification procedure based on exhaustive testing over a manually constructed set of base cases and reduction rules. The methodology for constructing this verification set is not clearly automated or formally justified, appearing ad-hoc for more complex tasks, which undermines the "provable" nature of the guarantee.
2.  **Highly Engineered Task Representations:** The learning problem is made substantially easier by domain-specific encoders that provide oracle-like information to the model. For instance, the quicksort encoder directly inputs the boolean result of a comparison, and the bubble sort encoder is given an explicit flag indicating whether a counter has reached the input length. This offloads critical algorithmic logic to the hard-coded feature extractor, calling into question what the neural network is actually learning and limiting the generality of the approach.

**3. Supporting Arguments for Decision**

My decision is based on a meticulous examination of the paper's methodology and claims, which reveals several critical flaws.

*   **On the "Provable" Guarantee:** The term "proof" implies a level of formal mathematical certainty that is not met here. The method is more akin to model checking or exhaustive testing. The central challenge lies in constructing the complete verification set `V`. For addition, this is claimed to be done analytically. However, for other tasks, the paper states, "we randomly generate input candidates until they completely cover the base cases and reduction rules" (Appendix A.6). This is methodologically vague. How can the authors be certain that a randomly generated set provides complete coverage of an analytically-derived set of rules, especially if the latter is difficult to enumerate in the first place? The paper provides insufficient detail to reproduce this critical step or to be confident in its completeness. Without a formal, transparent, and verifiable method for constructing `V`, the claim of a "provable guarantee" is unsubstantiated.

*   **On the Domain-Specific Encoders:** The design of the encoders constitutes a major weakness. True algorithmic learning should involve the model discovering relationships from raw or minimally-processed inputs. Instead, the encoders perform significant parts of the computation.
    *   For **Quicksort**, the encoder input `MLP([Qarray(pj) <= Qarray(phi), ...])` directly tells the model whether one element is less than or equal to another. The core logic of the `PARTITION` step—the comparison—is thus not learned but provided as an input feature.
    *   For **Bubble Sort**, the encoder uses `i3 == length` to terminate the algorithm. This introduces a direct dependency on the input length, which the paper itself criticizes as a source of poor generalization in prior work. It is unclear how `length` is provided to the system.
    *   These design choices reduce the NPI's role from learning an algorithm to learning a sequence of controller states given highly informative, pre-computed features. This significantly diminishes the impressiveness of the learning results.

*   **Inconsistent Experimental Details:** The paper presents results for two different versions of the bubble sort task. The version in the main paper (Table 1) uses one encoder, while the version in the appendix used for verification (Table 4) uses a different one that is even more engineered (`MLP([Q(1, i1) <= Q(1, i2), ...])`). This change is not justified, making the results difficult to interpret and compare. Which model was actually verified? Why was the change necessary for verification? This lack of clarity is concerning.

**4. Additional Feedback for Improvement**

Should the authors choose to revise this work, I offer the following suggestions. These points are not central to my decision but are intended to improve the paper's quality.

*   **Clarity of Contribution:** The paper states that NPI "naturally supports recursion" and the core change is modifying training traces. The authors should be more precise in positioning their contribution. The primary contribution is conceptual: highlighting the importance of recursion for generalization and verification. The current framing sometimes makes the work seem like an incremental improvement to NPI.
*   **Technical Details:** The mention of "tail recursion optimization" in Section 2.2 is hand-wavy. Was this actually implemented in the NPI framework? If so, how? If not, it should be removed. Furthermore, the pseudocode in Algorithm 1 is ambiguous; it is unclear how the environment state `e` is passed to and returned from sub-calls (e.g., by value or by reference).
*   **Formatting and Proofreading:** The paper suffers from a lack of polish. The title is in all caps. The references section contains conspicuous formatting errors (e.g., `et al\.,? \Q2016\E, et al\.,`), which suggests the bibliography was not properly compiled or checked before submission. Meticulous proofreading is essential for a high-quality publication.

**5. Questions for the Authors**

To clarify my understanding and assess the work more accurately, I would appreciate answers to the following questions:

1.  Could you provide a more detailed, step-by-step description of how the verification set `V` is constructed for a non-trivial task like topological sort? Specifically, how do you formally guarantee that your randomly generated problem instances `SV` fully exercise every condition in the analytically derived `V`?
2.  Could you justify the design of the domain-specific encoders? What would the performance of your recursive models be if they were given more raw inputs (e.g., the numerical values at pointer locations for quicksort, rather than the result of the comparison)?
3.  Why were two different encoders used for the bubble sort experiments in the main paper and the appendix? Please clarify which set of results corresponds to the model that was successfully verified.
