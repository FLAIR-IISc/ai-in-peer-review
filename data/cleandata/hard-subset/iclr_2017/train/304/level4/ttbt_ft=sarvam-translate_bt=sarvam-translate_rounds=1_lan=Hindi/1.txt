This paper argues that being able to handle recursion is crucial for neural programming architectures – that handling recursion allows for robust generalization for test cases outside the domain and learning from smaller amounts of training data. Much of this paper is based on Reed and De Freitas's paper on neural programmer interpreters from ICLR 2016, which learns from program traces – a paper that trains NPI models on traces containing recursive calls. The authors show a way to verify correctness by evaluating the learned program on only a small set of base cases and reduction rules, and effectively demonstrate that the NPI architecture is fully capable of estimating the बबलसॉर्ट and Tower of Hanoi problems.

What I like is that the idea is very simple, and as the authors note, the only change is in the execution trace that the training pipeline can see. I'm not really sure what the correct conclusion is – does this mean that we have effectively solved the neural programming problem when execution traces are available? (and was the problem too easy to start with?). For example, a large input domain (as one of the reviewers also noted) is MNIST digits, and we can imagine a problem where NPI needs to predict how to sort the MNIST digits from highest to lowest. In this setting, having an execution trace will effectively separate the digit recognition problem from program logic estimation – and therefore the problem will not be more difficult than recognizing MNIST digits and learning बबलसॉर्ट from symbols. What is a problem where we have access to the execution trace but cannot estimate it using the proposed method?
