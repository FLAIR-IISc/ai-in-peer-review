This paper argues that being able to handle recursion is crucial for neural programming architecture – that handling recursion allows for robust generalization for test cases outside the domain and learning from small amounts of training data. Much of this paper is based on Reed and De Freitas' paper on the Neural Programmer Interpreter from ICLR 2016, which learns from program traces – a paper that trains an NPI model on traces containing recursive calls. The authors show a way to verify correctness by evaluating the program learned on only a small set of base cases and reduction rules, and effectively demonstrate that the NPI architecture is fully capable of estimating बबलसॉर्ट and Tower of Hanoi problems.

What I like is that the idea is very simple, and as the authors note, the only change is in the execution trace that the training pipeline can see. I'm not really sure what the correct conclusion is – does this mean we have effectively solved the neural programming problem with the execution trace available? (and was the problem too easy to start with?). For example, a large input domain (as one of the reviewers also noted) is MNIST digits, and we can imagine a problem where NPI needs to predict how to sort the MNIST digits from highest to lowest. In this setting, having an execution trace will effectively separate the digit recognition problem from program logic inference – and therefore the problem will not be more difficult than recognizing MNIST digits and learning बबलसॉर्ट from symbols. What is a problem where we have access to the execution trace but cannot predict it using the proposed method?
