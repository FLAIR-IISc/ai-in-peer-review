This paper argues that handling recursion is crucial for neural programming architecture. Handling recursion allows for robust generalization in test cases outside the domain and helps learn from a small amount of training data. Much of this paper is based on Reed and De Freitas' paper on Neural Programmer Interpreters from ICLR 2016, where the program learns from traces. This paper trains the NPI model on traces containing recursive calls. The authors demonstrate how to evaluate learned programs using only a small number of base cases and reduction rules to verify accuracy, and impressively show that the NPI architecture is capable of accurately estimating problems like Bubblesort and the Tower of Hanoi.

I liked that the concept is very simple, and as the authors note, the only change is the training pipeline, which changes the execution traces it sees. I'm not really sure what to learn from this – does it mean that we have effectively solved the neural programming problem once the execution traces are available? (And was the problem too easy to start with?). For example, a larger input domain (as one reviewer mentioned) is the MNIST digits, and we can imagine a problem where NPI needs to estimate a method to sort the MNIST digits from highest to lowest. In this situation, the problem of recognizing digits and estimating program logic will be separate if there are execution traces – and therefore the problem will not be more difficult than learning to recognize MNIST digits and learn to sort bubbles from symbols. What problems exist where we have access to execution traces but cannot estimate using the proposed method?
