

# Review

## Summary
This paper proposes Neural Architecture Search (NAS), a method using a recurrent neural network (the "controller") to generate and optimize neural network architectures. The controller generates architectural hyperparameters as a sequence of tokens, and the accuracy of the generated architecture (the "child network") is used as a reward signal to update the controller via policy gradients. The authors demonstrate their method on CIFAR-10 image classification and Penn Treebank language modeling, reporting improvements over previous human-designed architectures.

## Decision
Reject. The paper contains multiple technical inconsistencies and overstatements that undermine its scientific rigor, despite presenting an interesting approach.

## Supporting Arguments

1. **Technical Inconsistencies in Methodology**: The paper contains a critical typo in the REINFORCE update equation (Section 3.2): "m m∑ k=1" should be "1/m ∑". This is not a minor formatting error but a substantive mathematical mistake that would affect implementation and reproducibility. Additionally, the paper claims "the baseline b is an exponential moving average of the previous architecture accuracies" without specifying the decay rate, which is essential for replication.

2. **Overstated Claims**: The paper claims "Neural Architecture Search can design good models from scratch, an achievement considered not possible with other methods." This is factually incorrect—NAS methods existed prior (e.g., Zoph & Le, 2016), and the paper fails to properly contextualize its work within this literature. The statement "the controller can design a 15-layer architecture that achieves 5.50% error rate on the test set" is misleading without comparison to contemporary human-designed architectures (e.g., ResNet, DenseNet), which were already achieving <5% error on CIFAR-10.

3. **Problematic Comparisons**: The paper claims "our model with 40 filters added is 1.05x as fast as the DenseNet model that achieves 3.74%, while having better performance," but fails to provide actual speed metrics (e.g., milliseconds per image). The comparison is further invalidated by the admission that "the DenseNet model that achieves 3.46% error rate uses 1x1 convolutions to reduce its total number of parameters, which we did not do, so it is not an exact comparison." This admission undermines the entire comparison.

4. **Inconsistent Terminology and Formatting**: The paper inconsistently capitalizes "Neural Architecture Search" (sometimes as "Neural Architecture Search"). The references contain numerous formatting inconsistencies (e.g., "In NAACL,," with an extra comma, inconsistent use of "et al." vs. full author lists), which is unprofessional for a top-tier conference.

5. **Lack of Technical Depth**: The paper states "filter height in [1, 3, 5, 7]" without explaining why filter height=1 is included (a filter height of 1 would be a linear filter, not a convolutional filter). Similarly, the paper claims "the controller RNN is a two-layer LSTM with 35 hidden units on each layer" without justification for this specific architecture choice.

## Additional Feedback

1. **Clarify Comparisons**: Provide direct, fair comparisons with prior work (e.g., NASNet, ENAS) using identical evaluation protocols, including speed metrics (e.g., inference time per image, FLOPs), not just accuracy.

2. **Fix Methodological Details**: Correct the REINFORCE equation typo and specify all hyperparameters (e.g., baseline decay rate, filter height=1 justification).

3. **Contextualize Novelty**: Properly situate the work within the existing NAS literature, acknowledging prior work and clearly articulating how this approach differs and improves upon it.

4. **Improve Reference Formatting**: Ensure consistent reference formatting (e.g., remove extra commas, standardize "et al." usage).

5. **Define "Good" Architectures**: Clarify what constitutes a "good" architecture (accuracy, speed, parameter count, or a combination) and how these trade-offs are evaluated.

## Questions for Authors

1. Why does the search space include filter height=1, which is not a standard convolutional filter? How does this affect the architectures generated?

2. What is the exact computational complexity metric used when claiming "the new cell, with the same computational complexity, achieves an improvement of 0.5 test set BLEU"?

3. How does the paper's method compare to Zoph & Le (2016) in terms of search efficiency (e.g., number of architectures evaluated to reach comparable performance)?

4. What is the exact schedule for increasing the number of layers in the child networks during controller training (Section 4.1), and why was this schedule chosen?

5. Why does the paper claim "the controller can design a 15-layer architecture that achieves 5.50% error rate on the test set" when the best human-designed architecture at the time (ResNet-110) achieved ~3.5% error on CIFAR-10? This claim appears misleading without proper context.
