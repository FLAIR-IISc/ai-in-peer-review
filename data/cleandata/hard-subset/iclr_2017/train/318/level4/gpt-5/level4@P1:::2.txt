Summary of contributions
- The paper proposes a neural architecture that builds and manipulates a graph-structured, differentiable memory on the fly from text. Graph construction and editing are embedded in the inference process, while parameters for graph transformations and sentence encodings are learned across tasks.
- This appears to be the first implementation of a differentiable memory explicitly represented as a graph (rather than a list, stack, or unstructured vector store). The paper demonstrates the approach primarily on bAbI tasks.
- The idea is original, technically sound to the extent I could verify, and thought-provoking. The revised version is substantially clearer than the initial submission.

Decision: Accept
Key reasons:
1) Novelty and potential impact: an explicit, differentiable graph memory is a compelling direction beyond memory networks and stacks.
2) Feasibility demonstrated: preliminary results show the approach can match strong baselines on bAbI, making this a promising line worth publishing despite current limitations.

Supporting arguments
- What problem is tackled? Learning to construct, update, and use graph-structured representations as an internal state for reasoning over text. This targets tasks where relational structure is central (e.g., entities, relations, paths).
- Is the approach well motivated and situated in the literature? Yes. The paper builds on GG-NNs/GGS-NNs by making graphs an intermediate, mutable state (not just input), and contrasts to memory networks and other differentiable memory schemes. While the approach is more complex than memory networks, the motivation for using graphs—expressivity for relational structure and distributed computation over nodes—is convincing.
- Do the results support the claims? Partially. The method achieves performance on bAbI comparable to strong memory network baselines, but not a clear overall gain. Given the increased complexity, the empirical advantage is not yet established. Nonetheless, the feasibility of learnable graph-structured memory is supported. The current evidence does not show superior learning/generalization capacity versus simpler alternatives, and performance seems worse than more traditional rule-induction approaches (the paper should provide the exact comparison).

Strenghts
- Clear conceptual novelty: a fully differentiable, editable graph as memory.
- Integration of graph construction with inference rather than relying solely on pre-specified graphs.
- Preliminary results indicate the framework can learn to parse text into a useful graph and reason over it.
- The revised manuscript shows substantial improvement in clarity.

Weaknesses
- Empirical gains are not demonstrated relative to simpler memory mechanisms on bAbI; complexity is substantially higher.
- The paper does not yet establish improved generalization capacity; evidence remains preliminary.
- Clarity remains a concern: the model description is dense and, in parts, difficult to follow without careful reconstruction.
- Comparisons to traditional rule-induction systems (mentioned) are not sufficiently documented in the current draft; the claim that performance is worse needs precise citation and numbers.

Additional feedback to improve the paper (helpful suggestions, not necessarily tied to the decision)
- Clarity and organization: streamline the description of transformations and clearly separate training-time supervision from test-time inference. A small running example that shows how a sentence updates the graph step-by-step would greatly aid readability.
- Complexity analysis: provide time/space complexity with respect to number of nodes/edges and discuss practical limits. A brief profiling (nodes/edges per bAbI instance, runtime per step) would help.
- Ablations: report the contribution of key design choices (e.g., direct reference vs none, intermediate propagation, edge-update variants). This will help isolate what matters in practice.
- Fairness of comparisons: clearly indicate supervision levels (strong supervision for graph construction vs end-to-end baselines) and, where possible, include end-to-end versions of your model to contextualize results.
- Traditional rule induction: if you claim inferior performance vs rule-based induction, include explicit references, dataset splits, and numbers. If not available, rephrase to avoid unsupported comparisons.
- Error analysis: add qualitative examples where the graph memory helps (e.g., pathfinding-like reasoning) and where it fails (e.g., compositionality, coreference).
- Reproducibility: include implementation details, hyperparameters, and training setup (e.g., optimizer, learning rates, regularization) and, if possible, release code.

Questions for the authors
1) Supervision: How much graph-level supervision is required for the model to learn meaningful structures? Can you quantify performance degradation when reducing or removing graph supervision?
2) Direct reference: How is the direct-reference mechanism implemented in practice (dictionary, string match)? How robust is it to vocabulary variation and noise? What happens if it is disabled at test time?
3) Complexity: What are the typical graph sizes in bAbI experiments? What is the per-step computational/memory cost, and how does it scale? Any observed bottlenecks?
4) Comparisons: Which “traditional rule induction” system(s) are you referencing, and what are the exact comparative results on the same splits?
5) Generalization: Do models trained on shorter bAbI contexts generalize to longer contexts? Any experiments varying story length or number of entities?
6) Stability: How sensitive are results to hyperparameters (e.g., number of propagation steps, hidden size, edge types)? Please provide sensitivity plots or ranges.
7) Failure modes: What kinds of errors dominate when the model fails? Are they due to incorrect graph construction or reasoning over a correct graph?
8) Extensibility: Could your approach naturally incorporate sparsity (e.g., sparse edge updates) or local attention to mitigate quadratic scaling? Any preliminary results?
