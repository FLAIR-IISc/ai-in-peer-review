1. Summary and contributions
The paper introduces Gated Graph Transformer Neural Networks (GGT-NNs), an extension of GGS-NNs that treats a graph as a differentiable intermediate memory. The model incrementally constructs and transforms a graph from text via a set of learned, differentiable graph transformations (node addition, node state updates with optional direct reference, edge updates, message passing, and aggregation). It then uses this graph to answer queries or produce graph outputs. The approach is demonstrated on bAbI tasks (mostly near or at high accuracy, with notable strength on pathfinding and induction), and on two rule-learning settings (a 1D cellular automaton and families of 4-state Turing machines), where the model successfully learns graph update rules and generalizes to longer rollouts. The paper is original, technically sound to the extent presented, and the idea of an explicit, differentiable graph-structured memory is compelling and potentially broadly useful.

2. Decision: Accept
Primary reasons:
- Originality and potential impact: An explicit, differentiable graph memory that is constructed and transformed from text is novel and convincingly motivated; it goes beyond treating memory as unstructured vectors and enables interpretable, graph-structured state.
- Evidence of promise: Strong results on graph-suited tasks (e.g., pathfinding) and successful rule discovery in cellular automata and Turing machines suggest the approach is practically useful and general.

3. Supporting arguments
- Problem addressed: Learning to build and manipulate a graph-structured latent state from sequences of sentences and to use that state for reasoning and output generation. This targets structured reasoning tasks better aligned with graphs than lists or matrices.
- Positioning and motivation: The method is well-situated relative to GNNs/GGS-NNs, memory networks, DNCs, and related architectures. The motivation for making the memory explicitly graph-structured (interpretability, natural inductive bias for many tasks) is clear.
- Empirical support:
  - bAbI: With strong supervision and direct reference, the model achieves high accuracy on most tasks, particularly excelling on pathfinding (task 19) and induction (task 16), which aligns with the graph inductive bias. Performance on positional reasoning (task 17) lags, plausibly due to entity explosion and shorter context. Overall, performance is competitive with supervised baselines but not uniformly superior to all memory-network variants.
  - Rule discovery: The model learns Rule 30 and families of 4-state Turing machines from graph sequences and generalizes to longer horizons, indicating it can internalize and apply structured update rules.
- Limitations and fairness of comparison: The strongest bAbI results rely on substantial supervision (gold graphs and direct reference), whereas many baselines are end-to-end. The paper acknowledges this and shows that removing direct reference degrades performance. The added complexity (and quadratic scaling with graph size) raises efficiency concerns. Ablation studies isolating the contribution of each transformation and the supervision level would further substantiate claims.
- Clarity: The revised version is much clearer than the initial one, though the presentation remains dense. The transformation definitions are careful and technically detailed; still, key choices (supervision protocol, direct-reference mechanism, and training-time vs test-time graph access) would benefit from more explicit exposition in the main text.

4. Suggestions to improve the paper (meant to help; not central to the accept decision)
- Clarify supervision: Precisely describe what graph annotations are available during training and absent at test time; quantify how performance changes under weaker supervision (e.g., no edge labels, no node types, or noisy direct references).
- Report efficiency: Provide parameter counts, average graph sizes, runtime/memory usage, and scaling behavior with number of nodes/edges. Discuss practical limits and any sparsity or batching tricks.
- Expand ablations: Isolate the effect of each transformation (Tadd, Th, TC, Tprop, Trepr), number of propagation steps, and hidden size; show the benefit of direct reference; and report variance across runs.
- Fair comparisons: When possible, compare supervised GGT-NN to supervised baselines and end-to-end GGT-NN to end-to-end baselines on the same data regimes (e.g., 1k vs 10k examples).
- Error analysis: Analyze failures on task 17 and discuss strategies (e.g., compositional entity representations, type hierarchies) to mitigate entity explosion.
- Interpretability: Provide examples of extracted graphs during inference and illustrate how message passing solves tasks like pathfinding.
- Writing/organization: Streamline notation, move some appendix essentials (e.g., supervision protocol and direct-reference construction) into the main text, and add a small running example to ground Algorithm 1.

5. Questions for the authors
- Supervision dependency: How far can accuracy be maintained if full graph supervision is reduced or removed? Can the model learn useful graphs under weak or semi-supervised settings (e.g., only answers provided)?
- Direct reference: How are node types and multi-word entities resolved in general (beyond bAbI)? What is the quantitative gain from direct reference across tasks, and can a learned mention detector replace string matching?
- Scalability: What are typical node/edge counts at train/test time? What is the time and memory footprint per step, and how does it scale? Have you tried sparse edge updates or attention to reduce quadratic costs?
- Propagation depth: How sensitive are results to the number of propagation steps? Do deeper propagation steps help tasks that require multi-hop reasoning?
- Rule extraction: For the cellular automaton and Turing machines, can you extract human-interpretable rules from the learned transformations (e.g., by probing fset/freset or message functions)? How robust is long-horizon generalization beyond 30 steps?
- Comparative regimes: For bAbI, can you report results with 10k examples under the same supervision settings, and also an end-to-end variant without graph supervision, to clarify the gap to memory networks and DNCs?
- Ablations: What happens if you remove Tadd (fixed nodes), or TC (fixed edges), or replace Trepr with simpler pooling? Which components are most critical?

Overall, despite higher complexity and reliance on supervision for the strongest results, the core idea—explicitly differentiable graph memory built and used on the fly—is original and promising. The empirical evidence on graph-suited tasks and rule learning supports acceptance.
