Summary
The paper proposes Gated Graph Transformer Neural Networks (GGT-NNs), a framework that extends GG-NN/GGS-NN by making the graph itself a differentiable object with soft node types (xv), node existence strengths (sv), and a connectivity tensor C over edge types. The authors define a set of differentiable graph transformations—node addition (Tadd), node-state update (Th, with a direct-reference variant), edge update (TC), propagation (Tprop), and aggregation (Trepr)—that can be composed into a pipeline to incrementally build and update a graph from text, and then answer queries. The model is evaluated on bAbI tasks (with strong supervision using “knowledge graphs” exposed by the generator), and on rule-discovery tasks (1D cellular automaton Rule 30 and 2-symbol 4-state Turing machines), showing high accuracy and some length generalization. The approach emphasizes interpretable, explicitly graph-structured internal states.

Decision: Reject
Key reasons: (1) Experiments rely heavily on privileged/strong supervision (ground-truth graph and entity name matching), making comparisons to end-to-end baselines unfair and limiting the practical value; (2) Reporting lacks critical details and quantitative results (several referenced tables/figures are not present here, no hyperparameters, limited ablations), harming rigor and reproducibility; (3) Numerous technical/notation/formatting issues (including incorrect GRU equations, sloppy references, and “Transformer” naming confusion) undermine clarity and polish.

Supporting arguments
- Novelty/positioning: The core idea—differentiable structural updates (adding/removing nodes/edges with graded confidences) atop GG-NN-style propagation—is interesting and useful. However, the presented “direct-reference” mechanism and reliance on known entity-type names in bAbI reduce the novelty on the language grounding side. The “Transformer” term is misleading; there is no attention-based Transformer architecture here.
- Motivation and related work: The motivation to maintain an explicit, interpretable, graph-structured state is well argued. However, the related work discussion downplays how much of the performance may come from strong supervision and explicit entity alignment. Comparisons to contemporaneous graph-construction-from-text and dynamic-graph learning approaches are limited; fairness is explicitly acknowledged but not remedied experimentally.
- Methodological correctness: The transformation primitives are clearly defined, and Appendix B improves specificity. Yet there are troubling notation/definition errors:
  - Section 2.1 GRU equations are incorrect/incomplete (h̃ uses “Wx + U(r h(t−1))” with no dependence on x(t); biases inconsistent; the elementwise product symbol is missing).
  - Several LaTeX/notation slips (e.g., σ(W[a xv]) vs σ(W[a x]) types, inconsistent subscripts/superscripts, mis-placed biases, and undefined symbols in Algorithm 1) compromise precision.
  - The loss for nodes uses set-matching across orderings (good), but the text then replaces edges and new nodes with ground truth for further processing, i.e., teacher forcing of structure—this significantly eases learning and obscures the model’s true predictive ability without privileged correction.
- Empirical support:
  - bAbI: The model is trained with access to the generator’s knowledge object, including graph structure and entity type names (used for direct reference). This is a strong, unrealistic supervision signal and not comparable to end-to-end models trained from text only. The paper notes this but still highlights performance comparisons. Moreover, actual complete numerical results are missing from the provided content (Tables 1/2 not shown; only a few numbers appear). The modified graph for temporal tasks using added “record/history” nodes changes the task formulation, further complicating comparisons.
  - Rule-learning tasks: The idea is appealing and the evaluation metric (exact next-graph match across timesteps) is strict. But again, key quantitative details are absent (Table 3 not shown; “majority” of extended tasks is vague), and training/architecture specifics are missing. It’s unclear how much teacher forcing is used here.
  - Ablations: The only substantive ablation is direct reference on/off and optional per-sentence propagation. No analysis of the impact of the supervision regime (e.g., learning without substituting ground-truth nodes/edges), number of node types, edge-type design, or sparsity constraints is provided.
  - Efficiency/scalability: The quadratic time/space complexity is acknowledged but not quantified. No runtime/memory numbers, maximum graph sizes, or sparsification strategies are empirically explored.
- Clarity/polish:
  - Numerous reference formatting errors, duplicated author lists, and missing venues.
  - Typos and lapses (e.g., “state of the word,” inconsistent algorithm numbering, broken symbols).
  - The term “Transformer” is likely to confuse readers in the current literature context.

Additional feedback (to help improve the paper)
- Provide full quantitative tables, error bars, and exact hyperparameters (hidden sizes, number of propagation steps, optimizer, learning rates, batch size, early stopping, regularization). Include runtimes and memory footprints.
- Evaluate in a strictly end-to-end setting without access to ground-truth graphs or entity-type names; compare fairly to end-to-end baselines. If strong supervision is central, explicitly position the work as a structured learning setting with privileged information, and include matched baselines (e.g., MemNN with supervision and a GG-NN with oracle graphs).
- Ablate the supervision components: (i) no substitution with ground-truth nodes/edges (no teacher forcing), (ii) partial supervision, (iii) weak/latent alignment instead of direct reference.
- Clarify the mapping from text to node types without using generator names; use learnable mention-to-entity linking and assess robustness to paraphrase/synonyms.
- Tighten the math: fix GRU equations, define all symbols, check dimensions, and clean up notation. Rename the model to avoid confusion with Transformer architectures.
- Discuss and experiment with sparsity (sparse adjacency/attention) to improve scalability. Report performance/efficiency trade-offs with graph size.
- Expand comparisons to dynamic-graph learning and text-to-graph construction methods; include a GGS-NN baseline with a learned text-to-graph front end if feasible.

Questions for the authors
1. Can you provide the complete Tables 1–4, with means/stdev over multiple runs and all hyperparameters? What are the exact D, N, Y, number of propagation steps, and optimizer settings?
2. How does performance change when you remove the substitution of predicted nodes/edges with ground-truth graphs during training (i.e., no teacher forcing)? Similarly, what if direct-reference signals are replaced by learned mention/entity linking?
3. On bAbI, can you report results without modifying the generator (no added history nodes) by using the Appendix D sequence-of-graphs variant, and compare fairly to end-to-end models?
4. For the rule-learning tasks, please provide exact accuracy numbers for 7/20/30 timesteps, and clarify whether any structural supervision/teacher forcing was used. How sensitive is generalization to graph length and to different rule distributions?
5. What are the typical graph sizes during training/inference? What is the per-epoch runtime and memory usage? Any attempts at sparsity or neighborhood sampling?
6. How many edge types are defined per dataset, and how sensitive are results to this ontology? Can the model discover useful relation types rather than relying on predefined types?
