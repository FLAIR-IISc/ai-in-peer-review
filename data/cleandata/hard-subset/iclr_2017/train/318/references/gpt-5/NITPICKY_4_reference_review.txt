Summary and contributions
The paper proposes Gated Graph Transformer Neural Networks (GGT-NNs), an extension of GG-NNs/GGS-NNs that can construct and update a differentiable graph-structured latent state via a set of trainable transformations: node addition, node state update (with an optional direct-reference variant), edge update, propagation, and aggregation. A recurrent text encoder supplies sentence-level inputs that drive graph construction and updates; a separate query phase reads from the final (or sequence of) graphs to answer questions. The paper further introduces a strongly supervised training regime that uses ground-truth graphs at every timestep (including a permutation-invariant loss for unordered node additions) and reports experiments on bAbI tasks, a 1D cellular automaton (Rule 30), and small Turing machines. The approach is appealing in that it explicitly models internal state as a graph and demonstrates that an NN can learn to transform such state across time and tasks.

Decision: Reject
Primary reasons:
1) The experimental evidence relies on strong supervision and non-trivial task modifications that invalidate comparisons to prior work and obscure the central claims.
2) Significant clarity, methodological, and presentation issues (notation inconsistencies, missing/ambiguous details, broken references/tables, incomplete pseudocode) impair reproducibility and confidence in the results.

Supporting arguments
Evaluation design and fairness
- Strong supervision: For bAbI, training uses full ground-truth graphs at each timestep, with teacher forcing that replaces new nodes/edges by gold values “for further processing.” This substantially simplifies learning and is not comparable to end-to-end baselines trained only on QA supervision (e.g., MemN2N, EntNet, DNC, DMN+). Even the MemNN baseline cited uses different “strong” supervision (relevant sentences), which is still incomparable to gold graphs. The paper’s comparisons and claims of superiority on certain tasks are therefore not credible.
- Task modifications: The bAbI knowledge objects were altered (e.g., history linked lists for task 3, renaming multi-word entities), and the model leverages an oracle-like direct-reference mechanism keyed to entity names. These changes likely make the tasks easier and further break comparability with standard results.
- Reporting: The core results (Tables 1–3) are not present in the submission; many statements (“reaches 95% on 19/20 tasks”, generalization to 20–30 steps for automata/TMs) cannot be verified. No variances, seeds, or sensitivity analyses are provided.

Methodological clarity and rigor
- Notation and equations: 
  • GRU equations have missing symbols (elementwise product), inconsistent indexing, and undefined dimensions (e.g., W without subscripts, φ applied to Wx where x has no time index).
  • Lnode uses “−max_π …” which is at odds with standard set cross-entropy (one expects min over permutations of a positive loss or max of log-likelihood without a negative sign). The current form appears wrong or at least unclear.
  • Propagation (Eq. 5) introduces f_fwd/f_bwd producing vectors in R^α, but α is not tied to subsequent matrices in Eqs. (6–8). Dimension consistency is not established.
  • Edge update toggling rule (C′ = (1−C)·c + C·(1−r)) is ad hoc, may not be idempotent, and can conflict with fractional edge strengths; justification/ablation is missing.
- Architecture details: Key hyperparameters (D, N, Y, T, number of new nodes per step, maximum graph size), optimizer, learning rates, gradient clipping, regularization, and stopping criteria are not specified. The computational/memory cost (which the paper itself notes as a limitation) is not quantified.
- Direct reference: It assumes a predefined mapping from words to node types; the semantics of “node type” vs. “entity instance” are unclear. In places the text suggests one node per entity type; elsewhere multiple nodes per type (e.g., “record” nodes). This ambiguity matters for generalization and for the correctness of Th,direct.

Support for claims
- The central claim that the model “learns complex update rules” is weakened by teacher forcing that injects gold structure at each step. The rule discovery tasks would be compelling, but without full tables, error analysis, or ablations (e.g., no teacher forcing, no direct-reference, varying propagation depth), it is difficult to assess the true capability.
- The pathfinding result is highlighted, but again leverages strong supervision and task modifications; a fair comparison would decode from text with no gold graphs and show generalization to larger graphs/path lengths.

Presentation issues
- Numerous formatting/typographical errors: broken line wraps (e.g., Section 4), truncated algorithms (Algorithm 1/2 contain undefined steps, e.g., T_queryh; missing code blocks), cross-references to absent tables/figures, and a severely corrupted bibliography (duplicated author strings, punctuation placeholders such as “?, \Q2016\E”). This is beyond minor polish and impedes readability.
- Reproducibility: Although a GitHub link is mentioned, the paper must be self-contained enough to evaluate without external code.

Suggestions for improvement (not all required for acceptance)
- Provide a clean end-to-end setting: no gold graphs, no direct-reference oracle; report accuracy, learning curves, and compute. If strong supervision is used, clearly separate and label those results and avoid comparisons to end-to-end baselines.
- Remove task modifications or report results both with and without them; justify any changes.
- Clarify all dimensions and correct equations (GRU, loss for unordered sets, propagation). Provide a principled derivation for the edge update rule, or ablate simpler alternatives (e.g., independent Bernoulli logits).
- Add ablations: with/without teacher forcing; with/without Th,direct; varying Tprop; using sparse C; replacing f_fwd/f_bwd by fixed linear maps.
- Report full experimental details (hyperparameters, seeds, variance), wall-clock and memory usage vs. graph size, and error analyses (e.g., where task 17 fails).
- Improve the exposition and fix all reference/formatting issues; ensure all tables/figures are present.

Questions for the authors
1) What are the exact definitions and counts of “node types” (N) across tasks? Are types per-entity, per-class, or shared across instances? How many nodes per type typically exist?
2) Can you report results without replacing nodes/edges by gold (i.e., no teacher forcing), both on bAbI and the rule-discovery tasks?
3) How is direct reference implemented when entities are unseen or have multi-word names? How does performance change without name-based supervision?
4) Please reconcile the Lnode objective: should it be min over permutations of a positive loss? Provide the exact formula you implement.
5) Provide complete hyperparameters and compute/memory usage, and variance across seeds.
6) For the pathfinding task, what is the distribution of path lengths at test time, and how does accuracy scale with path length and graph size?
