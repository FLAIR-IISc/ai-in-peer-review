### Summary

The paper introduces the Gated Graph Transformer Neural Network (GGT-NN), a model designed to learn and manipulate graph-structured internal states from unstructured input, such as natural language. The core contribution is a set of five differentiable graph transformations: node addition, node state update, edge update, propagation, and aggregation. These transformations can be composed to form a network that incrementally builds a graph from a sequence of sentences and then uses this graph to answer a query. The author demonstrates the model's capabilities on three distinct problem domains: the bAbI question-answering dataset, learning the update rules of a 1D cellular automaton, and simulating arbitrary 4-state Turing machines. The paper claims that by using an explicit graphical representation, the GGT-NN can outperform other models on tasks naturally suited to graphs (e.g., pathfinding) and that its internal state is more interpretable than that of other memory-augmented networks.

### Decision and Key Reasons

**Decision:** Reject

**Key Reasons:**

1.  **Severe Presentation and Formatting Deficiencies:** The manuscript is in a state that is unacceptable for peer review. It lacks an abstract, and all figures (1-7) and tables (1-4) referenced in the text are missing. This makes it impossible to verify any of the quantitative claims or to visualize the proposed model and results. Furthermore, the reference section is malformed and largely unreadable. These omissions and errors fundamentally undermine a rigorous assessment of the work.
2.  **Lack of Clarity and Methodological Rigor:** The paper suffers from numerous minor technical and notational issues that accumulate to create significant ambiguity. Key equations contain typos and non-standard notation. More importantly, the framing of the supervision signal is potentially misleading. The model is presented as learning to "construct" a graph, but details in the appendix suggest that a substantial amount of manual, task-specific engineering was used to design the target graph schemas, a critical detail that is not sufficiently addressed in the main text.

### Supporting Arguments

**1. Unacceptable Manuscript Preparation:** A submission to a top-tier conference should be polished and complete. This paper fails on several fundamental counts:
*   **Missing Components:** The absence of an abstract, all figures, and all tables is a critical flaw. The entire empirical evaluation rests on Tables 1, 2, 3, and 4, none of which are present. The model's operations and outputs, meant to be illustrated in Figures 1-7, cannot be understood visually. Algorithm 1 is not formatted as a proper algorithm but rather as poorly structured inline text.
*   **Formatting Errors:** The bibliography is completely broken, appearing as raw, unprocessed output from a reference manager (e.g., `Andrychowicz and Kurach.,? \Q2016\E, Andrychowicz and Kurach.,`). This is not a minor cosmetic issue; it makes it impossible to properly identify and consult the cited literature.
*   **Unprofessional Style:** The consistent use of the first-person singular ("I") throughout the paper is unconventional for a formal research publication and detracts from its professional tone.

**2. Methodological and Technical Flaws:** Beyond the presentation, the paper contains numerous small errors and ambiguities:
*   **Inconsistent and Erroneous Equations:** In Section 2.1, the GRU update equation `h(t) = z h(t−1) + (1− z) h̃(t)` is missing time indices for the update gate `z`, which should be `z(t)`. The use of a simple circle for element-wise multiplication is non-standard. The loss functions in Section 4.1 are written imprecisely; for instance, `C∗ · ln(C)` should be explicitly written as a sum over the elements of the matrices. The notation for `Lnode` is similarly ambiguous.
*   **Overstated Contribution Regarding Graph Construction:** The paper claims the model can "construct the correct graph from the input sentences." However, Appendix C.1 reveals that the target graphs for bAbI tasks involved significant manual pre-engineering. For example, relation names like "is in" were disambiguated into different edge types based on context, and for temporal tasks, a linked-list "record" structure was manually added to the graph schema. This suggests the model is less about learning a graph structure *de novo* and more about learning to populate a heavily constrained, task-specific graph template. This reliance on a "strong supervision" signal that includes significant prior knowledge is a methodological weakness that is not adequately discussed.
*   **Missing Hyperparameter and Implementation Details:** The `Tadd` transformation adds nodes "up to some maximum number of nodes." This maximum number is a crucial hyperparameter that could heavily influence performance, yet it is never discussed or specified for the experiments.

### Additional Feedback

While the paper is not acceptable in its current form, the core idea of composable, differentiable graph transformations is promising and worthy of further development. Should the author choose to revise this work, I would strongly recommend the following:

1.  **Complete Overhaul of the Manuscript:** The paper must be rewritten to include all standard components (abstract, figures, tables), with professional formatting for equations, algorithms, and references. A standard bibliographic tool (e.g., BibTeX with a standard style file) should be used.
2.  **Transparent Framing of Supervision:** Be much clearer and more upfront about the degree of manual engineering involved in creating the target graphs. An ablation study examining the impact of this detailed supervision (e.g., comparing performance with generic vs. task-specific edge types) would greatly strengthen the paper.
3.  **Rigor in Technical Descriptions:** All equations should be carefully checked for correctness and standard notation. The description of the loss function for unordered sets (`Lnode`) should be clarified, explaining precisely how the `max over π` is handled computationally.
4.  **Full Experimental Details:** Provide a complete description of the experimental setup, including all key hyperparameters, model dimensions, and optimization details, to ensure reproducibility.

### Questions for the Author

1.  Could you please clarify the exact process for designing the ground-truth graphs for the bAbI tasks? Specifically, what rules or heuristics were used to create the different edge types (e.g., "gettable is in actor") and to add auxiliary structures like the history lists for Task 3?
2.  In the node addition transformation (`Tadd`), how is the maximum number of new nodes per step determined, and was this value tuned as a hyperparameter for each task?
3.  How is the maximization over permutations `π` in the `Lnode` loss function implemented? Is an exact solution computed (e.g., via the Hungarian algorithm), or is an approximation used?
