

# Review of "A Set of Tasks for Evaluating End-to-End Goal-Oriented Dialog Systems"

## 1. Summary of Contributions

The paper proposes a set of five synthetic tasks (T1-T5) for evaluating end-to-end goal-oriented dialog systems, grounded in a restaurant reservation domain with a structured knowledge base. The authors introduce an open resource (with a broken URL) that aims to provide a reproducible testing framework for assessing dialog systems' capabilities in handling dialog management, KB querying, and entity manipulation. The authors supplement their synthetic data with two real-world datasets (DSTC2 and Concierge) to validate that their synthetic tasks serve as a good proxy for real-world performance. They evaluate multiple baselines, including rule-based systems, classical IR methods, supervised embeddings, and Memory Networks, with an emphasis on how well models handle out-of-vocabulary entities. The paper makes a strong contribution toward standardizing evaluation for goal-oriented dialog systems.

## 2. Decision

Reject. The paper contains two major flaws that undermine its core contribution: a broken dataset URL that prevents reproducibility, and a contradiction between the text and results regarding Task 4 performance.

## 3. Supporting Arguments

The two most significant issues that warrant rejection are:

1. **Broken Dataset URL**: The paper claims to provide an open resource at "http://fb.ai/babi" (Section 3.1.2), but this URL is non-functional and appears to be a placeholder. This is a critical flaw because the paper's main contribution is an open resource for reproducible evaluation. Without a working URL, researchers cannot access the dataset, making the paper's central claim of "open resource" meaningless. This directly contradicts the paper's stated goal of "favoring reproducibility and comparisons."

2. **Contradiction in Results**: Section 5 claims that "T4 (providing information) becomes solvable because matches can be made to the results of the API call" when using Memory Networks with match type features. However, Table 2 shows that the per-dialog accuracy for T4 with match type features is 0.0%, identical to the version without match type features. If the task becomes "solvable" as claimed, the per-dialog accuracy should be greater than 0.0%. This contradiction undermines the credibility of the paper's results and claims.

Additional minor but noteworthy issues include:

- Inconsistent terminology: The paper uses "goal-directed objective" (Section 1) when it should use "goal-oriented task" or "goal-directed task."
- Inconsistent description of user requests: Section 3.1.1 states that "a user request implicitly defines a query that can contain from 0 to 4 of the required fields," but Section 3.1.2 states "creating a user request by sampling an entry for each of the four required fields," creating confusion about the number of fields.
- Training set size: The authors state "Training sets are relatively small (1,000 examples) to create realistic learning conditions," but 1,000 examples is extremely small for training neural networks on dialog tasks, and this claim is not adequately justified.

## 4. Additional Feedback

To improve the paper, the authors should:

1. Fix the dataset URL to a functional, permanent URL (e.g., a GitHub repository or a dataset repository like Hugging Face) before resubmission.
2. Re-examine the results for Task 4 to resolve the contradiction between the text and Table 2.
3. Justify the small training set size (1,000 examples) more thoroughly, or consider increasing the dataset size to better reflect realistic training conditions.
4. Clarify the inconsistency between the description of user requests (0-4 fields vs. 4 required fields).
5. Fix minor formatting issues in references (e.g., "et al\.,?" should be "et al.,").

## 5. Questions for Authors

1. Can you provide a working URL for the dataset that will persist beyond the conference deadline? If the current URL is a placeholder, what is the actual URL where the dataset will be hosted?

2. In Table 2, the per-dialog accuracy for Task 4 (providing information) is 0.0% for Memory Networks with match type features. How does this align with the claim in Section 5 that "T4 (providing information) becomes solvable" with these features? Please clarify this contradiction.

3. The paper states that "Training sets are relatively small (1,000 examples) to create realistic learning conditions." What evidence do you have that 1,000 examples is a realistic size for training end-to-end dialog models, particularly given that the authors evaluate Memory Networks with up to 4 hops?

4. In Section 3.1.2, you state that "We split types of cuisine and locations in half, and create two KBs." How exactly was this splitting performed? Was it random, or based on some criteria? This is important for understanding the OOV challenge.
