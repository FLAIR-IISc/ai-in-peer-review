- Proposes a systematic, reproducible, lightweight benchmark for end-to-end goal-oriented dialog systems.  
- Identifies a gap: lack of standardized evaluation for goal-oriented (task-oriented) dialog compared to chit-chat.  
- Designs five restaurant-domain tasks (API calls, updating requests, displaying options, providing info, full dialogs) grounded in a KB and reproducible.  
- Validates synthetic tasks on two real-world datasets (DSTC2 and a Concierge service) to show generalization.  
- Evaluates multiple architectures (rule-based, IR, supervised embeddings, memory networks) providing broad comparative analysis.  
- Introduces "match type features" to handle entities and OOV words as a notable contribution.  
- Finds memory networks with match type features perform best overall but still struggle to interpret API results for displaying options (Task 3).  
- Uses rigorous evaluation (per-response and per-dialog accuracy), careful splits, and ablation studies to support conclusions.  
- Decision: Accept — motivated benchmark and comprehensive, rigorous experiments.  
- Suggestion: Clarify relationship between per-response and per-dialog accuracy with concrete failure examples.  
- Suggestion: Expand discussion of benchmark limitations (unexpected user behavior, error handling, negotiation).  
- Suggestion: Strengthen related work comparison, explicitly contrasting with other benchmarks (e.g., Dodge et al. 2016).  
- Suggestion: Provide more detail on the Concierge dataset challenges (typos, off-topic behavior) and their effect on performance.  
- Question: Clarify whether models have access to the KB at test time and how that affects the OOV evaluation.  
- Question: Explain why match type features slightly degrade performance on Task 2 (updating API calls).  
- Question: Describe how the larger candidate set in Concierge was handled and its impact on model performance.  
- Question: Elaborate on the specific difficulties models face interpreting API results to present options and whether this is fundamental or addressable with more data/architectural changes.
