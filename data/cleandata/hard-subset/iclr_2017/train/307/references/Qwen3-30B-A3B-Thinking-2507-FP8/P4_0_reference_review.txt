

# Review

## 1. Summary of Contribution
The paper proposes a new evaluation framework for end-to-end goal-oriented dialog systems, addressing a critical gap in the literature. The authors introduce a set of five carefully designed tasks within the restaurant reservation domain that test specific capabilities required for successful goal-oriented dialogue (issuing API calls, updating requests, displaying options, providing information, and conducting full dialogs). They provide both synthetic data for controlled testing (with in-vocabulary and out-of-vocabulary settings) and real-world data (DSTC2 and a concierge service dataset) to validate the framework's effectiveness. The paper demonstrates that this resource enables reproducible evaluation and serves as a good proxy for real-world performance, with Memory Networks emerging as a strong baseline that still has limitations in handling certain dialog capabilities.

## 2. Decision
Accept. The paper makes a valuable contribution to the field by providing a well-designed, reproducible evaluation framework that addresses a clear gap in the literature. The proposed tasks are well-motivated and systematically test specific capabilities needed for goal-oriented dialog, and the authors provide both synthetic data for controlled experiments and real-world data to validate their framework.

## 3. Supporting Arguments
The paper successfully addresses a critical problem: the lack of standardized evaluation resources for end-to-end goal-oriented dialog systems. The authors clearly articulate why existing datasets (DSTC, ATIS, etc.) are inadequate for evaluating end-to-end models, as they are designed for specific components (like dialog state tracking) rather than full end-to-end evaluation.

The proposed framework is scientifically rigorous. The five tasks are carefully designed to test specific capabilities (dialog management, KB querying, entity handling), with clear definitions of success conditions. The authors provide a thorough experimental evaluation using multiple baselines (rule-based, IR, supervised embeddings, Memory Networks) across both synthetic and real-world data, with appropriate metrics (per-response and per-dialog accuracy). The results demonstrate that the synthetic tasks serve as a good proxy for real-world performance, as the relative performance of models is consistent across synthetic and real datasets.

The resource is well-documented and made publicly available (http://fb.ai/babi), which will be valuable for the community. The paper also makes a strong case for the importance of this work by showing that while end-to-end models can handle some aspects of goal-oriented dialog (like issuing API calls), they still struggle with others (like interpreting KB results to display options), providing clear directions for future research.

## 4. Additional Feedback
- The paper would benefit from a more detailed discussion of the framework's limitations and potential adaptations for other domains. While the authors mention the restaurant domain, a brief analysis of how the framework might generalize to other domains would strengthen the contribution.
- A more in-depth analysis of failure cases, particularly for Memory Networks on Tasks 3 and 5, would provide additional insights into the remaining challenges.
- The authors could include a brief discussion of how their framework might be extended to handle more complex dialog structures or multi-domain scenarios.

## 5. Questions for Authors
1. How would you adapt your framework for a different domain (e.g., hotel booking or flight reservation)? What specific changes would be needed to the task definitions and KB structure?
2. The Concierge dataset appears to have more variability in user requests and dialog structures than your synthetic tasks. Could you elaborate on how the relative performance of models (e.g., Memory Networks vs. supervised embeddings) compares between the synthetic tasks and Concierge, and whether this suggests any additional challenges that need to be addressed?
3. In Task 3 (displaying options), the paper notes that Memory Networks with match type features still fail to solve the task. What specific aspects of the KB results interpretation do they struggle with, and how might this be addressed in future work?
4. How does the framework handle dialog management when users provide multiple requests in a single utterance (e.g., "I want a French restaurant in London that's expensive and has a table for six")? Would this require modifying the task definitions?
