- Paper introduces a reproducible, lightweight benchmark of five synthetic restaurant-reservation tasks to evaluate end-to-end goal-oriented dialog models.  
- Tasks target specific capabilities: issuing API calls, updating requests, displaying options, providing information, and full dialogs.  
- Benchmark includes per-response and per-dialog accuracy metrics and an OOV test set to assess generalization to unseen entities.  
- Main contribution is a resource (not a new model) that fills a gap in standardized evaluation for goal-oriented dialog.  
- Validation shows task performance patterns align with results on real-world datasets (DSTC2 and Concierge), supporting the synthetic tasks as useful proxies.  
- Experiments compare multiple baselines (rule-based, IR, supervised embeddings, Memory Networks) and analyze response- and dialog-level errors.  
- Finding: Memory Networks (with match type features) can handle some tasks (API calls, OOV entities) but struggle with interpreting/displaying KB query results (Task 3).  
- Decision: Accept.  
- Key reasons for acceptance: addresses a significant gap with a well-motivated, scientifically rigorous, and reproducible benchmark.  
- Benchmark design strength: task decomposition enables fine-grained error analysis and the OOV sets target a critical challenge.  
- Reproducibility and dataset availability (http://fb.ai/babi) are major strengths that will facilitate future work.  
- Suggestion: provide more explicit justification for choosing these five tasks over other possible dialog phenomena.  
- Suggestion: discuss scalability and how tasks could extend to other, more complex domains (e.g., travel booking).  
- Suggestion: more thoroughly discuss limitations of synthetic data, especially deterministic user behavior vs. real users.  
- Suggestion: expand on why per-dialog accuracy is the more important metric and how single errors can derail dialogs.  
- Suggestion: discuss computation trade-offs of deeper Memory Networks (3–4 hops) vs. practical inference cost.  
- Question: how does performance scale with more training data—would Task 3 improve with larger datasets?  
- Question: what specific aspects of the Concierge dataset (beyond language diversity) make it harder—lack of structured KB, user variability, etc.?  
- Question: what specific error types in Task 3 prevent higher per-dialog accuracy despite decent per-response accuracy?  
- Question: given similar relative performance patterns but lower absolute scores on Concierge, what does this imply about transferring models trained on synthetic data?  
- Question: how would results differ in a generation setting (producing novel responses) versus the current ranking/candidate selection setting?
