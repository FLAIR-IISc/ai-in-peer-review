

# Review

## Summary
This paper introduces a novel dataset and set of tasks designed to systematically evaluate end-to-end goal-oriented dialog systems. The authors propose a set of five synthetic tasks within the restaurant reservation domain, grounded in an underlying knowledge base (KB), to test specific dialog capabilities such as dialog management, KB querying, and entity handling. They also supplement these with real-world datasets (DSTC2 and a concierge service dataset) to validate their synthetic tasks as a proxy for real-world performance. The authors demonstrate that their Memory Network baseline with match-type features outperforms other baselines on these tasks, though per-dialog accuracy remains low. The authors' key contribution is providing a reproducible, well-structured testbed for evaluating goal-oriented dialog systems.

## Decision
Reject. The paper contains multiple technical inconsistencies and insufficient methodological detail that undermine the scientific rigor of the claims.

## Supporting Arguments

1. **Critical methodological omissions**: The paper claims to test "generalization to new entities (restaurants, locations and cuisine types) unseen in any training dialog" but fails to specify how the OOV test sets were constructed. The description states "we split types of cuisine and locations in half" but doesn't clarify whether the split was random, systematic, or based on any semantic criteria. This is a fundamental flaw in the experimental design, as the ability to generalize to unseen entities depends entirely on how the OOV set was constructed. Without this information, the OOV results are scientifically meaningless.

2. **Lack of statistical rigor**: The paper states "Memory Networks with match type features give two performance gains" but provides no statistical significance testing for these gains. Claims like "the pattern of relative performances of methods is the same on Concierge and on our series of tasks" are made without any quantitative evidence (e.g., correlation coefficients, p-values) to support this assertion. This is a critical omission in an empirical paper.

3. **Inconsistent terminology and formatting**: The paper uses inconsistent capitalization ("bAbI" vs. "babI"), inconsistent reference formats ("Lemon et al., 2006; Wang and Lemon, 2013" vs. "Weston et al., 2015b"), and missing figure/table references ("Figure 1", "Table 1" are mentioned but not included). These formatting inconsistencies undermine the paper's professionalism and reproducibility.

4. **Unjustified evaluation metrics**: The paper uses "per-dialog accuracy" as a key metric but fails to justify why this metric is appropriate for goal-oriented dialog evaluation. The paper claims "per-dialog accuracy counts the percentage of dialogs where every response is correct" but doesn't explain why a single incorrect response should invalidate an entire dialog (e.g., in a restaurant booking scenario, a single incorrect response might not prevent successful booking). This metric choice significantly impacts the interpretation of results.

5. **Missing critical details**: The paper states "the model can provide correct responses for greeting messages, asking to wait, making API calls and asking if there are any other options necessary" but provides no examples of these correct responses. Without concrete examples, it's impossible to verify the claims about the model's capabilities.

## Additional Feedback

1. **Clarify OOV construction**: The paper must provide a detailed explanation of how the OOV test sets were constructed, including whether the split was random, semantic, or based on other criteria. This is essential for reproducibility.

2. **Add statistical significance testing**: Include statistical tests (e.g., t-tests, paired comparisons) to support all performance claims, particularly regarding the impact of match-type features.

3. **Define metrics clearly**: Provide a formal definition of "per-dialog accuracy" and justify why this metric is appropriate for goal-oriented dialog evaluation.

4. **Fix formatting inconsistencies**: Standardize all terminology, capitalization, and citation formats. Ensure all referenced figures and tables are included.

5. **Include concrete examples**: Provide specific examples of correct and incorrect responses for the Memory Network model to illustrate the claims about its capabilities.

## Questions for Authors

1. Can you provide a detailed description of how the OOV test sets were constructed, including the specific criteria used to split the cuisine types and locations? Was the split random, semantic, or based on some other criterion?

2. Can you provide statistical significance tests for the performance differences between Memory Networks with and without match-type features across all tasks?

3. Why is "per-dialog accuracy" (requiring all responses to be correct) the appropriate metric for evaluating goal-oriented dialog systems? How does this metric correlate with actual user satisfaction or task completion in real-world applications?

4. Can you provide specific examples of the "correct responses for greeting messages, asking to wait, making API calls" mentioned in Section 5, to verify these claims?
