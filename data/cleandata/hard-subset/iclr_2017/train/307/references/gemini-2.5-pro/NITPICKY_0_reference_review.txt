### **Summary of Contributions**

The paper introduces a set of tasks and an open-source dataset aimed at evaluating end-to-end goal-oriented dialog systems. The primary contribution is a reproducible and lightweight testbed, inspired by the bAbI tasks, designed to diagnose the capabilities and weaknesses of current models in a controlled restaurant reservation setting. The authors create five synthetic tasks to probe specific abilities like issuing API calls, handling knowledge base (KB) facts, and managing dialog state. A key aspect is the inclusion of an out-of-vocabulary (OOV) test set to evaluate generalization to unseen entities. To bridge the gap between synthetic and real-world data, the authors also adapt the DSTC2 dataset and evaluate on a proprietary "Concierge" dataset, showing that performance patterns on their synthetic tasks are a reasonable proxy for performance on real dialogs. The paper provides baseline results for several models, including IR methods, supervised embeddings, and Memory Networks, concluding that current end-to-end models, while promising, still fall short of reliable per-dialog goal completion.

### **Recommendation: Reject**

While the motivation for this work is strong and the goal of creating a diagnostic benchmark for goal-oriented dialog is highly valuable, the paper is undermined by a critical methodological flaw in its evaluation protocol and a significant number of inconsistencies, ambiguities, and formatting errors.

**Key Reasons:**
1.  **Data Leakage in Evaluation Setup:** The experimental design, as described in Section 3.1.2, includes all bot utterances and API calls from the training, validation, **and test sets (plain and OOV)** in the candidate pool for the ranking task. This constitutes a form of data leakage, as the model is evaluated on its ability to pick the correct response from a list that is guaranteed to contain it. This dramatically simplifies the task compared to a realistic scenario and may inflate performance metrics, making it difficult to assess the true capabilities of the models. The conclusions drawn from these experiments are therefore questionable.
2.  **Lack of Methodological Clarity and Reproducibility:** Several crucial details are either omitted or poorly explained. The process of converting the DSTC2 data, particularly the generation of API calls from dialog states and the artificial injection of ratings, is glossed over. The "match type features" are described in a convoluted manner, and the fundamental architectural choice of representing KB facts as "user utterances" in the Memory Network is relegated to a footnote in the appendix (Footnote 2), obfuscating a key implementation detail.

### **Supporting Arguments**

**Methodological Flaws:**

*   **Candidate Set Construction:** The statement in Section 3.1.2, "Candidates are ranked from a set of all bot utterances and API calls appearing in training, validation and test sets," is a major concern. A benchmark paper must have an unimpeachable evaluation methodology. By including test set answers in the candidate pool, the evaluation measures recognition rather than recall or generation. This setup does not reflect a real-world deployment scenario and potentially masks the difficulty of the tasks. The validity of the entire results table (Table 2) rests on this flawed premise.
*   **Artificial Determinism:** The paper notes that in Task 1, the bot's questioning order is deterministic. In the processing of DSTC2 data, ratings were added to make bot suggestions "consistently predictable". While simplification is the point of a diagnostic dataset, this level of induced determinism reduces the tasks' resemblance to real-world challenges, where systems must handle variability and ambiguity. This should be more prominently discussed as a limitation.
*   **"End-to-End" Misnomer:** The best-performing model, MemNN with "match type features," seems to depart from a purely end-to-end philosophy. These features require an external KB lookup for every word in every candidate to check for entity type matches. This is a form of engineered feature that relies on symbolic matching, moving the solution away from pure representation learning. The paper should be more precise in its terminology.

**Clarity and Presentation Issues:**

*   **Crucial Details in Appendix:** The Memory Network implementation (Appendix A) is critical to understanding the paper's main model. Footnote 2 within this appendix contains a highly non-obvious design choice: "API calls are stored as bot utterances..., and KB facts resulting from such calls as user utterances...". Why are structured, system-retrieved KB facts treated as user input? This is counter-intuitive and requires justification in the main text, not in a footnote.
*   **Missing Information:** The text refers to "Table 1" multiple times (e.g., in Sections 3.1.2, 3.2, 3.3) for dataset statistics, but this table is absent from the submitted paper.
*   **Sloppy Referencing:** The bibliography is riddled with inconsistencies and errors. For example:
    *   Many entries contain stray characters like `et al\.,?`.
    *   The `Kim et al.` reference contains LaTeX-like code (`\Q2016\E`).
    *   The `Bai et al.` and `Dodge et al.` references appear twice at the very end of the list with different and incorrect formatting.
    Such a lack of attention to detail in a submission to a top-tier conference is concerning and detracts from the paper's credibility.

### **Additional Feedback for Improvement**

*   The core idea of a diagnostic benchmark is excellent. The paper would be much stronger if the evaluation methodology were corrected. The candidate set at test time should consist only of responses seen during training, or a much larger set of plausible distractors that does not include the ground truth by default.
*   The description of the "match type features" in Section 4.5 needs to be rewritten for clarity. A simple, concrete example would be beneficial.
*   The rationale for treating KB facts as user utterances in the MemNN memory needs to be explicitly stated and defended in the main methods section.
*   Please thoroughly proofread the manuscript to fix the numerous formatting errors, especially in the reference list, and ensure all mentioned tables are included.

### **Questions for the Authors**

1.  Could you please justify the decision to include candidates from the test set in the ranking pool during evaluation? Have you considered an alternative evaluation where the candidate pool is fixed to only those responses seen in the training set? How would you expect the results to change under such a protocol?
2.  Could you elaborate on the process used to convert the DSTC2 dataset? Specifically, how were API calls and their outputs synthesized from the annotated dialog states, and what rules were used to inject restaurant ratings?
3.  What is the reasoning behind modeling KB facts as "user utterances" in the memory representation for the Memory Network? Does this imply the model learns to treat structured data returns and natural user language identically?
