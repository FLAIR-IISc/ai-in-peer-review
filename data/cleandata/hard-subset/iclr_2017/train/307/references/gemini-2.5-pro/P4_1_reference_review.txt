### **1. Summary**

This paper introduces a new benchmark suite for evaluating end-to-end, goal-oriented dialog systems. The authors argue that while end-to-end models have shown promise in open-domain chit-chat, their capabilities in structured, goal-oriented settings are less understood, partly due to a lack of suitable, open, and reproducible evaluation frameworks. To address this, the paper presents a set of five synthetic tasks in the restaurant reservation domain, grounded in a knowledge base (KB). These tasks are designed to be progressively more complex, isolating and testing specific capabilities such as issuing API calls, updating requests, interpreting KB results to present options, providing extra information, and conducting a full dialog. A key contribution is the inclusion of an out-of-vocabulary (OOV) test set to evaluate model generalization to unseen entities. The authors provide a thorough empirical evaluation of several models, including classical IR baselines, supervised embedding models, and Memory Networks. To validate the benchmark's relevance, results are also presented on two datasets derived from real user interactions: the Dialog State Tracking Challenge 2 (DSTC2) and a proprietary concierge service dataset. The findings show that Memory Networks outperform other baselines but still struggle to achieve high per-dialog accuracy, particularly on tasks requiring interpretation of KB results, indicating significant room for improvement in end-to-end models for this setting.

### **2. Decision and Key Reasons**

**Decision:** Accept

1.  The paper introduces a well-designed, open, and reproducible benchmark that fills a clear and important gap in the evaluation of goal-oriented dialog systems. The decomposition of the problem into distinct subtasks provides a valuable framework for detailed model analysis and targeted research.
2.  The empirical study is comprehensive and rigorous. It establishes strong baselines, demonstrates the utility of the proposed tasks for identifying model weaknesses, and validates the benchmark's relevance by showing that performance patterns on the synthetic tasks correlate with those on real-world data.

### **3. Supporting Arguments**

**Clarity:** The paper is exceptionally well-written and easy to follow. The motivation is clearly articulated in the introduction, contrasting the needs of goal-oriented dialog with the focus of prior end-to-end systems on chit-chat. The task designs are explained precisely, and Figure 1 provides an intuitive example of a full dialog flow. The description of the models and experimental setup is detailed and sufficient for reproducibility.

**Technical Soundness:** The methodology is sound. The creation of the synthetic dataset is principled, based on a structured KB and a user simulator, which allows for controlled experiments. The design choice to create parallel in-vocabulary and out-of-vocabulary (OOV) test sets is a major strength, as it directly addresses the critical challenge of generalization to new entities—a common failure mode for embedding-based models. The evaluation metrics, per-response and per-dialog accuracy, are appropriate for assessing goal-oriented success.

The experimental evaluation is robust. The paper compares a sensible range of models, from simple IR methods (TF-IDF, Nearest Neighbor) to learned models (Supervised Embeddings) and a then-contemporary end-to-end architecture (Memory Networks). The inclusion of a rule-based system for the DSTC2 task provides an important point of reference, demonstrating that on more complex, real-world data, learning-based approaches can surpass hand-coded logic. The analysis of results is insightful, correctly identifying where models succeed (e.g., issuing API calls) and where they fail (e.g., interpreting KB results to display options, leading to low per-dialog accuracy). This supports the central claim that the benchmark is effective for error analysis. The validation on DSTC2 and the Concierge dataset strongly supports the claim that the synthetic tasks are a good proxy for real-world performance, as the relative ranking of the models remains consistent.

**Novelty:** The primary contribution is the dataset and evaluation framework itself, which is a novel and valuable resource for the community. While inspired by prior work like the bAbI tasks, its application to goal-oriented dialog, with specific subtasks for KB interaction, dialog management, and OOV generalization, is a novel and impactful contribution. It moves beyond simple question-answering towards more complex, multi-turn interactions that involve state tracking and interaction with external knowledge sources. The proposed "match type features" are a simple but clever technique to improve OOV handling, further demonstrating the paper's focus on addressing practical challenges in end-to-end dialog.

### **4. Additional Feedback and Questions**

The following points are intended to help improve the paper and are not critical to the decision.

**Feedback:**
*   **Evaluation Setting:** The evaluation is performed in a ranking setting over a fixed set of all possible candidate responses. While this is a standard and practical approach for evaluation, it differs from a real-world deployment where a model would need to generate a response from scratch. It would be beneficial for the authors to briefly discuss this limitation and the potential challenges of moving towards a generative evaluation on these tasks.
*   **User Simulator Complexity:** The user simulator is based on a fixed set of patterns. This ensures control and reproducibility but may not capture the full diversity of real user language. A short discussion on the limitations of this pattern-based approach and potential future work involving more complex, perhaps learned, user simulators would strengthen the paper.
*   **Concierge Dataset Details:** The description of the Concierge dataset is clear, but one detail could be expanded. The paper notes that API call results are not included in the dialog history. This is a significant departure from the synthetic tasks. It would be interesting to briefly speculate on how this affects the modeling challenge—for instance, does it require the model to implicitly learn and store entity information in its parametric weights rather than retrieving it from memory?

**Questions for the Authors:**
1.  The candidate set for ranking is comprised of all bot utterances from the combined training, validation, and test sets. Could you clarify the size of this candidate set? Also, does including test set candidates during the training-time ranking process not risk some form of information leakage, even if the model is only trained on training dialogs?
2.  The "match type features" (Section 4.5) are shown to be effective, especially for OOV entities. The paper states they are "added to the candidate representation." Was this choice deliberate? Did you experiment with adding similar features to the input/memory representations as well, which might help the model's controller (query vector) to better focus on relevant entity types during memory lookups?
3.  On the DSTC2 task (T6), the Memory Network with match type features does not improve over the one without. This contrasts with the OOV tasks where the features provided a significant boost. Do you have any hypotheses for why this might be the case? Does it suggest that the type of OOV challenges in DSTC2 are different from those in your synthetic OOV split?
