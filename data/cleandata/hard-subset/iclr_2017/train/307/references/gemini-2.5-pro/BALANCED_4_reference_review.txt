### **Summary of Contributions**

This paper introduces a valuable new resource for the community: a suite of tasks designed to test and analyze end-to-end goal-oriented dialog systems. The authors are motivated by the observation that while end-to-end models have shown promise in open-domain chit-chat, their capabilities in structured, goal-oriented settings are not well understood. To address this, they propose a set of six tasks in the restaurant reservation domain. Five of these are synthetically generated, inspired by the bAbI QA dataset, and are carefully designed to isolate and evaluate specific system capabilities such as issuing API calls (Task 1), updating them (Task 2), presenting options from a knowledge base (Task 3), providing extra information (Task 4), and conducting a full dialog (Task 5). A key strength of this benchmark is the inclusion of an out-of-vocabulary (OOV) test set to measure generalization to unseen entities. To ground these synthetic tasks in reality, the authors also include a sixth task based on the real human-bot dialogs from the DSTC2 dataset and further validate their findings on a proprietary dataset of human-human conversations from a concierge service. The paper provides a thorough evaluation of several baselines, from classical IR methods to supervised embeddings and Memory Networks, offering insightful analysis into their respective strengths and weaknesses. A key finding is that while modern end-to-end models achieve encouraging per-response accuracy, their per-dialog accuracy remains very low, highlighting the significant challenges that remain in building reliable goal-oriented agents.

### **Decision and Key Reasons**

**Decision: Accept**

1.  **Significant and Timely Contribution:** The paper introduces a high-quality, open-source, and much-needed benchmark for goal-oriented dialog. The lack of standardized and reproducible evaluation frameworks has been a significant bottleneck for progress in this area. The bAbI-style decomposition of the problem into sub-tasks is an excellent design choice that facilitates targeted model development and fine-grained error analysis.
2.  **Thorough and Insightful Empirical Analysis:** The paper presents a rigorous experimental evaluation with strong baselines that establishes the difficulty of the tasks and provides valuable insights. The analysis effectively highlights the gap between per-response and per-dialog performance, pinpointing a core challenge for future research. Furthermore, the validation on real-world datasets (DSTC2, Concierge) convincingly demonstrates that the synthetic tasks are a good proxy for real-world challenges, reinforcing the value of the proposed benchmark.

### **Supporting Arguments**

The primary contribution of this work is the dataset and task suite itself, which is a substantial asset for the community. Goal-oriented dialog is of immense practical importance, but research has been hampered by a lack of suitable, open benchmarks. Existing datasets are often proprietary, noisy, small-scale, or not designed for end-to-end training. This work directly addresses this gap. The design is thoughtful, particularly the inclusion of an OOV split, which forces models to generalize beyond memorizing entities seen in training dialogs—a crucial capability for real-world deployment.

The experimental methodology is sound and the analysis is compelling. The authors compare against a sensible and comprehensive set of baselines, including rule-based systems, classic IR, learned embeddings, and an end-to-end Memory Network. This provides a clear picture of where different architectural paradigms succeed and fail. The paper's conclusion that current end-to-end models struggle with multi-turn consistency (low per-dialog accuracy) despite reasonable single-turn performance (per-response accuracy) is an important and well-supported takeaway. The simple but effective "match type features" proposed to improve OOV handling is a nice secondary contribution that further strengthens the Memory Network baseline.

Finally, the paper is exceptionally well-written and clear. The motivation is well-articulated, the task descriptions are precise, and the results are presented and interpreted lucidly. This work sets a clear path forward, providing both the tools (the dataset) and the direction (the analysis of model failures) for the field to tackle the challenges of building robust, end-to-end, goal-oriented dialog systems.

### **Additional Feedback for Improvement**

These points are intended to be constructive suggestions and do not detract from my positive assessment of the paper.

*   **Complexity of Simulated Language:** The user simulation relies on a fixed set of 43 patterns. While this is useful for creating a controlled testbed, it is linguistically simple compared to real users. It would be beneficial to add a brief discussion in the conclusion or future work section about the limitations of this templated approach and how the benchmark could be extended in the future, perhaps by using paraphrastic generation to increase linguistic diversity.
*   **Alternative OOV Handling:** The "match type features" are a practical and effective solution for handling OOV entities when a KB is present. For a more complete picture, it might be interesting to briefly discuss or contrast this with other potential approaches for OOV handling that don't rely on an explicit type system, such as character-level or subword-based models, even if experiments are outside the scope of this paper.
*   **From Ranking to Generation:** The evaluation is framed as a ranking task over a candidate set. This is a standard and reasonable choice, but a short discussion on the additional challenges that a full generation-based evaluation would entail for these tasks could provide useful context for future work aiming to solve them in a generative setting.

### **Questions for the Authors**

I would appreciate it if you could clarify the following points during the author response period.

1.  The performance on Task 3 (Displaying options) is particularly low (0% per-dialog accuracy). This task requires sorting and iteratively presenting options based on user feedback. Could you provide a more detailed error analysis for this specific task? Is the model failing at retrieving the correct restaurant names, sorting them by rating, tracking which options have already been offered, or a combination of these?
2.  The user simulator has a 25% chance of accepting a suggested option (unless it's the last one). How sensitive are the results, particularly for Tasks 3 and 5, to this specific probability? Have you explored how dialog length and task difficulty change if this parameter is varied?
3.  For the Concierge dataset, you note that API calls do not have arguments and that operators do not always use an external service. How does the model learn to handle cases where no API call is necessary? Is there an explicit "no_api_call" action in the candidate set, or is the model expected to learn to select a regular chat response instead?
