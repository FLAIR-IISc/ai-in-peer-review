### 1. Summary of Contributions

This paper introduces a valuable open-source benchmark for evaluating end-to-end, goal-oriented dialog systems. The authors argue that existing resources are insufficient for systematically assessing the capabilities of such systems, particularly in comparison to traditional, slot-filling approaches. To address this, they propose a suite of six tasks within the domain of restaurant reservations. The first five tasks are synthetically generated to test distinct, crucial abilities: issuing API calls (T1), updating API calls (T2), presenting sorted options from a knowledge base (KB) (T3), providing extra information from a KB (T4), and conducting a full dialog (T5). The synthetic data is generated in a controlled manner, allowing for clear error analysis and the creation of out-of-vocabulary (OOV) test sets to evaluate generalization to unseen entities. The sixth task (T6) is derived from the real human-bot dialogs of the DSTC2 dataset, and an additional real-world, human-human dataset ("Concierge") is also used for evaluation. The paper provides a thorough empirical evaluation of several baseline models—including rule-based, information retrieval, supervised embedding, and Memory Network models—on this new benchmark. The results demonstrate the utility of the tasks for diagnosing model weaknesses and confirm that while modern end-to-end models show promise, they still struggle with key aspects of goal-oriented dialog, especially in achieving full dialog success.

### 2. Decision and Key Reasons

**Decision:** Accept

**Key Reasons:**
1.  **Valuable Resource Contribution:** The paper provides a well-designed and much-needed public benchmark for a critical research area. The breakdown of the complex problem of goal-oriented dialog into a set of distinct subtasks is a significant contribution that will enable more focused research, systematic model development, and clearer error analysis for the community.
2.  **Strong and insightful Experimental Analysis:** The paper presents a comprehensive set of experiments with appropriate baselines. The analysis clearly supports the central claims: that the proposed tasks are useful for gauging model capabilities, that performance on synthetic data is a reasonable proxy for performance on real data, and that current end-to-end models, while promising, still have significant shortcomings in reliability for goal-oriented interactions.

### 3. Supporting Arguments for the Decision

The paper's contribution is both technically sound and highly relevant to the field.

**Clarity:** The paper is exceptionally well-written. The motivation for the new benchmark is clearly articulated, and the design of each task is described in sufficient detail to be easily understood. The use of figures and tables effectively communicates the task structure and experimental results. The claims are stated directly and are well-supported by the subsequent analysis.

**Technical Soundness:** The methodology is robust. The design of the synthetic tasks is principled, isolating specific skills required for goal-oriented dialog. The creation of an OOV test set is a critical feature that allows for a rigorous assessment of a model's ability to generalize, a known weakness of many embedding-based approaches. The selection of baseline models is comprehensive, ranging from classic IR methods to a strong end-to-end neural architecture (Memory Networks). The evaluation metrics—per-response and per-dialog accuracy—are well-chosen, as the stark difference between them effectively highlights the difficulty of achieving robust, multi-turn success. The inclusion of experiments on real-world datasets (DSTC2 and Concierge) validates the central premise that insights gained from the synthetic tasks are applicable to more realistic settings.

**Novelty and Impact:** While the concept of creating synthetic "toy" tasks to probe AI capabilities is not new (e.g., the bAbI tasks), its application to decompose the challenges of goal-oriented dialog is a novel and impactful contribution. Currently, progress in this area is hampered by a lack of standardized, reproducible, and interpretable evaluation frameworks. This work directly addresses that gap. By open-sourcing the data and tasks, the authors provide a durable resource that can foster reproducible research and serve as a standard testbed for future models. The finding that even a strong model like Memory Networks fails on seemingly simple tasks like presenting sorted options (T3) provides a clear and actionable challenge for future work.

### 4. Additional Feedback for Improvement

These points are intended as constructive feedback and are not critical to the decision.
*   The evaluation is performed in a ranking setting over a candidate set. While this is a standard and practical approach, a brief discussion of its limitations would be helpful. For example, how might the size and composition of the candidate set influence results? Acknowledging that this setup does not test pure generative capabilities would add context for the reader.
*   The "match type features" are shown to be crucial for handling OOV entities. This approach could be viewed as a form of feature engineering. It would be interesting to see a short discussion on the trade-off between injecting this kind of explicit knowledge versus developing models that can learn such matching behavior in a more purely end-to-end fashion.
*   The paper notes that the rule-based system for Task 6 was optimized on the validation set. Clarifying the complexity of this process (e.g., approximate person-hours to build and tune) would provide a stronger reference point for the effort required compared to training the machine learning models.

### 5. Questions for the Authors
*   The candidate set for evaluation is composed of all bot utterances and API calls from the combined training, validation, and test sets. Could you state the size of this candidate set? Does this construction methodology not carry a risk of leaking information from the test set, potentially making the ranking task easier than if candidates were drawn only from the training set?
*   For the OOV evaluation, Section 4.5 states that match type features can be used "as long as it has access to a KB with the OOV entities." To clarify, during OOV testing, is the model provided with the new KB that contains the unseen entities, or must it identify them without this explicit resource?
*   The 0% per-dialog accuracy on Task 3 (Displaying options) across all models is striking. The task description mentions that the bot must list restaurants sorted by rating. Is this sorting requirement the primary reason for failure? Have you performed any analysis to see if models can suggest the correct *set* of restaurants but in the wrong order?
*   Could you elaborate on the MemN2N architecture used? Specifically, Appendix A mentions encoding time and speaker information by extending the vocabulary. Was this done by adding dedicated tokens to the bag-of-words representation for each memory slot? A little more detail on this implementation would be helpful.
