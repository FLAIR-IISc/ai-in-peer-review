This paper introduces a set of tasks aimed at evaluating end-to-end goal-oriented dialog systems. The primary contribution is an open-source testbed, in the spirit of the bAbI tasks, designed for the restaurant reservation domain. This testbed consists of five synthetically generated tasks that systematically test different capabilities, such as issuing API calls, dialog management, and using a knowledge base (KB). The authors supplement this with a sixth task derived from the real-world DSTC2 dataset and further validate their approach on a proprietary dataset from a human-operated concierge service. The paper provides baseline results for several models, including IR methods, supervised embeddings, and Memory Networks, demonstrating that while modern end-to-end models show promise, significant challenges remain, particularly in handling OOV entities and interpreting KB results.

**Decision:** Reject

**Key Reasons:**
1.  A critical methodological flaw in the experimental setup concerning the candidate set definition appears to constitute a significant data leak, calling into question the validity of the reported results.
2.  The paper suffers from a substantial number of clarity issues in its methodology, particularly regarding data processing and task design, which severely hinders reproducibility. This is compounded by numerous formatting errors and citation inconsistencies that suggest a lack of careful preparation.

**Supporting Arguments:**

**1. Methodological Flaws and Lack of Clarity:**

*   **Candidate Set Definition and Data Leakage:** In Section 3.1.2, the paper states, "Candidates are ranked from a set of all bot utterances and API calls appearing in training, validation and test sets (plain and OOV) for all tasks combined." This is a major experimental flaw. If the model is evaluated on its ability to rank candidates, and the candidate set includes responses that *only* appear in the test set, the model has access to test-set information during training and validation. This fundamentally compromises the evaluation protocol. At best, this description is extremely unclear; at worst, it invalidates the reported accuracies. The scientific rigor of the paper hinges on this detail, which is not adequately addressed.

*   **Vague Data Processing Procedures:** The conversion of external datasets is under-specified. For the DSTC2 data (Section 3.2), the authors state, "...given the dialog states we created the API calls to the KB and their outputs which we added to the dialogs." The procedure for this inference is not detailed. Similarly, they "added ratings to the restaurants," but the method for assigning these ratings is omitted. These steps are crucial for reproducibility, and their absence makes it impossible for others to replicate Task 6 or to fully understand its difficulty. The same lack of detail applies to the Concierge dataset (Section 3.3), where an aggressive anonymization process removes named entities like locations. For a restaurant booking task, removing location information seems counter-intuitive and its impact on the task's viability is not discussed.

*   **Oversimplification and Mischaracterization of Tasks:** The paper claims Task 3 requires the model "to learn to sort" (Section 3.1.1). However, the bot's behavior is described as deterministically "listing the restaurant names sorted by their corresponding rating (from higher to lower)." The model does not need to learn a general sorting algorithm; it merely needs to learn to produce a fixed, ordered sequence of outputs based on the KB state. This is a subtle but important mischaracterization of the required capability.

*   **Unjustified Modeling Choices:** In Appendix A, the authors note that for the Memory Network, "KB facts resulting from such calls [are stored] as user utterances." This is a peculiar design choice. KB results are system-generated information, not user input. This choice is not justified and could have unintuitive effects on the learning process, as the model might learn to treat system facts and user utterances identically.

**2. Formatting and Presentation Issues:**

*   **Citation Errors:** The reference section is riddled with errors that appear to stem from a faulty BibTeX compilation. Multiple entries contain extraneous characters (e.g., "Bai et al\.,?", "Kim et al\.,? \Q2016\E"). A reference to "Henderson et al., 2014a" ends with a stray parenthesis. Most egregiously, a malformed, partial sentence from the main body of the paper has been inserted into the reference list ("...), we represent each utterance as a bag-of-words..., Dodge, Dodge,? \Q2016\E, Dodge,").
*   **Inconsistencies and Typos:** There are numerous minor inconsistencies throughout the paper. For instance, "rule based" is used interchangeably with "rule-based," and "per dialog" with "per-dialog." Appendix C mentions "best validation validation sets." While individually minor, the high frequency of such errors detracts from the paper's quality and suggests a rushed submission process.

**Additional Feedback:**

Despite the recommendation for rejection, the core idea of creating a diagnostic, bAbI-style benchmark for goal-oriented dialog is highly valuable. The community needs such resources to enable systematic progress. The breakdown of a complex dialog into subtasks is a strong and commendable design principle.

To improve the paper for a future submission, I strongly recommend the following:
1.  Completely clarify the candidate set construction. If there is a data leak, the experiments must be rerun with a corrected protocol (e.g., the candidate set for evaluation on the test set should not include any responses that do not appear in the training or a generic vocabulary).
2.  Provide detailed, step-by-step descriptions of the DSTC2 and Concierge data processing pipelines in an appendix.
3.  Rephrase the claims about the tasks to more accurately reflect the skills being tested (e.g., avoiding the "learning to sort" phrasing).
4.  Conduct a thorough proofread of the entire manuscript to fix all formatting, citation, and grammatical errors.

**Questions for Authors:**

*   Could you please confirm the exact composition of the candidate set used during evaluation? Specifically, when evaluating a model on a test example, are responses that appear *only* in the test set included in the set of candidates to be ranked?
*   Could you elaborate on the heuristic or method used to generate API calls and assign ratings when converting the DSTC2 dataset?
*   What was the rationale for representing KB facts as "user utterances" in the Memory Network's memory implementation? Have you explored the alternative of treating them as system-generated facts with a distinct speaker embedding?
