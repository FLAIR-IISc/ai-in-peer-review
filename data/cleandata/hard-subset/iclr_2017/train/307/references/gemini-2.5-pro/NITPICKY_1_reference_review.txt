### Review

**1. Summary of Contributions**

This paper introduces a new benchmark suite of tasks designed to evaluate end-to-end, goal-oriented dialog systems. The primary contribution is a set of five synthetically generated tasks in the restaurant reservation domain, grounded in a knowledge base (KB), which are designed to systematically test different capabilities such as dialog management, KB querying, and handling of out-of-vocabulary (OOV) entities. The authors supplement these synthetic tasks with two datasets derived from real user interactions: a modified version of the DSTC2 dataset and a proprietary dataset from an online concierge service. The paper provides an empirical evaluation of several models on this benchmark, including classical IR methods, supervised embeddings, and Memory Networks, offering an analysis of the strengths and weaknesses of current end-to-end approaches. The authors claim their resource facilitates reproducible comparisons and provides a useful proxy for performance on real-world tasks.

**2. Decision and Key Reasons**

**Decision:** Reject

**Key Reasons:**

1.  **Critical Flaw in Evaluation Protocol:** The experimental setup suffers from a form of test set leakage. The candidate set used for ranking at evaluation time is constructed from all bot utterances and API calls across the training, validation, *and test sets* (both plain and OOV). This means the model is evaluated on its ability to pick the correct response from a list that is guaranteed to contain it, and which includes responses it has never seen in training. This fundamentally invalidates the reported accuracy metrics, as performance is likely inflated and does not reflect a model's true generative or predictive capability in a realistic setting.

2.  **Contradiction of Reproducibility Goals:** While a stated aim is to provide a resource that "favors reproducibility and comparisons," several key components of the work are not reproducible. The "Concierge" dataset is proprietary, preventing verification of the claim that the synthetic tasks are a good proxy. Furthermore, the methodology for converting the DSTC2 dataset and for creating the rule-based baseline is described too vaguely to be replicated.

**3. Supporting Arguments for Decision**

The core idea of creating a diagnostic, bAbI-style testbed for goal-oriented dialog is commendable. However, the execution is undermined by significant methodological and reporting flaws.

*   **Invalidity of Quantitative Results due to Candidate Set Construction:** In Section 3.1.2, the paper states: "Candidates are ranked from a set of all bot utterances and API calls appearing in training, validation and test sets (plain and OOV) for all tasks combined." This is a critical error. In a real-world deployment, a dialog system does not have access to a pre-compiled list of ground-truth responses for the conversations it will encounter. A valid evaluation protocol would involve ranking candidates drawn from the training set vocabulary only, or perhaps from a predefined grammar, but certainly not from the test set itself. This leakage makes it impossible to trust the absolute performance numbers reported in Table 2 or to fairly compare the different models, as their ability to handle unseen outputs is not being properly tested.

*   **Lack of Reproducibility:**
    *   **Proprietary Data:** The evaluation on the proprietary "Concierge" dataset (Section 3.3) is used to support a central claim: "the pattern of results observed in our tasks is indeed a good proxy for what would be observed on real data." Since this dataset is not public, this claim is unverifiable by the research community. This directly conflicts with the paper's goal of creating an open and reproducible resource.
    *   **Opaque Data Processing:** The conversion of the DSTC2 dataset is non-trivial but is described in a single, high-level paragraph (Section 3.2). Key details are missing, such as the exact rules used to "create the API calls to the KB and their outputs" from the dialog states, or the method for adding artificial ratings. Without these details, the resulting "Task 6" cannot be precisely reproduced.
    *   **Opaque Baselines:** The rule-based system for Task 6 is described as "28 rules and... 21 patterns" (Section 4.1), but the rules themselves are not provided. This makes the baseline a black box and prevents a fair comparison or replication of the result where MemNNs outperform it.

*   **Task Design Simplifications:** The synthetic tasks contain several simplifications that limit their realism. For instance, in Task 1, the bot is said to ask for information "in a deterministic order," which significantly constrains the problem. In Task 3, user behavior is hard-coded ("Users have a 25% chance of accepting," "Users always accept the option if this is the last remaining one"). While simplifications are necessary for controlled experiments, these feel overly restrictive and may not encourage the development of models robust to more varied dialog flows.

**4. Additional Feedback for Improvement**

*   **Formatting and Presentation:** The paper lacks the final polish expected for a top-tier conference.
    *   There are inconsistencies in hyphenation (e.g., "endto-end" in Section 1 vs. "end-to-end" elsewhere).
    *   The bibliography contains numerous formatting errors, such as stray question marks, incorrect BibTeX escape sequences (`\Q2016\E`), and inconsistent author formatting (e.g., `Bai et al\.,?`). A thorough cleaning of the `.bib` file is required.
    *   The notation for the supervised embedding model's scoring function, `(Ax)>By`, is non-standard. Standard matrix notation (e.g., `(Ax)^T(By)`) should be used for clarity.

*   **Methodological Clarity:**
    *   In Appendix A, the decision to map KB facts to "user utterances" is highly counter-intuitive and should be explicitly justified in the main text.
    *   A concrete example of how "match type features" are applied (Section 4.5) would greatly improve comprehension.

*   **Deeper Analysis:** In Section 5, the paper notes that adding match type features hurts the MemNN performance on Task 2 but offers no explanation. A discussion of such negative results would strengthen the analysis.

**5. Questions for the Authors**

1.  Could you please clarify the reasoning behind including ground-truth responses from the test set in the candidate pool during evaluation? Have you performed experiments with a more realistic candidate set (e.g., limited to training set responses), and if so, how do the results change?
2.  To align with the paper's goal of reproducibility, would you be willing to release the scripts and detailed rules used to convert the DSTC2 data into your task format, as well as the 28 rules comprising your baseline system for Task 6?
3.  What is the justification for modeling KB facts returned from an API call as "user utterances" in the memory of the Memory Network? This seems to conflate external knowledge with user input. What were the alternatives you considered for injecting KB information?
