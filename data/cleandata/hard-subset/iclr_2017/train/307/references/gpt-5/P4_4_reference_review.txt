1. Summary of contributions
- The paper introduces an open, reproducible testbed for end-to-end goal-oriented dialog centered on restaurant booking. It comprises five synthetic tasks (API call issuance and updates, option display with sorting, information provision from KB returns, and a full combination) plus two evaluations on real dialogs (a reformulated DSTC2 and a proprietary concierge dataset).
- The tasks explicitly test capabilities crucial to goal-directed dialog: implicit state tracking without explicit annotations, interaction with a symbolic KB, reasoning over KB returns (including sorting by rating), and handling entities unseen in training dialogs (OOV).
- The authors evaluate strong baselines: TF–IDF, nearest neighbor retrieval, supervised embeddings, and Memory Networks (MemN2N), including a principled extension with match type features to better handle entities and exact matches. They propose per-response and per-dialog accuracy, the latter capturing goal completion.
- Results show MemN2N outperforms other baselines and, with type features, handles OOV entities and KB lookups better; still, per-dialog performance on the hardest tasks remains low. Similar performance patterns transfer to the real datasets, supporting the testbed’s external validity. The resource aims to enable error analysis and progress tracking in a controlled setting.

2. Decision: Accept
Key reasons: (i) Clear, well-motivated benchmark that fills a gap for evaluating end-to-end, KB-grounded goal-oriented dialog with OOV considerations; (ii) Methodologically sound experimentation with insightful analyses showing strengths and limitations, and evidence of transfer to real data.

3. Supporting arguments
- Problem and motivation: Traditional slot-filling/POMDP systems are strong but costly to port; existing end-to-end evaluations focus on chit-chat. The paper targets the missing middle: a reproducible, controlled benchmark for goal-oriented dialog that stresses KB usage and entity handling.
- Technical soundness: Task design is systematic, decomposing capabilities; OOV split is thoughtfully constructed by holding out entity sets in training dialogs while exposing KB entries. Evaluation as next-utterance classification is standard and isolates response selection from generation issues. Metrics (per-response and per-dialog) capture both local and global performance.
- Placement in literature: The work situates itself among bAbI QA tasks, DSTC datasets, and non-goal dialog corpora, arguing for reproducibility and diagnostic evaluation in goal-oriented settings. The addition of match type features addresses known weaknesses of embeddings for entities.
- Evidence: Comparative results are comprehensive across models, tasks, OOV splits, and two real-world datasets. Observations (e.g., multi-hop benefits, failures on sorting/options, gains from type features) are consistent and align with the claimed capabilities and limitations. The rule-based vs ML comparison on real data further supports the case for end-to-end learning in noisier settings.

4. Additional feedback (to help improve the paper; not necessarily decision-critical)
- Clarify candidate set construction: including all bot utterances and API calls from train/val/test across tasks is common in next-utterance classification but can blur experimental boundaries. Please motivate this choice and discuss potential effects (e.g., candidates originating from test-only dialogs).
- Strengthen baselines where possible: For DSTC2, consider comparisons to stronger neural state trackers adapted to the ranking setup or release the rule-based system to allow reproduction. Reporting a generative baseline (e.g., seq2seq with copy) would contextualize ranking results.
- Metric refinements: In addition to per-dialog accuracy, report goal success rate (e.g., whether the correct booking is made even if some earlier responses are imperfect) and a breakdown by error type (API formation, sorting, entity carryover).
- Task realism: The deterministic bot prompts and fixed linguistic patterns ease learning. Consider adding controlled stochasticity (varied question order, paraphrasing, disfluencies) and richer KB schemas to better bridge to real-world variability.
- DSTC2 conversion: Injecting ratings to enable sorting is practical; please discuss the impact on realism and potential biases introduced by this heuristic.
- OOV setting: Since systems have access to both KBs at training time, the OOV challenge is “dialog-OOV but KB-known.” Clarify this design and consider an ablation where OOV entities are unseen in both dialog and KB at training (testing pure generalization via types).
- Analysis depth: Provide more ablations on match type features (which types matter most), hop counts vs performance, and memory size sensitivity. Include qualitative error analysis on Tasks 3/5 (e.g., failures in sorting vs option progression logic).

Questions for the authors
- Candidate set:
  - Do candidate responses include test-only utterances and API calls never observed in training/validation dialogs? How does performance change if candidates are restricted to train/val-only?
- DSTC2 conversion:
  - How were ratings assigned to restaurants, and could this introduce label shortcuts (e.g., always pick highest-rated)? What fraction of DSTC2 turns is affected?
- Sorting and KB reasoning:
  - Can you quantify MemN2N’s errors on Task 3: incorrect sorting vs failing to advance options vs stopping criteria? Does adding explicit supervision for ranking improve outcomes?
- OOV handling:
  - In the OOV setup, since the KB with unseen entities is available at training time, how much of the gain comes from exact matching versus learned type associations? What happens if the OOV KB is withheld until test?
- Realism and robustness:
  - How sensitive are models to increased linguistic variability (more templates, paraphrases) or noise (typos, ASR-like errors) in the simulator? Any experiments injecting such noise?
- Generality:
  - Are code and data generation scripts released to instantiate new domains (e.g., travel, movies)? What is required to extend the KB schema and regenerate tasks?
- Baselines:
  - Did you try pointer/copy mechanisms or key–value memory networks for KB grounding? If so, how did they compare on Tasks 3–5?

Overall, this work provides a valuable, well-executed benchmark and a careful empirical study that is likely to catalyze progress on goal-oriented, KB-grounded dialog.
