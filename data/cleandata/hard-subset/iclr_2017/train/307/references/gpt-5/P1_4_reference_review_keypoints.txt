- Proposes an open, lightweight, reproducible benchmark for end-to-end goal-oriented dialogue focused on restaurant reservation, decomposed into five synthetic tasks plus a DSTC2 conversion and a concierge dataset.

- Benchmark explicitly tests implicit state tracking, KB querying/exploitation, reasoning (e.g., sorting), and OOV entity handling; uses per-response and per-dialog accuracy.

- Provides strong baselines (IR, supervised embeddings, MemN2N) and introduces match-type features for entity/OOV handling.

- Results: MemN2N outperforms retrieval/embedding baselines, solves T1–T2 and T4 with type features, improves on OOV splits, but struggles on option display (T3) and full dialogs (T5); per-dialog accuracy remains low.

- Findings generalize to DSTC2 and proprietary concierge data, supporting the benchmark’s external validity.

- Decision: Accept.

- Acceptance reasons: timely and well-motivated benchmark filling a gap; careful empirical study revealing informative failure modes; open, reproducible resource with likely community impact.

- Targets central question of replacing slot-filling with end-to-end models and argues for controlled, decomposed tasks to enable diagnosis.

- Positions work clearly against POMDP/slot-filling, chit-chat corpora, and QA-style bAbI tasks; motivates match-type features due to embedding limits for entities/OOVs.

- Methodology: clear simulation (KB schema, API semantics, templates), OOV splits, apt evaluation metrics, tuned baselines, and MemN2N ablations.

- Supported claims: end-to-end models can learn API calls and exploit KB facts with types; typed exact matching aids OOV handling; option presentation and full-dialog completion remain challenging.

- Insight/impact: task decomposition and OOV settings help analyze bottlenecks (KB reasoning, multi-step decisions); TF–IDF retrieval underperforms learned models in this setting.

- Suggestion: Clarify candidate set construction, justify inclusion of test-only responses, and report results with train/val-only candidate sets.

- Suggestion: Report variance across runs, confidence intervals, and significance testing for brittle per-dialog accuracy.

- Suggestion: Consider a generation evaluation or discuss how ranking maps to deployable surface realization.

- Suggestion: Discuss deterministic synthetic templates and plans to introduce paraphrase/noise variation to avoid overfitting.

- Suggestion: Analyze potential bias from DSTC2 conversion (ratings enforcing deterministic ordering) and whether models exploit this.

- Suggestion: Release an anonymized or synthetic variant of the Concierge dataset to foster reproducibility on noisier human dialogs.

- Suggestion: Add stronger neural baselines (seq2seq with copy/pointer, Key-Value Memory Networks, retrieval-augmented models).

- Suggestion: Provide detailed error analysis on T3/T5 (sorting, stopping criteria, reliance on ratings vs names) and ablations on memory features.

- Suggestion: Report sensitivity to candidate set size and negative sampling strategies; clarify negatives used for supervised embeddings.

- Suggestion: Ensure long-term availability of datasets, DSTC2 conversion scripts, and MemN2N baseline code with hyperparameters.

- Question: Do models have access to full test candidate sets at inference; how do results change with train/val-only candidates and what is average candidate set size per task?

- Question: How does performance change if OOV KB is only available at test time or if type lexicons are incomplete?

- Question: How were DSTC2 ratings assigned and could this create artifacts or encourage heuristic exploitation?

- Question: How sensitive are MemN2N results to number of hops and memory size; does attention focus on API results and are there qualitative examples showing multiple hops help?

- Question: How robust are models to increased linguistic variability (paraphrases) and noise (typos/ASR errors)?

- Question: Can you report per-dialog success rates conditioned on dialog length/number of turns to quantify compounding error effects?
