Summary of contributions
- Problem and goal: The paper targets the gap between chit-chat dialog modeling and goal-oriented, KB-grounded dialog, asking whether end-to-end models can replace slot-filling systems in narrow domains, and where they fall short.
- Main contribution: A lightweight, reproducible benchmark suite for goal-oriented dialogs (restaurant booking), decomposed into five synthetic subtasks (API call issuing, updating, option display with sorting, KB-based info retrieval, full dialogs), plus two realistic datasets (DSTC2 converted to the task format and a Concierge dataset).
- Evaluation design: Per-response and per-dialog accuracy; an OOV test split that forces generalization to unseen entities by splitting KB entities; ranking-based evaluation with a unified candidate set; explicit error analysis through task decomposition.
- Modeling and baselines: IR baselines, supervised embeddings, MemN2N with multi-hop attention; a simple yet effective “match type” feature to enable exact typed entity matching and handle OOV entities; a rule-based system for DSTC2.
- Empirical findings: MemN2N (especially with match types) outperforms IR and supervised embeddings; models can learn to manage dialog state and issue/update API calls, handle some OOV entities, but still struggle with interpreting KB outputs and executing multi-step behaviors (e.g., sorting and proposing options), yielding low per-dialog accuracy. The same relative ranking of methods appears on the real datasets, suggesting the synthetic tasks are a useful proxy.

Decision: Accept
Key reasons: 
1) The benchmark and OOV design fill an important gap for evaluating end-to-end, KB-grounded, goal-oriented dialog, with strong emphasis on reproducibility and interpretability. 
2) The empirical analysis is careful and substantiates claims: it identifies where current end-to-end models succeed and fail, offers a simple, effective method (match types) for OOV handling, and validates trends on real data.

Supporting arguments
- Significance and timeliness: A standardized, decomposed benchmark for goal-directed dialog with KB interaction and OOV stress tests is valuable for the community, where evaluation has been noisy and not directly comparable. The tasks have already become influential in practice.
- Motivation and placement: The paper is well-motivated and well-situated within POMDP/slot-filling, chit-chat seq2seq, DSTC, and prior bAbI/QA testbeds, articulating why goal-directed evaluation differs from chit-chat and why decomposition is useful.
- Soundness and rigor: The data generation is transparent; OOV construction is principled; metrics target both local and end-goal success; hyperparameters and variants are documented; attention visualizations and ablations on hops support reasoning claims. The DSTC2 conversion and Concierge evaluation bolster external validity.
- Clarity and reproducibility: The paper is clear, with detailed model descriptions and appendices. Public release of Tasks 1–6 enables reproducibility; the Concierge dataset is anonymized but (unclear if) publicly released.

Caveats and limitations (do not outweigh the contribution)
- Language and policy simplicity: Synthetic dialogs use limited patterns, deterministic bot behavior, and a fixed order of information requests. While this aids interpretability, it may encourage template matching and underestimate real complexity.
- Ranking-based evaluation: Selecting from a known candidate set differs from free-form generation at deployment; it can be easier and can leak knowledge by including test candidates in the pool. This should be acknowledged as a limitation of the task format.
- Baseline breadth: Modern generative baselines with copy/pointer mechanisms and structured KB-grounded models are not compared. Given the paper’s focus on the benchmark rather than SOTA, this is acceptable but leaves open how far stronger architectures would go on these tasks.
- Real-data generality: DSTC2 and Concierge analyses are useful, but the Concierge set is proprietary; broader public release (if possible) would strengthen impact.

Additional feedback to improve the paper
- Provide a variant evaluation where the candidate set is drawn only from training to reduce leakage and better simulate deployment; report sensitivity to candidate set size.
- Add or discuss generative baselines with copy/pointer and constrained decoding for API calls; these are natural competitors for OOV and entity-heavy tasks.
- Expand language variability or include noise injection (paraphrases, ASR-like noise) to bridge the synthetic–real gap further, and report robustness.
- Include a data-scaling study (e.g., from 1k to larger training sizes) and an ablation on KB size/density to characterize data efficiency and generalization.
- Clarify release plans for the Concierge dataset or provide a synthetic variant that mimics its variability to enable public benchmarking.
- Separate action selection (dialog policy) from surface realization in the candidate set to disentangle policy competence from templating.

Questions for the authors
- Candidate set construction: You state candidates include all bot utterances and API calls from train/val/test across all tasks. Does this introduce information leakage at test time? What changes if the candidate set is restricted to training-only utterances?
- DSTC2 conversion: How often do the original bots in DSTC2 deviate from deterministic behavior, and how does that impact per-dialog accuracy? Could you share a breakdown of failure modes there?
- Match type features: Can you quantify how much of the OOV gains come from type matching vs. exact lexical matching? Would a pointer/copy mechanism achieve similar or better results?
- Sorting and option display (Task 3): Did you try explicit supervision for sorting (e.g., auxiliary losses) or architectural biases (e.g., differentiable sorting or attention over ranked KB facts)?
- Release and licensing: Will Tasks 1–6 and code be fully released? Can any portion of Concierge be shared, or can you provide a synthetic “Concierge-like” variant?
- Beyond ranking: Have you tried a generative decoding setup on these tasks, and if so, how does performance compare under per-response and per-dialog metrics?
