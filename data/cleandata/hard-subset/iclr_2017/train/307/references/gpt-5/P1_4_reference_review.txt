Summary and contributions
- The paper proposes an open, lightweight, and reproducible benchmark suite for end-to-end goal-oriented dialogue, centered on restaurant reservation. It decomposes the overall goal into five synthetic tasks (issuing/updating API calls, displaying options with sorting, providing KB-grounded information, and full dialogs), plus a converted real-world task from DSTC2 and an additional evaluation on a proprietary concierge dataset.
- The benchmark explicitly tests capabilities crucial to goal-directed dialogue: implicit state tracking, querying/exploiting a KB, reasoning over results (e.g., sorting by rating), and handling OOV entities. It includes both per-response and per-dialog accuracy to reflect goal completion.
- The paper provides strong baselines: IR methods, supervised embeddings, and Memory Networks (MemN2N), including a useful enhancement (“match-type features”) to handle entities and OOVs. The results demonstrate that MemN2N outperform retrieval/embedding baselines, can solve some tasks (T1–T2; T4 with type features), improve on OOV splits, but still struggle on option display (T3) and full dialogs (T5), and have low per-dialog accuracy overall. Patterns largely carry over to DSTC2 and concierge data, supporting the benchmark’s external validity.

Decision: Accept
Key reasons: (1) The benchmark is timely, well-motivated, and fills a clear gap for evaluating end-to-end goal-oriented dialogue with controlled, interpretable subtasks and OOV conditions; (2) The empirical study is careful and reveals informative failure modes that will guide future research; (3) The resource is open, reproducible, and likely to have significant community impact.

Supporting arguments
- Problem and motivation: The paper squarely targets a central question—can end-to-end models replace traditional slot-filling on narrow goal-oriented tasks?—and argues convincingly for controlled, decomposed tasks to enable diagnosis and reproducibility, addressing evaluation challenges in chit-chat.
- Placement in literature: It clearly distinguishes prior work on POMDP/slot-filling, chit-chat corpora, and QA-style bAbI tasks, highlighting the lack of suitable goal-oriented end-to-end benchmarks. The conversion of DSTC2 and comparison to related retrieval settings (e.g., Ubuntu) are appropriate. The “match-type” idea is well-motivated by limitations of low-dimensional embeddings for entities and OOVs.
- Methodology and rigor: The simulation is specified with clear KB schema, API semantics, user/bot templates, and OOV splits. Evaluation metrics (per-response and per-dialog) are apt for goal completion. Baselines are reasonable and tuned; MemN2N variants include ablations with/without type features and multiple hops. Results are coherent across synthetic and real settings, supporting claims: (i) end-to-end models can learn to issue/update API calls and exploit KB facts (with types), (ii) OOV handling benefits from typed exact matching, and (iii) option presentation and full-dialog completion remain challenging.
- Insight and impact: The benchmark’s task decomposition and OOV setting provide a useful scaffold to analyze bottlenecks (reasoning over KB results, multi-step decisions). The observation that retrieval TF–IDF underperforms NN/learned models in goal-oriented settings is instructive.

Suggestions for improvement (to help the paper; not decision-critical)
- Evaluation protocol:
  - Clarify candidate set construction. You state candidates include all bot utterances and API calls from train/val/test (plain and OOV) across all tasks. This implies test-only responses are in the candidate bank at test time. Please justify this choice, discuss implications for realism, and consider reporting results with candidate sets restricted to train/val responses.
  - Provide variance across multiple runs and significance where comparing models. Add confidence intervals for per-dialog accuracy, which can be brittle.
  - Consider complementing ranking with a generation evaluation (even if auxiliary), or at least discuss how ranking results might translate to deployable systems requiring surface realization.
- Data and realism:
  - Synthetic templates make the bot behavior deterministic in T1–T5. Discuss potential overfitting to surface cues and plans to introduce noise/variation (e.g., paraphrases, ASR noise, non-deterministic slot order) while preserving control.
  - The DSTC2 conversion adds ratings to enforce deterministic option ordering. Please analyze potential bias this introduces and whether models exploit this determinism rather than perform generalizable reasoning.
  - The Concierge dataset is proprietary; consider releasing a small anonymized subset or a synthetic variant with similar characteristics to foster reproducibility on noisier human-human dialogs.
- Baselines and analysis:
  - Include stronger neural baselines targeting entity handling and KB grounding, e.g., seq2seq with copy/pointer mechanisms, Key-Value Memory Networks, or retrieval-augmented models with exact-match and span-copy features.
  - Provide more detailed error analysis on T3/T5 (e.g., failures in sorting, stopping criteria, using ratings vs names), and ablations on the time and speaker features in memory.
  - Report sensitivity to candidate set size and negative sampling strategies; for supervised embeddings, clarify N negatives and sampling distribution.
- Release and documentation:
  - Ensure long-term availability of the datasets, conversion scripts for DSTC2, and the MemN2N baseline code with defaults/hyperparameters to match reported numbers.

Questions for the authors
1. Candidate set: Do models have access to the full set of test candidates at inference? How do results change if candidates are restricted to train/val-only responses? What is the average candidate set size per task?
2. OOV setup: You allow access to the OOV KB at training time for typing/matching. Can you quantify performance if the KB is only available at test time (no prior exposure), or if type lexicons are incomplete?
3. DSTC2 conversion: How exactly were ratings assigned, and could this create artifacts (e.g., single deterministic “best” option)? Did you observe model reliance on positional or surface heuristics versus true rating-based sorting?
4. MemN2N details: How sensitive are results to the number of hops and memory size? Do you observe attention focusing on API results in T3/T4 as intended? Any qualitative examples where multiple hops clearly help?
5. Robustness: How do models perform with increased linguistic variability (more templates/paraphrases) or with noise (typos/ASR errors) in the synthetic tasks?
6. Metrics: Could you report per-dialog success rates conditioned on dialog length/number of turns to understand compounding error effects?

Overall, this paper provides a valuable, much-needed benchmark and a careful baseline study; it is well-positioned to catalyze progress in end-to-end goal-oriented dialogue.
