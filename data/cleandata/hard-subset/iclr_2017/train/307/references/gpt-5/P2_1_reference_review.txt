1. Summary of contributions
This paper introduces an open, reproducible benchmark suite to evaluate end-to-end goal-oriented dialog systems. The core contributions are:
- A set of five synthetic but controlled tasks (restaurant booking) that decompose goal-oriented dialog into key capabilities: slot elicitation and updates, API calling, option ranking based on KB attributes, and KB-grounded information provision, plus a combined full-dialog task. The tasks deliberately include OOV entity evaluation via a KB split to test generalization to unseen entities.
- Two complementary real-world datasets: a reformulation of DSTC2 into the proposed task format, and a proprietary “Concierge” human-human dataset, used to validate whether insights from the synthetic tasks transfer.
- Systematic evaluation of baselines spanning rule-based systems, classical IR, supervised embeddings, and end-to-end Memory Networks (MemN2N), including a simple but effective “match type features” mechanism to handle entities and OOVs. Evaluation uses both per-response and per-dialog accuracy, the latter aligning with task completion.
- Findings: MemN2N outperform IR and supervised embedding baselines and can learn nontrivial skills (API calls, limited KB use, OOV handling with types), but still fall short on tasks requiring interpreting and ranking KB results and on full dialogs, especially in per-dialog accuracy.

Overall, the paper fills a notable gap by providing a reproducible, interpretable testbed for goal-oriented dialog and careful empirical analysis of current end-to-end methods’ strengths and limitations.

2. Decision: Accept
Key reasons:
- Significance and timeliness: The paper provides a much-needed, controlled benchmark to evaluate goal-oriented dialog, with task decomposition and OOV testing that enable principled diagnosis—something missing from existing resources focused on state tracking or chit-chat.
- Empirical rigor and insight: Broad, well-justified baselines and clear metrics reveal actionable limitations of end-to-end models, with consistent patterns across synthetic and real datasets, supporting the benchmark’s external validity.

3. Supporting arguments
- Problem and motivation: Clearly articulated. Traditional slot-filling systems lack scalability; chit-chat datasets are a poor proxy for transactional tasks. The proposed tasks directly target essential capabilities (state tracking, API interactions, sorting/option presentation, KB grounding, and OOV handling).
- Positioning in literature: Well placed relative to POMDP/state tracking, bAbI QA, and non-goal-oriented dialog corpora. The paper explains why those are insufficient for end-to-end goal-oriented evaluation and how this benchmark addresses reproducibility and interpretability gaps.
- Methodology and evaluation: Strong. The simulation is transparent; the OOV split is principled; the ranking setting makes evaluation unambiguous; both per-response and per-dialog metrics are informative. The inclusion of both synthetic and real data, with a rule-based baseline for DSTC2, strengthens conclusions.
- Results support claims: Yes. The analyses show where MemN2N succeed (API calls, some OOV handling via types) and where they fail (interpreting KB outputs, option ranking, achieving full-dialog success), aligning with the stated goals of gap analysis rather than SOTA improvement.
- Reproducibility: Data availability and detailed hyperparameters are provided; tasks are deterministic and controllable, which supports consistent comparisons.

Limitations (acknowledged but not fatal):
- The ranking setup with a fixed candidate set simplifies response generation and may not reflect real deployment. However, it enables clear, reproducible comparisons and isolates reasoning skills.
- The simulator’s deterministic bot prompts and limited linguistic variability reduce realism. The authors mitigate this by adding DSTC2 and Concierge and by using OOV splits.
- The “match type features” presuppose access to typed KB entity lists at test time; while pragmatic, this favors models that can exploit exact type matching.
- The real-data conversion (DSTC2) involves post-hoc creation of API calls and ratings; this introduces assumptions and noise. Still, it is used to validate relative trends, not to claim SOTA on DSTC2’s original task.

4. Additional feedback to improve the paper
- Include modern competitive baselines that better handle entities and copying (e.g., pointer-generator, copy-augmented seq2seq, key-value memory networks), to strengthen conclusions about current limitations.
- Report ablations on candidate set size and composition to quantify sensitivity of ranking-based results to search space.
- Deepen error analysis: categorize failures on T3/T5 (e.g., sorting mistakes vs. not conditioning on KB results vs. dialog control).
- Consider adding controlled linguistic variability and non-determinism in the simulator (e.g., multiple paraphrases and question orders) to reduce overfitting to script patterns while keeping reproducibility.
- Clarify whether training jointly on all tasks versus per-task affects performance and transfer.
- If feasible, release code for the simulator and evaluation scripts alongside data to further promote reproducibility.

5. Questions for the authors
- Candidate set construction: Do candidates include all utterances across train/val/test (including OOV test) and across all tasks? How do you prevent leakage from test candidates shaping model learning or biasing evaluation?
- Joint vs. separate training: Did you try multi-task training across Tasks 1–5 to test transfer (e.g., from API-call skills to T5)? Any gains?
- Sorting and KB reasoning: On T3, what proportion of errors are due to incorrect sorting vs. not conditioning on the correct KB subset vs. dialog flow errors?
- Sensitivity to simulator determinism: If you randomize the order of slot elicitation in T1, how do the models degrade?
- Real-data conversion: For DSTC2, adding ratings to enable option ranking imposes a heuristic. How robust are results to different rating assignments?
- Match type features: At test time, do you assume full coverage of OOV entities in the KB? How would performance change if the KB lacked some OOV entities or had noisy typing?
- Per-dialog accuracy: Did you analyze how many turns typically fail per dialog and where? Could curriculum learning or constrained decoding improve per-dialog success?

In sum, the paper meets ICLR criteria well: it tackles a relevant and clearly defined problem, is well-motivated and situated in the literature, and provides rigorous, reproducible empirical support for its claims. I recommend acceptance.
