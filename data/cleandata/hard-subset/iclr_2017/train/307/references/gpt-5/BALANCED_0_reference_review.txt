Summary of contributions
- The paper proposes a reproducible, lightweight benchmark suite for end-to-end goal-oriented dialog, centered on restaurant booking. It decomposes the overall task into five synthetic subtasks (API issuing, updates, option display with sorting, KB grounding for extra info, and full dialogs) plus two evaluations on real-world data (a conversion of DSTC2 and a proprietary “Concierge” dataset).
- The benchmark explicitly tests capabilities critical for goal-directed dialog: implicit state tracking, KB querying, interpreting KB results (including sorting), and handling OOV entities via access to a KB.
- The authors provide baselines spanning IR methods, supervised embeddings, and Memory Networks (MemN2N), and introduce “match type” features to better handle entities and OOV words.
- Empirically, MemN2N substantially outperforms IR and supervised embeddings, solves simpler tasks (T1–T2), benefits from multi-hop reasoning and type features, but still fails on more complex aspects (e.g., option sorting and full dialogs). Similar method rankings carry over to DSTC2 and Concierge, suggesting the synthetic tasks are a useful proxy for real settings.

Decision: Accept
Key reasons:
1) The paper fills a clear gap by providing a well-structured, reproducible benchmark for goal-oriented end-to-end dialog with principled OOV evaluation and links to real data.
2) The experimental analysis is careful and informative, offering actionable insights into where current end-to-end models succeed and fail (notably, reasoning over KB results and multi-step dialog control).

Supporting arguments
- Problem clarity and motivation: The paper directly targets the difficulty of evaluating end-to-end goal-oriented dialog systems beyond chit-chat. The decomposition into subtasks mirrors real requirements (state completion, KB calls, interpreting results, and entity grounding), and the OOV split is thoughtfully designed to test generalization to unseen entities—an acknowledged Achilles’ heel of embedding methods.
- Positioning in literature: The work is well-situated relative to POMDP-based slot-filling, bAbI QA tasks, DSTC datasets, and recent end-to-end dialog approaches. The link to prior bAbI tasks and the explicit distinction between QA and dialog (requiring question-asking and KB use) are helpful.
- Scientific support for claims: Results are reported with per-response and per-dialog accuracy, emphasizing task success rather than turn-level correctness alone; the gap between these metrics is revealing. Ablations (with and without match type features; number of hops) strengthen claims about what capabilities help. The inclusion of both synthetic and real data (DSTC2 and Concierge) supports the proxy claim via consistent method rankings.
- Reproducibility and utility: The synthetic tasks and KB are made available; the annotation scheme, simulation details, and candidate-ranking evaluation are clearly described. This is valuable infrastructure for the community.

Areas for improvement (not blockers)
- Synthetic language and determinism: The simulator uses fixed bot patterns and deterministic question ordering for missing fields. While this controls difficulty, it also reduces realism and may inflate performance on T1. Introducing non-deterministic strategies, paraphrases, and noise would strengthen robustness claims.
- Evaluation setup: Framing as next-utterance ranking with a candidate set that includes test utterances is standard for retrieval, but limits conclusions about generative models and deployment feasibility. Complementary generation-based evaluation (with copy mechanisms/pointers) and success-rate user simulations or human evaluation would broaden impact.
- Baseline coverage: The baseline set omits strong sequence-to-sequence models with attention/copy/pointer networks or neural retrieval cross-encoders that could handle entities and sorting (e.g., explicit pointer/copy for KB grounding). Including them would sharpen conclusions about where MemN2N stands.
- Analysis depth: The error analysis is mostly aggregate. More fine-grained breakdowns (e.g., which step fails in T3/T5; failure modes for OOV vs. in-vocab per entity type; sensitivity to number of options; sorting mistakes vs. selection mistakes) would guide model design.
- Real-data limitations: The Concierge dataset is proprietary and lacks KB outputs, constraining interpretability and reproducibility of those results.

Questions for the authors
- Candidate set: You state the candidate set includes all bot utterances and API calls from train/val/test (plain and OOV). Could you quantify how often gold responses are near-duplicates? Have you tried restricting candidates to train/val only to test robustness?
- Access to KB at test time: For OOV evaluation, are all KB entities known at test time for type matching? How would performance change if only a subset of KB entities were known (more realistic setting)?
- Sorting in Task 3: Did you attempt architectural modifications (e.g., explicit differentiable sorting/selection, pointer networks) or supervision on the ranking of options? Any negative results to share?
- DSTC2 conversion: Are the derived API calls and added ratings made available with the release? How sensitive are results to the rating heuristic?
- Match type features: Can you report ablations disentangling “exact match” vs. “type” cues? For example, does adding only entity-type tags without exact match still help OOV?
- Hyperparameters and stability: How sensitive are MemN2N results to the number of hops, memory size, and embedding dimension? Any variance/sensitivity analysis across random seeds?
- Generalization beyond restaurants: Have you tried cross-domain transfer (e.g., swapping slot types or a new KB schema) to test whether models learn higher-level dialog control vs. domain-specific patterns?

Suggestions to improve the paper
- Broaden baselines with modern copy/pointer mechanisms and retrieval cross-encoders; include at least one generative model evaluated on success rate (with constraints to avoid generic responses).
- Introduce controlled linguistic variability and non-determinism in the simulator (synonyms, paraphrases, stochastic question order), and report robustness.
- Add a deeper error analysis per capability (state tracking, KB grounding, sorting, OOV per entity type), and qualitative failure cases for T3/T5.
- Consider releasing an additional synthetic track with noisy ASR-like perturbations and typos to match DSTC2/Concierge conditions.
- Report confidence intervals or multiple runs for key results, and discuss statistical significance of main improvements.

Overall, this is a timely and useful benchmark contribution with clear insights about the limitations of current end-to-end models in goal-oriented dialog. The resource and analyses are likely to spur follow-up research, warranting acceptance.
