1. Summary of contributions
The paper proposes an open, reproducible testbed for evaluating end-to-end goal-oriented dialogue systems. It decomposes the restaurant-reservation task into five synthetic, controlled subtasks that probe specific capabilities: implicit state tracking (issuing and updating API calls), interpreting and reasoning over KB returns (sorting and selecting options), and retrieving KB attributes. It further adds two datasets with real language: a conversion of DSTC2 into the same task format and a proprietary “Concierge” human-human dataset for external validation. The authors benchmark a spectrum of baselines (IR, supervised embeddings), a rule-based system, and an end-to-end Memory Network (MemN2N) with a simple but effective extension for entity handling (match type features). The study shows that MemN2N can learn to issue and update API calls and handle OOV entities when augmented with types, but still struggles with multi-step reasoning over KB results, yielding low per-dialog success. The dataset is positioned as a reproducible, lightweight resource that enables targeted error analysis and can serve as a proxy for performance trends on more realistic data.

2. Decision
Accept. Key reasons: (i) a well-motivated and clearly presented benchmark suite that fills a gap for goal-oriented, end-to-end evaluation with OOV and KB grounding; (ii) comprehensive and rigorous baseline study that identifies where current models work and where they fail, with corroboration on real data.

3. Supporting arguments
- Problem addressed: The paper targets the lack of standardized, reproducible evaluation resources for end-to-end goal-oriented dialogue with KB grounding. It asks whether such systems can match traditional slot-filling in narrow domains and, if not, where they fall short.
- Motivation and placement: The work is well-situated relative to slot-filling/POMDP systems, non-goal-oriented corpora (Twitter/Reddit/movie dialogs), QA testbeds (bAbI), and DSTC datasets. The design explicitly emphasizes reproducibility, decomposed capabilities, and OOV generalization—areas underserved by existing data.
- Technical soundness: 
  - Task design is explicit, controlled, and tied to a concrete KB with transparent API semantics. The OOV split is principled: train/test KBs share only abstract attributes but not entities, enabling a clean test of entity generalization via type features.
  - Evaluation uses both per-response and per-dialog accuracy; the latter appropriately reflects task completion.
  - Baselines cover strong IR and embedding methods; MemN2N is a plausible end-to-end baseline for multi-hop reasoning; rule-based results establish an upper bound on synthetic tasks.
  - The match type feature mechanism is simple, justified, and empirically impactful for OOV and attribute retrieval.
  - Experimental methodology is reported with hyperparameters and ablations (e.g., hops), and findings on synthetic tasks align with results on DSTC2 and Concierge, strengthening external validity.
- Evidence supporting claims: Results clearly show (a) MemN2N learns API issuance/updates; (b) type features improve OOV handling and attribute retrieval; (c) sorting/option display remains challenging; (d) relative method rankings transfer to real datasets. Claims about strengths/weaknesses and the usefulness of the benchmark are supported by the reported metrics and analyses.

4. Additional feedback to improve the paper (suggestions to help authors; not all affect the acceptance decision)
- Clarify and broaden baselines:
  - Include sequence-to-sequence models with attention and copy/pointer mechanisms, which are directly relevant for entity handling and could be competitive on OOV and KB-attribute retrieval.
  - Consider key–value memory networks or KB-aware “soft retrieval” variants to test whether structured key/value access reduces the gap on Tasks 3 and 5.
  - Report simple ablations for MemN2N (e.g., without time features, without speaker tags) to quantify each design choice.
- Evaluation:
  - Discuss the implications of the ranking setup with a global candidate set including test items. Analyze how results might change under generation with lexical constraints, or under a candidate set restricted to in-domain templates.
  - Provide statistical significance or confidence intervals for main comparisons, at least on DSTC2 and Concierge.
  - Add error breakdowns for Tasks 3/5 to isolate failures (e.g., sorting errors vs. acceptance handling vs. stopping criterion).
- Task realism:
  - Consider introducing controlled nondeterminism and linguistic variability in the synthetic templates to reduce the predictability of bot prompts and to better approximate real-world variability.
  - Explore multi-domain or compositional domains to test portability of learned skills.
  - For Task 3, clarify tie-breaking policy in sorting and evaluate robustness when ratings are close or identical.
- Resource clarity:
  - Ensure release of code for data generation and all baseline implementations.
  - For DSTC2 conversion, discuss potential artifacts introduced by adding ratings and deviations from original bot behavior.

Questions for the authors
- Candidate set construction: Does the candidate pool include all API call strings and bot utterances from both train and test splits (including OOV entities)? Could this create mild test-time leakage in the ranking setting? Have you evaluated with candidate sets restricted to training-only candidates?
- OOV setup: At training time, models have access to the KB containing OOV entities. In realistic deployments this is reasonable, but can you quantify how performance changes if the KB lexicon is incomplete or noisy? How sensitive are type features to lexicon coverage?
- Task 3 specifics: How are rating ties resolved during sorting? Are errors more often due to incorrect ordering, failure to stop when the user accepts, or mis-tracking which options have already been proposed?
- DSTC2 conversion: Since ratings were injected to induce determinism in option selection, could this bias the evaluation against methods that rely on conversational cues rather than ratings? Can you share results under an alternative policy (e.g., original sequence/order) to test robustness?
- Robustness to template variation: How do results change if you increase the number of user/bot templates or introduce paraphrases and noise? Any experiments varying template entropy?
- MemN2N hops and attention: Can you provide a broader ablation over the number of hops across all tasks and analyze attention distributions when the model fails (especially Task 3)?
- Concierge dataset: Are there plans to release an anonymized subset or a synthetic variant that mirrors its statistics and variability to enable public benchmarking?
