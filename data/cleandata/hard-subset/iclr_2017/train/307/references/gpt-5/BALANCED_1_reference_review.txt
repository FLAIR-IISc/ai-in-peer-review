1. Summary of contributions
The paper introduces an open evaluation suite for end-to-end goal-oriented dialog in the restaurant reservation domain. It decomposes the problem into five synthetic tasks that test core capabilities: dialog management (asking for and updating slot values), interacting with a knowledge base (issuing API calls, sorting returned options by rating, extracting attributes), and combining them in full dialogs, with both in-vocabulary and out-of-vocabulary (OOV) test sets. The authors further extend evaluation to real interactions by (i) converting DSTC2 into their task format and (ii) evaluating on a proprietary human-human concierge dataset. They benchmark rule-based, IR baselines, supervised embedding models, and Memory Networks (MemN2N), and propose “match type” features to better handle entities and OOVs. Results show MemN2N outperform IR and supervised embeddings, are competitive with a rule-based system on DSTC2, but still fall short on multi-step reasoning tasks (e.g., ranking options and maintaining per-dialog success). Overall, the work aims to provide a reproducible, interpretable testbed and an analysis of where end-to-end approaches succeed and fail in goal-directed settings.

2. Decision: Reject
Key reasons:
- Limited methodological novelty and analysis: the main contribution is a synthetic dataset and straightforward baselines; the MemN2N variant and match-type features are incremental, and the experimental setup (ranking-only, deterministic templates) constrains what we can conclude about end-to-end dialog modeling.
- Limitations in evaluation design undermine the central claims: synthetic tasks are highly templated and deterministic; the ranking formulation and candidate construction raise concerns about generalization; stronger contemporary neural baselines (e.g., copy/pointer mechanisms) are missing.

3. Supporting arguments
- Question/Problem: The paper tackles whether end-to-end neural models can replace traditional slot-filling systems in narrow-domain, goal-oriented dialogs by proposing a controlled, decomposed benchmark and assessing core competencies (state tracking, KB interaction, entity grounding).
- Motivation and placement: The need for reproducible, interpretable goal-oriented dialog evaluation is well argued and connected to bAbI tasks and DSTC datasets. The focus on KB-grounded operations and OOV entities is appropriate and timely.
- Empirical support: The results convincingly show (i) IR methods underperform, (ii) supervised embeddings are insufficient for multi-step KB use, (iii) MemN2N benefits from multi-hop attention and type features, and (iv) per-dialog accuracy remains low even when per-turn accuracy looks promising. The extension to DSTC2 and Concierge suggests the synthetic tasks can predict relative method trends.
- Concerns on rigor:
  - Synthetic tasks are deterministic with limited linguistic variability (20 bot patterns), and the bot’s behavior is fixed (e.g., deterministic slot inquiry order), making it unclear how methods would fare under more realistic, stochastic policies or paraphrases.
  - Ranking-only evaluation with a global candidate set spanning train/val/test and all tasks conflates recognition/generation and may induce leakage effects (models can exploit surface matches among fixed candidate strings). It also sidesteps whether models can produce compositional API argument strings.
  - The proposed match-type features depend on access to KB lexicons at test time, partially sidestepping the OOV challenge and advantaging methods over settings where only dialog transcripts are available. This blurs the line between end-to-end learning and feature engineering.
  - Missing baselines: No comparison to sequence-to-sequence with attention + copy/pointer networks, retrieval-augmented models, or structured memory architectures designed for entity grounding. Given that Tasks 3–4 require sorting and slot-centric retrieval, such baselines are important.
  - Limited error analysis and ablations: Beyond hop counts, we lack detailed ablations on the role of time features, speaker indicators, memory size, candidate set size, or the precise contribution of KB facts vs. dialog history.

4. Additional feedback (to help improve the paper; not all items are part of the accept/reject basis)
- Strengthen baselines:
  - Include copy/pointer or pointer-generator models for both ranking and generation settings; entity-copying should help Tasks 3–4 and OOV.
  - Evaluate transformer-based retrieval/generation with constrained decoding of API arguments.
  - Add a neural belief-tracking baseline (e.g., RNN/Transformer DST) with explicit state representation and KB querying to contrast implicit vs. explicit state modeling.
- Evaluation design:
  - Provide both ranking and generation results; for generation, use constrained decoding for API calls and compute exact-match metrics for arguments.
  - Report slot-wise and step-wise metrics (e.g., accuracy of each slot in API calls, correctness of option sorting, precision/recall for phone/address extraction) and error taxonomies to pinpoint failure modes.
  - Avoid pooling candidates across tasks and splits; ensure no candidate leakage and report candidate set size sensitivity.
  - Add significance testing and multiple random seeds.
- Dataset realism:
  - Increase linguistic diversity (paraphrases, varied bot policies, typos, ASR-like noise) and introduce stochasticity in slot request order and surface forms.
  - Scale the KB and include distractor facts to assess robustness of attention mechanisms and sorting performance.
  - Consider multi-domain extensions and cross-domain transfer to test generalization.
- Analysis:
  - Deeper ablations on match-type features: what happens if KB lexicons are incomplete? How sensitive are results to entity coverage?
  - Examine why match-type features slightly hurt T2 and articulate best practices for combining exact-match cues with embeddings.
  - Provide qualitative traces showing how models attend to KB facts during sorting and information provision; verify that sorting is learned (not memorized) by perturbing ratings.
- Reproducibility:
  - Release code, data generation scripts, and exact hyperparameters/seeds; clarify licensing for datasets and Concierge usage constraints.

Questions for the authors
- Candidate set construction: Are candidates pooled across tasks and splits? Could this inflate ranking results via surface overlap? Have you tried per-task, per-split candidate pools and measured the delta?
- OOV protocol: During training, do models have access to KB lexicons that include OOV entities appearing only at test time? If so, how do you justify the OOV designation and how would performance change if the KB lexicon were also split?
- Match-type features: How are entity boundaries and normalization handled (e.g., phone numbers, multiword cuisines)? How robust is typing to noisy surface forms in DSTC2/Concierge?
- Sorting in Task 3: Can you demonstrate that models learn to sort rather than memorize a single ranked list (e.g., via counterfactual rating swaps within the same dialog instance)?
- Failure analysis: What are the dominant error types on T3/T5? Does the model fail at (a) issuing correct API calls, (b) mapping facts back to surface forms, (c) maintaining dialog state, or (d) stopping criteria when options are accepted?
- DSTC2 conversion: Adding ratings to enable deterministic ranking may introduce artifacts. How sensitive are results to this augmentation? Could you report results without ratings (e.g., top-k heuristic) or with multiple valid options?
- Scalability: How do methods scale with larger KBs, longer dialogs, and larger candidate sets? Any measurements of memory/computation and latency?
- Generalization: Did you try training on synthetic tasks and testing zero/few-shot on DSTC2/Concierge to evaluate transfer?

Overall, the benchmark is valuable and thoughtfully designed, and the empirical patterns are informative. Addressing the evaluation and baseline gaps would substantially strengthen the paper and its impact.
