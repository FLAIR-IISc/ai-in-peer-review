Summary of contributions
- Problem: Evaluate whether end-to-end dialog models can handle goal-oriented, transactional interactions that require dialog management and KB use.
- Contributions: (i) a lightweight, reproducible testbed comprising five synthetic tasks for restaurant booking (API call issuance/updates, option display with sorting, info provision, and a full pipeline), (ii) a converted version of DSTC2 as Task 6 in the same format, and (iii) additional evaluation on a proprietary “Concierge” dataset. The setup includes an OOV split designed to test generalization to unseen entities.
- Methods: Baselines (TF-IDF, Nearest Neighbor, supervised embeddings) vs. end-to-end Memory Networks (MemN2N) with and without “match type features” that encode entity types and exact match cues.
- Findings: MemN2N outperforms other baselines and can handle API calls and some OOV cases with match types, but per-dialog accuracy remains low on tasks requiring KB interpretation/sorting (T3, T5) and on real data (DSTC2/Concierge).

Decision: Reject
Key reasons: The experimental design has several methodological weaknesses (heavy templating; ranking with a candidate set constructed from test utterances; giving systems access to both KBs including the OOV KB at training; limited, dated baselines; lack of statistical rigor), and the paper exhibits numerous technical/formatting issues. These undermine the strength and generality of the claims despite the potentially useful dataset idea.

Supporting arguments
- Task design and ecological validity:
  - Synthetic tasks rely on a very small set of deterministic patterns (20 bot templates; bot asks in fixed order), making the problems brittle and arguably too “gameable” by template priors. This limits external validity and makes it hard to conclude about real-world goal-oriented dialog.
  - Claim that Task 3 tests sorting is not rigorously established; no controlled analysis showing that models truly learned a compositional sorting procedure vs. memorizing patterns over a small candidate set.
- Evaluation protocol concerns:
  - Ranking rather than generation; candidate set is formed from “all bot utterances and API calls appearing in training, validation and test sets (plain and OOV) for all tasks combined.” Including test utterances in the candidate pool conflates task difficulty and risks leakage of target forms. This setup also departs from realistic deployment where responses must be generated or retrieved from a fixed library not derived from the test set.
  - Per-dialog accuracy is reported but no confidence intervals or statistical significance across runs; no variance/seed reporting.
- OOV setup and KB access:
  - Systems “have access to the training examples and both KBs” during training, yet OOV test dialogs are generated from the second KB. This undermines the intended OOV evaluation because entity strings from the OOV KB are available a priori (e.g., to the match-type mechanism), reducing the novelty of test entities.
  - Match-type features rely on exact matching into the KB; this partially sidesteps the OOV problem by providing oracle-like type signals for unseen entities rather than demonstrating true generalization.
- Baselines and comparisons:
  - Baselines are limited and dated for the claims: no pointer/copy mechanisms, no span-copy/slot-tagging hybrids, no transformer-based retrieval/generation models, no neural belief trackers, and no graph/KB-aware models. The conclusions about the frontier of end-to-end methods are therefore not well supported by the comparisons.
  - The rule-based baseline for DSTC2 is weak (40.7% per-response val acc) and hand-tuned; stronger non-neural baselines (e.g., CRF/RNN slot trackers + template NLG, or retrieval with better features) are missing.
- DSTC2 conversion:
  - The conversion adds ratings not present in the original data to create determinism of “best” options. This alters the task and prevents fair comparison to prior DSTC2 results, limiting the value of the evaluation.
- Reproducibility and completeness:
  - Concierge dataset is proprietary and not released, yet used for key claims about realism.
  - No ablations beyond “match types on/off” and hops. For example, no analysis of time/speaker features, candidate-set size sensitivity, or alternative matching schemes.
- Clarity/formatting issues:
  - Numerous reference formatting errors (“?,” “\Q2016\E,” duplicates), typos, and repeated fragments (e.g., “Dodge, Dodge”), which detract from professionalism and suggest the manuscript is not polished.
  - Some implementation details are relegated to appendices without sufficient main-text transparency (e.g., memory construction nuances, negative sampling settings for embeddings).

Additional feedback to improve the paper (helpful, not all decision-critical)
- Redesign evaluation:
  - Provide both ranking (with a fixed candidate library that excludes test-only targets) and generation metrics; add human evaluation for generated outputs on real data.
  - Report multi-run means/stdev and significance tests.
- Strengthen OOV claims:
  - Do not expose the OOV KB to models during training. Alternatively, justify and quantify the impact of exposing it (ablation).
  - Evaluate models that do not rely on KB string matching (e.g., pointer/copy networks with subword units) and compare to match-type features.
- Baselines:
  - Add modern encoder-decoder with copy, retrieval-augmented transformers, neural belief trackers, span-based slot extractors with constrained decoding, and structured decoders for API calls.
  - Include a strong rule-based/slot-filling pipeline baseline on DSTC2 with standard trackers.
- Task realism:
  - Increase linguistic variability (more paraphrases, noise, typos) and non-determinism in bot behavior in synthetic tasks; control difficulty by varying surface diversity and dialog branching.
  - Provide diagnostics that directly test sorting/compositional reasoning (e.g., controlled generalization tests).
- Reproducibility:
  - Release code, data generation scripts, converted DSTC2, and fixed candidate sets; document seeds; add thorough hyperparameter tables and ablations.
- Writing:
  - Fix references and all formatting/typographical issues; tighten method descriptions in the main text; clarify candidate-set construction and evaluation splits.

Questions for the authors
1. Candidate set: Does it include responses that appear only in the test set (plain/OOV)? Please justify this choice and provide results with a candidate library constructed solely from training data.
2. OOV KB exposure: Why are models given access to both KBs during training? Can you report results where the OOV KB is withheld entirely, and quantify the effect on OOV performance?
3. Match-type features: Are type matches allowed for any token in the global KB, including entities that never appear in training dialogs? How do you prevent this from leaking test-only surface forms?
4. Sorting: How do you verify that T3 improvements reflect true sorting ability rather than memorization over small candidate spaces? Any controlled generalization tests (e.g., unseen rating distributions)?
5. DSTC2 conversion: How were ratings injected and how deterministic is the policy? Could you release the exact conversion procedure and assess sensitivity to that design choice?
6. Variance: What is the variability across seeds for your main results, especially per-dialog accuracy on T3/T5/T6?
7. Concierge: Can you release an anonymized subset and/or provide a detailed schema and annotation to make those results reproducible?
