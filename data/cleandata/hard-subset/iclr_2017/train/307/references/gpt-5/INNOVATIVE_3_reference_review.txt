1. Summary
This paper introduces a lightweight, open, and reproducible testbed for end-to-end goal-oriented dialog, focused on restaurant reservation. The authors decompose the overall task into five synthetic subtasks covering dialog management, KB querying, interpreting KB results (including sorting), OOV generalization to unseen entities, and a combined full-dialog task. They further convert DSTC2 into the same format and evaluate on a proprietary human-human “Concierge” dataset to test transfer beyond synthetic language. They benchmark IR baselines, supervised embeddings, and MemN2N, proposing simple but impactful “match type” features to handle entities and OOVs. Results show that end-to-end models can issue and update API calls and handle unseen entities with type features, but still struggle with interpreting KB outputs and sustaining full-dialog success. The dataset and analysis aim to catalyze progress by enabling controlled, interpretable evaluation with per-response and per-dialog metrics.

2. Decision: Accept
Key reasons:
- Novel and timely contribution: a principled, decomposed benchmark for goal-directed dialog with explicit KB grounding and OOV evaluation. This is the kind of bold, field-shaping resource that can reorient research beyond chit-chat toward actionable dialog.
- Clear empirical insights: The paper convincingly diagnoses what current end-to-end models can and cannot do, and the type features provide a simple, generalizable idea to handle entities.

3. Supporting arguments
- What problem is tackled? The gap between non-goal-oriented end-to-end dialog success and the requirements of goal-directed dialog (tracking constraints, querying KBs, interpreting symbolic outputs, handling unseen entities), and the lack of a reproducible, interpretable benchmark to evaluate these abilities.
- Is the approach well motivated and well placed? Yes. The need to move from chit-chat to transactional dialog is well argued; prior resources (DSTC, ATIS, MTurk-based collections) either constrain modeling to state tracking, lack reproducibility, or are noisy/hard to analyze. The bAbI-like decomposition makes capabilities testable and comparable. Related work is covered comprehensively.
- Do results support the claims? Yes. The synthetic tasks isolate capabilities; MemN2N’s strengths on API call issuance/updates and weaknesses on interpreting KB outputs and full dialogs are consistent across synthetic and real data (DSTC2 conversion, Concierge). OOV splits and type features substantiate claims about entity handling. While the domain and language are simplified, the transfer pattern to real data supports the testbed’s relevance.

Strengths
- A coherent, decomposed benchmark with explicit KB grounding and OOV generalization—high potential community impact.
- Sensible evaluation design: per-response and per-dialog metrics; ranking setting to reduce generation confounds; small training sizes to mimic realistic data scarcity.
- Inclusion of real-world datasets to validate synthetic findings.
- Simple, effective “match type” features highlighting entity/OOV limitations in current embeddings.
- Transparent baselines and ablations; memory hops analysis; error examples.

Limitations (acceptable given the contribution)
- Synthetic language templates and deterministic bot policies limit linguistic variability; Task 3’s deterministic sorting and acceptance rule may over-structure behavior.
- The ranking formulation sidesteps full NLG; some capability (e.g., copying entities) is partially externalized via candidates and type features.
- DSTC2 conversion introduces design choices (adding ratings; generating API outputs) that may bias difficulty and comparability.
- The proprietary Concierge dataset cannot be released (as implied); this weakens reproducibility on the most realistic setting.

4. Additional feedback (to help improve the paper; not gating the decision)
- Clarify and release details: simulator code, KBs, exact candidate pools, and splits (including OOV protocol), to maximize reproducibility. If Concierge cannot be released, consider releasing a small anonymized subset or detailed statistics and annotation schema for API-event tagging.
- Expand error analysis: categorize MemN2N failures on T3/T5 (e.g., ranking mistakes vs. stopping criteria vs. entity mismatches), and report targeted metrics (e.g., API argument accuracy, sorting accuracy).
- Baselines: include a copy/pointer mechanism or pointer-generator seq2seq baseline to test entity handling without type features; also consider a Transformer-based retrieval model if feasible.
- Ablations: report the effect of time/speaker features, memory size truncation, and training set size; show sensitivity to the number of negative candidates and candidate set composition.
- DSTC2 conversion: detail how ratings were injected and whether multiple valid options existed; analyze robustness to ASR noise and non-deterministic bot behavior.
- Discuss generalization: plans for extending beyond restaurants, and whether the decomposition transfers (e.g., travel, movie tickets). A multi-domain variant would amplify impact.
- Consider a small human evaluation on Concierge turn-level appropriateness to complement NUC accuracy.

Questions for the authors
- Release: Will you release the simulator, both KBs, and precise scripts for generating plain and OOV splits? What is the license?
- Candidate pool: Does the candidate set include all responses across all tasks and splits? How do you prevent candidate leakage that could unintentionally simplify tasks?
- OOV protocol: During training, do models have read access to the OOV KB (for type features) but no dialogs from that KB? Please clarify the exact information exposure.
- DSTC2 conversion: How were ratings assigned, and could that introduce label bias? Are there turns with multiple acceptable responses, and how is that handled in ranking evaluation?
- Match type features: Do they rely solely on string lookup in the provided KB, or any additional NER? How would they extend to domains without a clean KB lexicon?
- Scaling: How does performance vary with training size (e.g., 1k vs. 10k examples)? Any signs of closing the T3/T5 gap with more data or more hops?
- Realism: Have you tried relaxing deterministic policies in the simulator (e.g., varying question order) to reduce overfitting to templates, and how does that affect model rankings?
