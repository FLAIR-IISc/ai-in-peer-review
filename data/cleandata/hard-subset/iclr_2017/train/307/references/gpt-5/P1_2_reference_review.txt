1) Summary of contributions and claims
- Problem: The paper targets evaluation of end-to-end dialog systems in goal-oriented, task-completion settings, a gap where traditional slot-filling systems excel but end-to-end models are under-explored and hard to evaluate.
- Contributions:
  - A lightweight, reproducible benchmark suite for goal-oriented dialog in a restaurant reservation domain: five synthetic tasks (issuing/updating API calls, displaying options including sorting by rating, providing KB-grounded information, and full dialogs).
  - An explicit OOV test protocol via a split KB (disjoint cuisines/locations), to assess generalization to unseen entities.
  - A conversion of DSTC2 into the same task/ranking format (Task 6), plus evaluation on a real human-human “Concierge” dataset to check whether synthetic performance patterns transfer to noisier real data.
  - Baselines spanning IR (TF-IDF, Nearest Neighbor), supervised embeddings, MemN2N (with multi-hop attention), and a simple rule-based system; introduction of match-type features to help memory networks and other baselines handle entities and OOVs.
  - Findings: MemN2N > supervised embeddings > IR across tasks; match-type features significantly improve OOV and KB grounding; T1/T2 are mostly solved, T3/T5 remain challenging; results on DSTC2 and Concierge echo the relative ordering and highlight remaining gaps (low per-dialog accuracy).
- Claims: The dataset provides a controlled, interpretable, reproducible way to stress-test core capabilities necessary for goal-oriented dialog. End-to-end models can learn non-trivial operations (API calls, handling updates, limited OOV handling) but still fall short on reliably completing dialogs, especially when interpreting and presenting KB results.

2) Decision: Accept
Key reasons:
- The benchmark/resource is timely, well-motivated, and fills a clear gap for evaluating goal-oriented end-to-end dialog with KB grounding and OOV stress tests.
- The empirical study is careful and insightful: multiple baselines, ablations (match-type features, number of hops), OOV split, plus validation that patterns transfer to real data (DSTC2, Concierge).

3) Supporting arguments
- Specific question/problem: Can end-to-end models replace traditional methods in narrow-domain, goal-oriented dialog? If not, where do they fail? The tasks are designed to decompose required abilities (state tracking, API issuing/updating, sorting and presenting options, KB-grounded answering), enabling diagnosis.
- Motivation and positioning: The paper is well situated relative to POMDP/slot-filling systems, non-goal-oriented end-to-end dialog, bAbI QA, and DSTC. It articulates why existing corpora and evaluation paradigms are insufficient for reproducibility and capability diagnosis.
- Scientific rigor and correctness:
  - Clear task definitions, controllable simulation with a KB, and explicit OOV protocol.
  - Distinct metrics (per-response, per-dialog) that reflect task completion difficulty and error compounding.
  - Hyperparameter tuning on dev sets; analysis of multi-hop attention and match-type features; inclusion of a rule-based upper bound on synthetic tasks and a rule-based baseline on DSTC2.
  - Cross-dataset validation: the same performance ordering and qualitative failures observed on synthetic tasks appear on DSTC2 and Concierge, lending credence that the synthetic tasks probe relevant capabilities.
- Novelty/impact: The dataset and evaluation framework should catalyze research on entity-centric reasoning, KB interaction, and robustness to OOVs in goal-oriented dialog. The match-type augmentation provides a simple, practical mechanism to mitigate common entity-handling failures in embedding models.

4) Additional feedback to improve the paper (helpful suggestions)
- Clarify the candidate ranking setup: You state that candidates include all bot utterances and API calls appearing in train/val/test for all tasks combined. Using test candidates in the candidate set can be seen as easing the problem and risks inflating absolute numbers. Please justify this design choice and discuss its implications; consider reporting results with candidate sets restricted to train/val.
- Expand baseline coverage: Include strong sequence modeling baselines with copy/pointer or key–value mechanisms (e.g., pointer networks, copy-augmented seq2seq) that are known to help with entities and OOVs; this would make the analysis more contemporary and strengthen conclusions about what capabilities remain lacking.
- Provide more granular error analysis: For T3/T5 failures, categorize typical error modes (sorting mistakes, stopping criteria, mismatched entity/attribute grounding, etc.) and relate them to the model architecture limitations (e.g., bag-of-words utterance encoding for MemN2N).
- Discuss realism/coverage: The simulator uses 43 user and 20 bot patterns with deterministic bot behavior and a small set of slots. Please comment on how diversity (paraphrases, disfluencies, mixed intents) might alter difficulty and how the framework could be extended (e.g., stochastic policy variants, additional slots, non-deterministic prompting).
- Report significance and variance: Add confidence intervals or multiple runs to quantify variability, especially since training sets are small (~1k).
- KB visibility and OOV protocol: You allow access to both KBs at training time for typing OOV entities. While useful to test exact-match/type handling, it reduces difficulty. Consider an alternative condition where the OOV KB is not available at training and discuss performance implications.
- Reproducibility: Ensure public availability of code, simulator, data generation scripts, and exact candidate lists to facilitate fair comparison.

Questions for the authors
- Candidate set: Do results change materially if candidates are restricted to those seen in training/validation? Can you report both?
- OOV setting: Since models can see the OOV KB types at training (for typing features), how much of the OOV gain stems from the type lexicon vs. learned generalization? Can you ablate “unknown KB at train time”?
- Sorting and stopping in T3: What is the typical number/distribution of options? Did you try architectures with explicit comparison operations or numeric embeddings for ratings?
- MemN2N bag-of-words: Did you test position- or n-gram-aware encodings, or key–value memory networks, to better bind attributes to entities in the returned KB facts?
- DSTC2 conversion: How often do the original system’s errors (ASR/NLU) contradict the deterministic rules you impose (e.g., sort by rating)? Do such inconsistencies materially cap the achievable accuracy?
- Concierge: What is the candidate set size and diversity? Are there domain drifts (requests outside booking) that systematically mislead models? Any delexicalization tried?
- Hyperparameter tuning: Were hops and other hyperparameters tuned per-task using dev from the same distribution? Can you comment on cross-task generalization of a single setting?

Overall, this is a valuable, well-motivated resource paper with solid empirical analysis and clear takeaways that will be useful to the community. I recommend acceptance.
