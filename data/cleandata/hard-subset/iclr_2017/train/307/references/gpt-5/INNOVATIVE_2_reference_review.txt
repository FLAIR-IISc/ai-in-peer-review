1) Summary of contributions
The paper introduces an open, reproducible testbed for end-to-end goal-oriented dialog. It decomposes restaurant-booking interactions into five synthetic, controlled tasks (API call issuance, updates, option listing with sorting, information retrieval from KB results, and full dialogs), plus a converted DSTC2 set (Task 6) and an additional real human-human “Concierge” dataset used for external validation. The design explicitly probes capabilities crucial for task completion: implicit state tracking, KB access, entity/OOV handling, and basic reasoning (e.g., sorting options). The authors benchmark rule-based systems, classical IR, supervised embeddings, and Memory Networks (with and without match-type features for entity handling). They report that MemNNs outperform other learning baselines, can learn API-calling and handle OOV entities with match-types, but still struggle on end-to-end goal completion, especially when interpreting KB results and managing full dialogs. The datasets and conversions are released to foster reproducibility and progress.

2) Decision
Accept. Key reasons:
- Novel and useful benchmark: A bold, well-scoped testbed for goal-oriented dialog with principled task decomposition and OOV evaluation. This resource fills a clear gap and is likely to shape and accelerate research.
- Clear empirical insights: Systematic comparisons reveal where current end-to-end models succeed and fail (e.g., progress on API calls vs. difficulty with option ranking and KB grounding), validated on both synthetic and real settings.

3) Supporting arguments
- Motivation and placement: The paper is well positioned relative to prior work in slot-filling/POMDP systems and non-goal-oriented end-to-end dialog. It targets a missing piece: reproducible, interpretable, goal-centric evaluation for end-to-end models, akin to bAbI for QA but with dialog-specific structure and KB grounding.
- Task design and novelty: The five synthetic tasks isolate core skills, include entity and OOV stress-tests, and require execution-like behaviors (API calls, sorting). The OOV split by cuisine/location types is a thoughtful way to probe generalization and entity handling.
- Empirical methodology: The comparison suite (rule-based, IR, supervised embeddings, MemNNs) is sensible and exposes interesting trends: TF-IDF is weak in goal-directed settings; supervised embeddings are decent but brittle; MemNNs deliver sizable gains; match-type features are crucial for OOV and KB lookup. The multi-hop analysis provides evidence that iterative memory access helps.
- External validation: Converting DSTC2 into the same framework and evaluating on Concierge lends credibility that relative trends transfer to real data, addressing the common criticism of synthetic-only benchmarks.
- Impact: An open, lightweight, consistent benchmark for goal-oriented dialog is likely to become a community reference and stimulate more principled progress.

Limitations (not fatal given the contribution is primarily a benchmark and analysis):
- Language and policy determinism: Synthetic user/bot templates and fixed API-call orders simplify the problem and may encourage overfitting to patterns; real-world language variability and non-determinism remain underexplored in the core tasks.
- Retrieval evaluation: Using ranking over a global candidate pool (including test candidates) departs from generation and might mask challenges in surface realization and out-of-candidate responses.
- Baseline coverage: Missing strong contemporary generative baselines with copy/pointer mechanisms, key-value memory networks, or graph-structured KB reasoning, which are directly relevant to entity grounding and OOV handling.
- DSTC2 adaptation artifacts: Injecting ratings to ensure deterministic option ordering could bias the task; the rule-based DSTC2 baseline is relatively modest and may understate the challenge.
- KB assumptions: The KB/API design is simplified (e.g., single party size per restaurant). While appropriate for a testbed, it limits realism.

4) Suggestions to improve the paper (helpful, not decision-critical)
- Add stronger neural baselines: seq2seq with attention + copy/pointer, key-value memory networks, and retrieval-augmented generators. These are likely to stress the entity/OOV dimension more meaningfully than match-types alone.
- Report sensitivity to candidate-set scope: Compare per-turn candidate restriction (contextually plausible actions) vs. global candidate pools; analyze how candidate size affects per-response and per-dialog accuracy.
- Deepen error analysis for Tasks 3 and 5: Where exactly do MemNNs fail in sorting, option selection, and grounding returned KB facts? Provide confusion patterns and attention visualizations; analyze the effect of additional hops.
- Explore more realistic variability: Add stochastic bot policies, paraphrase augmentation, ASR noise, typos, and non-deterministic API behaviors to bridge synthetic tasks further toward real-world conditions.
- Consider a generation track: Provide BLEU-free evaluation (e.g., NUC-like) with constrained action schemas, or success-based metrics with human evaluation for full dialogs.
- Clarify and document data release: Ensure all synthetic tasks and the DSTC2 conversion scripts are available; state the status of Concierge (if non-releasable, provide a smaller public analog).

Questions for the authors
- Candidate set: Do all experiments rank over the union of candidates from train/val/test and all tasks? How do results change if candidates are restricted per-turn to plausible actions?
- Sorting behavior: How is sorting learned in Task 3? Did you analyze attention/hop trajectories that evidence ranking/comparison, and how robust is it to small perturbations (e.g., tied ratings)?
- OOV handling: Match-type features rely on KB lexicons. How would performance change if the KB did not include test-time entities’ surface forms (e.g., partial/noisy lexicons)? Could copy/pointer mechanisms reduce dependence on curated types?
- DSTC2 conversion: Adding ratings for deterministic option selection could alter dialog dynamics. Can you quantify how much this modification impacts difficulty versus the original transcripts?
- Realism of the API schema: How sensitive are results to relaxing simplifying assumptions (e.g., multiple party sizes per restaurant, partial matches, multi-criterion ranking)? Any plans to expand tasks accordingly?
- Release details: Will code, data generators, and evaluation scripts be released to ensure reproducing all numbers, including OOV splits and match-type processing?

Overall, the paper offers a timely, novel benchmark and clear empirical takeaways. Despite imperfections, its potential impact and the clarity of the task decomposition justify acceptance.
