Summary and contributions
- The paper introduces a lightweight, reproducible testbed for evaluating end-to-end goal-oriented dialog systems in the restaurant reservation domain. It decomposes the problem into five synthetic tasks (API call issuance, updates, option display with sorting, KB-based information provision, and full dialogs), adds an OOV evaluation split that forces generalization to unseen entities, and supplements with two datasets based on real interactions (converted DSTC2 and a proprietary concierge dataset).
- The work benchmarks several families of methods (IR baselines, supervised embeddings, MemN2N with/without match-type features, and a hand-engineered rule-based system on DSTC2), evaluates both per-response and per-dialog accuracy, and provides ablations and qualitative analyses.
- Key findings: MemN2N can learn non-trivial behaviors (API calls, handling OOV entities with type features) and outperforms IR and supervised embedding baselines, but struggles with tasks that require reasoning over KB outputs (sorting options, aggregating facts), leading to low per-dialog success. Relative method ordering observed on synthetic tasks transfers to real data (DSTC2, concierge), supporting the proxy value of the synthetic suite.

Decision: Weak accept
Key reasons:
1) The paper fills an important gap by providing a well-scoped, reproducible benchmark and clear evaluation protocol for end-to-end goal-oriented dialog; the synthetic tasks isolate core sub-abilities and the inclusion of real data strengthens external validity.
2) The empirical analysis is careful and yields actionable insights (e.g., benefits/limits of match-type features, multi-hop attention helps, acute failure modes on KB interpretation and sorting), which are valuable to the community.

Supporting arguments
- Problem and motivation: Well placed in the literature. The limitations of slot-filling/POMDP pipelines and the lack of suitable end-to-end goal-oriented datasets are correctly identified. The benchmark is a natural extension of bAbI-style QA to dialogs with KB grounding and task completion metrics.
- Methodological soundness: Tasks are designed to probe specific capabilities (state tracking, KB query formation and update, interpreting KB results, OOV handling), with explicit per-response and per-dialog metrics. The OOV split is thoughtful and highlights weaknesses in standard embedding methods; match-type features are a reasonable step toward entity handling.
- Baselines and analysis: The range of baselines is appropriate for the paper’s aims. The ranking setup makes model comparisons straightforward and reproducible. The ablations (hops, match types) and qualitative attentions are informative. The conversion of DSTC2 into the task format and the concierge evaluation show similar relative trends, lending credibility to the synthetic suite as a proxy.
- Claims support: The claims are modest and matched by results: end-to-end models are promising but not yet competitive with handcrafted methods in narrow domains for reliable task completion; the benchmark exposes concrete failure modes.

Areas for improvement (not blockers)
- Synthetic language and determinism: The simulator uses a small set of templates (20 bot, 43 user) and deterministic bot behavior (e.g., fixed slot query order), which risks overfitting to superficial patterns and underestimates the complexity of dialog management. Adding paraphrase diversity, stochastic slot orders, and noise (e.g., typos, self-corrections) would improve robustness.
- Ranking vs generation: The evaluation is ranking over a candidate set including outputs from train/val/test across tasks. While this ensures tractability and comparability, it limits realism and can obscure difficulties in NLG and content planning. Reporting results with a generative model (e.g., seq2seq/transformers with copy/pointer mechanisms) and/or constrained decoding would broaden insights.
- KB reasoning depth: Task 3 requires sorting by rating, but the chosen MemN2N architecture is not well-suited for algorithmic operations like sorting. Including baselines with explicit differentiable reasoning over structured KB outputs (e.g., key-value memories, pointer/copy networks) or symbolic/neuro-symbolic hybrids would clarify whether failures are architectural or data-related.
- OOV and type features: While match-type features are effective, they rely on exact KB matches and privileged type information; this can partially bypass the core challenge. It would help to report results with stricter OOV settings (e.g., entities unseen and not included in KB at training time) and with alternative mechanisms (learned copy/pointer, span selection).
- DSTC2 conversion: Injecting ratings to enforce a deterministic option order departs from the original dataset design and prevents comparison with DSTC2 leaderboards. Consider reporting an additional variant aligned to DSTC2’s original outputs, or quantify sensitivity to the injected heuristic.

Questions for the authors
1) Candidate set composition: You include all bot utterances and API calls from train/val/test (plain and OOV) across tasks as candidates. Could you quantify how sensitive results are to the size/content of the candidate set? Do models overfit to candidate idiosyncrasies?
2) Multiple valid responses: In DSTC2 and Concierge, there can be several acceptable next utterances. How often does the “single gold” assumption penalize plausible outputs? Can you report an alternative metric (e.g., recall@k, success rate with equivalence classes)?
3) Sorting and KB interpretation: Do you have diagnostics on whether MemN2N is failing the sorting operation vs failing to attend to the relevant KB facts? Any experiments with key–value memories or pointer networks?
4) OOV setting: Models have access to both KBs during training. How would performance change if the OOV KB were only available at test time (no backprop through OOV entities), or if only partial type lexicons were available?
5) Language diversity: What is the entropy/diversity (e.g., type–token ratio) of user/bot utterances in the simulator? Any experiments with increased paraphrase variability or noise injection to test robustness?
6) Reproducibility: Will you release code for data generation, evaluation scripts, and trained model configs/hyperparameters to ensure exact reproducibility?

Additional feedback to strengthen the paper
- Provide per-task data statistics in the main text (counts, average turns, vocabulary/candidate sizes) and report learning curves to show sample efficiency.
- Add stronger modern baselines tailored to entities and KBs (e.g., copy/pointer, span extraction over KB outputs, key–value memories), and a simple hybrid with an explicit symbolic API-calling module plus learned response selection.
- Report success rate per dialog length buckets; long-dialog failure is particularly informative.
- Include more detailed error analysis: confusion between entity types, wrong field updates, failure to stop after acceptance, etc., plus attention visualizations on failure cases.
- Consider a human-in-the-loop evaluation on a small subset to assess whether per-dialog accuracy correlates with task success as perceived by human judges.

Overall, this is a useful, well-motivated resource paper with careful analysis. With some extensions to reduce synthetic determinism and broaden baselines, it can become a standard benchmark for grounded, goal-oriented dialog.
