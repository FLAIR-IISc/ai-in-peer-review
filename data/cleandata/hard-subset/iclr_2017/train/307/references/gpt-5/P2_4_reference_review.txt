1) Summary and claimed contributions
The paper proposes an open, reproducible benchmark for end-to-end goal-oriented dialog, centered on restaurant booking. It decomposes the overall task into five synthetic subtasks (slot elicitation, API issuance and updating, reading/querying KBs, sorting/displaying options, returning KB attributes) plus two evaluations on real data (converted DSTC2 human–bot dialogs and a proprietary Concierge human–human set). It introduces an OOV split that forces generalization to unseen entities (locations, cuisines, restaurants) and evaluates several baselines (rule-based, classical IR, supervised embeddings) and end-to-end Memory Networks (MemN2N), including a simple but effective “match-type” feature mechanism for entities. The work aims to (i) enable consistent, interpretable comparisons, (ii) diagnose which capabilities are missing, and (iii) assess whether synthetic tasks correlate with performance on real dialog.

2) Decision: Accept
Primary reasons:
- Well-motivated benchmark addressing a clear gap: reproducible, task-decomposed evaluation for goal-oriented dialog with KB grounding and OOV entities.
- Careful experimental setup with diverse baselines and metrics; insightful findings (e.g., MemN2N can issue/modify API calls but struggles with KB interpretation/sorting; match-type features are crucial for OOV).
- Evidence that performance patterns transfer from synthetic to real data, supporting the benchmark’s external validity.

3) Supporting arguments
On the core questions:
- What problem is tackled? The paper targets an evaluation deficit for end-to-end goal-oriented dialog systems: lack of controlled, reproducible, interpretable tasks requiring dialog management plus symbolic KB use and generalization to unseen entities. This is a concrete, important problem for the community.
- Is the approach well motivated and placed in the literature? Yes. The paper positions itself vis-à-vis POMDP/slot-filling systems (handcrafted, narrow-domain), chit-chat datasets (not goal oriented), prior dialog QA tasks (not full dialog), DSTC (state-tracking focus), and crowdsourcing-based evaluations (weak reproducibility). The design mirrors the bAbI philosophy, adapted to goal-oriented dialog with KB operations and OOV challenges, and compares to strong baselines.
- Do results support the claims with rigor? The evaluation uses per-response and per-dialog accuracy (the latter aligned with task completion). The baseline suite is appropriate: IR (TF-IDF, nearest neighbor), supervised embeddings, and MemN2N with/without match-type features. The OOV split is clearly defined. Findings are consistent and interpretable: (i) IR is weak here (goal-directed dialog moves quickly, diminishing lexical overlap), (ii) supervised embeddings do moderately well but fail on KB reasoning, (iii) MemN2N advances substantially, particularly on API issuance/updating, and (iv) match-type features unlock OOV handling and T4. The pattern replicates on DSTC2 and Concierge, suggesting the synthetic tasks are informative. The methodology is transparent and reproducible.

4) Additional feedback to improve the paper (not all are decision-critical)
- Evaluation setup:
  - Candidate ranking vs generation: clarify why ranking is chosen and outline a path to include generation (e.g., copy/pointer mechanisms, constrained decoding) in future releases. Consider reporting an auxiliary metric like Recall@k or a smaller candidate pool study to show robustness.
  - Using a candidate set drawn from train/valid/test (including test) could be debated; justify this choice and report sensitivity to candidate set construction.
  - Provide more granular error analysis (per-turn confusion breakdowns on T3/T5; typical failure modes in sorting, option acceptance/termination).
- Dataset design:
  - The deterministic order of slot prompts in T1 simplifies the problem; consider adding paraphrastic variation and policy randomness to prevent overfitting to templates.
  - For DSTC2, you add ratings to support option ordering. Please quantify any artifacts this introduces and the fraction of turns where the original bot behavior was non-deterministic.
  - For Concierge, removing named entities via NER may obscure the role of entity handling. Consider releasing a version with de-identified but typed entities (e.g., placeholders with type tags) to test type-aware generalization on natural text.
  - Report learning curves (1k–10k+) to show data efficiency and whether scaling training reduces T3/T5 gaps.
- Modeling:
  - Given the centrality of entities, compare to pointer/copy networks or key–value memory networks; your match-type feature is a simple surrogate for copying—an ablation against a genuine copy mechanism would strengthen conclusions.
  - For T3 (sorting), analyze whether failures are due to inability to learn ordering vs. long-context reasoning limits; e.g., probe models with explicit supervision for ranks or with differentiable sort-like modules.
  - Include sensitivity to number of hops and memory size, and report stability across random seeds.
- Reproducibility and release:
  - Ensure code, data generation scripts, KBs (both splits), and the DSTC2 conversion are publicly available with licenses.
  - Document the KB entity lexicons used for match-typing and whether models may access both KBs at train time (as stated). Clarify whether that setting is meant to reflect realistic deployment.

Questions for the authors
- Candidate set construction: Why include candidates from test sets, and how do results change if candidates are restricted to training (plus KB/API templates)? Any leakage concerns?
- KB access: Systems “have access to both KBs” during training. Does this artificially ease OOV by exposing unseen entities? How would performance change if the second KB were hidden until test time but the type lexicon were available?
- Sorting in T3: Do MemN2N attention patterns indicate any learned ordering? Did you try auxiliary losses (e.g., ranking supervision) or architectural variants (key–value memories) to explicitly support sorting?
- DSTC2 conversion: Adding ratings to support deterministic option selection could bias behavior. Can you quantify how often the original dialogs disagreed with “pick highest rating,” and how you handled such cases?
- Robustness: How sensitive are results to paraphrase diversity (increasing user/bot pattern counts), noise (ASR-like corruption), and longer dialogs?
- Generalization: Do models trained on T1–T4 transfer zero-shot to multi-domain variants (e.g., hotels) if given only the new KB schema and type lexicon?

Overall, this is a timely, well-executed benchmark paper with clear motivation, solid baselines, and useful insights that I expect will catalyze progress on goal-oriented dialog.
