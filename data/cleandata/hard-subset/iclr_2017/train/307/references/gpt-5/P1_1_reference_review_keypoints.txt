- Problem: Tests whether end-to-end neural dialog models can handle goal-oriented tasks requiring state tracking, KB querying, and reasoning over retrieved facts, and presents a reproducible testbed.
- Contribution 1: Five synthetic restaurant booking dialog tasks that decompose API call handling, KB sorting/proposal, KB-fact extraction, and full dialogs, with in-vocabulary and OOV splits to probe entity generalization.
- Contribution 2: Conversion of DSTC2 into the same framework (Task 6) and evaluation on a proprietary human-human “Concierge” dataset to assess realism and transferability.
- Contribution 3: Baselines spanning IR, supervised embeddings, and end-to-end Memory Networks with a match-type feature, plus per-response and per-dialog accuracy metrics and ablation studies (hops, type features).
- Contribution 4: Empirical findings that MemNNs can learn API issuance/updates, benefit from match-type features and multiple hops, yet struggle with reasoning over KB results (e.g., sorted option display) and full-dialog correctness; trends transfer to DSTC2 and Concierge.
- Decision: Accept.
- Key reason 1: Provides a well-motivated, reproducible benchmark filling a gap for end-to-end goal-oriented dialog evaluation with thoughtful task design and aligned metrics.
- Key reason 2: Careful, insightful empirical study comparing multiple baseline families, yielding actionable analyses and consistent results across synthetic and real data.
- Supporting argument — motivation/positioning: Nicely contrasts slot-filling/POMDP pipelines with neural end-to-end approaches and situates the work relative to bAbI QA, DSTC, and recent dialog research.
- Supporting argument — technical design: Tasks explicitly target transactional dialog capabilities and use an OOV split to stress realistic entity generalization; ranking evaluation focuses on decision-making rather than surface generation.
- Supporting argument — empirical rigor: Multiple baselines, ablations, two metrics, and cross-dataset validation; match-type feature effectively handles entities/OOVs; synthetic trends mirror real data.
- Supporting argument — clarity/reproducibility: Task construction and evaluation are clearly described, dataset public, and hyperparameters/additional results in appendices.
- Caveat 1: Synthetic language is limited (43 user/20 bot templates) and bot behavior is deterministic, reducing linguistic and strategy diversity.
- Caveat 2: Ranking evaluation uses a global candidate set drawn from train/val/test, simplifying output space versus true generation and potentially easing the task.
- Caveat 3: DSTC2 conversion choices (added ratings, assumed deterministic ordering) may introduce biases relative to the original task.
- Caveat 4: Comparisons are limited to MemNNs vs. IR/embedding baselines; newer architectures with copy/pointer or seq2seq could broaden insights.
- Feedback 1: Clarify and justify construction of the global candidate set and assess potential leakage; consider reporting results with candidate sets constrained by dialog state.
- Feedback 2: Report robustness across random seeds, provide variance/confidence intervals, and run significance tests for key comparisons.
- Feedback 3: Expand analysis of failures on T3/T5 (diagnose sorting vs. grounding vs. stopping), include attention analyses and minimal task variants to isolate error sources.
- Feedback 4: Add or reference experiments with generative models or pointer/copy mechanisms for OOV entities and KB grounding.
- Feedback 5: For DSTC2 conversion, detail how ratings were assigned, quantify non-deterministic bot behavior, and consider tolerant evaluation for tied-option orders.
- Feedback 6: Consider releasing expanded language patterns, multi-domain variants, and a sanitized Concierge subset to enhance reproducibility.
- Question 1: Does including all candidates from train/val/test risk inflating per-response accuracy; can results be reported with pruned candidate sets constrained by dialog state or action type?
- Question 2: For OOV setup, how realistic is providing both KBs at training time; how sensitive are results if the OOV KB is only available at test time?
- Question 3: Do type/match features require full KB access or only schema information?
- Question 4: For DSTC2 conversion, how were ratings assigned, how often are ties present, and were alternative tie-breaking policies tested?
- Question 5: For Task 3 sorting failures, can errors be attributed more to sorting, mis-grounding, or stopping criteria; any attention-weight evidence?
- Question 6: How do results scale with more training data (beyond 1k examples); do MemNNs and supervised embeddings close the per-dialog gap?
- Question 7: Have generative MemNNs or seq2seq models with copy been tried on these tasks, and how do they compare?
- Question 8: Will code, data, and the specific Concierge evaluation splits be made available to ensure exact replication?
