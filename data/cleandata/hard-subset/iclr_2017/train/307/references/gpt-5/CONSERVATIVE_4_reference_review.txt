Summary
The paper proposes a lightweight, reproducible testbed for evaluating end-to-end goal-oriented dialog systems. It introduces five synthetic tasks in a restaurant-booking domain (with a KB, API calls, sorting options, and OOV entity handling), plus two additional datasets derived from real dialogs (a converted DSTC2 split and a proprietary concierge dataset). The authors benchmark several baselines (rule-based, IR, supervised embeddings) and a Memory Network variant with “match type” features to better handle entities. They report encouraging per-response accuracy but low per-dialog accuracy, arguing that end-to-end models are not yet ready to replace traditional slot-filling in goal-directed settings and that their synthetic tasks reflect performance patterns on real data.

Decision: Reject
Key reasons:
1) Methodological and evaluation concerns limit the interpretability and external validity of the results (synthetic templates, deterministic behaviors, and a ranking setup with candidates drawn from train/val/test—including OOV test—sets).
2) Empirical support is not strong enough: the baselines are dated and do not include established strong methods for goal-oriented dialog or modern architectures; the link between synthetic performance and real-world performance is insufficiently demonstrated.

Supporting arguments
- Evaluation design issues:
  - Candidate leakage: The candidate set for ranking includes all bot utterances and API calls from training, validation, and both test splits (plain and OOV). This is a strong assumption that inflates performance and diverges from realistic deployment where test-only responses would not be known a priori. It undermines the claim that results reflect true predictive ability.
  - Ranking over fixed candidates rather than generation: While convenient, this sidesteps core challenges of response generation, surface realization, and robustness. Conclusions about end-to-end dialog capability are therefore limited.
  - Synthetic determinism and limited linguistic diversity: Only 43 user patterns and 20 bot patterns, fixed order of slot requests (Task 1), simplified KB assumptions (e.g., single party size per restaurant). Models can exploit artifacts and heuristics. This weakens the value of the benchmark as a proxy for real-world dialog complexity and variability.
  - OOV setup relies on access to typed KB lexicons at test time to fire “match type” features. This is plausible in some applications but is a strong assumption; it blurs the line between learned generalization and lookup-based matching.

- Insufficient baselines and comparisons:
  - The baselines (TF-IDF, nearest neighbor, supervised bag-of-embeddings, MemN2N with hand-crafted match-type features) are weak by current standards. Missing comparisons include established slot-filling/belief tracking pipelines, strong DSTC2-era trackers, sequence-to-sequence with attention and copy/pointer mechanisms, key-value memories, retrieval-augmented models, and modern transformer-based architectures with schema-guided NLU/NLG. Without these, it is difficult to judge how challenging or representative the tasks really are.
  - On DSTC2, the rule-based system constructed here is quite simple and underperforms. There is no comparison against published DSTC2 baselines/trackers, which were specifically designed for that dataset. This makes the claim that the synthetic tasks mirror real-world trends less convincing.

- Claims vs. evidence:
  - The primary claim is a useful benchmark that diagnoses capabilities (dialog management, KB use, sorting, OOV entities) and correlates with real dialog performance. However, the main empirical result is that per-dialog accuracy remains low across models; and the evidence that improvements on synthetic tasks translate to real data is suggestive rather than robust (no significance tests, no stronger baselines, and the concierge dataset is proprietary and not reproducible).
  - The “match type” feature engineering is not a substantive modeling advance and performs inconsistently (e.g., small drops on T2). It looks ad hoc and tailored to the tasks.

- Reproducibility and rigor:
  - No statistical significance or variance estimates are reported.
  - The DSTC2 conversion introduces ratings to enforce a sortable criterion; the details and scripts should be released for reproducibility and to verify that the conversion does not bias the task.
  - The concierge dataset is proprietary and cannot be used to replicate results; its role should be downplayed or complemented with public alternatives.

Additional feedback (to help improve the paper)
- Restrict candidate responses to those available from training only; report both ranking and generation results. Consider more realistic candidate sets that do not contain test-only utterances.
- Increase linguistic diversity and reduce determinism in the simulator (paraphrases, disfluencies, alternative slot orders, corrections), and add noise (ASR-like errors) in a controlled way.
- Include stronger contemporary baselines: transformer encoder-decoder with copy/pointer and constrained decoding for API calls; neural belief tracking + policy learning; key-value memory networks; retrieval-augmented generation; schema-guided NLU/NLG; and published DSTC2 baselines.
- Provide statistical tests, multiple random seeds, and ablations (e.g., removing access to KB type lexicons at test time) to clarify where gains come from.
- Release all scripts for DSTC2 conversion; consider adding a public real-world dataset (e.g., MultiWOZ-style or other open goal-oriented corpora) formatted in your framework to replace or supplement the proprietary concierge set.
- Report more granular error analyses (e.g., failures in sorting, entity grounding, coreference, state tracking, API arguments) and learning curves to assess data efficiency.

Questions for the authors
1) Why is the candidate set built from all splits (including test and OOV)? How do results change if candidates are restricted to training-only?
2) At test time, what exact resources are assumed available for match type features? Do you assume full KB lexicons for all entity types, including OOV entities? How does performance change if only training-seen entities/types are available?
3) Can you compare against published DSTC2 baselines (e.g., RNN/LSTM trackers, hybrid pipelines) and recent transformer-based models with copy mechanisms?
4) Are the DSTC2 conversion scripts and rating assignment procedure public? Could the rating injection bias the task (e.g., deterministic top-choice behavior)?
5) Do you have generation results (not just ranking) and human evaluations on the real datasets?
6) Can you provide variance across random seeds and significance testing, and ablations isolating the impact of each design choice (hops, time features, speaker flags, match types)?
