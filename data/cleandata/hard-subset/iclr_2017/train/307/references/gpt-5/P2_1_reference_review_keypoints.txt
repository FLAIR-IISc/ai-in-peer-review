- Introduces an open, reproducible benchmark suite for end-to-end goal-oriented dialog evaluation.  
- Defines five controlled synthetic tasks (slot elicitation/updates, API calling, KB-based option ranking, KB-grounded info, and full-dialog) with OOV entity splits.  
- Provides two real-world datasets: a reformulated DSTC2 and a proprietary Concierge human-human dataset to validate transfer.  
- Systematically evaluates baselines: rule-based, IR, supervised embeddings, MemN2N, and a “match type features” method for OOVs.  
- Uses both per-response and per-dialog accuracy, aligning per-dialog with task completion.  
- Finds MemN2N outperform IR and embedding baselines, can learn API calls and some OOV handling, but fails on KB interpretation, option ranking, and full-dialog success.  
- Paper fills a gap by offering a reproducible, interpretable testbed and empirical analysis of end-to-end methods’ limits.  
- Decision: Accept.  
- Key reason: Significant, timely contribution providing controlled diagnostic tasks and OOV testing for principled evaluation.  
- Key reason: Empirical rigor with broad baselines and clear metrics showing consistent patterns across synthetic and real data.  
- Problem and motivation clearly articulated: existing datasets don’t target end-to-end goal-oriented capabilities.  
- Well positioned relative to prior work (POMDP/state tracking, bAbI, chit-chat corpora).  
- Strong methodology: transparent simulator, principled OOV split, and unambiguous ranking evaluation.  
- Results support claims: analyses show specific strengths and weaknesses of end-to-end models.  
- Reproducibility supported by available data and detailed hyperparameters; tasks are deterministic and controllable.  
- Limitation: Ranking setup with fixed candidate set simplifies generation and may not reflect real deployments.  
- Limitation: Simulator’s deterministic bot prompts and limited linguistic variability reduce realism.  
- Limitation: Match type features require typed KB entity lists at test time, favoring certain models.  
- Limitation: DSTC2 conversion involves post-hoc API call and rating creation, adding assumptions/noise.  
- Suggestion: Add modern baselines that handle copying/entities (pointer-generator, copy-augmented seq2seq, key-value memory networks).  
- Suggestion: Report ablations on candidate set size/composition to quantify sensitivity.  
- Suggestion: Deepen error analysis for T3/T5 (sorting vs. KB conditioning vs. dialog control).  
- Suggestion: Add controlled linguistic variability and non-determinism (paraphrases, varied question orders).  
- Suggestion: Clarify effects of joint vs. per-task training on transfer.  
- Suggestion: Release simulator and evaluation code alongside data to enhance reproducibility.  
- Question: How are candidate sets constructed and how is leakage from test candidates prevented?  
- Question: Were multi-task (joint) trainings across Tasks 1–5 tried and did they improve transfer?  
- Question: On T3, what proportion of errors are sorting mistakes vs. wrong KB conditioning vs. dialog flow errors?  
- Question: How sensitive are models to simulator determinism (e.g., randomized slot elicitation order)?  
- Question: For DSTC2 conversion, how robust are results to different rating assignments for option ranking?  
- Question: Do match type features assume full KB coverage of OOVs at test time, and how does noisy/incomplete typing affect performance?  
- Question: Did you analyze per-dialog turn failures and whether curriculum learning or constrained decoding could help?
