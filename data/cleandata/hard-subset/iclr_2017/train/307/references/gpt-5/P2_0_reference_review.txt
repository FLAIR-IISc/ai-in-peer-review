1) Summary of contributions
The paper introduces an open, reproducible benchmark suite for evaluating end-to-end goal-oriented dialog systems. The suite decomposes a restaurant-reservation task into five synthetic subtasks (API call issuance, API update, option display with sorting, knowledge retrieval, and full dialogs), adds an OOV evaluation split to stress generalization to unseen entities, and includes two “real” datasets: a converted DSTC2 set (Task 6) and a proprietary concierge chat dataset. The authors benchmark rule-based systems, IR baselines, supervised embeddings, and end-to-end Memory Networks (MemN2N), with a simple yet effective extension (“match type” features) to handle entities and OOV words. Results show: (i) synthetic tasks are completely solvable by rules but remain challenging for learned models; (ii) MemN2N outperform IR and embedding baselines, especially with multiple hops and entity type features; (iii) per-response accuracy can be decent but per-dialog accuracy remains low on harder tasks; and (iv) the relative method ranking on synthetic tasks correlates with that on DSTC2 and concierge data. The main contribution is the benchmark and analysis rather than a novel model.

2) Decision: Accept
Key reasons:
- The benchmark is well-motivated, carefully constructed, and fills a significant gap for reproducible evaluation of goal-oriented dialog with KB grounding and OOV stress tests.
- The empirical study is thorough and yields actionable insights (e.g., where end-to-end models succeed/fail, the utility of hops and type features, and the mismatch between per-response and per-dialog success), with evidence that synthetic tasks reflect trends on real data.

3) Supporting arguments
- Problem clarity and motivation: The paper targets a core open question—can end-to-end models replace traditional slot-filling/POMDP approaches in narrow-domain goal-oriented settings? Existing “chit-chat” corpora and metrics are ill-suited; this resource provides controllable tasks that exercise dialog management, KB grounding, sorting, and OOV entity handling. This is well placed in the literature, bridging bAbI QA tasks and dialog state tracking datasets.
- Methodological soundness: The task design isolates competencies; OOV splits are principled (disjoint cuisine/location sets). The retrieval-based evaluation (Next-Utterance-Classification) is appropriate for controlled comparison and per-dialog accuracy captures goal completion. Baselines span non-learning (TF-IDF, NN), learned retrieval (supervised embeddings), and a competitive end-to-end model (MemN2N), with ablations (hops, match types).
- Claims supported by evidence: Results convincingly show (a) MemN2N > supervised embeddings > IR across tasks; (b) multi-hop improves; (c) match type features are crucial for OOV and for KB fact retrieval (Task 4); (d) Tasks 3/5 remain unsolved, highlighting current model limitations in reasoning over KB results (sorting, option display); (e) relative rankings persist on DSTC2 and concierge, suggesting the synthetic tasks are a useful proxy.
- Impact and practicality: A widely-usable benchmark with clear diagnostics is valuable to the community. The OOV setting and the separation of per-response vs. per-dialog metrics will likely steer model design (e.g., pointer/copy, entity linking, symbolic operations).

4) Additional feedback to improve the paper (helpful, not central to decision)
- Candidate set construction: Using a global candidate pool that includes all responses from train/val/test (including test-only candidates) is common in retrieval tasks but can raise concerns. Clarify why this does not leak target information (parameters are not fit on test text) and report sensitivity to alternative candidate pools (e.g., train+val only).
- Realism of the simulator: While determinism is a feature for diagnostics, note more explicitly how the fixed bot templates and small user template set could limit linguistic diversity. Consider reporting robustness to paraphrase augmentation or noise injection (ASR-like errors) to better bridge to real data.
- Entity handling: Match type features rely on exact KB matches; in practice, NER/linking errors occur. It would strengthen the paper to test noisy entity recognition (or to discuss integration with learned NER/linking) and to report the effect on OOV performance.
- Broader baselines: Although MemN2N is a reasonable choice, including an encoder–decoder with attention and copy/pointer mechanisms (or key–value memory networks) would provide a stronger baseline, especially for Tasks 3/5 and OOV handling.
- Metrics on real data: For DSTC2/concierge, report per-dialog accuracy (if defined) or an alternative notion of task success to better reflect goal completion. If not feasible for concierge, state this limitation explicitly.
- Sorting behavior analysis: Provide qualitative/quantitative error analysis specific to Task 3 (e.g., is the model failing to rank by rating, failing to stop after acceptance, or misinterpreting user rejections?).
- Release details: Please be explicit about what is released (Tasks 1–6 data, code, evaluation scripts) and any licensing constraints. If the concierge set cannot be released, consider releasing a sanitized subset or detailed generation recipes to replicate its properties.

5) Questions for the authors
- Candidate pool: Does including test utterances in the candidate set change results compared to using only train+val candidates? Can you report this sensitivity?
- OOV protocol: Models have access to both KBs during training. How would performance change if the OOV KB were only available at test time (closer to a true “new entities at deployment” scenario)?
- Entity typing: Are type features derived purely from string lookup against the KB, or do you rely on an NER? How robust is performance if entity surface forms vary (misspellings, morphological variants)?
- Task 3 failure modes: What are the dominant errors (sorting vs. control flow vs. grounding)? Did you try architectural variants (e.g., key–value memories, supervised attention over ratings) to help with sorting?
- DSTC2 conversion: You add ratings to enable deterministic option selection. Could this alteration bias the task? How different are the converted dialogs from originals in terms of system behavior variability?
- Per-dialog success: On DSTC2 and concierge, what fraction of dialogs are fully correct (if definable)? If not definable for concierge, can you approximate with a proxy (e.g., correct API usage + closing act)?
- Reproducibility: Will you release code to generate the synthetic dialogs, the exact candidate sets, and the trained MemN2N configurations/hyperparameters to facilitate direct comparison?

Overall, this is a timely, well-motivated benchmark contribution with careful analysis. Acceptance is warranted given its likely impact on research into end-to-end, KB-grounded dialog.
