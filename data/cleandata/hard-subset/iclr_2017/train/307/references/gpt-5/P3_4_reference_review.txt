Summary and contributions
The paper proposes an open evaluation suite for end-to-end goal-oriented dialog in the restaurant reservation domain. Inspired by the bAbI philosophy, it decomposes the end goal into five synthetic tasks (issuing/updating API calls, presenting sorted options based on KB results, providing requested attributes, and full dialogs), and supplements them with two datasets from real interactions (a DSTC2 conversion and a proprietary concierge chat dataset). The benchmark is designed to be lightweight, reproducible, and to enable fine-grained error analysis (including an explicit OOV split to stress entity handling). The paper evaluates several baselines, notably Memory Networks (MemNNs) with/without match type features, against rule-based, TF-IDF, nearest neighbor, and supervised embeddings. Key findings: MemNNs can learn to issue and update API calls and benefit substantially from match type features for OOV entities, but still struggle on tasks requiring using KB results to rank options and complete multi-turn dialogs; per-dialog accuracy remains low even when per-response accuracy is decent. The synthetic tasks’ outcomes reflect relative trends on real data, suggesting they can serve as a proxy.

Decision: Accept (borderline)
Reason: The paper fills a clear gap by providing a reproducible, interpretable benchmark focused on goal-oriented dialogs with KB interaction, and offers a careful, multi-dataset evaluation that surfaces concrete shortcomings of current end-to-end models. While the simulated language is templated and some modeling choices can be debated (ranking setup, typed features), the resource and analysis are valuable to the community and have already influenced subsequent work.

Supporting arguments
- Problem definition and motivation: Strong. There is a well-articulated need for an end-to-end, goal-oriented testbed with controllable structure (including KB reasoning) and OOV entities. The paper situates itself well against POMDP/slot-filling, chit-chat corpora, DSTC, and earlier bAbI-style tasks, and justifies the decomposition into subtasks and the per-dialog metric.
- Methodological clarity and reproducibility: Good. The simulator and datasets are described in detail (domain schema, API semantics, templating, OOV split via disjoint cuisines/locations). The evaluation setup and metrics are clear. Hyperparameters and ablations (hops, match types) are reported; code/data availability is indicated.
- Empirical support: Convincing trends across synthetic and real datasets. The experiments are thorough for the intended baseline scope and support the claims: MemNNs handle implicit state and API calls; match type features are essential for entities and OOV; ranking options from KB results and robust full-dialog success remain challenging. Observing similar relative performance ordering on DSTC2 and Concierge increases external validity.
- Impact: High. A public, decomposed, KB-grounded goal-oriented benchmark (with OOV emphasis) is a useful community contribution and has clear potential to catalyze research.

Caveats
- Synthetic language/template constraints: The simulation uses a small set of bot and user patterns and deterministic question ordering. This eases the tasks and can bias models toward superficial heuristics; it also narrows linguistic variability relative to real usage. The authors acknowledge this and attempt to bridge to real data, but the gap remains.
- Ranking setting and candidate set: The evaluation ranks among a global candidate set that includes utterances from train/val/test and all tasks. This simplifies the problem compared to generation (and even to constrained policy realization), and can inflate results by providing gold responses that never appear in training. It limits conclusions about deployable systems.
- Match type features: The OOV gains are largely due to injecting type-level exact-match features keyed to a provided KB. While reasonable for entity-heavy tasks, it partially reintroduces hand-engineered signals; the paper could better contextualize this against alternative neural mechanisms (copy/pointer networks, key-value memories) that also address entity handling.
- Real-data evaluation: The DSTC2 transformation adds ratings and cannot be compared directly to prior DSTC results; the concierge dataset is proprietary and not released, limiting reproducibility of that part. Still, these sections are framed as external validity checks rather than core contributions.

Suggestions to improve the paper (not prerequisites for acceptance)
- Broaden baselines: Include contemporary models more tailored to entity handling, e.g., pointer/copy mechanisms, key-value memory networks, and encoder-decoder policy models with copying. This would clarify whether match-type features are essential or can be learned.
- Robustness variants of the simulator: Add non-deterministic question ordering, larger/paraphrased templates, lexical noise, and ambiguous/missing KB results to more closely resemble real dialogs; report sensitivity analyses.
- Generation metrics: Complement ranking with generative evaluation (constrained decoding or surface realization) and report task success with slot accuracy and API-argument exact match. This would strengthen claims about deployability.
- Fair candidate construction: Clarify and justify including test utterances in the candidate pool; provide ablations where the candidate pool is restricted to training-only responses.
- Release concierge data (or a de-identified subset) if possible, or provide more detail and scripts to simulate similar noise/variability.

Questions for the authors
- Candidate set: You state candidates are all bot utterances/API calls from train/val/test (plain and OOV) for all tasks combined. Why include test-side utterances? How sensitive are results to restricting the pool to training-only candidates?
- Match type features: What KB access is assumed at test time for typing? How do results change if the KB is incomplete or noisy? Can the model recover without types using a copy mechanism?
- Determinism: In Tasks 1–2, the bot asks for missing slots in a fixed order. How do models perform if this order is randomized, or if user updates conflict/overlap?
- Sorting in Task 3: Can you analyze why MemNNs fail—insufficient attention capacity, need for explicit differentiable sorting, or failures in reading multi-fact evidence? Any ablations on number of options returned?
- DSTC2 conversion: Adding ratings to enforce option ordering may bias the task. Did you verify that this does not introduce artifacts (e.g., leaking answers)? Could you compare against the original DSTC success metric?
- Generalization: Beyond OOV entities within the same schema, can models generalize to unseen slot types or different KB schemas (e.g., hotels)? Any preliminary cross-domain experiments?
- Release artifacts: Will you release the simulator, scripts for DSTC2 conversion, and MemNN training code to ensure full reproducibility?

Overall, this is a well-motivated benchmark paper with careful analysis and useful insights. Despite limitations in realism and evaluation setup, it is a valuable resource for the community.
