Summary and contributions
- The paper introduces an open evaluation suite for end-to-end goal-oriented dialog centered on restaurant reservations. It decomposes the problem into five synthetic tasks (slot elicitation, updating, option display with sorting, knowledge use, and full dialogs), plus a converted DSTC2 benchmark (Task 6) and results on a proprietary concierge dataset.
- It provides baseline comparisons across rule-based systems, IR baselines (TF-IDF, Nearest Neighbor), supervised embeddings, and Memory Networks (MemN2N), and proposes “match type” features to handle entity exact matches and OOV entities.
- The main empirical takeaway is that MemN2N variants outperform simple baselines and can handle some goal-oriented behaviors (issuing/updating API calls, using OOV entities via type features), while still failing on tasks that require sorting and interpreting KB outputs (notably Task 3 and the full dialogs).

Decision: Reject
Key reasons: (1) The evaluation protocol and task design contain several artificial choices that limit external validity and potentially bias results (global response ranking over a candidate set compiled from test data; highly templated, deterministic language; heuristic conversions of DSTC2). (2) The empirical study is underdeveloped methodologically (missing stronger baselines and ablations; unclear hyperparameters/variance; OOV setup and type features undermine the “no domain knowledge” claim).

Supporting arguments
- Unrealistic ranking setup: All models select from a closed candidate set comprising all bot utterances and API calls from training/validation/test and across tasks. This leaks test responses into the output space, departs from deployment realities, and reduces the task to retrieval. It likely inflates absolute performance and penalizes models that could generate valid but unseen paraphrases. For DSTC2 and Concierge, multiple responses can be acceptable; the single-reference ranking metric may be brittle.
- Synthetic language and deterministic policies: The simulator uses 43 user and 20 bot patterns, with the bot speaking in invariant templates and deterministic slot query order. This creates a highly stylized distribution with limited linguistic variability, making gains less informative for real-world goal-oriented dialog.
- OOV setup and KB exposure: Models “have access to the training examples and both KBs.” The OOV test uses entities unseen in training dialogs but present in the KB. This blurs what “OOV” means: tokens can be unseen in dialogs but available as KB entries and thus discoverable by “match-type” features. Embedding-based models are declared “natively impossible” to handle OOV, but this ignores subword/char approaches; more importantly, the protocol confounds OOV with availability of a typed gazetteer.
- Match-type features require domain typing: The approach relies on pre-typed KB schemas and entity gazetteers, which contradicts the “no domain knowledge” and “end-to-end” framing. This is an important caveat not sufficiently discussed.
- Weak/dated baselines for the core capability gaps: Copy/pointer mechanisms (for entity handling), key-value memory networks, neural belief trackers, or even simple attention-based seq2seq with copy are absent. Since the central challenge is symbol manipulation and KB grounding (e.g., sorting; entity carryover), omission of pointer/copy baselines is a missed opportunity and weakens claims about MemN2N’s relative strengths/weaknesses.
- DSTC2 conversion introduces heuristics: Adding ratings and forcing a highest-rating policy to disambiguate options imposes non-native behavior. It is unclear how multi-valid responses are handled; the rule-based baseline is limited (28 rules) and tuned on validation, making comparisons fragile.
- Metrics and statistical rigor: No variance across runs, no confidence intervals or significance tests; hyperparameter search details are sparse in the main text (N negatives, margin m, dimension, early stopping), and results hinge on validation-driven choices (e.g., using last utterance vs. history), raising concerns of overfitting to the specific tasks.
- Ambiguities in task specification: Task 3 does not specify tie-breaking when ratings are equal. Task 2 requires asking “if done with updates,” which seems scripted; how multiple acceptable wordings are judged is unclear. Using a global candidate set “for all tasks combined” may further conflate task-specific distributions.
- Reproducibility and availability: While the synthetic tasks are released, the Concierge dataset is proprietary and not available, limiting reproducibility of those results; code for all baselines and conversion scripts is not clearly committed to release in the paper. The references and appendices contain formatting errors, broken citations, and footnote artifacts, which complicate reproducibility.
- Formatting/clarity issues: Numerous citation glitches (e.g., “Bai et al\.,? 2009”, “Kim et al\.,? \Q2016\E”), repeated author names, stray footnote text in body, and inconsistent punctuation suggest the submission needs a thorough copy-edit.

Additional feedback to improve the paper
- Rework evaluation to avoid using test utterances in the candidate set. Consider generation with constrained decoding plus exact-match constraints, or restrict candidates to training-only. Report sensitivity analyses.
- Add stronger baselines tailored to entity manipulation: pointer-generator, copy-augmented seq2seq, key-value MemNNs, neural belief trackers, or span-prediction over KB facts. Include subword or char-level embeddings for OOV handling to fairly assess the “natively impossible” claim.
- Clarify OOV protocol: vocabulary construction, UNK handling, whether KB-only entities expand the vocabulary, and how match-type features are constructed without dialog exposure. Consider a stricter OOV setting where entities are absent from both dialog and KB at training.
- For DSTC2, document how multiple valid responses are resolved. Provide a multi-reference or semantic equivalence handling. Avoid injecting arbitrary ratings if possible, or isolate their effect.
- Report multiple runs with confidence intervals, full hyperparameters (N negatives, margin, d, hops, learning rates), candidate set sizes per task, and training/inference times.
- Expand error analysis for Tasks 3–5 to pinpoint failure modes (sorting mistakes vs. grounding mismatches vs. dialog management).
- Fix all references/formatting issues; ensure release of code and conversion scripts; clarify whether Concierge can be released or remove it.

Questions for the authors
1) What is the exact size of the candidate set per task at test time, and how does accuracy change if the set excludes test and cross-task utterances?
2) How do you handle cases with multiple acceptable bot responses (especially in T6 and Concierge)? Is there any multi-reference evaluation?
3) Precisely define the OOV setup: Are tokens from the OOV KB added to the vocabulary during training? How are UNKs handled in embeddings? Do match-type features use only KB membership or also training dialogs?
4) For Task 3, what tie-breaking is used for equal ratings? How often does this occur?
5) Why does adding match-type features hurt T2 slightly? What ablations isolate the contribution of types vs. attention hops?
6) Did you try pointer/copy mechanisms or subword embeddings? If so, how do they compare on OOV tasks?
7) For DSTC2 conversion, can you provide details and code, and quantify how much the added rating heuristic changes the label distribution vs. originals?
8) How many runs per model are reported, and what is the variance across seeds?
