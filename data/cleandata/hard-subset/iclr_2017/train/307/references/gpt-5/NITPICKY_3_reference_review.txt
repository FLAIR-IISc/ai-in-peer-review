Summary and contributions
- The paper proposes an open evaluation resource for end-to-end goal-oriented dialogue in a constrained restaurant reservation domain. It decomposes the problem into five synthetic tasks (API issuing, updating, option presentation with sorting, KB fact retrieval, full dialogues) plus a converted DSTC2 task and an additional real “Concierge” dataset.
- It frames evaluation as next-utterance ranking with per-response and per-dialog accuracies, and reports baselines: TF–IDF, nearest neighbor, supervised embeddings, and Memory Networks (MemN2N), with a match-type feature extension to better handle entities and OOV tokens.
- Experiments suggest MemN2N outperform IR and embedding baselines; match-type features help markedly on OOV and KB-fact retrieval. Nevertheless, end-to-end models struggle on tasks requiring sorting and multi-step reasoning (T3/T5). The pattern of relative performance on synthetic tasks is claimed to transfer to DSTC2 and Concierge.

Decision: Reject
Key reasons:
1) Experimental design issues that substantially confound conclusions: ranking over a candidate pool that includes test and OOV responses across all tasks; strong templating/determinism in simulated data; and granting access to both KBs at training time while evaluating OOV. These choices make the setting less realistic and bias results.
2) Insufficient empirical rigor and baseline coverage for a benchmark paper: limited comparisons (e.g., no competitive copy/pointing/seq2seq baselines), no statistical significance/variance, and limited error analyses support the main claims only partially.

Supporting arguments
- Candidate pool construction: The paper states candidates are “from training, validation and test sets (plain and OOV) for all tasks combined.” Including test-only responses in the candidate set is a major leakage in a ranking setup and artificially simplifies OOV selection (the model is asked to rank strings it would never have generated). It also biases the comparisons against generative models and towards methods that can exploit string-level matching/type features. This undermines the core claim that the benchmark faithfully evaluates goal-directed ability.
- OOV setup: Systems have access to both KBs at training, while OOV evaluation uses the disjoint KB. Combined with match-type features that consult the KB at test time, this makes OOV “generalization” partly a lookup problem rather than learning to generalize. The conclusion that “embedding methods cannot generalize” is too strong given this design; a fairer setup would prevent using test-only KB entity inventories during training, or at least disentangle effects with ablations.
- Synthetic tasks: The simulator uses 20 deterministic bot patterns, a fixed slot-asking order, and templated language. This invites learning surface heuristics and yields fragile conclusions about dialog management. The sorting task (T3) is a useful stressor, but it is not demonstrated that models actually learned sorting versus memorizing canonical outputs across a small template space.
- Baselines: Absent are strong modern baselines for entity-heavy goal-oriented dialog, e.g., pointer/copy-augmented seq2seq models, key–value memory networks, retrieval+copy hybrids, or transformer retrieval baselines with exact-match features. The rule-based baseline for DSTC2 is weak and hand-tuned on validation, and the paper cannot compare with DSTC leaderboards due to task conversion. Claims of outperforming a “dedicated” baseline are thus limited in significance.
- Metrics and rigor: No reporting of confidence intervals/variance over multiple seeds on small training sets (1k dialogs). Per-dialog accuracy is emphasized but not complemented with error breakdowns per capability (e.g., slot grounding vs. sorting vs. KB fetch vs. copy). Claims like “MemNNs can issue API calls and manipulate OOV words” are plausible but would benefit from targeted probes.
- Reproducibility and openness: While Tasks 1–5 are stated to be available, the paper also relies on a proprietary Concierge dataset for conclusions about realism; absence of public release weakens the case that the benchmark is an open, reproducible proxy for real-world performance.
- Clarity/formatting: Numerous reference and LaTeX artifacts (“? 2016”, “\Q2016\E”, dangling parentheses, footnotes inlined mid-paragraph, duplicated fragments) and missing figures/tables in the current draft impede readability and confidence. These are not fatal scientifically, but they are pervasive.

Additional feedback to improve the paper
- Fix the ranking evaluation design: restrict the candidate pool to responses observed in training (or train+val) only, or separate retrieval (from a large fixed pool) from generation, and avoid including test-specific strings. Report results under both “closed” (train-only) and “open” (train+KB lexicon) candidate pools to quantify the effect.
- Revisit the OOV protocol: do not expose the test KB surface forms during training; if match-type features consult the KB at test, clearly separate “KB-available” versus “KB-unknown” conditions. Provide ablations where the KB lexicon is withheld to assess true compositional generalization.
- Strengthen baselines: include copy/pointer networks, key–value memory networks, and a modern retrieval baseline (e.g., dual-encoder transformers) with exact-match/type features. Also include a simple finite-state/slot-filling baseline on T1–T2 to calibrate difficulty.
- Report robustness: averages over multiple random seeds; significance tests; error breakdowns by capability (slot elicitation, KB lookup, sorting, entity copying).
- Clarify the DSTC2 conversion and Concierge evaluation: define success in Concierge (no KB outputs), report per-dialog metrics if applicable, and discuss domain/task mismatch. If Concierge is not releasable, mark it clearly as illustrative and avoid using it to support general claims.
- Improve documentation: release the simulator, data generation scripts, and conversion code; provide precise templates, random seeds, and KB splits.

Questions for the authors
1) Candidate set: Do you include any responses that only appear in test/OOV sets in the candidate pool? If so, can you report results when restricting candidates to train-only responses?
2) OOV protocol: During training, do models have access to the surface forms of entities from the test-only KB via match-type features or any preprocessing? Can you ablate this to isolate true OOV generalization without KB lexicon exposure?
3) Sorting in T3: How do you verify that models learned to sort by rating rather than memorizing template orders? Any controlled probes (e.g., adversarial shuffles, out-of-template orders)?
4) Baselines: Why not include pointer/copy models or key–value memory networks? Do you expect match-type features to approximate copying, and can you compare directly?
5) Variance: What is the run-to-run variability across seeds for per-dialog accuracy on T3/T5/T6?
6) DSTC2 conversion: Can you release the conversion code and detail how errors in ASR/bot behavior were handled? How does your evaluation relate to DSTC official metrics?
7) Concierge dataset: Will it be released? How is per-dialog success defined there given missing KB outputs?
