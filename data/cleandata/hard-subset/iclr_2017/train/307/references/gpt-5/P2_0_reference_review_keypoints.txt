- Paper contribution: introduces an open, reproducible benchmark suite for end-to-end goal-oriented dialog with five synthetic subtasks, OOV split, and two real datasets (DSTC2-converted and concierge).
- Task design: decomposes restaurant-reservation into API call, API update, option display/sorting, knowledge retrieval, and full dialogs to isolate competencies.
- OOV evaluation: includes disjoint cuisine/location sets to stress generalization to unseen entities.
- Datasets: provides synthetic tasks plus a converted DSTC2 set and a proprietary concierge chat dataset (concierge possibly not releasable).
- Baselines and models: benchmarks rule-based systems, IR baselines, supervised embeddings, and MemN2N with a “match type” feature for entity/OOV handling.
- Key result 1: synthetic tasks are solvable by rules but remain challenging for learned models.
- Key result 2: MemN2N outperforms IR and embedding baselines, especially with multiple hops and entity type features.
- Key result 3: per-response accuracy can be decent but per-dialog accuracy is low on harder tasks.
- Key result 4: relative method rankings on synthetic tasks correlate with DSTC2 and concierge results, suggesting synthetic tasks are a useful proxy.
- Main contribution: the benchmark and analysis rather than a novel model.
- Decision: Accept — benchmark fills a gap, is well-constructed, and the empirical study is thorough and actionable.
- Problem clarity/motivation: addresses whether end-to-end models can replace slot-filling/POMDP in narrow goal-oriented settings and bridges bAbI and dialog state tracking work.
- Methodological soundness: principled OOV splits, controlled task design, appropriate retrieval evaluation, and a range of baselines with ablations.
- Evidence supports claims: multi-hop and match-type features help; Tasks 3/5 remain unsolved; synthetic trends match real data.
- Impact/practicality: the benchmark, OOV setting, and per-dialog metrics will steer model design toward entity handling and symbolic operations.
- Feedback — candidate pool: clarify inclusion of test utterances in candidate pool and report sensitivity to alternative pools (train+val only).
- Feedback — simulator realism: clarify limits from deterministic templates and small user templates; consider paraphrase/noise robustness tests.
- Feedback — entity handling: test noisy NER/linking or discuss integration with learned entity linkers since match types assume exact KB matches.
- Feedback — broader baselines: add encoder–decoder with attention+copy or key–value memory baselines for Tasks 3/5 and OOV handling.
- Feedback — real-data metrics: report per-dialog accuracy or task success for DSTC2/concierge or state limitations if infeasible.
- Feedback — sorting analysis: provide qualitative/quantitative error analysis specific to Task 3 (ranking vs. control flow vs. grounding).
- Feedback — release details: explicitly state what will be released (data, code, eval scripts) and licensing; consider sanitized concierge alternative if needed.
- Questions — candidate pool: does including test utterances in candidates change results versus train+val-only?
- Questions — OOV protocol: how does performance change if the OOV KB is only available at test time?
- Questions — entity typing: are type features from KB string lookup or NER, and how robust to surface-form variation?
- Questions — Task 3 failures: what are dominant error modes and were architectural variants (key–value memory, supervised attention) tried?
- Questions — DSTC2 conversion: does adding ratings and making selection deterministic bias the task relative to originals?
- Questions — per-dialog success: what fraction of DSTC2/concierge dialogs are fully correct or approximated by proxies?
- Questions — reproducibility: will code to generate synthetic dialogs, exact candidate sets, and trained MemN2N configs/hyperparameters be released?
