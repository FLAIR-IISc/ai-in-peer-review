1. Summary and contributions
The paper introduces a publicly available benchmark suite for end-to-end goal-oriented dialog in the restaurant reservation domain, modeled after bAbI-style tasks. It decomposes the problem into five synthetic tasks (API issuance, updates, option presentation with sorting, information provision from KB results, and full dialogs), plus two “real” datasets: a reformatted DSTC2 and an anonymized concierge dataset. The work aims to create a lightweight, reproducible way to probe specific capabilities: implicit state tracking, KB querying, reasoning over KB outputs (including sorting), and handling out-of-vocabulary (OOV) entities. The authors benchmark rule-based systems, IR baselines, supervised embeddings, and MemN2N, and propose “match-type” features to mitigate entity/OOV issues. They report that MemN2N with match-types perform best overall, solve API issuance and information-provision, but still struggle on option sorting and full dialogs, with low per-dialog accuracy even when per-response is decent. The dataset and analyses are intended to catalyze progress and provide error analysis by design.

2. Decision
Accept. Key reasons: (1) The paper provides a well-motivated, reusable, and thoughtfully designed benchmark for end-to-end goal-oriented dialog with controlled diagnostics and OOV generalization tests; (2) The empirical analysis is careful and reveals concrete failure modes of current end-to-end systems, offering a clear research agenda.

3. Supporting arguments
- Problem relevance and positioning: The gap between chit-chat models and goal-oriented dialog is real, and the paper is well-situated among POMDP slot-filling work and non-goal-oriented end-to-end dialog. The link to DSTC2 and the concierge dataset strengthens external validity.
- Methodological clarity and reproducibility: The tasks are clearly specified, with explicit KB schema, simulation details, and evaluation metrics (per-response and per-dialog). OOV tests are thoughtfully constructed. Releasing both the synthetic and converted datasets (and describing candidate sets and memory setup) promotes reproducibility and fair comparison.
- Diagnostic value: By decomposing capabilities (API calls, updates, KB interpretation, sorting, OOV), the benchmark yields insightful error analysis (e.g., MemN2N handles API issuance but fails to sort options robustly). The observed ranking of methods (MemN2N > supervised embeddings > IR) aligns across synthetic tasks and real data (DSTC2/Concierge), supporting claims that the synthetic suite is a useful proxy.
- Technical contributions: While not a novel model, the match-type feature augmentation is a simple, interpretable, and effective mechanism to handle entity exact match and OOV issues, substantially improving OOV and KB-reading tasks (especially Task 4).

4. Additional feedback (to help improve the paper)
- Evaluation protocol and candidate set: Clarify whether the union of candidates from train/val/test (including OOV test) is available during training as negatives or only at inference. Ranking against a candidate pool that contains delexicalized templates plus arbitrary entity instantiations can overstate OOV robustness vs. generative settings. Consider reporting results with candidate pools restricted to training-time candidates only, and an evaluation that requires composing responses with unseen entities via copy mechanisms.
- Baseline coverage: Including strong sequence models with copy/pointer mechanisms, key-value memory networks, or neural module networks tailored for KB reasoning would strengthen conclusions. In particular, Task 3 (sorting) invites architectures that can perform explicit symbolic sorting or use differentiable sorting/pointer networks; even a simple heuristic “max-by-rating” module would be an informative upper bound beyond the rule-based system.
- Entity handling realism: Match-type features assume high-quality KB linking and exact string matches. It would help to report performance with automatic NER/linking (no oracle matches) to better reflect real-world conditions, and to quantify sensitivity to noisy entity mentions (typos, morphology).
- DSTC2 conversion: You add ratings to enable deterministic option ranking. Please analyze the impact of this augmentation: is model performance sensitive to rating distributions or the deterministic choice rule? Also, the rule-based baseline is helpful, but comparisons to adapted RNN baselines used in DSTC literature (e.g., using the transcripts but not state annotations) would further validate the claims.
- Metrics and error analysis: Per-dialog accuracy is informative but harsh; add complementary metrics such as API call field accuracy, entity F1, decision accuracy for “when to stop proposing,” and breakdowns by dialog phase. Present qualitative error typologies on Task 3/5 to guide future model design.
- Data realism: The synthetic language patterns are quite limited (43 user, 20 bot patterns). Consider adding paraphrase augmentation or a crowd-sourced paraphrase layer to mitigate lexical overfitting. Reporting generalization to paraphrased templates would be valuable.
- Access to OOV KB: In OOV settings, the model has access to the OOV KB during training (for type matching), though not to dialogs. This is a reasonable assumption for production, but please also report a stricter setting without access to OOV KB entries until test time to tease apart the reliance on the entity lexicon.

5. Questions for the authors
- Candidate pool: Are test-only candidates included during training-time negative sampling, or only at evaluation? How does performance change if the candidate pool is restricted to training candidates?
- OOV setup: Do models have access to the OOV KB lexicon during training, or only at inference? Can you report results when OOV KB items are hidden until test?
- Sorting in Task 3: What proportion of errors stem from incorrect top-1 selection vs. failure to advance through the ranked list or to stop when the user accepts? Any evidence that additional hops help specific sub-steps?
- DSTC2 conversion: How often do the original bot responses violate deterministic rules (e.g., ask order, option order)? Do such inconsistencies account for a significant performance ceiling?
- Entity recognition: Are match-type features based on exact string match against the KB, or was any normalization/NER applied? How robust are results to spelling errors or minor variations?
- Concierge dataset: Since no KB outputs are provided, what fraction of turns are “decision/action” vs. “surface realization”? Could you decouple the tasks (API-call decision classification vs. response ranking) to better understand where models struggle?

Overall, this is a timely, well-motivated benchmark and analysis paper that, despite limitations (synthetic patterns, ranking-only evaluation, limited baselines), will likely be influential in structuring progress on goal-oriented dialog.
