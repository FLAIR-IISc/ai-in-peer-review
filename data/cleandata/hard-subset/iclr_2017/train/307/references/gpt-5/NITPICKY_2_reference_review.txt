Summary
This paper proposes a lightweight, reproducible testbed for end-to-end, goal-oriented dialog in the restaurant reservation domain. It decomposes the task into five simulator-generated subtasks (API call issuing, updates, option display with sorting, KB-based information provision, and full dialogs) plus two evaluations on real data (DSTC2-converted and a proprietary concierge dataset). The authors benchmark IR baselines, supervised embeddings, and end-to-end Memory Networks (with and without “match type” features for entities). They report that MemNNs outperform baselines and can learn non-trivial behaviors (e.g., issuing API calls, handling OOV entities with type features), but still fall short on tasks requiring reasoning over API results (sorting options) and achieving high per-dialog success. The resource is intended as a shared, interpretable evaluation proxy that correlates with performance on real dialogs.

Decision: Reject
Key reasons:
1) Evaluation design and leakage concerns: the ranking setup uses a candidate pool built from train/validation/test (including OOV) across all tasks and grants access to both KBs during training, undermining the validity of claims about OOV generalization and inflating the realism of the evaluation.
2) Methodological and reporting gaps: limited ablations, no variance/significance, weak coverage of competitive neural baselines (e.g., copy/pointer mechanisms, seq2seq/transformers), and an overreliance on exact-match ranking in noisy real dialogs.
3) Numerous presentation/formatting issues and inconsistencies that obscure rigor and reproducibility (malformed references, LaTeX artifacts, ambiguous details).

Supporting arguments
- Candidate set contamination: The candidate pool includes responses from test sets and all tasks, which is not a realistic deployment setting and can bias results. It also conflates tasks and gives the model knowledge of response forms it should not have. This weakens per-response and per-dialog numbers and complicates comparison to methods that would otherwise need to generate or retrieve under tighter constraints.
- OOV setup design choices: The model is allowed access to both KBs during training. Combined with match type features that trigger when the entity is present in the KB, this substantially simplifies the OOV problem (the model can “know” all OOV entities’ types a priori). This reduces the external validity of the OOV results and blurs the distinction between generalization to unseen lexical items and simple KB lookups with string matching.
- Metrics for real data: Exact-match ranking in DSTC2-converted and Concierge is brittle due to multiple valid responses and non-deterministic behavior. The paper acknowledges this but still reports only per-response and per-dialog accuracies, with no complementary metrics (e.g., API-call argument accuracy, slot-F1, option-sort correctness, entity-level F1, or success-rate annotations). This makes it hard to substantiate claims about “goal achievement.”
- Missing baselines and ablations: No comparison to strong contemporary end-to-end models for goal-oriented dialog that incorporate copying/pointer or key–value memory mechanisms, hierarchical RNNs, or (now-standard) attention-based sequence models. Ablations on number of hops, memory variants, training size sensitivity, and type features are thin; no variance across seeds or significance tests are provided.
- Simulator design choices reduce ecological validity: Deterministic bot prompting order in T1 and the forced acceptance behavior in T3 simplify the problem and may bias learned behaviors. Sorting by rating is a good stressor, but a toy sort in a small candidate space with fixed patterns may not transfer to realistic KB reasoning.
- DSTC2 conversion introduces annotation bias: Adding ratings post hoc and assuming “highest rating” governs behavior to make responses predictable can diverge from the original data’s intent, undermining comparability and interpretability.
- Reproducibility and data availability: The proprietary Concierge dataset is not released, yet it features prominently; this undermines the repeatability of those results. Core code/hyperparameter details are deferred to appendices; some details remain ambiguous (e.g., tokenization of multi-word entities, address/phone handling, normalization).
- Presentation quality concerns: Several references are malformed or duplicated; LaTeX artifacts remain (“\Q2016\E,” stray parentheses, duplicated author names, unmatched footnotes). Typos and hyphenation issues (“endto-end”) appear repeatedly. These are distracting and hinder verification.

Additional feedback (to help improve the paper)
- Evaluation
  - Avoid building candidate sets from test utterances and across tasks. Provide per-task candidate pools constructed only from training/validation or adopt constrained generation with normalization and multi-reference matching.
  - Report complementary metrics: API-call field accuracy, sort correctness for T3, entity-level F1 for T4, and overall task-completion/success (human-judged where needed) for T5/6/Concierge. For DSTC2/Concierge, consider weakly supervised or reference set matching to reduce exact-match brittleness.
  - Provide variance over multiple seeds and significance tests.
- OOV protocol
  - Preclude access to the OOV KB during training; at test time, provide only the KB needed for inference. Clarify whether OOV benefits are due to type matching against a known KB vs. generalization of the model’s parameters.
  - Explicitly handle multi-token entities and normalization (e.g., phone/address formats).
- Baselines and modeling
  - Include stronger baselines: sequence-to-sequence with attention and copy/pointer mechanisms; key–value memory networks; hierarchical encoders; simple retrieval-augmented generation. Provide ablations of match type features vs. more general copy mechanisms.
  - Examine multi-task vs. per-task training; report whether a single model can solve all tasks with shared parameters.
- Simulator realism
  - Relax deterministic behaviors (question order, acceptance probabilities) and introduce paraphrase diversity on the system side. Explore larger training sets to assess data scaling and whether sorting remains hard because of capacity or data scarcity.
- Data and artifacts
  - Release the DSTC2 conversion scripts, all code, and as much of the Concierge preprocessing pipeline as possible. If Concierge cannot be released, consider a comparable public corpus or a synthetic proxy with equivalent noise characteristics.
- Writing/formatting
  - Fix all malformed references and LaTeX artifacts. Ensure consistent terminology and hyphenation, and remove duplicated or stray sentences in Appendix A.

Questions for the authors
1) How large is the candidate set per task, and how do performance and runtime scale with its size? What happens if you restrict candidates to training-only utterances?
2) What is the exact protocol for allowing access to both KBs during training? If you withhold the OOV KB until test time, how do OOV results change?
3) How are multi-token entities (e.g., “New York,” addresses) handled in type matching? Are tokens lowercased/normalized? How do you treat formatting variants for phone numbers?
4) Are models trained separately per task or jointly? If jointly, what is the multi-task training setup and performance impact? If separately, why is that acceptable for a “general” dialog evaluation?
5) Could you report entity-level metrics (slot F1, API-call field accuracy) and sorting accuracy for T3? For T6/Concierge, can you provide multiple references or lenient matching to account for non-determinism?
6) How stable are results across random seeds and hyperparameters (e.g., number of hops, embedding dimension)? Please provide mean±std over multiple runs.
7) Why exclude modern copy/pointer baselines, which seem directly relevant to the OOV and entity issues you identify?
