1. Summary of contributions
- The paper introduces a lightweight, reproducible testbed for evaluating end-to-end goal-oriented dialog systems in the restaurant reservation domain. It decomposes the overall task into five synthetic subtasks (API issuing, API updating, option display with sorting, KB fact retrieval, and full dialogs), plus a conversion of DSTC2 into the same format (Task 6), and a proprietary human-human “Concierge” dataset for external validity.
- A key design choice is an OOV split that stresses entity generalization by ensuring novel restaurants, cuisines, and locations at test time while allowing access to the KB, thereby testing whether models can use symbols they have not seen in training dialogs.
- The authors benchmark IR baselines, supervised embeddings, MemN2N (with and without simple “match type” features for entities), and a rule-based system. They report both per-response and per-dialog accuracies, highlighting that current end-to-end models often achieve reasonable local accuracy but poor end-to-end task completion.
- Empirically, MemN2N outperform IR and supervised embeddings, particularly on issuing/updating API calls; adding match type features substantially improves OOV handling and enables answering KB lookups (Task 4). Performance gaps on option display (sorting) and full dialogs persist, and similar patterns hold on DSTC2 and the Concierge dataset.

2. Decision
Accept. The main reasons are:
- Novel, clear, and useful benchmark design that decomposes goal-oriented dialog into interpretable subtasks, with a principled OOV evaluation protocol. This is a valuable resource likely to catalyze progress and more meaningful comparisons.
- The paper convincingly shows where current end-to-end systems succeed and fail on goal-oriented tasks, providing a roadmap for future model innovations.

3. Supporting arguments
- Problem significance: End-to-end dialog models have largely been evaluated on chit-chat; goal-directed tasks require KB grounding, state tracking, and decision-making. The proposed tasks directly target these capabilities.
- Motivation and positioning: The paper situates itself well relative to POMDP/slot-filling systems (data-efficiency but domain-bound), existing dialog corpora (limited for end-to-end training or not reproducible), and bAbI QA tasks (not full dialog). The case for a reproducible, controlled, and interpretable testbed is strong.
- Technical soundness: The task design is thoughtful (e.g., deterministic policies where appropriate for learnability; sorting in Task 3; explicit KB grounding; per-response vs per-dialog metrics). The OOV split is well-motivated and probes a critical weakness of embedding-based models. Baselines are appropriate and tuned; ablations and hyperparameters are documented. The match type feature is a simple but effective idea that substantially improves OOV handling in a realistic way (KB available at test time).
- Evidence: Results across synthetic tasks, converted DSTC2, and Concierge show consistent relative ordering of methods, suggesting the synthetic tasks are informative proxies. Analyses of hops and attention examples increase interpretability.

4. Feedback to improve the paper (helpful, not all decision-critical)
- Candidate-ranking vs generation: The evaluation frames response selection as ranking from a fixed candidate set that includes utterances from test data across tasks. This is pragmatic and reproducible, but it would be valuable to discuss implications (e.g., ceiling effects, reliance on candidate coverage) and provide a path to a generative evaluation with copy/pointer mechanisms.
- Determinism and diversity: Some subtasks enforce deterministic system behavior (e.g., question order). Consider adding variants with controlled stochasticity and paraphrasing to better reflect real-world variance while retaining interpretability.
- Option display (Task 3): Sorting is central here; please clarify tie-breaking and whether equal ratings occur. A variation with non-unique ratings and explicit tie policies would prevent overfitting to trivial heuristics.
- OOV protocol: The choice to provide both KBs at training (while dialogs use only one) is reasonable to enable type matching. Make explicit how this setup maps to deployment scenarios, and report results if the KB is restricted to the training half at train time to isolate the effect of KB exposure.
- DSTC2 conversion: You introduce ratings to enable deterministic option ordering. Please quantify how often this alters the original dialog flow and whether it biases evaluation. A small human verification sample would be reassuring.
- Error analysis: Add a compact breakdown of dominant error types per task (e.g., sorting failures vs entity copy vs state carryover), ideally with a few confusion examples, to guide model development.
- Baselines: Including a seq2seq/Transformer with copy/pointer networks and retrieval-augmented models would strengthen the baseline suite, especially on Tasks 3–5 and OOV.
- Metrics: Consider reporting goal success with length-normalized or partial credit, and latency/turn-efficiency, to complement per-dialog accuracy.

5. Questions for the authors
- Candidate set construction: You state candidates include all bot utterances and API calls from training, validation, and test (plain and OOV) for all tasks. Does this introduce any bias by exposing test-only phrasings during training-time scoring function learning, or is the candidate set only used at test time? Please clarify the exact training procedure and justify this design.
- Sorting in Task 3: How are ties handled and do ties occur in practice? If so, what is the ground-truth ordering policy for equal ratings?
- OOV KB access: At train time, models can see both KBs. Have you measured performance if the training-time KB is limited to the “seen” half (to simulate not knowing future entities), and type features are computed only from that? How much does the extra KB exposure contribute?
- DSTC2 conversion fidelity: How often did the deterministic highest-rating rule contradict the original system behavior? Do you have statistics on such mismatches and their effect on evaluation?
- Generalization beyond restaurants: Can the simulator and task generator be easily reparameterized to other domains/KB schemas, and will you release code for doing so?
- Per-dialog failure modes: For Tasks 3 and 5, what proportion of failures are due to a single persistent error early in the dialog vs scattered local mistakes? Any evidence that increasing memory hops further helps?

Overall, this is a timely and ambitious benchmark contribution with clear potential to steer the community toward more rigorous and goal-driven dialog evaluation. I recommend acceptance.
