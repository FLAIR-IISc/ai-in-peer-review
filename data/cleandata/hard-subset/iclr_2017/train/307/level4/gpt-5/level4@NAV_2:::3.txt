1. Summary of claims and contributions
- The paper presents a new evaluation resource for end-to-end, goal-oriented dialog, instantiated in a restaurant-booking domain. Conversations are generated by simulating user requests against a knowledge base (KB) of restaurants with attributes (e.g., cuisine, price range), and mapping tuples to natural language via rule-based templates.
- The task is framed as next-response selection: at each turn, models rank a fixed set of candidate system utterances (including API calls), and are assessed with per-response and per-dialog accuracy.
- The authors compare several baselines and show that Memory Networks outperform simple bag-of-words and classic IR baselines on this setup.

2. Decision
Accept.
Key reasons:
- A reusable, controlled testbed for goal-oriented dialog is valuable for comparability and diagnostics.
- The empirical study is careful and shows clear gains of an end-to-end model over strong, simple baselines under a well-defined evaluation protocol.

3. Supporting arguments
- Problem addressed: assessing whether end-to-end models can handle core capabilities required in transactional dialog (tracking user constraints, issuing and updating API calls, and using returned KB facts), beyond chit-chat settings.
- Motivation and placement: The case for end-to-end models is strong in open-domain dialog; this work examines whether those benefits translate to narrow domains where traditional slot-filling approaches have historically done well.
- Evidence: The ranking setup allows unambiguous automatic evaluation. Results indicate consistent improvements of Memory Networks over bag-of-words style baselines on both per-response and, to a lesser extent, per-dialog metrics, supporting the claim that end-to-end models can learn non-trivial operations in this goal-oriented context.

4. Additional feedback for improvement (helpful suggestions; not the basis of the decision)
- Ranking versus generation: While ranking simplifies evaluation, it limits generalization to new domains if the candidate set is derived from train/val/test responses. Including a generation-based experiment (even with simple constrained decoding) would strengthen claims about scalability and domain transfer.
- Candidate set construction: Please clarify how candidate responses would be obtained in practical deployments, particularly when porting to a new domain where such a list may not exist a priori. A discussion of how this aligns with the “minimal manual effort” motivation would be helpful.
- Template variability and release: Since dialogs are produced via a finite set of NL templates applied to (cuisine, price, etc.) tuples, the language variability is inherently bounded. Publishing the full set of templates/rules used to render user and system utterances would support reproducibility and let others probe robustness to paraphrase.
- Scope and positioning: The paper makes a compelling case for a controlled benchmark. A brief discussion of when a hand-engineered, domain-specific system might still be preferable (e.g., when extremely high reliability is required) would help contextualize the contribution.

Questions for the authors
- How is the candidate response set constructed in practice, especially for a new domain? If it is the union of all utterances across splits, how would that translate to a real deployment without substantial manual curation?
- Do you have plans or preliminary results for a generation setting (e.g., seq2seq or retrieval-augmented generation) to evaluate whether end-to-end models can generalize beyond a fixed candidate pool?
- Can you release the full list of natural-language generation templates used by the simulator for both user and system sides? If not, could you provide statistics on template coverage and variability?
- Given the focus on narrow-domain tasks, how do you envision end-to-end models complementing or replacing hand-engineered systems where domain knowledge is readily available and user satisfaction targets are strict?
