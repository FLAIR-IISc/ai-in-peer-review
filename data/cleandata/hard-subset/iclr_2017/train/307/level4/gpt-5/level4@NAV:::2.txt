Review- The push to deploy chatbots for nearly all human–computer interactions surged in 2016, with claims that they could handle dialogues far beyond small talk. This paper offers a serious and much-needed reality check. While it is primarily relevant to Dialog/Natural Language venues (to inform software engineers about the limits of current chatbots), it is also suitable for Machine Learning venues (to remind researchers of the need for more realistic validation of ML for dialogue), and I view it as highly significant.

Two key conjectures underlie this work and are likely to spur further research. Although not written explicitly, Antoine Bordes articulated them during a NIPS workshop talk covering this work. Given the metrics used in the paper:
1) The performance of end-to-end ML approaches remains inadequate for goal-oriented dialogues.
2) When comparing algorithms, relative performance on synthetic data is a good predictor of performance on natural data. This would notably diverge from prior observations, but the authors have made a strong effort to align synthetic and natural conditions.

Although the algorithmic novelty is a relatively simple addition to memory networks (match type), this is the first deployment and evaluation on a goal-oriented dialogue, and the experimental protocol is excellent. The paper is very clear and accessible to readers beyond ML and dialogue specialists. I was especially impressed by the succinct appendix that summarizes memory networks, followed by tables that illuminate the effect of the number of hops.

While this paper sets the state of the art in exploring more rigorous metrics for dialogue modeling, it also highlights how brittle and somewhat arbitrary these metrics still are. This is intended as guidance for future work rather than a request for revision.

First, they use per-response accuracy (essentially next-utterance classification from a fixed list). Table 3 makes clear how unreasonable this can be in practice: the crucial pieces are a correct API call and a reasonably short dialogue, yet this would yield only 1/7 accuracy because the six bot responses needed to reach the API call must also be exactly correct.

Would per-dialog accuracy, requiring all responses to be correct, be better? Table 2 shows how sensitive it is to the experimental protocol. I was initially puzzled that subtask T3 accuracy (0.0) was much lower than the full dialogue T5 (19.7), until the authors pointed me to the task definitions (3.1.1), where T3 requires showing 3 options while T5 requires showing only one.

For the concierge data, what would happen if “correct” meant being the single best, not merely among the top-5?

While I cannot fault the authors for using standard dialogue metrics and even proposing new ones that are arguably too pessimistic, I can suggest one representation that might yield more meaningful metrics for goal-oriented dialogues. Suppose I sell Virtual Assistants as a service, paid upon successful completion of a dialogue. What metric maximizes my revenue? In the restaurant setting, the loss would likely be a weighted sum of errors in the API call, the number of turns to reach that API call, and the number of user-rejected options. Such a loss cannot be measured on canned dialogues and would require either real human users or a realistic simulator.

Another issue closely tied to representation learning that the paper does not address sufficiently is vocabulary mismatch between the user and the knowledge base. For the match type method to tag “Indian” as a “type of cuisine,” that word must appear verbatim in the KB. I can envision cases where the KB uses obfuscated terminology, and we would prefer ML to learn these associations rather than relying on hand-specified mappings.
