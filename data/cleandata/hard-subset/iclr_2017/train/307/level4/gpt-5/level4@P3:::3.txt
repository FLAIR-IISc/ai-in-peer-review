Summary of contributions
- The paper proposes an open, reproducible testbed for evaluating end-to-end goal-oriented dialog systems. It decomposes the restaurant reservation domain into five synthetic tasks that exercise core capabilities (state tracking, issuing and updating API calls, sorting and presenting options, and retrieving KB-grounded facts), plus two evaluations on real data (a reformulated DSTC2 set and an online concierge dataset).
- The tasks are designed to probe both language understanding and symbolic manipulation, including an out-of-vocabulary (OOV) setup to test generalization to unseen entities. Evaluation is done via next-utterance ranking with per-response and per-dialog accuracy.
- The authors benchmark rule-based, IR, supervised embedding, and Memory Network models (with and without match-type features). MemN2N achieves encouraging per-response performance and learns non-trivial behaviors (e.g., issuing API calls, handling OOV entities with type features), but per-dialog success remains limited—highlighting open challenges.
- The paper argues that the synthetic tasks yield diagnostic signals that transfer to real datasets (DSTC2 and Concierge), thereby supporting the testbed’s relevance.

Decision: Accept (weak accept)
- Key reasons: The paper provides a much-needed, carefully designed, and reproducible benchmark for goal-oriented dialog; it offers clear empirical insights into what current end-to-end architectures can and cannot do, including OOV generalization and KB grounding, and validates result patterns on real data.

Supporting arguments
- Problem: The work tackles a specific and important question—how to systematically assess whether end-to-end dialog models can replace or complement traditional slot-filling systems in goal-directed settings. This is timely and under-resourced.
- Motivation and placement: The testbed is well-motivated relative to existing chit-chat datasets and dialog state tracking benchmarks. It fills a gap: controlled, task-decomposed, KB-grounded evaluation for end-to-end models, with OOV stress tests. The comparison to rule-based, IR, and supervised embeddings is appropriate and informative.
- Claims and evidence: Results substantiate the main claims. MemN2N substantially outperforms IR and supervised embedding baselines, especially with match-type features, and can issue/update API calls and retrieve KB facts. The notable gap between per-response and per-dialog performance honestly reflects current limitations (e.g., option ranking and multi-step reasoning), aligning with the paper’s conclusions. Consistent method rankings across synthetic tasks, DSTC2, and Concierge support the claim that the synthetic suite is a useful proxy for real-world behavior.
- Rigor: The OOV setup is well thought-out, and the ablations around match-type features and number of memory hops add insight. The rule-based baseline on DSTC2 is a helpful foil. While the ranking formulation simplifies evaluation, it allows clean comparisons and reproducibility.

Additional feedback to improve the paper (non-blocking)
- Ranking vs. generation: The ranking setup enables reproducible evaluation, but future extensions should include sequence-generation baselines (e.g., seq2seq, pointer or copy mechanisms, KB-augmented decoders), especially to test scaling to new domains where candidate pools are not easily curated.
- Candidate set design: Ranking over all responses drawn from train/val/test (including OOV) eases evaluation but blurs deployment realism and may leak test-only utterances into the candidate pool. Consider variants that restrict candidates to training-only or to turn-type-specific subsets; report sensitivity analyses.
- Language variability: The simulator’s limited template diversity and deterministic question order risk overestimating model robustness. Publishing the templates/rules and providing higher-variability (paraphrased/noisy) variants would strengthen the resource and encourage more realistic modeling.
- Metrics: Per-dialog accuracy is appropriate, but complementary measures (e.g., task success rate under noisy ASR-like perturbations; slot accuracy; API-argument F1; option-order correctness) would give finer-grained diagnostics.
- Error analysis: Expand qualitative analysis on failure modes for Tasks 3 and 5 (sorting, multi-hop aggregation over KB returns, handling user acceptance/decline loops). This could guide the community toward architectural changes (e.g., explicit differentiable sort/rank, program induction, or symbolic-neural hybrids).
- Domain generality: Discuss how to port the framework to other transactional domains (travel, retail) and whether match-type features generalize when entity ontologies change. A brief case study or guidelines would increase impact.
- Reproducibility: Please release the simulator, KBs, NL templates, rule-based DSTC2 system, and scripts for candidate generation. This will be key to adoption.

Questions for the authors
1) Candidate responses: For the synthetic tasks you rank over a candidate set comprising utterances from train/val/test (plain and OOV). In deployment, such a set is unavailable for unseen domains. Can you justify this choice and report results when candidates are restricted to training-only? Can you provide per-turn constrained candidate sets (e.g., API-calls-only when applicable) and compare?
2) Generation baselines: Did you attempt sequence generation (e.g., seq2seq with attention, pointer-generator, or copy mechanisms with KB linking)? If so, how do per-dialog metrics compare? If not, what challenges did you encounter?
3) Template and KB release: Will you release the simulator templates/rules and full KBs (including the OOV split) to enable reproducibility and extension? If not, can you at least enumerate the template patterns and deterministic dialog policies used?
4) Determinism and ordering: How sensitive are results to the deterministic slot-question order (Task 1) and to the acceptance process for options (Task 3)? Have you tried randomizing these to test robustness?
5) OOV mechanism: For match-type features, what KB access is assumed at test time, and how are entity mentions matched (exact string, normalization)? How does performance change with entity surface variation (misspellings, synonyms)?
6) DSTC2 conversion: The reformulation adds ratings and enforces consistent option ordering. Could this introduce artifacts that advantage certain models? Can you quantify how often the converted optimal action disagrees with the original system behavior?
7) Data efficiency: Training sets are relatively small (1k). Can you report learning curves per task and model (including match-type ablations) to show sample efficiency and saturation?
8) Human evaluation: For real-data tasks (Concierge, DSTC2), did you perform any human-in-the-loop evaluation (e.g., next-utterance preference or task success) to complement ranking metrics?

Overall, this is a valuable benchmark contribution with clear analyses and baselines. The work will likely stimulate progress on KB-grounded, goal-oriented dialog, and the suggested clarifications and releases would further strengthen its utility.
