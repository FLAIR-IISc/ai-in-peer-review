Attempts to use chatbots for nearly all forms of human–computer interaction were a major trend in 2016, accompanied by claims that they could handle many dialog types beyond simple chit-chat. This paper offers a much-needed reality check. While it is primarily relevant to Dialog/NLP venues (to inform software engineers about the current limitations of chatbots), it is also appropriate for Machine Learning venues (to encourage more realistic validation of ML methods applied to dialog). I therefore consider this work highly significant.

Two important conjectures underlie this paper and are likely to spur further research. Although not explicitly written, Antoine Bordes clearly articulated them during a NIPS workshop presentation covering this work. Given the metrics used in the paper:
1) The performance of end-to-end ML approaches is still insufficient for goal-oriented dialog.
2) When comparing algorithms, relative performance on synthetic data is a good predictor of performance on natural data. This would be a notable departure from prior observations, but the authors make a strong effort to align synthetic and natural conditions.

The original algorithmic contribution is a relatively simple addition to memory networks (match type), but this is the first time such models are deployed and tested on a goal-oriented dialog, and the experimental protocol is excellent. The paper is very clear and accessible beyond the core ML and dialog communities. I was particularly impressed by the concise appendix that effectively summarizes memory networks, followed by tables that clearly show the impact of the number of hops.

While this paper represents the state of the art in pursuing more rigorous metrics for dialog modeling, it also underscores how brittle and somewhat arbitrary these metrics remain. This is a recommendation for future research rather than a request for revision.

First, the paper uses per-response accuracy (i.e., next-utterance classification from a fixed list). Table 3 vividly illustrates how impractical this can be: what truly matters is issuing the correct API call and keeping the dialog reasonably short, yet per-response accuracy would be only 1/7 because the six bot responses prior to the API call must also be exactly correct.

Would per-dialog accuracy, where all responses must be correct, be better? Table 2 shows how sensitive it is to the experimental protocol. I was initially puzzled that accuracy for subtask T3 (0.0) is much lower than for the full dialog T5 (19.7), until the task definitions (3.1.1) clarified that T3 requires displaying three options while T5 requires displaying only one.

For the concierge data, what would happen if “correct” meant being the single best option rather than simply among the top five?

While I do not fault the authors for using standard dialog metrics—and for proposing new ones that may even be overly pessimistic—I can suggest a representation that could yield more meaningful metrics for goal-oriented dialogs. Suppose I sell Virtual Assistants as a service and am paid upon successful completion of a dialog. What metric would maximize my revenue? In the restaurant domain, the loss would likely be a weighted sum of errors in the API call, the number of turns needed to reach that API call, and the number of user-rejected options. However, such a loss cannot be computed from canned dialogs and would require either real human users or a realistic simulator.

Another issue, closely related to representation learning and not adequately addressed here, is vocabulary mismatch between the user and the knowledge base. For the match type algorithm to map “Indian” to “type of cuisine,” that exact term must appear in the KB. I can easily imagine cases where the KB uses obfuscated or idiosyncratic terminology, and we would prefer ML methods to learn these associations rather than relying on hand-specified mappings.
