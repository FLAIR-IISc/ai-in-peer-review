- Introduces an open, lightweight testbed for end-to-end goal-oriented dialog focused on restaurant reservation.  
- Decomposes the task into five controlled subtasks and adds two real-world evaluations (DSTC2 conversion and an "online concierge" dataset).  
- Benchmarks rule-based, IR, supervised embeddings, and Memory Networks (with a "match type" feature) using per-response and per-dialog accuracy.  
- Shows end-to-end models learn non-trivial behaviors (slot elicitation, API calls) but struggle with KB grounding (ranking/presentation) and per-dialog success, especially under OOV.  
- Provides a strong, well-motivated experimental protocol, clear write-up, and public data to catalyze research.  
- Decision: Accept.  
- Key reason 1: Provides a reproducible benchmark and error decomposition needed for goal-oriented dialog evaluation.  
- Key reason 2: Empirical analysis credibly highlights limitations and shows synthetic task performance predicts real-data trends.  
- Methodology strength: Principled task design testing state tracking, API formation/revision, KB grounding, sorting, and OOV generalization.  
- Evidence: Memory Networks outperform IR and supervised embeddings; match type aids KB grounding/OOV; per-dialog success remains low on hard tasks.  
- Clarity: Paper is exceptionally clear with a helpful appendix and ablations (e.g., hops).  
- Suggestion — metrics: Report task success rate with semantic tolerance, API slot-wise accuracy/weighted slot error, efficiency (turns), cumulative utility penalties, and Top-k (Hits@k).  
- Suggestion — T3 vs T5: Normalize option-presentation metrics (e.g., MRR or success within N offers) to reconcile task-definition differences.  
- Suggestion — OOV/entity linking: Discuss/experiment with entity linking, character/subword features, and paraphrase augmentation.  
- Suggestion — statistical validation: Quantify synthetic-to-real transfer (e.g., Spearman correlations with confidence intervals).  
- Suggestion — generality: Discuss portability of the simulator/tasks to other domains (travel, retail) and required reconfiguration.  
- Suggestion — robustness: Report variance across seeds, hyperparameter sensitivity, and small-data ablations to assess stability and sample efficiency.  
- Suggestion — human evaluation: Add small-scale human judgments on Concierge for task success and appropriateness.  
- Questions for authors: What is the candidate set size (especially Concierge) and can you report Hits@k?  
- Questions for authors: Do you have API call slot-wise accuracy and turns-to-success metrics, or can the simulator be extended to measure them?  
- Questions for authors: Can you quantify method-ranking correlations between T1–T5 and DSTC2/Concierge and test significance?  
- Questions for authors: How does performance change with slight entity surface-form variation; have you tried subword models or entity linking?  
- Questions for authors: For DSTC2 conversion, can you separate failures due to ASR noise versus policy inconsistency?  
- Questions for authors: Will you release code for data generation, DSTC2 conversion, and training scripts to ensure reproducibility?  
- Overall: High rigor, transparency, and a valuable open benchmark—recommended for acceptance.
