Summary
This paper provides a much-needed reality check on end-to-end approaches for goal-oriented dialog. It introduces a clear, reproducible testbed in the restaurant reservation domain that decomposes the task into five subtasks (plus two evaluations on real data: DSTC2 and a Concierge dataset). The work carefully evaluates Memory Networks (with and without a simple but impactful “match type” feature) against strong baselines (rule-based, IR, supervised embeddings), under both in-vocabulary and out-of-vocabulary conditions. The authors show that end-to-end models can learn nontrivial behaviors (e.g., issuing and updating API calls, handling some OOV entities) but still struggle with key operations such as interpreting API call results and robustly achieving dialog success. The paper is clearly written, the experimental protocol is excellent, and the appendix offers a concise, accessible summary of Memory Networks and the effect of multi-hop attention.

Decision: Accept
Key reasons:
- Significance and timeliness: The testbed and results provide a rigorous, reproducible yardstick for goal-oriented dialog, addressing a clear need in the community.
- Careful, well-motivated evaluation: The synthetic tasks are designed to mirror essential capabilities and are validated against real datasets (DSTC2 and Concierge), yielding coherent relative performance trends.

Supporting arguments
- Contributions and scope: While the algorithmic novelty is limited (adding match-type features to MemNNs), the contribution lies in the experimental framework and the careful comparative evaluation. This is the first deployment and thorough testing of end-to-end models in a goal-oriented setup with controlled subtasks, OOV stress tests, and real-world datasets.
- Clarity and accessibility: The paper is exceptionally well-written and accessible beyond dialog and ML specialists. I especially appreciated the concise Memory Network appendix and the clear ablations on the number of hops.
- Empirical insights and two key conjectures (implicit, but articulated by the authors elsewhere and consistent with the results here, given the chosen metrics):
  1) End-to-end ML approaches remain insufficient for reliable goal-oriented dialog, particularly in per-dialog success.
  2) Relative performance on the synthetic tasks appears predictive of relative performance on natural data when the synthetic conditions are carefully matched.
- Error analysis by design: The five subtasks isolate core competencies (dialog management, API call formation and updates, sorting options, using KB facts, handling OOV entities). Results show MemNNs excel at issuing/updating API calls (T1–T2), improve with match types for T4 and OOV settings, but still falter on displaying sorted options and full-dialog success (T3, T5). This detailed breakdown is valuable for guiding future research.
- Real-data validation: The pattern of relative performance across methods carries over from the synthetic tasks to DSTC2 and Concierge. This is particularly useful evidence that the proposed testbed is an informative proxy for more realistic conditions.

Additional feedback to improve the paper (helpful suggestions; not central to my decision)
- Evaluation metrics and protocol:
  - Per-response accuracy (next-utterance classification over a fixed candidate set) can be misleading in goal-oriented settings. As the examples (e.g., Table 3 in the appendix) suggest, what ultimately matters is issuing the correct API call in a reasonably short dialog; yet that yields low per-response accuracy because all preceding template responses must be exact.
  - Per-dialog accuracy is informative but highly sensitive to task design. The discrepancy between T3 (0.0) and T5 (19.7) stems from task definitions (T3 requires presenting three options; T5 may require only one). An ablation that varies the number of required options would better reveal robustness to this choice.
  - For Concierge, consider reporting top-1 vs top-k next-utterance accuracy (or MRR). If “correct” means “best” rather than “in top-5,” how do conclusions change?
  - Beyond standard dialog metrics, consider goal-focused utility metrics. In a “Virtual Assistant as a service” framing, a loss could combine (i) errors in the API call (slot errors), (ii) turns to reach the correct call, and (iii) the number of rejected options. This would better align evaluation with task success and cost, though it may require a richer simulator or human-in-the-loop evaluation.
- Entity handling and vocabulary mismatch:
  - Match-type features rely on exact KB matches. In practice, user vocabulary and KB terminology may not align (synonyms, paraphrases, obfuscated labels). Please discuss or experiment with learned mappings between user text and KB schemas (e.g., NER/EL with synonym expansion, fuzzy string matching, or a learned entity-linking layer) to reduce reliance on exact string overlap.
  - It would be helpful to quantify how performance degrades when the KB uses different surface forms than training dialogs (e.g., “South Asian” vs “Indian”).

Questions for the authors
- Synthetic-to-natural transfer: You argue (and show evidence) that relative performance on synthetic tasks tracks real-data performance. Could you provide quantitative correlation (e.g., rank correlation across methods) to strengthen this claim?
- Task design sensitivity: For T3/T5, how does performance vary with the number of required options, acceptance probabilities, or the presence of near-ties in ratings? An ablation would clarify how brittle the models are to these parameters.
- Concierge evaluation: Which ranking metrics did you compute (top-1 vs top-k, MRR)? If you evaluated only top-5, what happens under stricter top-1 evaluation?
- Candidate set effects: How sensitive are results to candidate set size/composition and negative sampling? Do conclusions change if candidates are constrained by a dialog-state filter?
- Entity mismatch: Have you tried training with alias tables or learned entity-linking to bridge user terms and KB entries? How would match-type features integrate with such a component?
- Beyond MemNNs: Did you attempt pointer/copy mechanisms or key-value memory variants for T3/T4 to better ground returned KB facts when presenting options?

Overall, this is a valuable, timely, and well-executed paper that will help the community assess and improve end-to-end methods for goal-oriented dialog. I recommend acceptance.
