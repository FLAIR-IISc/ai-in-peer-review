Review- SYNOPSIS:
This paper presents a new dataset for evaluating end-to-end goal-oriented dialogue systems. All interactions are set in the restaurant domain, where the task is to check availability and ultimately book a table based on parameters the user provides to the bot during the dialogue. The data are generated via simulation: a knowledge base is used to sample parameter tuples (cuisine, price range, etc.), followed by rule-based transformations to produce natural-language utterances. The learning objective is to rank a set of candidate responses at each subsequent turn, and evaluation is reported as per-response accuracy and per-dialog accuracy. The authors show that Memory Networks outperform simple bag-of-words baselines.

THOUGHTS:
I thank the authors for an interesting contribution. That said, I remain skeptical about the usefulness of end-to-end trained systems in a narrowly defined domain. In open-domain settings, it is hard to hand-code all states and responses at scale, so end-to-end methods are compelling. However, in constrained domains we typically understand the space well, and the primary goal is high user satisfaction. In such cases, doesn’t it make more sense to exploit domain knowledge to engineer the best possible system?

Given the restricted domain, I am also disappointed that the objective is to RANK rather than GENERATE responses, though I recognize this simplifies evaluation. I am furthermore unclear how these candidate responses would be obtained in practice. My understanding is that the models rank the set of all responses seen in train/validation/test (last sentence before Sec 3.2). If a key motivation for end-to-end training is effortless scaling to new domains without manual re-engineering, where do these candidate lists come from for a new domain? Generative approaches would likely generalize better across domains than ranking a hand-collected pool of generic replies, which I view as the weakest aspect of this work.

Finally, because the data are produced by simulating and expanding (cuisine, price, …) tuples using NL generation rules, the linguistic variability is necessarily constrained. While this enables generating unlimited data, I could not find the rule set used. It would be helpful to release this as well.

Overall, despite my reservations, I find this an interesting contribution worthy of acceptance at the conference.
