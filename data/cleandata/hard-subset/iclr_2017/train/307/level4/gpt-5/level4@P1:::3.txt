1. Summary of contributions
This paper introduces an open, reproducible testbed for evaluating end-to-end goal-oriented dialog systems in a controlled yet realistic setting. The authors design a suite of tasks in the restaurant reservation domain that progressively test core capabilities: dialog management (implicitly tracking state), issuing and updating API calls to a knowledge base (KB), interpreting KB results (including sorting and selecting options), and handling unfamiliar entities (OOV). The evaluation uses per-response and per-dialog accuracy in a ranking setup, which enables clear comparisons and error analysis. The paper benchmarks rule-based systems, classical IR baselines, supervised embeddings, and Memory Networks, and reports results on both synthetic tasks and real conversations from DSTC2 and a concierge service. Memory Networks outperform bag-of-words baselines and IR methods and learn non-trivial operations (e.g., issuing API calls and handling OOV entities with match-type features), though per-dialog success remains limited on harder tasks.

2. Decision: Accept
Key reasons:
- The paper provides a useful, well-structured, and openly available testbed for goal-oriented dialogs that facilitates reproducibility, ablation, and comparison across methods.
- The experimental study is thorough and informative across synthetic, semi-real (DSTC2), and real concierge data, revealing strengths and limitations of current end-to-end approaches and offering a clear roadmap for future research.

3. Supporting arguments
- Problem framing: The paper targets a concrete, important question—can end-to-end models, which have shown promise in chit-chat, handle the demands of goal-directed dialog? The task decomposition (T1–T5), OOV splits, and inclusion of DSTC2 and concierge logs make this question answerable in a reproducible way.
- Motivation and placement: The work is well positioned within dialog literature. It contrasts traditional slot-filling/POMDP systems with end-to-end methods, and identifies a gap: resources that allow systematic testing of end-to-end models in goal-oriented settings. The design is reminiscent of bAbI QA tasks, adapted to dialog.
- Empirical support: The results substantiate the claims. Memory Networks clearly outperform IR and supervised embedding baselines, especially on API-related tasks and OOV handling when augmented with match-type features. At the same time, per-dialog accuracy is still low on tasks requiring interpreting and presenting KB results (e.g., sorting options), which aligns with the paper’s conclusion that end-to-end systems are not yet ready to supplant well-engineered slot-filling systems in narrow domains. The similar performance patterns observed on DSTC2 and concierge data strengthen the external validity of the synthetic tasks as a proxy.
- Utility: The testbed’s design (e.g., modular tasks, OOV split, ranking-based evaluation) and availability should catalyze progress by enabling careful ablations and consistent comparisons.

4. Additional feedback and questions for the authors
These points are intended to help improve the paper and clarify the contribution; not all of them affect my accept decision.

- Ranking vs generation:
  - Concern: The choice to evaluate only response ranking, rather than generation, limits applicability to new domains and real deployments. In practice, where would the candidate set come from when scaling to a new domain—particularly if one of the motivations for end-to-end methods is to avoid handcrafting?
  - Request: Please justify this design choice more explicitly and discuss how the testbed could be extended to generation (e.g., constrained decoding, templated NLG, or hybrid rank/generate approaches). If possible, report preliminary generation results or a plan to provide generation-compatible annotations.

- Candidate set construction:
  - Concern: The current setup ranks over all candidates from train/val/test across tasks. How would this translate operationally outside the benchmark, especially for unseen domains or evolving systems?
  - Questions:
    - How do you envision constructing and maintaining candidate sets in practice without sacrificing scalability?
    - Could you provide statistics on candidate set size per task and its impact on accuracy and runtime?
    - Have you tried more realistic candidate pools (e.g., domain-specific templates plus a small number of generic responses)?

- Simulator variability and coverage:
  - Concern: Synthetic dialogs necessarily restrict linguistic variability and dialog policy diversity. The bot’s deterministic behavior (e.g., fixed slot order, fixed templates) can make tasks easier and may bias models toward pattern memorization.
  - Requests:
    - Please publish the full set of NL generation rules/templates and dialog policies (number, types, and examples) to support reproducibility and to let others expand the simulator.
    - Consider reporting results under increased variability (e.g., multiple paraphrases per bot act, randomized slot-order policies) to test robustness.

- Evaluation metrics:
  - Suggestion: While per-response and per-dialog accuracy are clear, adding goal success rate, number of turns to success, and error types (e.g., API call errors vs KB interpretation errors) would sharpen the diagnostic value. For DSTC2 and concierge, human evaluation of dialog coherence or task success (even on a subset) would be informative.

- OOV handling and match-type features:
  - Positive: Match-type features are an effective and simple addition; they materially improve OOV performance and T4.
  - Questions:
    - How sensitive are results to KB coverage and entity linking errors?
    - Can match-type features be combined with subword/character models to reduce reliance on exact matches?
    - Do these features generalize to domains with more heterogeneous entity schemas?

- Comparisons to traditional systems:
  - The rule-based baseline on DSTC2 is helpful. If feasible, adding a slot-filling or neural belief-tracking baseline with a learned policy/NLG would better contextualize end-to-end performance in narrow domains.

- Generalization beyond restaurants:
  - Suggestion: Comment on how the task suite and datasets could be adapted to other domains (e.g., travel, retail) and what would be required to maintain the OOV split and KB grounding.

Questions to clarify in a rebuttal
- How exactly is the candidate set defined per task and per dataset? Is it truly the union of all candidates across train/val/test, and if so, how do you prevent leakage between splits?
- What is the typical candidate set size at inference time, and how does performance vary with candidate pool size?
- Can you share or release the full list of user and bot templates/rules used in the simulator, and the code to generate dialogs?
- Would you consider adding a generation track to the benchmark? If not, what practical strategy do you recommend for constructing candidate sets when scaling to new domains?
- For T3/T5, have you tried explicit reasoning modules (e.g., differentiable sorting or pointer mechanisms) or memory-augmented controllers to improve option ranking and selection?
- On DSTC2, can you provide an analysis of errors due to ASR artifacts vs modeling limitations?
- How do the models behave when the bot’s question order is randomized or when paraphrase variability increases?

I appreciate the authors’ clarifications and additional results. I have updated my score accordingly.
