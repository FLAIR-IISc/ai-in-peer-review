1. Summary of contributions
The paper introduces a well-designed, reproducible testbed for evaluating end-to-end dialog systems in goal-oriented settings, instantiated in the restaurant reservation domain. It decomposes the overall goal into focused tasks (issuing and updating API calls, ranking options from KB results, retrieving attributes, and full dialogs), and augments synthetic tasks with converted DSTC2 dialogs and a real-world Concierge dataset. The authors evaluate several baselines and an end-to-end Memory Network, proposing a simple yet impactful extension (match type features) to handle entities and OOV words. The experimental protocol is careful and informative: it uses both per-response and per-dialog accuracy, analyzes OOV generalization, and compares synthetic vs. real data to assess whether synthetic performance patterns predict those on natural dialogs. The work is clearly written, accessible beyond dialog/ML specialists, and offers a compact, very effective summary of Memory Networks and the effect of hop count.

2. Decision
Accept. The paper provides (i) a high-quality, reproducible, and broadly useful testbed for goal-oriented dialog; and (ii) a careful empirical study that both tempers expectations about current end-to-end methods and offers evidence that synthetic-task comparisons can anticipate performance trends on real data.

3. Supporting arguments
- Significance and timeliness: Against the 2016 wave of broad chatbot claims, this work offers a rigorous reality check for goal-directed tasks, relevant to both dialog/NLP and ML audiences.
- Central empirical takeaways (two implicit conjectures the work substantiates): (1) end-to-end methods are still short of what is needed for reliable goal-oriented dialog; (2) relative ordering of methods on well-crafted synthetic tasks can be predictive of their ordering on natural dialogs—an important and non-trivial finding given past skepticism.
- Methodological contribution: Although the algorithmic novelty (match type features for MemNNs) is simple, it is the first careful deployment on goal-directed dialog with a compelling evaluation. The addition materially helps with entity handling and OOV generalization, and the paper presents informative analyses (e.g., the role of multiple hops).
- Experimental rigor and clarity: The task design supports granular error analysis, OOV stress-testing, and controlled comparisons. The findings are consistent across synthetic tasks, DSTC2, and the Concierge dataset. The MemNN outperforms an engineered rule-based system on DSTC2, which underscores the value of learning-based approaches even when hand-coding is non-trivial.
- Metric insights: The study highlights how brittle standard metrics can be. Per-response next-utterance classification can obscure task success (e.g., if only the API call truly matters). Per-dialog accuracy, while goal-aligned, is sensitive to task design choices (e.g., T3 requires ranking three options; T5 relaxes that to one), which explains the counterintuitive gaps (T3 ≈ 0 vs. T5 ≈ 19.7 in Table 2).

4. Suggestions for improvement (to help the authors; not critical to the decision)
- Metrics more aligned with task utility: Consider a success-oriented metric that reflects business-relevant outcomes (e.g., cost-weighted errors in API fields, number of turns to a correct API call, number of rejected options) rather than strict per-response or all-or-nothing per-dialog accuracy. While such a metric may require a realistic simulator or live users, even an approximate version could be informative.
- Top-k evaluation on Concierge: Since operator responses are varied and the candidate set is large, report how results change if “correct” means top-1 vs. best among top-k (k > 5), or use MRR/Recall@k to better capture near-miss quality.
- Vocabulary–KB mismatch: Match type features assume exact lexical overlap between user text and KB entities. Discuss or experiment with learned mappings (e.g., synonymy/normalization, ontology linking) so models can bridge cases where the KB uses obfuscated or alternate terminology and the user does not use exact KB strings.
- Make the two guiding conjectures explicit: Stating them clearly would strengthen the narrative and aid future comparisons.
- Reporting details: A short error analysis focused on where full-dialog failures accumulate (e.g., API field errors vs. option ordering vs. attribute retrieval) would further clarify where end-to-end models most struggle.

Questions for the authors
- Could you report top-k measures on Concierge (e.g., Recall@1/5/10, MRR) and analyze how often the gold response appears just outside the top-5?
- For per-dialog accuracy, how sensitive are the results to small relaxations (e.g., minor template variations counted as correct, or success defined by a correct API call within a bounded number of turns)?
- In T3/T5, how much of the failure is due to option sorting vs. tracking which options have already been proposed vs. stopping criteria? Any ablations isolating these factors?
- On OOV generalization, to what extent do gains come from type features alone versus learned generalization from seen entities? Have you tried incorporating learned entity normalizers or synonym dictionaries to mitigate exact-match reliance?
- Since models have access to both KBs at training time for OOV evaluation, could you clarify the risk of implicit leakage (e.g., memorizing KB strings) and how you ensured fairness across methods?
- How robust are the MemNN results to increased candidate set size and more diverse, non-deterministic bot language (as in DSTC2/Concierge)? Do additional hops consistently help in those settings?
- Would a simulator that probabilistically accepts any of the top-ranked semantically appropriate responses (rather than a single canonical utterance) materially change per-dialog success rates?
