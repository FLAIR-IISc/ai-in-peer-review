Chatbots were widely promoted in 2016 as a universal interface capable of handling dialogs far beyond small talk. This paper is a welcome reality check. While primarily relevant to dialog/NLP venues—where it can inform software engineers about current chatbot limitations—it also fits at machine learning venues, where it highlights the need for more realistic validation of ML methods on dialog tasks. I consider the work highly significant.

Two important conjectures underlie the paper and could spur further research. Although not stated explicitly in the text, Antoine Bordes articulated them in a NIPS workshop presentation on this work, and they align with the metrics used here:
1) The performance of end-to-end ML approaches remains insufficient for goal-oriented dialogs.
2) Relative performance on synthetic data is a good predictor of performance on natural data. This departs from earlier observations, but the authors made a strong effort to align synthetic and natural conditions.

The paper’s algorithmic contribution is a simple addition to memory networks (“match type”), yet this is, to my knowledge, the first deployment and evaluation of such models on a goal-oriented dialog task, and the experimental protocol is excellent. The writing is clear and accessible beyond ML and dialog specialists. I was particularly impressed by the concise appendix summarizing memory networks and by the tables that make the influence of the number of hops easy to understand.

While this paper represents the state of the art in exploring more rigorous metrics for dialog modeling, it also reminds us how brittle and somewhat arbitrary these metrics remain. This is a recommendation for future research rather than a revision request.

First, the paper uses per-response accuracy (next-utterance classification from a fixed set). Table 3 illustrates how misleading this can be in practice: what really matters is issuing a correct API call and keeping the dialog reasonably short, yet per-response scoring reduces this to roughly 1/7 accuracy because all six prior bot responses must also be exactly correct before the API call.

Would per-dialog accuracy—requiring every response to be correct—be preferable? Table 2 shows how sensitive it is to the protocol. I was initially puzzled that accuracy for subtask T3 (0.0) is far below the accuracy for the full dialog T5 (19.7), until the task definitions (Section 3.1.1) clarified that T3 demands presenting three options, whereas T5 requires only one.

For the concierge data, what would happen if “correct” meant returning the single best option, rather than being among the top five?

I do not fault the authors for using standard dialog metrics and even introducing new ones that are arguably too pessimistic. Still, I suggest a representation and metric more aligned with goal-oriented objectives. Suppose I offer Virtual Assistants as a service and get paid upon successful task completion. What metric maximizes revenue? For the restaurant domain, a plausible loss would weight: (a) the number of errors in the API call, (b) the number of turns to reach that API call, and (c) the number of user-rejected options. Such a loss cannot be computed on canned dialogs; it would require either real human users or a realistic simulator.

Finally, a representation-learning issue remains insufficiently addressed: what if the user’s vocabulary does not match the knowledge base? For the match-type mechanism to tag “Indian” as a cuisine, that exact token must appear in the KB. In practice, KBs may use obfuscated or alternative terminology. We would prefer ML to learn these associations automatically rather than rely on hand-crafted mappings.
