- Introduces a clean, reproducible restaurant-reservation testbed decomposing goal-oriented dialog into interpretable subtasks and two realistic evaluations (DSTC2, concierge).
- Provides a careful empirical study comparing rule-based, IR, supervised embeddings, and Memory Networks, plus a simple “match type” extension for entities.
- Shows end-to-end neural models can learn API-call behavior and KB usage but struggle on complex tasks, with synthetic task performance mirroring real datasets.
- Paper is clearly written, well-situated in literature, and offers an exemplary experimental protocol with useful Memory Network ablations.
- Decision: Accept.
- Primary reason 1: Fills a real gap by providing a reproducible, interpretable framework bridging synthetic and real goal-oriented dialog data.
- Primary reason 2: Provides a sober, evidence-based assessment of end-to-end methods and practical methodological advances (match type features).
- Problem/motivation: Timely examination of whether end-to-end dialog can replace slot-filling and what capabilities remain lacking.
- Positioning: Complements chit-chat and bAbI evaluations by focusing on dialog management, KB interaction, and OOV entities.
- Methodology: Five controlled tasks plus two realistic datasets enable rigorous, comparable evaluation and built-in error analysis; OOV split is insightful.
- Evidence: Memory Networks outperform IR and supervised embeddings; match type features improve OOV and KB lookup, but per-dialog accuracy stays modest on harder tasks.
- Per-response accuracy can be misleading; per-dialog accuracy is informative but sensitive to task design, complicating comparisons.
- Recommend task-success metrics aligned with application value (weighted API-call penalties, turns to first correct API call, number of rejected options).
- For concierge data, report top-k next-utterance classification (top-1/top-5/MRR) to assess retrieval utility.
- Match type features rely on exact KB matches; realistic vocabulary mismatches need learned entity linking or canonicalization approaches.
- Suggest finer-grained error analysis by capability (API formation, result interpretation, sorting, stopping) and by entity type.
- Request ablations isolating match type components (exact match vs. types vs. both) and interactions with number of hops.
- Report sensitivity to candidate set size and retrieval bias, especially for concierge data.
- Question: Have analogous tasks been piloted in other domains (travel, movies) and do method rankings persist?
- Question: How sensitive are results to candidate set size/content and would constrained-generation models show similar per-dialog patterns?
- Question: In T3/T5, which specific operations (reading/sorting KB, stop decision) cause most errors and would explicit supervision/pointer mechanisms help?
- Question: How are type features constructed for OOV entities at test time and how robust are they to noisy/missing KB strings?
- Question: In DSTC2, which noise sources (ASR vs. bot nondeterminism) most degrade performance, and do more hops help under noise?
- Question: For concierge evaluation, can you report top-k metrics and analyze near-miss candidates to assess practical retrieval utility?
- Question: Have you tried weakly supervised or embedding-based mappings between user terms and KB schema values, and what obstacles exist?
