1. Summary of contributions
This paper introduces a reproducible, lightweight testbed for evaluating end-to-end goal-oriented dialog systems. The authors design five synthetic tasks in the restaurant-booking domain that decompose a full interaction into subproblems: issuing and updating API calls, ranking options based on KB results, and retrieving entity attributes. The tasks are paired with a KB and an Out-Of-Vocabulary (OOV) split to test generalization to unseen entities. Evaluation is done via next-utterance ranking with per-response and per-dialog accuracy. The paper provides strong baselines (TF–IDF, nearest neighbor retrieval, supervised embeddings) and an end-to-end Memory Network (MemN2N) model, including a useful extension with “match-type” features to better handle entities and OOV tokens. The authors further validate on two datasets with real language: a reformatted version of DSTC2 and a proprietary “Concierge” dataset. The main empirical finding is that MemN2N significantly outperforms IR and supervised embedding baselines, learns nontrivial operations (e.g., issuing API calls, handling OOV entity mentions), but still struggles on tasks requiring option ranking and multi-turn completion, as seen in low per-dialog accuracy. Overall, the work offers a clear, well-positioned benchmark and baseline analysis for goal-oriented dialog.

2. Decision: Accept
Key reasons:
- The paper provides a much-needed, well-motivated, and reproducible benchmark that isolates critical capabilities for goal-oriented dialog, with OOV evaluation and comparisons to real-world data.
- The empirical analysis is careful and informative, establishing meaningful baselines and identifying where current end-to-end methods succeed and fail.

3. Supporting arguments
- Problem significance: Goal-oriented dialog requires skills beyond chit-chat (state tracking, KB queries, entity grounding). Existing datasets and evaluations often lack reproducibility or clear diagnostics. The proposed tasks and metrics directly target these gaps.
- Placement in literature: The work is well situated relative to POMDP-based slot-filling systems and recent end-to-end/neural dialog approaches. The test suite parallels bAbI-style diagnostics for QA but emphasizes dialog behaviors (including asking questions and using KB outputs).
- Empirical support: Results are consistent and transparent. MemN2N improves over IR and supervised embeddings across most settings, especially with match-type features for entities and OOVs. The observed per-response gains alongside low per-dialog accuracy convincingly support the claim that end-to-end models remain imperfect for robust goal completion. Patterns replicate on DSTC2 and Concierge, strengthening external validity.
- Reproducibility and clarity: Tasks, KB design, OOV protocol, and evaluation setup are clearly described. The decomposition enables targeted error analysis and future model development.

Concerns (not decision-critical):
- Ranking vs generation: The framework evaluates next-utterance ranking from a global candidate set (including test candidates), which simplifies evaluation but raises questions about deployment and generalization to new domains where candidate sets are not precompiled. This limits the claim of easy scalability compared to generative models.
- Candidate set construction: Using all bot utterances from train/validation/test provides a strong oracle set for ranking. It would be useful to better justify this choice and discuss implications for real-world use.
- Language variability: Synthetic dialogs rely on a finite set of templates/patterns, which constrains linguistic diversity. While the inclusion of DSTC2 and Concierge partly addresses this, publishing the generation rules would improve transparency and help the community assess coverage and bias.
- Narrow-domain skepticism: In tightly scoped domains, hand-engineered systems can be strong and interpretable. The paper’s contribution is primarily as an evaluation resource rather than a claim that end-to-end methods already surpass domain-specific engineering.

4. Suggestions for improvement (helpful, not central to my decision)
- Provide (and release) the full set of natural-language templates/patterns and the rule lists used in the simulator; include statistics on paraphrase diversity and coverage.
- Clarify how candidate responses would be obtained in a new domain. Consider experiments with restricted or domain-agnostic candidate sets, or add a generative baseline to quantify the ranking–generation trade-off.
- Report candidate set sizes per task and analyze sensitivity to candidate pool construction (e.g., without test utterances, or with augmented distractors).
- Add ablations for match-type features (per type), memory hops, and history length to better connect model components to capabilities.
- Consider human evaluation on a small subset of full dialogs to complement per-dialog accuracy, especially on the real-data tasks.
- Discuss error categories for Task 3/5 (option ranking/long dialogs) to guide model development (e.g., sorting mistakes, failure to stop, entity grounding).

Questions for the authors
- Candidate construction: In practice, how would you build the candidate set for a new domain without pre-existing test utterances? What is the impact on accuracy if candidates are limited to train/validation only, or to domain-generic templates?
- Generalization: Have you tested a generative model (e.g., seq2seq, pointer or copy mechanisms) on these tasks to compare scalability to new domains/entities?
- Templates: Can you release the NL generation templates/patterns and the simulator code? What is the approximate paraphrase diversity per intent?
- OOV handling: For OOV entities, how often do errors arise from entity grounding vs subsequent reasoning (e.g., sorting by rating)?
- Per-dialog failures: For Tasks 3 and 5, what are the dominant failure modes (incorrect sorting, not stopping after acceptance, wrong attribute retrieval)?
- DSTC2/Concierge details: What are the candidate set sizes and distribution of intents/responses for these datasets? How sensitive are results to ASR noise or utterance variability?
- Practical deployment: If ranking is retained, what strategy would you propose to maintain and update candidate inventories over time in a production system?
