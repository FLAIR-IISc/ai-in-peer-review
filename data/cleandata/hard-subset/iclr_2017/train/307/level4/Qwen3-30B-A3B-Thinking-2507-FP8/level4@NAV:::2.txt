

This paper provides an important reality check regarding the overhyped application of chatbots to diverse human-computer interactions, a trend prominent in 2016. While primarily relevant for Dialog/Natural Language venues (to inform software engineers of current chatbot limitations), its insights also hold significant value for Machine Learning venues (to emphasize the need for more rigorous validation of ML in dialog systems), warranting high significance.  

Two key conjectures underpin this work and are likely to stimulate further research. Though not explicitly stated, Antoine Bordes articulated them during a NIPS workshop presentation on this paper. Regarding the metrics employed:  
1) End-to-end ML approaches remain insufficient for goal-oriented dialogs.  
2) Relative performance on synthetic data serves as a strong predictor of performance on natural data—a notable departure from prior observations, validated through careful alignment of synthetic and natural conditions.  

While the algorithmic contribution involves a modest extension to memory networks (match type), this represents the first deployment and testing of memory networks in goal-oriented dialog. The experimental protocol is exemplary, and the paper’s clarity is outstanding, making it accessible beyond ML and dialog research. The appendix’s concise summary of memory networks, coupled with tables elucidating the impact of hop count, is particularly commendable.  

Though this work sets the state-of-the-art for rigorous dialog modeling metrics, it also highlights the brittleness and arbitrariness of current metrics—offering recommendations for future research rather than revision. Specifically:  
- Per-response accuracy (next-utterance classification) yields counterintuitive results (e.g., Table 3 shows a 1/7 accuracy requirement for correct API calls, demanding exact bot responses for all prior turns).  
- Per-dialog accuracy (all responses correct) is highly protocol-sensitive (e.g., T3’s 0.0% vs. T5’s 19.7% in Table 2, due to differing task definitions: T3 requires displaying 3 options, T5 only 1).  
- For concierge data, defining "correct" as "best among all options" (not just the top 5) merits exploration.  

While standard metrics are justified, a more meaningful alternative could align with real-world outcomes: if selling virtual assistants by successful dialog completion, revenue-maximizing metrics might weight API errors, dialog turns, and user-rejected options. However, such metrics require real users or realistic simulators, not canned dialogs.  

Additionally, the paper overlooks a critical issue in representation learning: vocabulary mismatches between user input and knowledge bases. For match type to map "Indian" to "type of cuisine," the exact term must exist in the KB. This is problematic when KBs use obfuscated terminology, as ML should ideally learn associations rather than relying on manual mapping.
