

Summary:
This paper introduces a valuable testbed for evaluating end-to-end goal-oriented dialog systems in a controlled, reproducible manner. The authors propose five restaurant reservation tasks that systematically test critical dialog capabilities including API call issuance, request updates, option display, information provision, and full dialog completion. They generate synthetic data using a knowledge base and natural language generation rules, while also validating their approach on real-world data from a Dialog State Tracking Challenge dataset and an online concierge service. The evaluation uses per-response and per-dialog accuracy metrics, demonstrating that Memory Networks outperform traditional baselines while highlighting persistent challenges in interpreting API results to display options.

Decision:
I recommend acceptance of this paper. The contribution is significant as it addresses a critical gap in the evaluation of end-to-end dialog systems for goal-oriented applications, where current datasets are often noisy, domain-specific, or designed for component-level evaluation rather than end-to-end training.

Supporting arguments:
The paper makes a compelling case for the need for a standardized evaluation framework. Current datasets for goal-oriented dialog systems either require manual annotation (like DSTC datasets), are designed for specific components (dialog state tracking), or lack reproducibility (e.g., using crowdsourcing platforms). The authors' synthetic testbed provides a controlled environment for benchmarking while maintaining the ability to validate results on real-world data. Their demonstration that Memory Networks can outperform traditional baselines on these tasks (though not perfectly) provides valuable insights into the current capabilities of end-to-end approaches. The consistent performance patterns observed across synthetic and real-world datasets (DSTC2 and concierge service) further validate the testbed's utility. The paper's focus on identifying specific weaknesses (e.g., interpreting API results to display options) rather than claiming state-of-the-art performance is particularly valuable for guiding future research.

Additional feedback:
To further strengthen the paper, I suggest the authors provide more detail about the natural language generation rules used to create user and bot utterances. This would enhance reproducibility and allow readers to better assess the variability in the synthetic data. Additionally, the paper could benefit from more explicit discussion of the testbed's limitations and how it might generalize to domains beyond restaurant reservation. A deeper error analysis would also help readers understand exactly where the models fail (e.g., specific types of API interpretation errors).

Questions for authors:
1. Could you provide more detail about the natural language generation rules used to create user and bot utterances? Specifically, how were the 43 user patterns and 20 bot patterns designed to ensure sufficient variability while maintaining the restaurant reservation context?

2. Given that the current approach relies on ranking against a fixed candidate set (all responses from training, validation, and test sets), how would this framework scale to new domains where the candidate set would need to be generated? The end-to-end approach's key advantage is ease of scaling to new domains without manual re-engineering, but this aspect isn't clearly addressed for candidate generation.

3. The paper notes that match type features significantly improve performance on OOV tasks. Could you elaborate on how this approach might be extended to other domains where the entity types might differ from the current restaurant reservation scenario? Would the same 7 special entity types be applicable across different domains, or would they need to be redefined?
