The use of chatbots for all forms of human-computer interaction was a major trend in 2016, with claims that they could handle various types of dialog beyond simple chit-chat. This paper represents a serious reality check on that ambition. While the paper is highly relevant for Dialog and Natural Language venues, where it can educate software engineers on the limitations of current chatbots, it is equally valuable for Machine Learning venues, as it highlights the need for more realistic validation of ML applied to dialog systems. Consequently, I consider this work to be of high significance.

Two important conjectures underlie this paper and are likely to inspire further research. While not explicitly stated in the text, Antoine Bordes articulated them clearly during a NIPS workshop presentation covering this work. Considering the metrics and tasks chosen in this paper, these conjectures are:
1) The performance of end-to-end ML approaches is still insufficient for goal-oriented dialogs.
2) When comparing algorithms, relative performance on synthetic data is a good predictor of performance on natural data. This would be a significant departure from previous observations, but the authors make a strong effort to align the conditions of the synthetic and natural data.

While the paper's primary algorithmic contribution is a relatively simple addition to memory networks (the 'match type' feature), this work marks the first time such models have been deployed and tested on a goal-oriented dialog task. The experimental protocol is excellent. The paper is exceptionally clear and accessible to a readership beyond ML and dialog researchers. I was particularly impressed by the concise summary of memory networks in the appendix, as well as the tables clearly illustrating the influence of the number of hops.

While this paper represents the state-of-the-art in the search for more rigorous metrics for dialog modeling, it also serves as a reminder of how brittle and somewhat arbitrary these metrics can be. This critique is intended as a recommendation for future research rather than a request for revision.

First, the authors use per-response accuracy, which treats the task as a next-utterance classification problem from a fixed list of candidate responses. Table 3 clearly illustrates how misleading this metric can be in practice. In a real-world scenario, the primary goals are a correct API call and a reasonably short dialog. However, under this metric, a successful seven-turn dialog would receive an accuracy of only 1/7, as the six intermediate bot responses leading to the final API call must also be exact matches.

Is per-dialog accuracy, where all responses in a dialog must be correct, a better alternative? Table 2 demonstrates its sensitivity to the experimental protocol. I was initially puzzled that the accuracy for subtask T3 (0.0%) was much lower than for the full dialog T5 (19.7%). This counterintuitive result is explained by the task definitions (Section 3.1.1), which reveal that T3 requires displaying three options while T5 only requires one. Regarding the concierge data, a question arises: what would the results be if 'correct' meant selecting the single best response, rather than a response from the 5-best list?

While I cannot fault the authors for using standard dialog metrics—and indeed, for proposing new ones, even if they are arguably too pessimistic—I believe a different perspective could lead to more meaningful metrics for goal-oriented tasks. Consider a commercial scenario: if a virtual assistant service is paid only upon successful task completion, what metric would best align with maximizing revenue? In the context of the restaurant-finding task, such a metric would likely be a weighted loss function incorporating the number of errors in the final API call, the number of turns required to reach it, and the number of options rejected by the user. However, such a loss function cannot be evaluated on static dialog datasets and would require interaction with either real human users or a realistic user simulator.

A final point, closely related to representation learning, is the issue of vocabulary mismatch between the user's query and the knowledge base (KB), which the paper does not fully address. Specifically, the 'match type' feature requires an exact string match; for the system to recognize 'Indian' as a 'type of cuisine', the word 'Indian' must exist verbatim in the KB. One can easily imagine scenarios where the KB uses non-standard or obfuscated terminology. In such cases, an ideal system should learn the association between the user's vocabulary and the KB's terminology, rather than relying on pre-specified, exact matches.
