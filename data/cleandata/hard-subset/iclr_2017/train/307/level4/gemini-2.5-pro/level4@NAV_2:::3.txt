This paper presents a valuable new testbed for evaluating end-to-end goal-oriented dialog systems. The authors introduce a series of tasks within the restaurant reservation domain, where a system must converse with a user to gather parameters, query a knowledge base, and ultimately book a table. The dataset is synthetically generated using a simulator and rule-based language transformations, and the evaluation is framed as a task of ranking candidate responses for each turn. The work provides a thorough baseline analysis, demonstrating that Memory Networks outperform classical information retrieval and supervised embedding models on the proposed tasks.

I recommend accepting this paper. It introduces a well-structured and useful resource for the community that facilitates reproducible research in a critical area. My primary reservations center on two key points: the motivation for purely end-to-end approaches in narrow domains and the limitations of a ranking-based evaluation framework.

My supporting arguments are as follows:

First, the paper’s motivation hinges on the scalability limitations of traditional, hand-crafted dialog systems. While this is a compelling argument for open-domain chit-chat, its applicability to narrow-domain, transactional settings is less clear. In a domain like restaurant booking, where the state space is well-understood and the primary goal is high user satisfaction and task completion, leveraging explicit domain knowledge to engineer a robust system is often the most practical approach. The paper does not fully justify why an end-to-end system, which forgoes this explicit knowledge, would be preferable in this context.

Second, the decision to frame the task as ranking from a pre-defined set of candidates, rather than generating responses, significantly limits the scope and practical implications of the work. One of the main promises of end-to-end systems is their ability to generalize and scale to new domains with minimal re-engineering. However, if deploying in a new domain requires first manually compiling a comprehensive list of all possible system utterances to serve as candidates, this reintroduces a substantial manual bottleneck. A generative model would present a more compelling case for the scalability and flexibility of the end-to-end paradigm.

The following points are offered as additional feedback to help improve the paper:

*   The data is generated via a simulator using a set of natural language generation rules. While this allows for the creation of large datasets, it also constrains the linguistic variability. For the sake of transparency and reproducibility, it would be beneficial to publish the set of rules used by the simulator, perhaps in an appendix or as supplementary material.
*   Could the authors elaborate on how the candidate response set would be obtained in a practical deployment for a new domain? The current methodology, which appears to pool all responses from the training, validation, and test sets, is not feasible in a real-world scenario and raises questions about the evaluation protocol.
