Attempts to utilize chatbots for every form of human-computer interaction have been a major trend in 2016, with claims that they could solve many forms of dialogs beyond simple chit-chat. This paper serves as a serious reality check, providing a much-needed assessment of the current state of chatbots. While its relevance is mostly pertinent to Dialog/Natural Language venues, where it can educate software engineers about the limitations of current chatbots, it can also be published at Machine Learning venues to educate researchers about the need for more realistic validation of ML applied to dialogs. I would consider this work to be of high significance.

Two important conjectures underlying this paper are likely to open up new avenues for research. Although they are not explicitly stated in the paper, Antoine Bordes clearly articulated them during a NIPS workshop presentation that covered this work. Considering the metrics chosen in this paper, these conjectures can be summarized as follows:

1) The performance of end-to-end ML approaches is still insufficient for goal-oriented dialogs.
2) When comparing algorithms, relative performance on synthetic data can be a good predictor of performance on natural data. This finding would be a significant departure from previous observations, but the authors made a concerted effort to match the synthetic and natural conditions.

While the paper's original algorithmic contribution consists of a relatively simple addition to memory networks (match type), it is the first time these are deployed and tested on a goal-oriented dialog. The experimental protocol is excellent, and the paper's clarity is excellent, making it accessible to a readership beyond ML and dialog researchers. I was particularly impressed by the concise appendix on memory networks, which summarized them effectively, followed by the tables that explained the influence of the number of hops.

This paper represents the state-of-the-art in the exploration of more rigorous metrics for dialog modeling, but it also highlights the brittleness and arbitrariness of these metrics. This observation is more a recommendation for future research than a suggestion for revision.

The authors use per-response accuracy (basically the next utterance classification among a fixed list of responses) as one of their metrics. However, as table 3 clearly shows, this can be problematic in practice: all that matters is a correct API call and a reasonably short dialog, which would only yield a 1/7 accuracy, as the 6 bot responses needed to reach the API call also have to be exact.

Would per-dialog accuracy, where all responses must be correct, be a better metric? Table 2 shows how sensitive it is to the experimental protocol. Initially, I was puzzled that the accuracy for subtask T3 (0.0) was much lower than the accuracy for the full dialog T5 (19.7), until the authors pointed me to the task definitions (3.1.1), where T3 requires displaying 3 options while T5 only requires displaying one.

For the concierge data, it would be interesting to explore what would happen if 'correct' meant being the best, not among the 5-best.

While I cannot fault the authors for using standard dialog metrics, and coming up with new ones that are actually too pessimistic, I can think of one way to represent dialogs that could result in more meaningful metrics in goal-oriented dialogs. Suppose I sell Virtual Assistants as a service, being paid upon successful completion of a dialog. What is the metric that would maximize my revenue? In this restaurant problem, the loss would likely be some weighted sum of the number of errors in the API call, the number of turns to reach that API call, and the number of rejected options by the user. However, such a loss cannot be measured on canned dialogs and would either require a real human user or a realistic simulator.

Another issue closely related to representation learning that this paper fails to address or explain properly is what happens if the vocabulary used by the user does not match exactly the vocabulary in the knowledge base. In particular, for the match type algorithm to code 'Indian' as 'type of cuisine', this word would have to occur exactly in the KB. I can imagine situations where the KB uses some obfuscated terminology, and we would like ML to learn the associations rather than humans to hand-describe them.
