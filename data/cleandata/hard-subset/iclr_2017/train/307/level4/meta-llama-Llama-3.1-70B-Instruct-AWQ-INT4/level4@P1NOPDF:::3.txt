SYNOPSIS:
This paper introduces a new dataset for evaluating end-to-end goal-oriented dialog systems. The dataset is generated in a restaurant setting, where the goal is to determine availability and eventually book a table based on parameters provided by the user to the bot as part of a dialog. The data is created by running a simulation using an underlying knowledge base to generate samples for various parameters (cuisine, price range, etc.) and then applying rule-based transformations to render natural language descriptions. The objective is to rank a set of candidate responses for each next turn of the dialog, and evaluation is reported in terms of per-response accuracy and per-dialog accuracy. The authors demonstrate that Memory Networks can improve upon basic bag-of-words baselines.

THOUGHTS:
I would like to thank the authors for their interesting contribution. However, I have some reservations about the utility of end-to-end trained systems in narrow-domain settings. In open-domain settings, there is a strong argument to be made that hand-coding all states and responses would not be scalable, making end-to-end trained methods a more sensible approach. Nevertheless, in narrow-domain settings, we typically have a thorough understanding of the domain, and the primary goal is to achieve high user satisfaction. In such cases, wouldn't it be more effective to leverage domain knowledge to engineer the best possible system?

Given the restricted domain, I am somewhat disappointed that the objective is to rank responses rather than generate them, although I understand that this simplifies evaluation. I am also uncertain about how these candidate responses would be obtained in practice. It appears that the models rank the set of all responses in the train, validation, and test sets (as mentioned in the last sentence before Section 3.2). Since a key argument for the end-to-end training approach is its ease of scaling to new domains without requiring manual re-engineering of the system, I wonder where this information would be obtained for a new domain in practice. Generating responses would allow for better generalization to new domains, as opposed to simply ranking a list of hand-collected generic responses. In my opinion, this is the weakest aspect of this work.

Furthermore, as the data is generated using a simulation by expanding (cuisine, price, ...) tuples using NL-generation rules, it inevitably constrains the variability in the training responses. Although this is a trade-off with the ability to generate unlimited data using the simulator, I was unable to find the list of rules used. It would be beneficial to publish this information as well.

Despite my reservations, I believe this is an interesting contribution worthy of publication at the conference.

I have updated my score following the clarifications and new results.
