### Summary of Contributions

This paper addresses the challenge of applying end-to-end neural network models to goal-oriented dialog systems. While such models have shown success in chit-chat settings, their efficacy in transactional dialogs that require interaction with knowledge bases (KBs) and explicit goal completion remains unclear. The primary contribution of this work is the introduction of a new public dataset and a set of diagnostic tasks designed to systematically evaluate the capabilities of end-to-end dialog systems.

Set in the context of restaurant reservations, the work presents five synthetic tasks of increasing complexity, designed to test specific skills such as issuing API calls, handling user updates, interpreting API results, and providing information from a KB. To validate these synthetic tasks, the paper also includes experiments on two datasets derived from real-world interactions: a conversion of the Dialog State Tracking Challenge 2 (DSTC2) dataset and a new dataset from an online concierge service. The authors provide a comprehensive evaluation of several baseline models on these tasks, including classical information retrieval methods, supervised embedding models, and end-to-end Memory Networks. They demonstrate that Memory Networks, augmented with a proposed "match type feature" for handling entities, achieve the strongest performance among the learning-based models, yet still fall short of solving the more complex tasks. The paper's results highlight specific weaknesses of current end-to-end models, particularly in reasoning over knowledge returned from API calls.

### Decision and Justification

**Decision:** Accept

**Key Reasons:**
1.  **Significant Resource Contribution:** The paper introduces a well-designed, public, and reproducible testbed for goal-oriented dialog, a critical area of research where standardized evaluation benchmarks are lacking. The bAbI-style task decomposition provides a valuable diagnostic tool for analyzing model strengths and weaknesses.
2.  **Thorough and Rigorous Evaluation:** The experimental setup is comprehensive, providing strong baselines and insightful analysis. The validation of the synthetic tasks against real-world datasets (DSTC2, Concierge) convincingly demonstrates their utility as a proxy for real challenges, strengthening the paper's claims and the value of the contributed dataset.

### Supporting Arguments for the Decision

The paper's main strength is its contribution of a new resource to the community. The lack of suitable, open, and reproducible benchmarks has been a significant barrier to progress in end-to-end goal-oriented dialog. By creating a series of tasks with controlled complexity and making the data publicly available, the authors provide a much-needed tool for systematic research and comparative analysis. The design, which breaks down the dialog process into sub-tasks (e.g., Task 1: API calls, Task 3: Displaying options), is particularly effective for error analysis, allowing researchers to isolate and address specific model deficiencies.

The empirical evaluation is sound and supports the paper's claims effectively. The choice of baselines is appropriate, ranging from non-learning IR methods to standard supervised embeddings and a more advanced end-to-end architecture (Memory Networks). The results clearly demonstrate the limitations of simpler models and the relative strengths of the Memory Network. The use of both per-response and per-dialog accuracy is a crucial evaluation choice, as it starkly illustrates the difficulty of achieving perfect goal completion—a single error can derail an entire dialog, a fact highlighted by the frequent combination of high per-response accuracy with near-zero per-dialog accuracy.

Furthermore, the inclusion of an Out-Of-Vocabulary (OOV) test set and the novel "match type feature" directly addresses the important and practical challenge of generalizing to unseen entities, a known weakness of many embedding-based models. The experiments on the DSTC2 and Concierge datasets are a critical component of the paper, as they show that the relative performance patterns of the tested models are consistent between the synthetic and real data. This consistency strongly supports the central premise that the proposed testbed is a valid and useful tool for driving progress on real-world problems.

### Additional Feedback for Improvement

While the paper is strong, the following points could enhance it further:
*   A more detailed error analysis for the full dialog task (Task 5) would be beneficial. For dialogs where the model failed, it would be insightful to know which sub-task (e.g., API call formation, option display, information provision) was the primary cause of the failure. This would further leverage the diagnostic power of the task design.
*   The paper proposes "match type features" as a way to handle entities and OOV words. The performance gains are clear, especially on OOV tasks. A brief ablation study showing the impact of matching against "input or memory" versus just one of them could provide a clearer picture of how this feature is being used by the model.
*   The candidate set for evaluation is defined as all possible bot utterances across all data splits. While this is a standard approach for ranking-based evaluation, it would be helpful to briefly discuss the potential implications of this setup compared to a real-world scenario where candidates might be generated dynamically or come from a much larger set.

### Questions for the Authors
*   Regarding the "match type features," you state they are added if an entity word appears in the candidate, the KB, and the "input or memory." For multi-word entities (e.g., cuisine type "french"), how is the match performed? Is it based on an exact string match of the full entity name?
*   The results for Task 3 (Displaying options) show that models struggle significantly, with the best model achieving 0% per-dialog accuracy. This task requires sorting restaurants by rating. Does your analysis of the Memory Network's attention weights give any indication that the model is attempting to learn this sorting/comparison operation, or does it fail in a more fundamental way?
*   On the DSTC2 and Concierge datasets, real-world language is much more variable than in the synthetic tasks. Could you elaborate on the main challenges encountered when converting the DSTC2 data into your task format, particularly in mapping noisy user utterances and varied system responses to structured API calls and KB results?
