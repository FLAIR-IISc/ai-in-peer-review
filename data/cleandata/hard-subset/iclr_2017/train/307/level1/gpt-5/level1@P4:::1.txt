1. Summary of contributions
- The paper introduces a controlled, reproducible testbed for evaluating end-to-end goal-oriented dialogue systems in the restaurant reservation domain. It decomposes the task into five synthetic subtasks (API issuing, API updating, option presentation with sorting, knowledge use, and full dialogues), provides OOV splits to test entity generalization, and adds two more realistic evaluations: a conversion of DSTC2 into the same framework and a proprietary “Concierge” dataset.
- It evaluates several baselines (TF-IDF, nearest neighbor, supervised embeddings) and an end-to-end Memory Network (MemN2N) with and without “match type” features to handle entities and OOVs. The study measures both per-response and per-dialog accuracy, revealing that per-response can be encouraging while per-dialog remains low, especially for tasks requiring reasoning over KB outputs (e.g., sorting) and full conversations.
- The results and analysis identify concrete strengths (issuing/updating API calls; handling OOVs with type matches) and weaknesses (interpreting KB outputs; multi-turn robustness), and show similar performance patterns across synthetic tasks and real data (DSTC2/Concierge), arguing the synthetic suite is a useful proxy for real-world evaluation.

2. Decision: Accept
Key reasons:
- Novel, well-constructed, and much-needed evaluation resource for goal-oriented dialog that emphasizes reproducibility, diagnostics, and OOV challenges.
- Sound and carefully executed empirical study providing clear insights into what current end-to-end methods can and cannot do in task-oriented settings.

3. Supporting arguments
- What problem is tackled? The paper addresses whether end-to-end neural dialog models—successful in chit-chat—can handle goal-oriented tasks requiring state tracking, API interaction, KB interpretation, and entity handling. It proposes a testbed that decomposes these capabilities and supplies OOV evaluation.
- Motivation and placement in literature: The motivation is well grounded. Prior corpora focus on chit-chat or isolated state tracking; reproducible, fine-grained, end-to-end goal-oriented evaluation resources are scarce. The work builds on MemN2N and related dialog/QA literature and identifies concrete gaps (entities, API use, sorting, goal completion) that are not assessed by chit-chat datasets. The inclusion of DSTC2 and a Concierge dataset strengthens external validity.
- Are claims supported? Yes. The tasks are clearly defined; evaluation metrics (per-response and per-dialog) align with the goal-oriented objective. The experimental setup compares classical IR, supervised embeddings, and MemN2N with/without type features, and includes rule-based for context. Results consistently show: (i) MemN2N > supervised embeddings > IR; (ii) type features substantially help with entities/OOV; (iii) tasks needing KB interpretation and sorting remain difficult; (iv) similar trends on DSTC2 and Concierge. The methodology is sound, with hyperparameters validated and ablations (type features, hops) discussed. Potential limitations are acknowledged (synthetic language, deterministic templates), and the primary claim is the value of the testbed plus diagnostic insights rather than SOTA on any single task.

Caveats:
- The synthetic tasks use templated language and deterministic bot behavior, which may limit linguistic variability; however, the authors mitigate this via DSTC2 and Concierge.
- The response selection is a ranking over a candidate set that includes all utterances across splits; while standard in retrieval-style evaluation, it differs from generative evaluation and may simplify the problem relative to generation.
- Baselines lack contemporary sequence-to-sequence with attention/copy/pointer mechanisms, which are relevant for entity-heavy tasks. Nonetheless, the core contribution is an evaluation framework, and the MemN2N baselines establish a credible starting point.

4. Additional feedback (to improve the paper; not necessarily affecting the decision)
- Baselines: Include generative baselines (seq2seq with attention; pointer/copy mechanisms; key-value memory networks) and compare to MemN2N, especially on OOV/entity-heavy tasks and Task 3 (sorting/option listing).
- Candidate set and evaluation: Clarify the construction of the candidate response set (use across splits and tasks), and discuss implications for realism vs. generation. Consider an additional generation-based evaluation or constrained decoding to test beyond retrieval.
- Task diversity: Increase user-side linguistic variability (more paraphrases, noise) and non-deterministic bot policy variants, to better stress language understanding and policy learning.
- Analysis: Provide more error analyses on Task 3 and 5 (e.g., where sorting or option stopping fails), and ablate type features per entity type. Report confidence intervals or variance across runs.
- DSTC2 conversion: Detail how ratings were injected and whether alternative heuristics change results. Provide a mapping consistency check to ensure no unintended biases.
- Scaling: Explore sample efficiency (vary train sizes), and multi-domain variants to assess transfer.
- Reproducibility: Release code, data generation scripts, and trained models to facilitate adoption.

5. Questions for the authors
- Candidate set: The candidate pool includes utterances from train/validation/test (plain and OOV) across tasks. Can you confirm that models do not see test-specific candidates during training beyond knowing the candidate vocabulary, and comment on how this setup affects generalization vs. a generation setting?
- DSTC2 conversion: How exactly are ratings assigned to returned restaurants, and could this heuristic influence the option-ordering task? Did you observe sensitivity of results to this choice?
- OOV handling: For OOV entities, are embeddings always unknown and only type features enable correct behavior, or do you also share subword features? How many test responses require unseen entity strings?
- Task 3 difficulty: What specific failure modes prevent per-dialog success (e.g., sorting errors, not stopping after user accepts, error propagation)? Did increasing hops beyond 4 or adding explicit supervision signals (e.g., attention supervision) help?
- Realism: How sensitive are results to increasing the number of user/bot templates, introducing paraphrases/typos, or relaxing the deterministic asking order in Task 1?
- Baselines: Did you attempt pointer/copy mechanisms or key-value memory networks? If so, how did they perform relative to match-type features?
- Generalization: When training on one KB partition and testing OOV on the other, how much performance degrades per entity type? Can you provide breakdowns (location vs. cuisine vs. restaurant name)?
- Metrics: Do per-dialog accuracy and per-response accuracy correlate with human judgments on DSTC2/Concierge samples? Any preliminary human evaluation to validate the proxy?
