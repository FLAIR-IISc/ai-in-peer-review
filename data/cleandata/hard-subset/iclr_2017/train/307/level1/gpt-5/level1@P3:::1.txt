Summary and contributions
The paper introduces an open, reproducible testbed for end-to-end training of goal-oriented dialog systems in the restaurant reservation domain. The testbed decomposes the task into five synthetic sub-tasks (API call issuance, updating calls, displaying sorted options, providing KB-grounded facts, and full dialogs), with additional evaluations on DSTC2 (converted to the same format) and on a proprietary “Concierge” dataset. The work highlights specific capabilities required for goal-directed dialogs: implicit state tracking, KB querying and response grounding, sorting and selection, and handling OOV entities. The authors provide strong baselines (TF-IDF, nearest neighbor IR, supervised embeddings) and a Memory Networks variant augmented with match-type features to better handle entities and OOVs. Results show MemNNs outperform other baselines, especially with type features, yet lag in per-dialog accuracy on tasks requiring multi-step reasoning over KB outputs (e.g., displaying options). The paper’s core contribution is a carefully designed, interpretable, and reproducible benchmark for analyzing strengths and weaknesses of end-to-end methods in transactional dialogs, along with a transparent empirical analysis.

Decision: Accept
Key reasons: (1) The dataset/testbed is novel, well-motivated, and fills a clear gap between chit-chat benchmarks and goal-directed dialog evaluation; (2) The empirical study is careful and insightful, demonstrating concrete limitations of current end-to-end models and providing reproducible baselines that should catalyze progress.

Supporting arguments
- Problem significance: The paper squarely targets the open question of whether end-to-end models can replace handcrafted slot-filling/POMDP systems in narrow domains. The decomposition into subtasks provides diagnostic leverage seldom available in existing corpora. Including OOV splits is particularly valuable for entity-centric tasks.

- Motivation and placement in literature: The authors situate the work well with respect to chit-chat datasets (Twitter/Reddit/Movie scripts), DSTC datasets, and POMDP/slot-filling systems. They correctly highlight evaluation difficulties for chit-chat, and the need for KB-grounded, goal-driven tests. The testbed is analogous in spirit to bAbI QA, but covers dialog policy and KB usage; this is an appropriate extension.

- Methodology and correctness: The simulation is clearly specified (KB schema, API calls, utterance pattern generation, stochastic user acceptance), with metrics that distinguish local vs global success (per-response vs per-dialog). The OOV split is principled (disjoint cuisines/locations). The ranking setup ensures unambiguous evaluation. Reported results are consistent with expectations: IR baselines struggle, supervised embeddings help but fail on KB-grounding steps, and MemNNs with type features improve OOV handling and factual grounding. The conversion of DSTC2 is described and justified, and the performance pattern transfers, lending credence to the testbed as a proxy for real tasks.

- Evidence supports claims: The claims are modest and well-supported: MemNNs can perform non-trivial operations (API calls, some OOV handling), but still falter on multi-step reasoning and option display; the testbed reveals these gaps; and patterns roughly replicate on DSTC2 and Concierge.

Caveats and limitations (do not change decision)
- Synthetic bias and determinism: Parts of the simulator (e.g., deterministic slot query order, templated bot language) risk overestimating progress and may simplify dialog management relative to real users/systems.

- Ranking vs generation: Framing response selection as ranking over a finite candidate set (collected across splits and tasks) eases the problem and may not reflect deployment realities. The candidate set also includes gold API calls and utterances, which can mask errors a generative system would face (e.g., surface form realization, pointer/copy).

- Baselines: While MemNNs are a reasonable end-to-end choice in 2016/2017, comparison to sequence-to-sequence baselines with copy/pointer mechanisms and delexicalization would be informative, especially for OOV entities and slot value propagation. Likewise, a key-value KB memory or explicit entity-centric attention could be competitive.

- Use of KB for OOV types: Allowing models to access the full KB (including OOV entities) during training is realistic in some settings but also blurs what “OOV generalization” means; it primarily tests exact-match/type handling rather than semantic generalization.

- DSTC2 adaptation: Adding ratings to enable deterministic option ranking is pragmatic but introduces a mismatch from original DSTC2 dynamics.

Constructive feedback to improve the paper
- Add stronger neural baselines: seq2seq with attention and copy/pointer; delexicalized models; key-value memory networks or entity-centric memory; and a simple RNN/LSTM belief tracker producing API calls, to position MemNNs more clearly.

- Report candidate set sizes and ablate ranking difficulty: e.g., restrict to in-domain candidates per task/turn vs global candidate pool; show sensitivity to candidate vocabulary size.

- Error analysis: Provide fine-grained breakdowns (e.g., failures in sorting vs entity linking vs tracking updates) and attention visualizations for T3/T5.

- Robustness: Include noise (typos/paraphrases) in synthetic data to bridge toward DSTC2/Concierge; experiment with non-deterministic bot policies.

- Generalization tests: Add cross-domain splits (e.g., hotels/flight) to test transfer; or demonstrate multi-domain extensions.

- Release plan: Clarify licensing and availability of all synthetic data and code; if Concierge cannot be released, provide detailed stats and possibly a smaller anonymized subset.

Questions for the authors
1) Candidate set construction: What are the exact sizes per task and per turn? Are candidates restricted to same-task utterances or pooled globally? How sensitive are results to this design?

2) OOV setup: Models have access to the KB containing OOV entities at training time. Have you tried disallowing that access to test purely compositional generalization? How do results change?

3) Sorting behavior (Task 3): Does the model truly learn to sort or memorize patterns of top-rated items? Have you tested with adversarial perturbations of KB ratings or with larger result sets?

4) Alternative architectures: Did you attempt seq2seq with copy/pointer or key-value memory networks? If so, how did they fare; if not, what are your expectations?

5) End-to-end policy learning: Could the framework support RL fine-tuning for action selection (e.g., when to ask vs propose vs confirm)? Any preliminary experiments?

6) DSTC2 conversion: Adding ratings is necessary for determinism—does this bias the task toward certain options? Did you try evaluating without enforcing sorted-by-rating behavior?

7) Per-dialog failures: Can you quantify the most frequent final failure modes in T5 (e.g., wrong option order, missed updates, wrong entity grounding)? This would guide future model design.
