1. Summary of contributions
- The paper proposes a lightweight, reproducible testbed for end-to-end goal-oriented dialog, centered on restaurant booking. It decomposes the problem into five synthetic tasks (API call issuance, updates, option ranking by rating, using KB returns, and full dialogs) plus two realistic datasets (a converted DSTC2 and a Concierge set) to probe capabilities such as dialog state tracking, KB querying, sorting, and handling OOV entities.
- It evaluates a set of baselines (IR, supervised embeddings) and an end-to-end Memory Network (MemN2N) variant, introducing “match-type” features to better handle entities and OOVs. Results are reported with both per-response and per-dialog metrics, emphasizing goal completion.
- Key findings: MemN2N significantly outperforms IR and embedding baselines on both synthetic and real data, learns non-trivial behaviors (issuing/updating API calls, entity handling with match types), but still fails on tasks requiring robust reasoning over KB returns (e.g., option display/sorting) and overall per-dialog success remains low. The synthetic tasks transfer in relative difficulty/pattern to DSTC2 and Concierge, suggesting utility as a proxy. Data and task generator (except Concierge) are to be publicly available.

2. Decision: Accept
Primary reasons:
- The work fills an important gap by providing a clear, reproducible benchmark for goal-oriented dialog, a domain where end-to-end models had lacked standardized evaluation.
- The empirical analysis is thorough and insightful, identifying concrete strengths/weaknesses of current end-to-end methods and proposing principled entity features that materially improve OOV performance.

3. Supporting arguments
- What problem is tackled? The paper asks whether end-to-end neural dialog systems—successful in chit-chat—can handle goal-oriented tasks requiring state tracking and KB manipulation, and offers a testbed to measure this rigorously. This is a timely and significant question.
- Motivation and positioning: The paper is well situated relative to chit-chat datasets (Twitter, Reddit), bAbI QA, and POMDP dialog frameworks. It clearly articulates why chit-chat evaluations don’t answer goal-oriented needs, and relates to DSTC challenges and user simulation literature. Compared to prior bAbI tasks, this adds dialog management, multi-turn interaction, API calls and KB use; compared to DSTC, it enables end-to-end learning and controlled analyses.
- Do results support claims?
  - The design isolates key competencies (state tracking, updates, sorting, KB lookup, OOV entities). Metrics (per-response and per-dialog) directly reflect success at completing tasks, revealing that high per-response does not imply successful dialogs—a valuable insight.
  - The MemN2N vs IR vs supervised embeddings comparisons consistently show expected ordering across synthetic and real datasets, supporting the claim that the synthetic suite is a reasonable proxy.
  - The match-type features address a well-identified weakness (exact matching and OOV entities) and demonstrably improve T4 and OOV performance. The ablations on hops indicate iterative reasoning helps.
  - Some caveats: language in synthetic tasks is templated (43 user / 20 bot patterns), the bot’s information-seeking order is deterministic in T1, and evaluation is ranking, not generation. These choices simplify the setting and could inflate reproducibility at the cost of realism. Still, the authors explicitly acknowledge these trade-offs and triangulate with DSTC2 and Concierge to mitigate.

4. Additional feedback to improve the paper (helpful suggestions, not necessarily affecting the decision)
- Broaden linguistic variability: Increase paraphrase diversity, non-deterministic policies for slot requests, and inject noise (typos, disfluencies) to better match real-world variability while preserving control.
- Generation and copy mechanisms: Complement ranking with sequence generation baselines incorporating attention/copy/pointer mechanisms (e.g., pointer networks) to compare with the proposed match-type features and test generalization to unseen entities.
- Stronger analysis of failure modes: Provide a fine-grained error taxonomy (e.g., sort failures vs KB retrieval vs coreference vs entity substitution), attention visualizations linked to errors, and per-turn confusion statistics, especially for T3 and T5.
- Statistical rigor: Report confidence intervals or significance tests where feasible, given relatively small training sets, and discuss variance across random seeds.
- Clarify OOV experimental protocol: Systems “have access to both KBs” during training; please clarify exactly what is accessible (entity lexicon vs dialog traces) and why this does not leak test information. Consider an OOV setting where the KB entity surface forms are unseen until test time.
- Candidate set construction: Since candidates are pooled from all tasks (including test), assess whether this biases ranking. Consider task-specific candidate sets or report sensitivity to candidate set size/composition.
- Release details: Ensure release of code, data generation scripts, and DSTC2 conversion recipes; if Concierge cannot be released, consider releasing a small, anonymized, synthetic-noisy subset to emulate its properties.

Questions for the authors
- OOV setup: What exactly is given to models at training regarding the OOV KB—are surface forms and types available as a lexicon? How would performance change if OOV surface forms were not known until test time?
- Determinism: How sensitive are the results to relaxing the deterministic question order in T1/T2? Have you run variants with randomized order or optional slot confirmations?
- Sorting in T3: Is the primary failure due to sorting, memory retrieval of ratings, or dialog control (continuing to next option)? Can you provide targeted diagnostics (e.g., evaluating just the top-1 option selection given facts)?
- Candidate sets: Since candidates include all bot utterances and API calls across splits and tasks, how do results change when restricting to task-specific candidates or scaling candidate pool size?
- DSTC2 conversion: Could you share the conversion script and quantify the noise introduced by adding ratings and aligning API outputs? How often does the original bot behavior deviate from deterministic policies?
- Match-type features vs copy: Did you compare to pointer/copy architectures? If not, can you comment on expected trade-offs and whether match-typing remains beneficial when copy is available?
- Generalization beyond restaurants: Have you tried re-instantiating the simulator in another domain (e.g., hotel booking) to test transfer, especially for the OOV/type handling claims?

Overall, this is a well-motivated and impactful benchmark paper with careful empirical study and actionable insights for the community. It is likely to catalyze progress on end-to-end goal-oriented dialog.
