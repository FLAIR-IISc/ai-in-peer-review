- Paper provides a systematic synthetic testbed to evaluate end-to-end learning for goal-oriented dialog, focusing on a restaurant reservation domain with five structured tasks.  
- Introduces five tasks that decompose capabilities needed for goal-oriented dialog: API calls, updating requests, displaying options, providing info, and full dialogs.  
- Shows Memory Networks with a novel "match type" feature can learn non-trivial operations on these tasks but still lag behind rule-based systems in per-dialog accuracy.  
- Demonstrates strong correlation between performance on the synthetic tasks and on real-world datasets (DSTC2 and concierge), making the testbed a useful benchmark.  
- Experimental protocol is very clear, with thorough ablation studies and insightful error analysis.  
- Decision: Accept — the testbed is a significant, reproducible contribution and a reality check on end-to-end methods for goal-oriented dialog.  
- Key contribution: rigorous evaluation of whether end-to-end models can compete with traditional methods in narrow-domain, goal-oriented tasks.  
- Evidence supports two conjectures: current end-to-end approaches are insufficient for goal-oriented dialog, and synthetic task performance predicts natural-data performance.  
- Synthetic testbed enables controlled evaluation of specific capabilities (e.g., OOV handling, API interpretation) and compares multiple baselines (rule-based, IR, supervised embeddings, MemN2N).  
- Analysis of metrics highlights that per-response accuracy is misleading for goal-oriented tasks and per-dialog accuracy is sensitive to experimental protocol.  
- Algorithmic contribution: simple but effective "match type" feature for Memory Networks and first thorough Memory Network evaluation on goal-oriented dialog.  
- Paper presentation is clear, with helpful appendices and informative tables (e.g., impact of number of hops).  
- Improvement: discuss broader implications for evaluation metrics and how findings should influence future evaluation frameworks.  
- Improvement: sketch concrete approaches for incorporating application-driven loss functions into the testbed.  
- Improvement: explore extensions to match type features to handle semantic associations between user language and KB terms (e.g., via embeddings).  
- Improvement: provide guidance on adapting the testbed to domains beyond restaurant reservations and discuss associated challenges.  
- Question: how would you change concierge evaluation to require a single best response instead of 5-best, and how would that affect model rankings?  
- Question: how would you design a more robust evaluation protocol to reduce sensitivity of per-dialog accuracy to protocol choices (T3 vs T5 issue)?  
- Question: give a concrete example of an application-driven loss (e.g., revenue-linked) for restaurant reservations and what extra data would be needed.  
- Question: how could match type features be extended to capture semantic similarity between entities (e.g., "Italian" vs "Italian cuisine") and what impact would that have on OOV performance?
