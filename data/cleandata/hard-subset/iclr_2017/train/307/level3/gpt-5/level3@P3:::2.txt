1) Summary of contributions and claims
- Problem tackled: The paper asks whether current end-to-end neural dialog systems can reliably handle goal-oriented tasks in narrow domains; what capabilities they have/ lack; and how to evaluate them in a reproducible way.
- Contributions: 
  - A clear, reproducible testbed for goal-oriented dialogs in restaurant booking: five synthetic tasks (API issuing, updating, ranking options, using KB facts, and a full dialog), a conversion of DSTC2 into the same format, and evaluation on a real online concierge dataset.
  - A simple but impactful extension to Memory Networks (“match type” features) that leverage KB entity types and exact matches to handle OOV entities and KB lookups.
  - Strong experimental protocol: ranking setup; both per-response and per-dialog metrics; OOV generalization; careful baselines (IR, supervised embeddings, MemNN with/without match type); thorough ablations (hops), appendix details, and clear tables.
- Main findings: 
  - Reality check: end-to-end models remain inadequate for reliable goal-oriented dialogs. Per-response accuracy can look promising while per-dialog success is still low, especially on tasks needing reasoning over KB results (T3, T5).
  - Two conjectures supported: (1) current end-to-end ML underperforms on truly goal-directed tasks; (2) relative performance on well-matched synthetic tasks predicts trends on natural data (DSTC2, Concierge).
  - The match-type augmentation is simple yet practically important for OOV entities and for providing extra info (T4), though it does not solve vocabulary mismatch or semantic association learning.

2) Decision
Accept.
Key reasons: 
- The paper provides a timely, much-needed benchmark and a well-designed experimental protocol that advances evaluation for goal-oriented dialog, offering a valuable reality check to both Dialog/NLP and ML communities.
- Clarity and rigor are excellent; the results convincingly support the core claims and will catalyze subsequent work.

3) Supporting arguments
- Specific question/problem: The paper directly targets the gap between chit-chat success and goal-directed performance, decomposing the latter into testable subtasks that isolate dialog management, API use, KB reasoning, and OOV handling.
- Motivation and placement: Well-situated against slot-filling/POMDP literature and prior end-to-end dialog/QA work. The need for reproducible, interpretable goal-oriented evaluation is clearly established.
- Do results support claims?
  - Yes: MemNNs outperform IR and supervised embeddings, yet still fall short on tasks requiring KB result interpretation and option ranking. Match type features improve OOV and T4, but do not resolve T3/T5.
  - The same relative ordering of methods holds on DSTC2 and Concierge, supporting the conjecture that synthetic task performance is a useful proxy for real data.
- Methodological strengths:
  - Transparent candidate-ranking setup; OOV splits; hop analysis showing benefits of iterative attention; comprehensive appendices.
- Evaluation caveats (acknowledged and important):
  - Common dialog metrics can be brittle. Per-response accuracy over-penalizes harmless intermediate variations when only API correctness matters; per-dialog accuracy is highly sensitive to task definitions (e.g., the counterintuitive T3 vs T5 outcomes).
  - On Concierge, the “correct among 5-best” vs “single best” decision materially affects conclusions.

4) Additional feedback (to help improve the paper)
- Metrics and objectives:
  - Consider application-driven losses focused on task success and cost: e.g., weighted sums of API-call errors, number of dialog turns, and user rejections—aligned with business metrics (e.g., revenue). This likely requires human users or well-validated simulators.
  - Report API-call correctness separately from surface-form response accuracy; treat intermediate turns differently from decision-critical actions.
  - For Concierge, clarify whether correctness should be top-1 (operator’s final choice) or top-k; justify with human annotations or downstream utility.
- Task design and realism:
  - Relax deterministic slot order and add paraphrase/ASR noise or mixed-initiative behavior to better reflect real interactions and reduce brittleness of per-dialog accuracy to protocol details.
  - Add evaluation with generation (not only ranking) or include both, to increase ecological validity.
- Modeling directions:
  - Address vocabulary/KB mismatch more directly: entity linking with fuzzy matching, subword/character models, pointer/copy mechanisms, key-value memory networks, or neural semantic parsing to bridge from user language to KB schema.
  - Learn semantic associations between user terms and KB entities (synonyms, paraphrases) rather than only exact matches; consider KB embeddings and ontology constraints.
  - Explore RL fine-tuning with success-oriented rewards once a reliable simulator/user study is available.
- Analysis:
  - Provide ablations isolating the contributions of time features, speaker tokens, and match types; include learning curves (data size sensitivity).
  - Examine candidate-set size effects on ranking and report calibration of confidence for decision-critical actions.

Questions to the authors
- Metrics/Protocol:
  - For Concierge, how would conclusions change under top-1 vs top-5 evaluation? Can you provide human judgments of “best” vs “acceptable” to ground this choice?
  - Can you report API-call correctness and decision-critical action accuracy separately from overall per-response accuracy?
  - Did you analyze sensitivity of per-dialog accuracy to minor task-definition changes (beyond T3 vs T5) to quantify brittleness?
- Modeling:
  - How much of the T3/T5 failure is due to ranking among many near-duplicate candidates versus genuine reasoning limits? Any experiments reducing candidate entropy?
  - Can you share ablations on match-type features: exact span vs fuzzy matches; effect per entity type; and performance when KB entity strings do not occur verbatim in dialog?
  - Did you try copy/pointer mechanisms or key-value memory variants, and if so, how did they compare?
- Data/Realism:
  - Have you considered injecting paraphrases, typos, or stochastic slot orders into the synthetic tasks to more closely mirror DSTC2/Concierge variability?
  - For RL, do you have plans for a user simulator calibrated to human behavior on these tasks, enabling application-driven rewards?
- Reproducibility:
  - Please clarify the candidate set construction across tasks and splits, and provide code/scripts to reproduce the ranking candidates and the match-type typing pipeline.
