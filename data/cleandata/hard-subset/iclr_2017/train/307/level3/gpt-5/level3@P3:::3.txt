Summary of contributions
- Proposes a clean, reproducible testbed for end-to-end goal-oriented dialog in a restaurant domain, grounded in a KB and driven by a rule-based simulator that generates both user and system utterances. The suite decomposes the problem into five subtasks (API call issuing/updating, option display with sorting, extra info retrieval, and full dialogs), plus a conversion of DSTC2 and an operator–user Concierge dataset.
- Frames evaluation as next-response ranking among a global candidate set, reporting both per-response and per-dialog accuracy to reflect goal completion. Includes OOV splits to explicitly test generalization to unseen entities.
- Benchmarks a strong end-to-end baseline (MemN2N) against TF-IDF, nearest neighbor IR, and supervised embeddings, and adds match-type features to handle entities. MemNNs consistently outperform bag-of-words/IR baselines, achieve solid per-response accuracy, learn non-trivial behaviors (API calls, entity manipulation), and beat a reasonable rule-based system on DSTC2. Nonetheless, per-dialog accuracy remains low on the hardest tasks, highlighting current limits.

Decision
Accept. Key reasons: (1) The dataset and task design are timely, carefully constructed, and enable controlled, reproducible evaluation with interpretable subtasks and an OOV regime. (2) The empirical study is thorough and honest, providing clear evidence that MemNNs outperform simpler baselines while still exposing important gaps, thus likely catalyzing future research.

Supporting arguments
- Problem relevance and scope: Goal-oriented dialog with KB access is central to assistants/bots, yet evaluation resources lag behind chit-chat tasks. This work offers a modular, analyzable proxy with both synthetic control and connections to real data (DSTC2, Concierge).
- Sound methodology: The tasks probe distinct capabilities (implicit state tracking, KB use, sorting, OOV handling). The OOV KB split is well-motivated and stresses a known weakness of embedding-based models. Reporting both per-response and per-dialog metrics surfaces the brittleness of end-to-end systems in multi-turn goal achievement.
- Empirical rigor: Multiple baselines (IR, supervised embeddings), a competitive end-to-end architecture (MemN2N), and a principled entity-typing augmentation support the claims. Ablations (hops, match-type features) are informative. Results on DSTC2 and Concierge echo patterns from simulation, lending external validity.
- Clarity and reproducibility: The paper is clearly written, model and training details are provided, and the dataset is positioned for open use. The breakdown maps naturally to error analysis.

Reservations and limitations (not blocking)
- Ranking vs generation: While ranking is appealing for comparability and stability, it presumes a precompiled candidate set and obscures fluency/control trade-offs. It also limits direct transfer to new domains unless one can source candidate responses.
- Narrow-domain end-to-end value: In highly structured, single-domain settings with deterministic policies, hand-engineered slot-filling systems may still yield higher user satisfaction. The paper’s own per-dialog results suggest current end-to-end methods remain fragile in longer transactions.
- Language variability: Rule-based NLG reduces diversity; generalization to paraphrases and noisy ASR-like inputs may be overestimated. Although DSTC2/Concierge partially address this, the core contribution still leans on synthetic data.

Additional feedback to improve the paper
- Publish the simulator’s NLG resources: Please release the full set of user and bot templates (43 and 20 patterns) and the templating rules. This will make variability explicit and allow the community to stress-test paraphrasing.
- Candidate ranking practicality: Discuss realistic pipelines for candidate construction in new domains. For example, can few-shot mined templates, retrieval from in-domain logs, or controllable NLG be used to seed and expand candidate banks?
- Add a generative track: Alongside ranking, consider a “copy+generate” evaluation with pointer/copy mechanisms for entities, constrained decoding for API calls, and automatic+human evaluation. This would better reflect deployment constraints without precompiled candidates.
- Robustness and scaling: Include data size scaling curves and noise robustness experiments (typos, paraphrases, disfluencies) on the simulated tasks to quantify brittleness vs. training size and variability.
- Error analysis for T3/T5: Provide a fine-grained breakdown (sorting errors, option continuation/stop errors, entity grounding) to pinpoint which reasoning steps fail.
- Metric extensions: Consider goal-success under limited tolerance (e.g., allow one non-critical error) and dialog efficiency metrics (turns to completion), which may correlate better with user satisfaction.

Answers to key questions
1. Specific problem: How to systematically evaluate end-to-end dialog systems for goal-oriented tasks requiring KB access and symbolic operations; what are their strengths/limitations relative to traditional methods.
2. Motivation and placement: Strong. The work fills a clear gap between chit-chat datasets and real transactional dialog, connects to DSTC literature, and leverages MemNN advances with entity typing.
3. Do results support claims: Yes. The study is careful and the data convincingly show both progress (MemNN gains, OOV handling via match types) and remaining gaps (low per-dialog success, difficulty with option display and KB interpretation).

Questions for the authors
- Candidate set construction: For Tasks 1–5, candidates include all bot utterances and API calls across train/val/test and OOV. How would this translate to a brand-new domain without prior logs? Can you quantify performance when candidates are restricted to training-only utterances?
- DSTC2 conversion: Since ratings were added to enable deterministic option ranking, how sensitive are results to this choice? Could alternative, non-deterministic policies be supported?
- OOV handling: At test time, models have access to both KBs. Does this create any leakage beyond type matching? What happens if the test-only KB is partially hidden (e.g., names unseen until API return)?
- Template release: Will you release the exact NLG templates and rules for both user and bot? If not, can you include them in an appendix or supplementary material?
- Generative baselines: Did you attempt a pointer-generator or copy-augmented seq2seq baseline on the ranking-as-classification setup (by reranking generated candidates)? Any preliminary findings?
- Scaling and hops: Can you provide learning curves and a sensitivity analysis on the number of hops per task, to better understand when multi-hop reasoning becomes essential?
- User-centric evaluation: Any plans to validate per-dialog accuracy against human judgment of goal success and satisfaction, especially on DSTC2/Concierge?
