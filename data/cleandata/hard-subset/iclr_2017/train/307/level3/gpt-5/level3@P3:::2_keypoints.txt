- Problem: Evaluate whether end-to-end neural dialog systems can reliably handle goal-oriented tasks in narrow domains and how to evaluate them reproducibly.
- Contribution: A clear, reproducible testbed for restaurant-booking dialogs with five synthetic tasks, DSTC2 conversion, and a real Concierge dataset.
- Contribution: A simple extension to Memory Networks (“match type” features) using KB entity types and exact matches to handle OOV entities.
- Contribution: Strong experimental protocol—ranking setup, per-response and per-dialog metrics, OOV splits, careful baselines, and thorough ablations.
- Main finding: End-to-end models remain inadequate for reliable goal-oriented dialogs; per-response accuracy can be misleadingly optimistic.
- Main finding: Tasks requiring KB reasoning and option ranking (T3, T5) show especially low per-dialog success.
- Main finding: Relative performance on synthetic tasks predicts trends on natural data (DSTC2, Concierge).
- Main finding: Match-type features materially help OOV handling and T4 but do not solve vocabulary mismatch or semantic association learning.
- Decision: Accept.
- Reason for acceptance: Provides a timely, well-designed benchmark and rigorous protocol that advances evaluation and offers a reality check to the community.
- Scope: Paper decomposes goal-oriented dialog into testable subtasks isolating dialog management, API use, KB reasoning, and OOV handling.
- Placement: Well situated against slot-filling/POMDP and end-to-end dialog/QA literature with clear motivation for reproducible evaluation.
- Result support: MemNNs outperform IR and supervised embeddings but still fail on KB interpretation and ranking tasks.
- Method strength: Transparent candidate-ranking setup, OOV splits, hop analysis, comprehensive appendices, and clear tables.
- Evaluation caveat: Per-response accuracy can over-penalize harmless intermediate variations; per-dialog accuracy is sensitive to task definitions.
- Specific caveat: Concierge evaluation changes notably depending on top-1 vs top-5 correctness criteria.
- Metric suggestion: Use application-driven losses aligned with task success and business metrics (API errors, dialog turns, user rejections).
- Metric suggestion: Report API-call correctness separately from surface-form response accuracy and treat intermediate vs decision-critical turns differently.
- Task-design suggestion: Relax deterministic slot order and add paraphrases/ASR noise or mixed-initiative behavior for realism.
- Evaluation suggestion: Include generation-based evaluation (not only ranking) to increase ecological validity.
- Modeling suggestion: Address vocab/KB mismatch via entity linking, fuzzy matching, subword/character models, pointer/copy mechanisms, or key-value memories.
- Modeling suggestion: Learn semantic associations between user terms and KB entities (synonyms, paraphrases) using KB embeddings or ontologies.
- Modeling suggestion: Explore RL fine-tuning with success-oriented rewards when a reliable simulator or user study is available.
- Analysis suggestion: Provide ablations isolating time features, speaker tokens, and match types; include learning curves and candidate-set size sensitivity.
- Reproducibility request: Clarify candidate-set construction and supply code/scripts for candidate ranking and the match-type typing pipeline.
- Question: How do conclusions change under top-1 vs top-5 evaluation on Concierge, and can human judgments ground this choice?
- Question: Can you report API-call correctness and decision-critical action accuracy separately from overall per-response accuracy?
- Question: How much of T3/T5 failure is due to ranking among near-duplicate candidates versus genuine reasoning limits, and are ablations available for match-type variants and copy/key-value models?
- Question: Have you considered injecting paraphrases/typos/stochastic slot orders into synthetic tasks and building a calibrated user simulator for RL experiments?
