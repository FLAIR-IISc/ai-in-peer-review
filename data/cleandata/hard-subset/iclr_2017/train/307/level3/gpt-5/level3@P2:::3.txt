Summary of claims and contributions
- The paper proposes an open, reproducible testbed for end-to-end goal-oriented dialog in a restaurant reservation setting. It decomposes the task into five interpretable subtasks (API call issuance, API updates, presenting sorted options from a KB, retrieving KB attributes, and full dialogs), plus an adapted version of DSTC2 and a real-world “Concierge” dataset.
- The evaluation is next-utterance ranking with per-response and per-dialog metrics, emphasizing goal completion.
- A Memory Network (MemN2N) baseline with match-type features outperforms IR and supervised embedding baselines, learns non-trivial operations (e.g., issuing API calls, using KB outputs), and shows some generalization to out-of-vocabulary entities when types are provided. Patterns observed on synthetic tasks mirror those on DSTC2 and Concierge, supporting the testbed’s external validity.
- The paper’s goal is not to beat handcrafted slot-filling systems in this narrow domain, but to provide a controlled proxy to expose strengths and weaknesses of end-to-end models.

Decision: Accept (borderline/weak accept)
Key reasons:
- The benchmark is timely, well-motivated, and likely to become an important resource for the community to study goal-oriented dialog in a controlled and interpretable way.
- The experimental analysis is careful and honest, revealing where end-to-end methods work and where they currently fail, and connecting synthetic findings to real data.

Supporting arguments
1) Problem significance and placement in literature
- The paper tackles a clear gap: there is no widely adopted, open, reproducible benchmark for end-to-end learning in goal-oriented dialog. Prior corpora (DSTC, ATIS) are tailored to state-tracking or constrained by licensing/crowdsourcing variability and do not offer the same controlled decomposition and OOV stress tests.
- The work is well-situated relative to POMDP/slot-filling systems, chit-chat datasets, and prior QA-style bAbI tasks.

2) Motivation and approach
- The design choices—simulation with a KB, controlled subtasks, OOV splits, and ranking-based evaluation—support fine-grained error analysis and rigorous comparison while remaining lightweight to use.
- Including DSTC2 and Concierge is a strong point: it grounds the proposed tasks in more realistic language and validates that relative performance patterns carry over.

3) Do the results support the claims?
- The empirical evidence supports the main claims: MemN2N outperforms IR and supervised embeddings; match-type features are crucial for OOV and for using KB outputs; models can issue and update API calls but struggle with option presentation and robust dialog-level success.
- The use of per-dialog accuracy is appropriate for goal completion and shows current limitations. The authors are transparent about these limitations.

Concerns and limitations (do not block acceptance but temper the contribution)
- Ranking vs generation: Constraining evaluation to ranking sidesteps generation challenges (lexicalization, handling unseen utterances, copy mechanisms) and depends on a precompiled candidate set. This weakens the claim of scalability to new domains unless one can supply high-coverage candidate lists without heavy authoring.
- Practicality of candidate sets: The work does not convincingly explain how, in a new domain, the candidate inventory would be constructed without reintroducing handcrafting.
- Language variability: The simulator’s small set of templates for user/bot utterances likely limits linguistic diversity and may overestimate model robustness. This is partially mitigated by DSTC2/Concierge, but the core simulated tasks remain narrow.
- Narrow-domain ceiling: While the goal is analysis rather than SOTA deployment, the fact that per-dialog accuracy remains low on more complex tasks highlights a gap to practical reliability.

Suggestions to improve the paper (for the camera-ready; not all are required for acceptance)
- Release simulator code, KBs, all NLG templates/patterns (user and bot), and the exact candidate lists for all tasks, including DSTC2 conversion scripts. This will enable replication, extension (e.g., paraphrase augmentation), and domain porting. The paper implies openness; please ensure the rule/template sets are included.
- Provide more detail on candidate sets: size per task, distribution of API-call vs NLG candidates, overlap across splits, and how candidates are constructed for DSTC2 and Concierge.
- Add ablations on: number of hops; memory size; effect of randomizing the order of missing-slot questions in Task 1; match-type features per entity type; sorting difficulty in Task 3 (e.g., perturb ratings or increase list length).
- Consider a generation track: report results for a seq2seq model with copy/pointer mechanisms and delexicalization; evaluate with success-rate and slot F1 rather than BLEU. Even a modest generation baseline would broaden the benchmark’s relevance.
- Stress-test language variability: add paraphrase augmentation, noise (ASR-like), typos, and non-deterministic bot behaviors in simulation to bridge further toward DSTC2/Concierge.
- Clarify the OOV setup: at test time, models can type-match against a KB containing unseen entities. Discuss how realistic this is and whether “cold-start KB entries” without textual exposure is an intended use case.

Questions for the authors
- Candidate responses: In a new domain, how do you foresee assembling comprehensive candidate lists without substantial handcrafting? Could you quantify how performance depends on candidate set size/coverage?
- Release details: Will you release the full simulator, all NL templates for users and bots, the KBs, and the exact candidate inventories used in evaluation (for T1–T6 and Concierge)? If not, what will be provided?
- Generation: Did you try any generative baselines (seq2seq with copy/pointer, pointer-generator, delexicalization)? If so, how do they compare on success rate and per-dialog accuracy?
- Determinism in Task 1: If you randomize the order of missing-slot prompts, how much does MemN2N performance drop? Does it still learn robust dialog management?
- Task 3 sorting: How do you verify the model truly learns to sort rather than memorize typical options? Any controlled tests with longer option lists or ties?
- OOV assumption: At test time, do models have access to the full KB (including unseen entities) for type matching? How would performance change if only embeddings from dialog data were available?
- DSTC2 conversion: Since ratings were injected to force a deterministic “best option,” how sensitive are results to this design? Could you report results without this augmentation?
- Data efficiency: Can you provide learning curves showing sensitivity to the number of training dialogs?
- Concierge labeling: API-use is a binary label without arguments—how consistent is this annotation, and how is ambiguity handled?

Overall, this paper offers a useful, well-scoped benchmark and a thoughtful analysis. Addressing the candidate-set practicality and releasing the full simulation/templates would substantially strengthen its impact.
