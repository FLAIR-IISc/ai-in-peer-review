Summary and contributions
The paper introduces an open, lightweight, and reproducible testbed for evaluating end-to-end goal-oriented dialog systems. Within a restaurant reservation domain, it defines five simulation-based tasks (API issuance, API updates, option display with sorting, information retrieval from KB facts, and full dialogs) plus two datasets from real interactions (DSTC2-derived human–bot dialogs and a human–human “Concierge” dataset). The work systematically probes capabilities like implicit state tracking, querying KBs, handling OOV entities, and sorting. It further provides strong baselines (IR, supervised embeddings) and an end-to-end Memory Network with a simple but effective match-type augmentation for entity handling. The experimental protocol is clear, well-motivated, and reproducible; the analysis yields actionable insights: end-to-end models handle simple dialog management (API calls/updates) but struggle to interpret KB results and maintain high per-dialog success; match-type features markedly improve OOV handling; and relative performance on synthetic tasks is predictive of results on more realistic data. Overall, it is a valuable reality check for the community.

Decision: Accept
Key reasons:
- A much-needed, well-designed benchmark and methodology that enable reproducible, fine-grained evaluation of goal-oriented dialog, with compelling evidence that synthetic tasks correlate with performance on real data.
- Thorough, clear experiments and analyses that surface concrete strengths/weaknesses of current end-to-end models; the match-type augmentation, while simple, is practically useful and well-studied here.

Supporting arguments
- Problem and motivation: The paper addresses a key gap—end-to-end dialog models have been mostly evaluated in chit-chat; it is unclear how they fare on transactional tasks requiring API use, KB reasoning, and entity handling. The proposed tasks directly test these abilities and provide built-in error analysis by design. The connection to the POMDP/slot-filling literature is well made, positioning the work within both dialog and ML communities.
- Approach positioning: The testbed is inspired by bAbI for QA, adapted thoughtfully to dialog (e.g., the bot must ask, not only answer). Including OOV splits, DSTC2 conversion, and a real Concierge dataset substantially strengthens external validity. Baselines span IR, supervised embeddings, and MemN2N; match-type features address known weaknesses with entities/OOV.
- Claims and evidence: Results are comprehensive (per-response and per-dialog) and consistent with the stated conjectures: (1) end-to-end models are not yet adequate for reliable goal-directed dialog; (2) relative gains on synthetic tasks mirror those on DSTC2 and Concierge. Ablations (e.g., hops) and comparisons (with/without match-types) are informative. The rule-based upper bound on synthetic tasks and the weaker rule-based performance on DSTC2 further underscore the value of learning-based approaches on real data.

Additional feedback (to improve the paper; not critical to the decision)
- Evaluation metrics and protocol:
  - Per-response accuracy can be misleading for transactional tasks; it equally penalizes low-importance turns and critical API errors. Consider reporting action/slot-specific accuracies and an aggregate, application-driven utility (e.g., weighted API error count, number of turns, user rejections), or task success under a budget of turns.
  - Per-dialog accuracy is extremely sensitive to task design choices (e.g., T3 requires ≥3 options while T5 allows ≥1). The T3 vs. T5 comparison can therefore be counterintuitive; harmonizing option constraints or reporting stratified results by number of options would aid interpretability.
  - On Concierge, consider top-k next-utterance classification (k=5,10) or human-judged acceptability; top-1 ranking is often too strict for open-ended human responses and may understate practical usefulness.
- Entity and vocabulary mismatch: The match-type feature relies on exact KB matches and does not learn semantic correspondences between user language and KB terms. Consider learned entity linking/normalization, character/subword models, pointer/copy mechanisms, and distributional alignment between user paraphrases and KB entities (e.g., type-consistent soft matching).
- Broader baselines: Adding pointer-generator/seq2seq with copy, key–value memory networks, or belief-tracking hybrids would contextualize MemN2N performance on KB retrieval and option presentation, especially for sorting/aggregation.
- Candidate set sensitivity: Since ranking is over a large candidate pool, report performance sensitivity to candidate set composition/size or to constrained legal-action subsets to approximate realistic policy spaces.
- Robustness and non-determinism: The deterministic bot policy in synthetic tasks favors pattern-matching; randomizing the order of information requests (or including controlled noise) would better assess robustness.

Questions for the authors
1) Concierge evaluation: What exactly constitutes a “correct” response—top-1 match to the single observed operator utterance? Did you measure top-k accuracy or human acceptability to account for multiple valid responses?
2) Match-type implementation: How are entities detected (case folding, tokenization, multiword spans)? Do you allow approximate matches or normalization (e.g., “St.” vs “Street”)? How are collisions handled across types?
3) OOV protocol: Models have access to both KBs during training. How would results change if the OOV KB were withheld until test time (simulating deployment to a new market)?
4) Candidate set: What is its exact size per task and how sensitive are results to that size? Do you have results when restricting to “legal” actions given the dialog state?
5) T3 vs. T5 difficulty: Can you provide analyses showing performance as a function of the number of KB options returned and acceptance probability? This would clarify the observed counterintuitive patterns.
6) Sorting evidence: Do the attention patterns or ablations (e.g., shuffling ratings) demonstrate that models truly learn to sort by rating rather than memorizing frequent options?
7) DSTC2 conversion: Bots sometimes make mistakes or behave non-deterministically. How did you treat inconsistent or erroneous gold responses in training/evaluation?
8) T2 drop with match-types: You note a slight performance decrease. Any hypothesis or ablation (e.g., interference between exact match cues and distributional cues) explaining this?
9) Hops and memory size: Can you provide a brief ablation on the number of hops vs. tasks (especially T3/T5) and on memory truncation to guide future work?
10) Toward application-aligned losses: Have you attempted reinforcement learning or cost-sensitive training that penalizes API errors more than phatic turns, or evaluated correlations with user success proxies?
