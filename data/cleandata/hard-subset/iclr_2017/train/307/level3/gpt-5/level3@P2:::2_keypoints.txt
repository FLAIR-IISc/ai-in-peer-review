- Introduces an open, lightweight, reproducible testbed for end-to-end goal-oriented dialog in a restaurant domain with five simulated tasks and two real datasets (DSTC2-derived and Concierge).
- Tasks probe implicit state tracking, KB querying, OOV entity handling, option display/sorting, and full dialogs.
- Provides baselines (IR, supervised embeddings) and an end-to-end Memory Network (MemN2N) with a match-type augmentation for entity handling.
- Experimental protocol is clear, well-motivated, and reproducible; analysis yields actionable insights.
- Main empirical finding: end-to-end models manage simple dialog actions (API calls/updates) but struggle to interpret KB results and to achieve high per-dialog success.
- Match-type features markedly improve handling of OOV entities.
- Performance on synthetic tasks is predictive of performance on more realistic datasets (DSTC2, Concierge).
- Decision: Accept.
- Key reasons: fills a needed gap with a well-designed benchmark/methodology enabling reproducible, fine-grained evaluation; thorough experiments revealing concrete strengths/weaknesses.
- Paper is well positioned relative to POMDP/slot-filling literature and adapts bAbI-style diagnostics thoughtfully to dialog.
- Inclusion of OOV splits, DSTC2 conversion, and a human–human Concierge dataset strengthens external validity.
- Ablations (e.g., hops, match-types) and comparisons (rule-based upper/lower bounds) are informative and consistent with claims.
- Recommendation: per-response accuracy can be misleading for transactional tasks; report action/slot-specific metrics and application-driven utility or task-success measures.
- Recommendation: per-dialog accuracy is sensitive to task design (T3 vs T5); harmonize option constraints or stratify results by number of options.
- Recommendation: on Concierge, use top-k next-utterance classification or human-judged acceptability rather than strict top-1.
- Recommendation: match-type relies on exact KB matches; consider learned entity linking/normalization, character/subword models, pointer/copy mechanisms, or soft type-consistent matching.
- Recommendation: add broader baselines (pointer-generator/seq2seq with copy, key–value memory networks, belief-tracking hybrids) to contextualize MemN2N.
- Recommendation: report sensitivity to candidate set composition/size and results when restricting to legal-action subsets.
- Recommendation: increase robustness in synthetic tasks by randomizing request order or adding controlled noise to better assess generalization.
- Questions for authors — 1) How is “correct” defined on Concierge; did you measure top-k or human acceptability? 2) How are entities detected and normalized for match-types (tokenization, multiword spans, abbreviations, collisions)? 3) OOV protocol: how would results change if OOV KBs were withheld until test time? 4) What is the exact candidate-set size per task and sensitivity to it; results when restricting to legal actions? 5) Can you analyze performance as a function of number of KB options and acceptance probability (T3 vs T5)? 6) Does attention/ablation evidence show models truly learn sorting by rating vs memorizing frequent options? 7) For DSTC2 conversion, how were inconsistent or erroneous gold responses handled? 8) Why does match-type slightly drop T2 performance — any ablation/hypothesis? 9) Provide ablation on number of hops and memory truncation effects, especially for T3/T5. 10) Have you tried RL or cost-sensitive training that penalizes API errors more than phatic turns, or evaluated correlations with user success proxies?
