**Summary**
This paper presents a valuable and timely contribution to the field of goal-oriented dialog systems. The authors introduce a well-designed testbed, comprising a set of synthetic and real-world tasks, to systematically evaluate the capabilities and limitations of end-to-end models. The central claim is that current end-to-end approaches, while promising, are not yet sufficient for robustly handling goal-oriented conversations. The paper rigorously tests this conjecture using Memory Networks, demonstrating their ability to perform non-trivial operations like issuing API calls but also highlighting their failures in interpreting KB results. The main algorithmic contribution is a simple but effective "match type" feature to handle out-of-vocabulary entities, which is thoroughly tested for the first time on these tasks. The work is significant for providing a crucial reality check to both the Dialog/NLP and broader ML communities.

**Decision: Accept**
I strongly recommend accepting this paper for two key reasons. First, it addresses a critical question about the practical viability of end-to-end dialog systems for goal-oriented tasks and provides a clear, reproducible benchmark for future research. Second, the experimental protocol and analysis are exceptionally clear and insightful, offering a valuable diagnosis of where current models succeed and fail.

**Supporting Arguments**
The paper’s strengths lie in its rigorous methodology and impactful conclusions. The authors convincingly show that performance on their well-designed synthetic data is a strong predictor of performance on real-world data (DSTC2, Concierge), validating the testbed's utility. The analysis of evaluation metrics is particularly insightful, correctly pointing out that common metrics like per-response accuracy can be misleading, as only the final API call or goal completion truly matters. The paper supports its claims with comprehensive experiments, including informative tables detailing the impact of model variations (e.g., number of hops, match type features), which strengthens the conclusions. While the algorithmic novelty is modest, the paper's primary contribution—a thorough and critical evaluation of an entire class of models on a crucial task—is a significant service to the community.

**Additional Feedback for Improvement**
While the paper is excellent, it could be strengthened by addressing two points. First, the evaluation metrics, while standard, are brittle. The authors could briefly discuss the potential for future work on more application-driven loss functions that directly penalize key failures (e.g., incorrect API calls) and reward efficiency (e.g., fewer turns), tying model training more closely to real-world utility. Second, the paper does not fully address the vocabulary mismatch problem between user language and KB terms. The "match type" feature requires exact lexical matches and cannot learn semantic associations (e.g., "cheap" vs "moderate"). Acknowledging this limitation more explicitly would frame an important direction for future work.

**Questions for Authors**
1.  Regarding the concierge data evaluation: Could you clarify if a "correct" response was defined as the single best response or one among a set of acceptable responses (e.g., top-5)? How might this choice influence the reported accuracy on such a noisy, human-generated dataset?
2.  The "match type" feature is a clever mechanism for handling OOV entities. Did you consider any approaches for learning semantic similarity between user utterances and KB entities when there is no direct lexical overlap?
