This paper introduces a valuable new resource for the community: a simulated, open-source dataset for training and evaluating end-to-end goal-oriented dialog systems. The authors create a testbed in the restaurant reservation domain, using a knowledge base and a rule-based simulator to generate dialogs. The tasks are thoughtfully designed to probe different system capabilities, such as issuing API calls, managing dialog state, and using KB information. The paper provides strong baseline results, demonstrating that Memory Networks can outperform classical IR and supervised embedding models, while also clearly showing the remaining challenges through per-response and per-dialog accuracy metrics.

I recommend that this paper be accepted. My decision is based on two key factors. First, the paper makes a significant contribution by releasing a reproducible and well-structured dataset that addresses a clear gap in the resources available for goal-oriented dialog research. This will facilitate standardized comparisons and help analyze model shortcomings. Second, the experimental analysis is thorough and provides insightful results, effectively benchmarking a modern neural architecture and revealing that while per-response accuracy is promising, achieving robust multi-turn goal completion remains a major hurdle.

My primary reservation concerns the framing of the core task as ranking from a pre-defined set of candidate responses. This approach limits the practical applicability of the findings, as it is unclear how such a candidate list would be generated in a real-world system for a new domain without significant manual collection, which runs counter to the goal of scalable end-to-end systems. Furthermore, I am skeptical about the immediate value of end-to-end training for narrow domains like this, where well-understood, hand-engineered systems may still offer superior performance and user satisfaction. The paper’s own results, showing low per-dialog accuracy, indirectly support this concern.

To improve the paper, I would strongly encourage the authors to publish the list of natural language generation rules used by their simulator. The current use of a small set of rule-based patterns may limit the linguistic diversity of the data, and making the rules public would increase transparency and allow for a better understanding of the dataset's inherent biases.

Finally, I would appreciate clarification on the following points:
1. The candidate response set is built from all utterances across the training, validation, and test sets. How do the authors envision a candidate set being generated in a live system deployed in a new domain, where future user and system utterances are unknown?
2. Could you provide more detail on the complexity and variety of the 20 bot-side natural language generation rules? This would help in assessing the difficulty of the language understanding and generation aspects of the tasks.
