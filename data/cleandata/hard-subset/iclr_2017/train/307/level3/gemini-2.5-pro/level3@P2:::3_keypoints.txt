- Introduces an open-source testbed for end-to-end, goal-oriented dialog in the restaurant reservation domain with five synthetic tasks plus DSTC2 and Concierge real-data tasks.
- Testbed probes abilities like issuing API calls, handling updates, and using KB information; simulated dialogs are generated from a KB using linguistic patterns.
- Provides thorough experimental evaluation of baselines (IR, supervised embeddings, Memory Networks); Memory Networks perform best but struggle on per-dialog accuracy.
- Decision: Accept.
- Primary reason 1: Supplies a much-needed open, well-designed benchmark that enables reproducible research and fine-grained analysis via a bAbI-style task breakdown.
- Primary reason 2: Offers comprehensive experimental analysis that clarifies gaps between current end-to-end models and reliability needed for practical goal-oriented systems.
- Main strength: Public dataset and lightweight testbed lower barriers to entry and facilitate standardized comparisons across approaches.
- Synthetic task design is thoughtful, allowing isolation of specific weaknesses (e.g., KB interpretation, out-of-vocabulary entities).
- Experimental insight: More sophisticated architectures outperform simple baselines; introducing "match type features" effectively helps with OOV entities.
- Key empirical finding: Models can have high per-response accuracy but very low per-dialog accuracy—single errors can derail entire conversations.
- Validation on DSTC2 and Concierge datasets supports the claim that the synthetic tasks serve as a reasonable proxy for real-world challenges.
- Caveat: Skepticism about short-term practicality of purely end-to-end systems for narrow-domain commercial apps, but paper succeeds as a diagnostic tool.
- Feedback 1: Framing the task as ranking candidate responses (vs. generation) significantly simplifies the problem and may reintroduce manual-effort bottlenecks.
- Feedback 2: Simulator uses limited linguistic patterns (43 user, 20 bot), limiting linguistic diversity; recommend releasing generation rules/patterns for transparency.
- Question 1: How should candidate response sets be constructed for new tasks without reintroducing manual effort?
- Question 2: Will the authors release the linguistic patterns and rules used by the simulator to improve transparency and reproducibility?
- Question 3: Do the authors see a path for hybrid systems combining end-to-end learning with rule-based components to ensure critical reliability (e.g., correct API calls, faithful KB presentation)?
