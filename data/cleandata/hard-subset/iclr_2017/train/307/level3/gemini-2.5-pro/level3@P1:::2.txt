### Summary of the Paper

This paper addresses the critical and timely question of whether modern end-to-end dialog systems, successful in chit-chat settings, are suitable for goal-oriented tasks. The authors make several significant contributions. First, they introduce a valuable new open-source testbed for goal-oriented dialog in the restaurant reservation domain. This testbed laudably breaks down the problem into five distinct subtasks that isolate key capabilities such as issuing API calls, handling updates, and using knowledge base (KB) results. The inclusion of an out-of-vocabulary (OOV) test set is particularly well-conceived, allowing for a rigorous assessment of generalization to unseen entities. Second, the paper provides a thorough empirical evaluation of various models, from classical IR baselines to supervised embeddings and Memory Networks (MemNNs). Finally, it introduces a simple but effective "match type" feature for MemNNs to better handle OOV entities by leveraging KB information. The authors convincingly demonstrate two main conclusions: (1) current end-to-end models are not yet robust enough to reliably handle goal-oriented dialogs, and (2) the relative performance of models on their synthetic tasks is a good predictor of performance on real-world data (from DSTC2 and an online concierge service), validating the utility of the testbed.

### Decision and Key Reasons

**Decision: Accept**

This is a clear accept. The paper provides a significant and timely "reality check" for the community on the state of end-to-end goal-oriented dialog. My decision is based on two key reasons:

1.  **A Valuable and Well-Designed Community Resource:** The paper's primary contribution is a public, reproducible, and well-designed suite of tasks for evaluating goal-oriented dialog systems. By decomposing the problem and providing OOV splits, the authors enable detailed error analysis and a clear path for measuring future progress. This is a far more rigorous approach than many existing evaluation paradigms and will be of great service to the field.
2.  **Thorough and Illuminating Experimental Analysis:** The empirical study is comprehensive and the conclusions are strongly supported. The paper offers clear insights into the specific failure modes of current systems (e.g., difficulty in interpreting and presenting KB results) and provides a strong baseline for future work. The validation of the synthetic data's utility against real-world datasets is a crucial result that strengthens the paper’s impact.

### Supporting Arguments for the Decision

The paper's experimental protocol is excellent. The comparison against a range of baselines—from simple TF-IDF and Nearest Neighbor to supervised embeddings and a hand-crafted rule-based system—provides a clear context for the performance of the proposed Memory Network model. The results table is particularly informative, showing a consistent trend where learning-based methods outperform IR baselines, and MemNNs outperform simpler embedding models, a trend that holds across both synthetic and real-world data.

The main algorithmic contribution, the "match type" feature, is straightforward but addresses a critical weakness of embedding-based models: handling specific, often OOV, entities. The experiments clearly show its effectiveness, particularly in the OOV setting and on tasks requiring the model to ground its response in KB facts (e.g., Task 4).

Most importantly, the paper’s conclusions are sober and well-grounded. While the per-response accuracies are encouraging, the near-zero per-dialog accuracies on the more complex tasks paint a clear picture of the remaining challenges. This work effectively reframes the problem from simply "building an end-to-end chatbot" to systematically solving the sub-problems required for a useful goal-oriented agent. The clarity of the writing and the helpful appendix, which includes implementation details and prediction examples, further enhance the paper's quality.

### Additional Feedback for Improvement

While the paper is strong, the following points could be considered to further improve it. These are suggestions for discussion and future work, not criticisms that detract from my decision.

1.  **Brittleness of Evaluation Metrics:** The paper relies on per-response and per-dialog accuracy, which are standard but can be brittle. Per-response accuracy over-penalizes incorrect conversational filler (e.g., "one moment please") when the ultimate success depends only on the correctness of the final API call. Conversely, per-dialog accuracy is extremely harsh; a single error in a long conversation results in a score of zero. The counterintuitive result where Task 5 (full dialog) has a higher per-dialog accuracy (1%) than Task 3 (displaying options, 0%) highlights how sensitive this metric can be to specific task definitions (e.g., T3 requires at least 3 options, making failure more likely). Future work could benefit from more nuanced, application-driven metrics.
2.  **Limitations of the "Match Type" Feature:** The proposed feature is a clever way to handle OOV entities but relies on exact string matches between the user utterance and the KB. This does not address the common vocabulary mismatch problem, where a user might say "british food" but the KB entity is simply "british". A discussion of this limitation and potential extensions (e.g., using character-level models, pre-trained embeddings, or a synonym dictionary) would strengthen the paper.
3.  **Exploring More Meaningful Loss Functions:** The current models are trained with a standard cross-entropy loss, which treats all errors equally. A more practical approach might involve a loss function tied to business goals. For example, an incorrect API call could be penalized far more heavily than a sub-optimal conversational response. A weighted loss function reflecting the cost of different errors (e.g., API errors, extra dialog turns, user rejections) could guide the model toward more commercially viable behavior. This would likely require a more sophisticated user simulator or human-in-the-loop training.

### Questions for the Authors

I would appreciate clarification on the following points to ensure my understanding is correct and to help improve the final version of the paper:

1.  Could you comment on the sensitivity of per-dialog accuracy to specific task definitions, as exemplified by the T3 vs. T5 results? Have you considered alternative, task-oriented metrics that might focus more directly on goal completion (e.g., accuracy of the final API call) rather than penalizing every conversational turn equally?
2.  The "match type" feature is a great addition for known KB entities. How do you envision extending this mechanism to handle semantic but not literal matches, such as synonyms or paraphrases (e.g., mapping "cheap" to "moderate" if no cheap options exist, or understanding "food from the UK" as "british cuisine")?
3.  For the Concierge dataset, the evaluation is based on ranking the correct response among a set of all possible operator utterances. Could you elaborate on why this ranking-based evaluation was chosen? How do you think the relative model performances would change under a stricter top-1 accuracy metric where the model must identify the single best response?
