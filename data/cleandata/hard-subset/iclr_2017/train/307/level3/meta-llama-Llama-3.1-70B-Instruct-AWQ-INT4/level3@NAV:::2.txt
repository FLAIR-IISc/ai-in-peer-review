**Summary of the Paper's Claims and Contributions**

This paper proposes a testbed for evaluating end-to-end dialog systems in goal-oriented applications. The authors introduce a set of five tasks in the context of restaurant reservation, which require manipulating sentences and symbols to conduct conversations, issue API calls, and use the outputs of such calls. They also provide a comparison of multiple methods, including rule-based systems, classical information retrieval models, supervised embeddings, and end-to-end Memory Networks, on these tasks. The results show that Memory Networks outperform other baselines, but still have limitations in handling goal-oriented dialog.

**Decision and Key Reasons**

I decide to **Accept** this paper, with two key reasons:

1. The paper provides a valuable reality check on the current state of end-to-end dialog systems in goal-oriented applications, highlighting their limitations and challenges.
2. The authors propose a thorough and well-designed testbed for evaluating end-to-end dialog systems, which can facilitate future research and development in this area.

**Supporting Arguments**

The paper's contributions are significant for both the Dialog/NLP and ML communities. The authors' conjectures about the inadequacy of end-to-end ML performance in goal-oriented dialogs and the predictive power of synthetic data are well-supported by their experimental results. The paper's experimental protocol and clarity are excellent, making it easy to follow and reproduce the results.

**Additional Feedback**

To improve the paper, the authors could:

1. Address the vocabulary mismatch between user language and KB terms, which is a significant challenge for end-to-end dialog systems.
2. Explore more meaningful, application-driven loss functions that tie to real-world revenue, which could require human users or realistic simulators.
3. Provide more analysis on the results, such as the impact of the number of hops on the performance of Memory Networks.

**Questions for the Authors**

1. How do you plan to address the vocabulary mismatch between user language and KB terms in future work?
2. Can you provide more details on the implementation of the match type features and how they are used in the Memory Networks model?
3. How do you think the results would change if you used a different evaluation metric, such as a weighted sum of API errors, dialog turns, and user rejections?
