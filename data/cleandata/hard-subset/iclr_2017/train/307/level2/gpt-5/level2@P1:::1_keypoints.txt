- Proposes a lightweight, reproducible testbed for end-to-end goal-oriented dialog focused on restaurant reservation, decomposed into five subtasks plus DSTC2-derived and Concierge datasets.
- Tasks evaluate API calling, API updates, option display with sorting, extra-info retrieval, and full dialogs to isolate dialog capabilities.
- Evaluates several baselines (rule-based, TF-IDF, nearest neighbor, supervised embeddings) and end-to-end Memory Network (MemN2N) with/without match-type entity features.
- Reports per-response and per-dialog accuracies and includes OOV splits to stress generalization to unseen entities.
- Finds MemN2N can learn API calls and updates and benefits from multi-hop attention.
- Shows match-type features substantially improve OOV performance and KB-lookup tasks.
- Demonstrates option display (T3) and end-to-end full dialogs (T5) remain challenging with low per-dialog accuracy.
- Observes similar relative method rankings on synthetic, DSTC2-derived, and Concierge datasets, supporting synthetic-to-real proxy validity.
- Releases data and resources with clear ablations and hyperparameters to encourage reproducibility.
- Decision: Accept (weak accept).
- Primary reason 1: Well-motivated, useful benchmark with clear task decomposition and solid baselines enabling reproducible comparison.
- Primary reason 2: Empirical study identifies concrete capability gaps (e.g., ranking/KB interpretation) and introduces effective match-type handling for entities/OOV.
- Problem framing: Filling the gap between chit-chat and goal-oriented dialog with interpretable subtasks is valuable and complements existing resources.
- Positioning: Builds on bAbI-style tasks and MemN2N but targets multi-turn state tracking, API use, and KB interpretation; related work coverage is reasonable.
- Evidence & rigor: Clear evaluation protocol, multiple baselines/ablations, attention visualizations, per-task breakdown, and cross-dataset checks.
- Acknowledged limitations: templated language and deterministic simulator, ranking-based evaluation instead of generation, low per-dialog accuracy on hard tasks, and a non-SOTA rule baseline on DSTC2.
- Novelty: Match-type feature integration and comprehensive goal-oriented evaluation suite are practical contributions despite known model components.
- Suggestion: Clarify candidate set construction and report results when candidate pool is limited to training candidates to rule out inflated accuracy.
- Suggestion: Increase language variability via paraphrases or noise to reduce overfitting while maintaining reproducibility.
- Suggestion: Add generative seq2seq/copy-pointer baselines for T3–T5 to compare entity grounding and ordering capabilities.
- Suggestion: Analyze T3 failures (incorrect fact attention vs tracking proposed options) and consider architectures with explicit list/set tracking or differentiable sorting.
- Suggestion: Explore curriculum or joint training across tasks to improve transfer and per-dialog performance on T5.
- Suggestion: Discuss DSTC2 conversion choices (adding ratings) and their realism impacts; report consistency for conversion heuristics if available.
- Suggestion: Add finer-grained metrics (turn-level F1 on API calls/slots, success rate conditioned on API correctness) to localize bottlenecks.
- Reproducibility reminder: Release code and exact candidate pools to avoid evaluation ambiguity.
- Questions for authors: Report candidate pool size per task and results with training-only candidates; evidence of memorization of test-only responses?
- Questions: How would OOV results change if models never saw the OOV KB during training (truly unseen entities)?
- Questions: For T3, is error due to incorrect ranking, tracking proposed options, or misinterpreting user acceptance; can a proposed-set memory help?
- Questions: Do match-type features cause false positives or depend on entity normalization; were character/subword features tried?
- Questions: Provide ablation of attention shifts across hops for T3/T4 and whether >4 hops helps or overfits.
- Questions: For Concierge dataset, how are intents disambiguated without logged API arguments and could weak supervision improve label fidelity?
