- Introduces an open, reproducible testbed for end-to-end goal-oriented dialog in a controlled restaurant-reservation domain.  
- Decomposes tasks into five synthetic subtasks (dialog management, API calls, KB-based sorting/display, attribute retrieval), plus a DSTC2-derived task and a proprietary concierge evaluation.  
- Compares multiple baselines (TF-IDF, nearest neighbor, supervised embeddings, rule-based, MemN2N) and augments MemN2N with match-type features for better entity/OOV handling.  
- Shows MemN2N achieves strong per-response accuracy, learns API construction and KB usage, and outperforms a hand-crafted slot-filling system on DSTC2.  
- Emphasizes per-dialog accuracy to better reflect task completion and notes remaining gaps on multi-step KB reasoning (e.g., sorting).  
- Decision: Accept.  
- Key reason 1: The testbed fills a reproducible, fine-grained evaluation gap for goal-oriented dialog research.  
- Key reason 2: Empirical results are thorough, well-analyzed, and generalize from synthetic tasks to real dialog data (DSTC2 and concierge).  
- Problem significance: Addresses the need to scale beyond handcrafted slot-filling and targets entity manipulation, API use, sorting, and OOV handling.  
- Placement/motivation: Well-situated relative to POMDP/slot-filling and chit-chat datasets; extends bAbI-style analysis to dialog-specific behaviors.  
- Methodological soundness: Tasks and OOV splits are clearly specified; ranking setup, metrics, and hyperparameters support reproducibility.  
- Modeling: MemN2N + match-type features appropriately target embedding-model weaknesses with entities.  
- Empirical ordering: TF-IDF < nearest neighbor < supervised embeddings < MemN2N; match-type features yield notable OOV improvements and enable Task 4.  
- Demonstrated failure modes: Tasks 3 and 5 reveal weaknesses in sorting and multi-step KB interpretation for end-to-end models.  
- Real-data validity: Method ordering persists on DSTC2 and concierge data; rule-based system fails to generalize as well to DSTC2.  
- Impact: Open resource with subtasks can catalyze work on entity handling, KB integration, and dialog management.  
- Concern: Simulator has deterministic behaviors (e.g., slot query order) that may simplify dialog management, though Tasks 3–5 and DSTC2 add non-determinism.  
- Concern: Evaluation uses next-utterance ranking with candidate pools containing train/val/test/OOV responses, which may favor retrieval methods and differ from generative/interactive setups.  
- Concern: Concierge dataset is proprietary, limiting full benchmarking despite open synthetic/DSTC2 resources.  
- Essential suggestion: Clarify candidate set construction, justify including test utterances, report sensitivity to candidate set size, and consider alternative candidate restrictions.  
- Essential suggestion: Provide ablations on MemN2N (hops, time/speaker, match-type) per task to link architecture choices to gains.  
- Essential suggestion: Expand discussion on simulation determinism and how Task 6/concierge mitigate realism concerns.  
- Nice-to-have: Add a pointer/copy baseline or discuss expected performance for entity/OOV handling.  
- Nice-to-have: Include noise-injected variants (paraphrases/ASR errors) to bridge synthetic and real conditions.  
- Nice-to-have: Provide an error taxonomy for Tasks 3 and 5 (sorting vs. option-tracking vs. stopping).  
- Clarification question: What is the candidate set size per task at test time, and how does performance change when varying it? Any leakage risk from shared templates?  
- Clarification question: In DSTC2 conversion, could non-deterministic or multi-criterion ranking be supported instead of deterministic ratings?  
- Clarification question: For OOV splits, how sensitive are results to withholding the OOV KB during training?  
- Clarification question: Are there failure analyses on Task 3 isolating sorting vs. dialog-control errors?  
- Clarification question: How stable are conclusions across random seeds for simulator generation and model training?
