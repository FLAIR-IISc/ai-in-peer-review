Summary and contributions
- The paper proposes a lightweight, reproducible testbed to evaluate end-to-end dialog systems in goal-oriented settings, with a focus on restaurant reservation. The test suite decomposes the overall goal into five subtasks (API calling, updates, option display with sorting, extra info retrieval, and full dialogs), plus a converted real dataset from DSTC2 and an additional realistic Concierge dataset.
- The authors evaluate several baselines (rule-based, TF-IDF, nearest neighbor, supervised embeddings) and an end-to-end Memory Network (MemN2N) with and without “match type” features for entities. They report both per-response and per-dialog accuracies, including OOV splits that stress generalization to unseen entities.
- Key empirical findings: MemN2N learns to issue and update API calls and benefits from multi-hop attention; match-type features substantially improve OOV and KB-lookup tasks; option display and end-to-end full dialogs remain challenging (low per-dialog accuracy). Similar relative performance trends observed on synthetic data are reported on DSTC2-derived and Concierge data.
- The work releases data/resources and provides clear ablations/hyperparameters, encouraging reproducibility and serving as a benchmark for the community.

Decision: Accept (weak accept)
Primary reasons:
1) Well-motivated and useful benchmark with clear task decomposition that enables analysis and reproducible comparison for goal-oriented dialog; accompanied by solid baselines and insights.
2) Empirical study identifies concrete capability gaps (e.g., option ranking/KB interpretation) and proposes a simple but effective mechanism (match types) for entities and OOV handling.

Supporting arguments
- Problem framing: The gap between chit-chat and goal-oriented dialog is real; decomposing goal-oriented dialogs into interpretable subtasks is valuable. The testbed complements existing resources (DSTC) by enabling end-to-end training/evaluation and built-in OOV stress tests.
- Positioning in literature: The work builds on bAbI-style tasks and MemN2N, but targets a different capability profile (multi-turn state tracking, API use, KB interpretation). Related work is fairly covered.
- Evidence and rigor: The evaluation protocol is clearly described, with multiple baselines and ablations (IR vs supervised embeddings vs MemN2N; effect of hops; match types; OOV splits). Results are consistent and analyzed with attention visualizations and per-task breakdown. The cross-check on DSTC2-derived and Concierge datasets supports the claim that the synthetic tasks are a reasonable proxy for real settings (at least in relative ranking of methods).
- Limitations (acknowledged by the authors): language templating and deterministic bot behavior in simulated tasks; ranking-based evaluation rather than generation; still low per-dialog accuracy on harder tasks; rule-based baseline on DSTC2 is not state-of-the-art and may underplay competitiveness of traditional pipelines. Despite these, the primary contribution is a benchmark and analysis, not SOTA performance.
- Novelty: While MemN2N is known, the match-type feature integration and the comprehensive, goal-oriented evaluation suite are new and practically useful.

Additional feedback (to help improve the paper; not all are decision-critical)
- Candidate set construction: You rank over a candidate pool that includes utterances from train/val/test across tasks. Please clarify why including test utterances is necessary and whether it inflates absolute accuracy. Reporting results with a candidate set built only from training (plus templated API calls) would strengthen claims.
- Language variability: The simulator uses 43 user and 20 bot patterns, with the bot deterministic. Consider adding paraphrase/noise variants (e.g., crowd-sourced augmentations or synthetic paraphrasing) to reduce overfitting to patterns and to stress robustness without sacrificing reproducibility.
- Generative baselines: Adding a modern seq2seq/attention model with copy/pointer mechanism would be informative, especially for T3–T5 where entity grounding and ordering matter. A copy-enabled model may narrow the gap on T4/T5 and provide complementary error modes to MemN2N.
- Sorting and list management: Since T3 hinges on sorting by rating and iteratively proposing options, an analysis of failure cases (e.g., attending to incorrect facts vs failing to track which options have been proposed) would be helpful. Consider evaluating architectures with explicit list/set tracking or differentiable sorting modules.
- End-to-end task complexity: Per-dialog accuracy remains low. Consider curriculum learning across tasks, or joint training on T1–T4 before T5, to see if compositional transfer helps T5.
- DSTC2 conversion: Adding ratings to enforce determinism is practical; please discuss how this affects realism and whether alternative policies (e.g., learning to rank with noisy labels) change conclusions. Also, report inter-annotator or heuristic consistency for the conversion, if available.
- Metrics: In addition to per-response/per-dialog accuracy, consider turn-level F1 on API-calls/slot-mentions, or success rate conditioned on API-call correctness, to more finely localize bottlenecks.
- Reproducibility: The dataset URL is provided; please ensure code and exact candidate pools are released to avoid ambiguity in evaluation.

Questions for the authors
1) Candidate set: What is the size of the candidate pool per task at test time, and how do results change if the candidate set is restricted to training candidates only? Any evidence of memorization of rare test-only responses?
2) OOV setup: Models are allowed access to both KBs during training. How would OOV results change if the model had no access to the OOV KB at training time (i.e., truly unseen entities at test, except for type schemas)?
3) T3 failure analysis: Is the primary source of error incorrect ranking of options, failure to keep track of which options have already been proposed, or misinterpretation of user acceptance? Can adding a “proposed-set” feature or memory slot alleviate this?
4) Match-type features: Do these features ever cause false positives (e.g., string collisions) and how sensitive are results to entity normalization? Have you tried character/subword features instead?
5) Multi-hop usage: Can you provide a small ablation showing how attention shifts across hops for T3/T4, and whether more than 4 hops helps or hurts due to overfitting?
6) Concierge dataset: Since APIs are not logged with arguments, how do you disambiguate different intents (e.g., Yelp for discovery vs verification)? Would weak supervision (distant signals from URLs) allow richer labels and improve evaluation fidelity?

Overall, this paper offers a timely, well-executed benchmark and analysis for end-to-end goal-oriented dialog. Despite limitations, it is likely to be useful to the community and to spur progress on the hard cases it surfaces.
